# GitLab CE 任务治理规则

## 结论

- `node-150` 上的 GitLab CE 任务系统用于管理指挥层任务，不用于给 bot 建立正式任务体系
- 项目早期阶段，CE 不给 `dev-bot`、`qa-bot`、`doc-bot` 直接挂正式任务
- bot 只作为执行工具或辅助资源出现，不作为 CE 任务系统的一等主体
- CE 中的正式任务必须沿用现有 `aedu/workbot` 任务体系，不再另造平行编号体系

## 第一原则

- CE 记录的是“我要推进什么、当前做到哪一步、卡在哪里、下一步是什么”
- CE 不是 bot 内部分工看板
- 文档、日报、tmux pane、bot 执行都不能替代 CE 的正式任务裁决职责

## 任务主体

### 指挥层

- CE 中的任务默认属于指挥层
- 指挥层负责定义目标、推进节奏、状态裁决和收口
- CE 中的“负责人”表达的是指挥推进责任，不表达 bot 身份

### bot

- bot 不是项目早期的正式任务主体
- bot 只在执行层参与：
  - 只读分析
  - 文档整理
  - 结构化输出
  - 辅助验收
- bot 分工只允许出现在评论、日报或执行记录中
- bot 不应直接占用 CE 正式任务编号

## 项目早期规则

- 项目早期只维护阶段目标和主线任务
- 不把内部文档动作、bot 分工动作、一次性整理动作直接录成 CE 正式任务
- 只有当某项工作需要持续跟踪、状态变更、阻塞管理和最终验收时，才进入 CE

## 编号与命名规则

- 正式任务只沿用当前 `aedu/workbot` 既有任务体系
- 当前主任务体系以 `[W1] ...` 这类任务为准
- 不允许临时新造平行体系，例如：
  - `CE-DOC-*`
  - `CE-QA-*`
  - `CE-OPS-*`

## 状态更新规则

- 优先更新已有正式任务，不新建替代任务
- 若已有任务能承接当前进展，应优先：
  - 写评论
  - 补阻塞说明
  - 更新标签或里程碑
  - 满足条件后再关闭
- 不允许“刚创建即关闭”的伪完成单

## 文档与 CE 的边界

- CE 负责管理正式任务状态
- 主状态文档负责解释系统状态、风险、规则和事实
- 日报负责记录当天执行过程和结论
- 若文档和 CE 状态冲突，应立即收口，不允许长期双口径并存

## 当前纠偏结论

- `CE-*` 这一批内部编号不应继续作为正式任务体系扩张
- 后续任务更新必须回到 `aedu/workbot` 现有正式任务体系
- 若需要记录 bot 执行痕迹，应写评论或日报，而不是再造 bot 专属任务单

## 后续约束

- 只要项目仍处于早期阶段，就继续执行“CE 不给 bot 直接挂任务”规则
- 等项目进入稳定执行期后，若确有必要引入 bot 级任务模型，必须先形成新决策，再升级规则
