## 二、开发任务单详情

### F1：教材章节树标准化功能

**对应文档依据**：STR-002、KB-001、STD-001、MAT-PHY-001

**目标**：把 PHY_PEP_G1_V1 的高中物理必修一章节树变成可存储、可编码、可引用的标准对象。

---

#### 任务卡 F1-T1：定义章节树节点模型

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立章节树最小对象模型，支持章/节/小节三级 |
| 输入 | STR-002、KB-001、MAT-PHY-001、当前固定执行口径 |
| 输出 | 章节树 schema |
| 核心字段 | `node_id`、`node_type`、`name`、`parent_id`、`subject`、`grade`、`edition`、`version`、`scope`、`status` |
| 验收标准 | 能表达章/节/小节三级、编码唯一、父子关系可校验、与固定口径字段兼容 |

---

#### 任务卡 F1-T2：实现固定口径校验

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 把当前冻结口径变成硬校验规则 |
| 输入 | 当前冻结口径（安徽/高中/高一/物理/PHY_PEP_G1_V1） |
| 输出 | 口径校验规则/校验函数 |
| 前置依赖 | F1-T1 |
| 验收标准 | 非安徽/高中/高一/物理/PHY_PEP_G1_V1 数据被拒绝、错误信息可定位字段、可被后续知识点/能力点/锚点复用 |

---

#### 任务卡 F1-T3：录入首版章节树样例

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 B |
| 目标 | 完成必修一首个章节的章节树样例 |
| 输入 | 必修一章节目录、F1-T1 schema、F1-T2 口径校验 |
| 输出 | 首版章节树样例数据（至少包含 1 个完整章节的章/节/小节结构） |
| 前置依赖 | F1-T1、F1-T2 |
| 验收标准 | 至少包含章、节、小节三级、所有节点通过口径校验、父子关系完整、编码无冲突 |

---

### F2：知识点标准入库功能

**对应文档依据**：KB-001、KB-004、STD-001、MAT-PHY-001

**目标**：把知识点从"文档描述"变成"标准条目"，能入库、能被引用、能被校验。

---

#### 任务卡 F2-T1：定义知识点对象模型

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立知识点最小入库模型 |
| 输入 | KB-001、KB-004、MAT-PHY-001、F1 输出 |
| 输出 | 知识点 schema |
| 核心字段 | `knowledge_id`、`name`、`chapter_node_id`、`definition`、`aliases`、`status`、`source_ref` |
| 验收标准 | 知识点可独立存储、必填字段完整、状态枚举清晰、与章节树节点可关联 |

---

#### 任务卡 F2-T2：定义知识点准入规则

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 实现知识点入库校验 |
| 输入 | F2-T1 schema、F1-T2 口径校验 |
| 输出 | 知识点准入规则/校验函数 |
| 前置依赖 | F1-T2、F2-T1 |
| 验收标准 | 无章节归属的知识点被拒绝、空定义被拒绝、同名冲突可检测、口径不匹配被拒绝 |

---

#### 任务卡 F2-T3：产出首批知识点样例

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 B |
| 目标 | 为首个章节生成第一批知识点条目 |
| 输入 | 首个章节内容、F2-T1 schema、F2-T2 准入规则 |
| 输出 | 首批知识点样例数据（至少 5-10 个知识点） |
| 前置依赖 | F1-T3、F2-T1、F2-T2 |
| 验收标准 | 所有知识点通过准入校验、每个知识点挂靠到章节树节点、定义清晰无歧义 |

---

### F3：能力点映射功能

**对应文档依据**：KB-004、STD-002、MAT-PHY-001

**目标**：把"知识点"进一步映射到"能力点"，形成最小教学语义层。

---

#### 任务卡 F3-T1：定义能力点对象模型

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立能力点最小模型 |
| 输入 | STD-002、KB-004 |
| 输出 | 能力点 schema |
| 核心字段 | `ability_id`、`name`、`description`、`level`、`category`、`status` |
| 验收标准 | 能力点可独立存储、能力分级清晰、类别可枚举 |

---

#### 任务卡 F3-T2：定义知识点→能力点映射规则

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立 knowledge -> ability 关系规则 |
| 输入 | F2-T3 知识点样例、F3-T1 schema |
| 输出 | 映射规则/关系 schema |
| 前置依赖 | F2-T3、F3-T1 |
| 核心规则 | 一个知识点可映射多个能力点、一个能力点可被多个知识点支撑、必须有主归属 |
| 验收标准 | 映射关系可存储、可查询、可校验 |

---

#### 任务卡 F3-T3：产出首批能力点样例

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 B |
| 目标 | 给首批知识点挂上能力点 |
| 输入 | F2-T3 知识点样例、F3-T2 映射规则 |
| 输出 | 首批能力点样例与映射数据 |
| 前置依赖 | F2-T3、F3-T1、F3-T2 |
| 验收标准 | 每个能力点至少挂靠一个知识点、映射关系完整、能力点类型覆盖理解/辨析/应用/建模/实验解释 |

### F4：锚点/证据挂接功能

**对应文档依据**：KB-004、STD-002、MAT-PHY-001

**目标**：给章节树、知识点、能力点挂上教材证据和引用锚点，让条目变成"可追溯资产"。

---

#### 任务卡 F4-T1：定义锚点对象模型

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立锚点最小模型 |
| 输入 | KB-004、STD-002、教材内容 |
| 输出 | 锚点 schema |
| 核心字段 | `anchor_id`、`anchor_type`、`source_ref`、`page_or_section`、`excerpt`、`target_type`、`target_id` |
| 锚点类型 | 章节位置、教材页码、定义、公式、例题、实验、图示、题型样例 |
| 验收标准 | 锚点类型可枚举、证据片段可存储、目标引用可追溯 |

---

#### 任务卡 F4-T2：定义锚点引用关系

| 字段 | 内容 |
|------|------|
| 建议负责人 | 开发 A |
| 目标 | 建立锚点与知识点/能力点的关系规则 |

| 输出 | 锚点关系 schema / 规则 |

| 核心规则 | 一个锚点可服务多个知识点/能力点、必须有主归属、引用可双向追溯 |

| 字段 | 内容 |

| 建议负责人 | 开发 B |

**目标**：把前四个功能串起来，形成第一份完整样例包，为后续批量复制提供模板。

## 六、为什么是这 5 个功能

| 功能 | 解决的问题 | 缺失后果 |
|------|-----------|----------|
| 章节树 | 内容挂哪里 | 少章节树，内容无归属 |
| 知识点 | 是什么 | 少知识点，只有目录没有知识 |
| 能力点 | 能做什么 | 少能力点，无法支撑后续教学消费 |
| 锚点 | 凭什么 | 少锚点，无法复核 |
| 样例包与校验 | 能不能跑通 | 少校验与样例包，无法验收 |

**多做则会乱**：
- 现在不该扩到 OBS / SIM
- 现在不该做大规模 GRAPH 正式建模
- 现在不该铺全册、全学科、全版本
- 现在不该回头补治理文档

**核心理念**：当前最缺的不是"更多内容"，而是"第一条可重复的生产流水线"。
