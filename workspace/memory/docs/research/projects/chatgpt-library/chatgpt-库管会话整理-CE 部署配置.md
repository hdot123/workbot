## 一、部署结论

**可以去部署 CE 系统**，但建议把"部署 CE 系统"和"开始大规模开发"分开理解。

### 可以部署的原因

1. 第一阶段功能已经拆成了可执行任务
2. 有明确的功能父卡和子任务卡结构
3. 有依赖关系、负责人建议、验收标准
4. 已经足够支撑排期、分派、跟踪、验收

**从"项目管理系统上线"的角度，条件已经够了。**

---

## 两个前提建议

### 前提一：先把任务结构录入规则定死

至少统一这几个字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| 编号 | 任务唯一标识 | F1-T1 |
| 名称 | 任务名称 | 定义章节树节点模型 |
| 所属功能 | 归属的功能块 | F1 章节树标准化 |
| 所属阶段 | 归属的项目阶段 | 第一阶段 |
| 前置依赖 | 前置任务编号 | F1-T1 |
| 负责人 | 建议负责人角色 | 开发 A |
| 交付物 | 任务输出物 | 章节树 schema |
| 验收标准 | 完成判断标准 | 能表达章/节/小节三级、编码唯一 |
| 状态 | 当前进度状态 | 待开始/进行中/待验收/已完成/阻塞 |

---

### 前提二：按"5 个功能父卡 + 15 个子任务卡"去建

- ❌ 不要一开始只建一个大任务
- ❌ 也不要把 15 个碎任务平铺得没有层级
- ✅ 否则 CE 一上线就会乱

## 二、CE 系统配置详情

### 2.1 任务层级结构

```
第一阶段（阶段）
├── F1 教材章节树标准化（功能父卡）
│   ├── F1-T1 定义章节树节点模型（子任务卡）
│   ├── F1-T2 实现固定口径校验（子任务卡）
│   └── F1-T3 录入首版章节树样例（子任务卡）
├── F2 知识点标准入库（功能父卡）
│   ├── F2-T1 定义知识点对象模型（子任务卡）
│   ├── F2-T2 定义知识点准入规则（子任务卡）
│   └── F2-T3 产出首批知识点样例（子任务卡）
├── F3 能力点映射（功能父卡）
│   ├── F3-T1 定义能力点对象模型（子任务卡）
│   ├── F3-T2 定义知识点→能力点映射规则（子任务卡）
│   └── F3-T3 产出首批能力点样例（子任务卡）
├── F4 锚点/证据挂接（功能父卡）
│   ├── F4-T1 定义锚点对象模型（子任务卡）
│   ├── F4-T2 定义锚点引用关系（子任务卡）
│   └── F4-T3 产出首批锚点样例（子任务卡）
└── F5 最小闭环校验与样例包（功能父卡）
    ├── F5-T1 定义最小闭环校验规则（子任务卡）
    ├── F5-T2 组装首个完整样例包（子任务卡）
    └── F5-T3 输出最小验收结果（子任务卡）
```

---

### 2.2 状态流转配置

```
待开始 → 进行中 → 待验收 → 已完成
            ↓
          阻塞
```

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| 待开始 | 任务已创建，尚未开始 | 任务创建时默认状态 |
| 进行中 | 开发已开始工作 | 负责人开始开发时切换 |
| 待验收 | 开发完成，等待验收 | 负责人提交验收时切换 |
| 已完成 | 验收通过 | 验收人确认通过后切换 |
| 阻塞 | 遇到阻塞无法继续 | 遇到依赖/资源问题时切换 |

---

### 2.3 任务完成定义（DoD）

给每张子任务卡补一个"完成定义"，否则开发容易说"差不多做完了"，但实际上不能验收。

| 任务卡 | 完成定义 |
|--------|----------|
| F1-T1 | schema 已提交，能表达章/节/小节三级，字段已冻结 |
| F1-T2 | 口径校验规则已实现，非安徽/高中/高一/物理/PHY_PEP_G1_V1 数据被拒绝 |
| F1-T3 | 首版章节树样例数据已录入，至少包含 1 个完整章节的章/节/小节结构 |
| F2-T1 | 知识点 schema 已提交，必填字段完整，状态枚举清晰 |
| F2-T2 | 知识点准入校验已实现，无章节归属/空定义/同名冲突可检测 |
| F2-T3 | 首批知识点样例已入库（5-10 个），每个知识点挂靠到章节树节点 |
| F3-T1 | 能力点 schema 已提交，能力分级清晰，类别可枚举 |
| F3-T2 | 知识点→能力点映射规则已定义，支持一对多关系，有主归属 |
| F3-T3 | 首批能力点样例与映射数据已完成，覆盖理解/辨析/应用/建模/实验解释 |
| F4-T1 | 锚点 schema 已提交，锚点类型可枚举（章节位置/页码/定义/公式/例题/实验/图示） |
| F4-T2 | 锚点引用关系已定义，支持双向追溯，可检测断链 |
| F4-T3 | 首批锚点样例数据已完成，每个知识点/能力点至少有一个锚点 |
| F5-T1 | 闭环校验脚本可执行，能检测孤立节点、断链，输出校验报告 |
| F5-T2 | 完整样例包已生成，包含节点、关系、校验结果，可完整展示闭环 |
| F5-T3 | 最小验收报告已输出，能明确回答"是否跑通"，列出阻塞点 |

## 三、部署动作顺序

### 推荐顺序

```
1. 部署 CE 系统
2. 建第一阶段 5 个功能父卡
3. 建 15 个开发子任务卡
4. 设状态流转：待开始 / 进行中 / 待验收 / 已完成 / 阻塞
5. 开始分派给开发
```

### 更稳妥的做法（推荐）

```
1. 部署 CE 系统
2. 建第一阶段 5 个功能父卡
3. 先只录入 F1 和 F2（共 6 个子任务卡）
4. 检查字段和流转是否顺手
5. 确认没问题，再补 F3-F5（共 9 个子任务卡）
6. 开始分派给开发
```

**理由**：这样风险更小，可以及时发现配置问题并调整。

---

## 四、注意事项

### ✅ 建议做的

- 第一批录入内容只包括这次确定的一阶段任务
- 按"5 个功能父卡 + 15 个子任务卡"的层级结构录入
- 统一任务字段格式
- 设置状态流转
- 给每个子任务卡定义完成定义（DoD）

### ❌ 不建议做的

- 顺手把后续阶段也一起灌进去
- 一开始只建一个大任务
- 把 15 个碎任务平铺得没有层级
- 在 CE 部署完成前就开始开发
