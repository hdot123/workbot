# 教材元数据表标准

> 文档编号：STD-010  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：教材标准表负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中的"教材元数据表"标准，包括教材基础信息、版本信息、适用范围、来源信息、结构状态与版本管理字段。

本文档重点回答以下问题：

1. 教材元数据表到底记录什么
2. 为什么教材元数据必须单独建表
3. 教材元数据如何服务知识底座、学科库、锚点、版本映射和学生绑定
4. 如何避免教材文件进来了，但系统根本不知道它到底是哪本书
5. 如何支撑后续多版本、多地区、多学科扩展

---

## 2. 教材元数据定义

教材元数据，是对一册教材在系统中的基础身份信息、版本信息、适用信息、来源信息和结构化状态的统一描述。

它不是教材正文内容，也不是章节树、知识点或锚点本身，而是教材作为"底座资产"的管理入口。

教材元数据的核心任务，是回答：

- 这本教材是什么
- 属于哪个学科、年级、册别、版本
- 适用于哪些地区
- 来源是什么
- 当前结构化做到什么程度
- 是否已进入系统主链路

---

## 3. 建设目标

### 3.1 建立教材资产主索引
让系统清楚知道自己有哪些教材资产。

### 3.2 支撑版本与地区适配
为地区绑定和版本映射提供基础对象。

### 3.3 支撑结构化过程管理
让每册教材的章节树、知识点、锚点等结构化进度可追踪。

### 3.4 支撑学生绑定
学生初始化时需要绑定教材版本，而不是直接绑某个 PDF 文件名。

### 3.5 支撑后续扩展
多地区、多出版社、多版本进入后，元数据表会成为底层交通枢纽。

---

## 4. 设计原则

### 4.1 一本教材一个主身份
每册教材必须有稳定唯一的主标识。

### 4.2 元数据与正文分离
元数据表只负责管理信息，不承载正文大块内容。

### 4.3 来源清楚
教材来源、文件、版本必须可追溯。

### 4.4 状态可管理
要能知道这本教材是否已入库、是否已结构化、是否可用。

### 4.5 支持版本化
不同出版社、不同版次、不同学科册别必须可明确区分。

---

## 5. 教材元数据对象范围

教材元数据表建议覆盖以下信息：

| 信息类别 | 说明 |
|----------|------|
| 教材身份信息 | 教材名称、出版社、版本等 |
| 学科与学段信息 | 学科、学段、年级等 |
| 版本与册别信息 | 版本号、册别、系列名等 |
| 地区适用信息 | 适用地区、政策映射等 |
| 来源与文件信息 | 来源引用、文件存储等 |
| 结构化状态信息 | 各标准表建设进度 |
| 生命周期状态信息 | 激活/归档/废弃状态 |

---

## 6. 教材身份信息

### 6.1 作用
标识"这本教材到底是哪一本"。

### 6.2 建议字段

| 字段 | 说明 |
|------|------|
| textbook_id | 教材唯一标识 |
| textbook_name | 教材全名 |
| textbook_short_name | 简称 |
| publisher | 出版社 |
| edition_name | 版本名称 |
| edition_year | 版本年份 |
| isbn | ISBN 号（如可得） |
| language | 语言 |

---

## 7. 学科与学段信息

### 7.1 作用
描述该教材属于哪个学科、哪个学段、哪个年级。

### 7.2 建议字段

| 字段 | 说明 |
|------|------|
| subject | 学科 |
| stage_level | 学段（小学/初中/高中） |
| grade_level | 年级 |
| course_type | 课程类型（必修/选择性必修/选修） |

---

## 8. 版本与册别信息

### 8.1 作用
区分不同教材版本和册别。

### 8.2 建议字段

| 字段 | 说明 |
|------|------|
| curriculum_version_id | 教材版本标识 |
| volume_no | 册别 |
| semester_hint | 学期提示（如适用） |
| version_label | 版本标签 |
| series_name | 系列名 |

### 8.3 原则
不能只靠教材名称区分版本。  
现实世界里同名不同版教材很多，不要低估混乱能力。

---

## 9. 地区适用信息

### 9.1 作用
说明教材在哪些地区、哪些口径下适用。

### 9.2 建议字段

| 字段 | 说明 |
|------|------|
| applicable_regions | 适用地区列表 |
| default_region_scope | 默认地区范围 |
| policy_mapping_refs | 政策映射引用 |
| curriculum_mapping_status | 课程映射状态 |

### 9.3 用途
支撑：
- 学生地区绑定
- 教材版本映射
- 试点学校接入

---

## 10. 来源与文件信息

### 10.1 作用
追溯教材原始文件和来源。

### 10.2 建议字段

| 字段 | 说明 |
|------|------|
| source_ref | 来源引用 |
| source_type | 来源类型 |
| source_date | 来源日期 |
| file_storage_ref | 原始文件引用 |
| file_checksum | 文件校验值 |
| file_format | 文件格式 |
| page_count | 页数 |

### 10.3 原则
来源和文件必须稳定可追溯。  
后面一旦教材文件换了但你没记录，整个锚点系统就会开始怀疑人生。

---

## 11. 结构化状态信息

### 11.1 作用
描述这本教材结构化做到哪一步。

### 11.2 建议字段

| 字段 | 说明 |
|------|------|
| chapter_tree_status | 章节树状态 |
| knowledge_point_status | 知识点状态 |
| ability_point_status | 能力点状态 |
| anchor_status | 锚点状态 |
| dependency_status | 依赖关系状态 |
| exercise_type_status | 题型状态 |
| misconception_status | 误区状态 |

### 11.3 建议状态值

| 状态值 | 说明 |
|--------|------|
| not_started | 未开始 |
| in_progress | 进行中 |
| completed | 已完成 |
| validated | 已校验 |

---

## 12. 生命周期状态信息

### 12.1 作用
描述该教材当前在系统中的可用性。

### 12.2 建议字段

| 字段 | 说明 |
|------|------|
| status | 生命周期状态 |
| active_flag | 激活标志 |
| archived_flag | 归档标志 |
| deprecated_flag | 废弃标志 |
| valid_from | 生效起始时间 |
| valid_to | 生效结束时间 |

### 12.3 建议状态值

| 状态值 | 说明 |
|--------|------|
| draft | 草稿 |
| active | 激活/可用 |
| deprecated | 废弃/不推荐使用 |
| archived | 归档/历史版本 |

## 14. 字段说明

### 14.1 textbook_id

教材资产主键。

建议格式：`TB_PHY_PEP_G1_V1_B1`

编码结构：`TB_学科_出版社_学段年级_版本_册别`

### 14.2 curriculum_version_id

教材版本对象标识，可被多个标准表引用。

建议格式：`PHY_PEP_G1_V1`

### 14.3 source_type

建议值：

| 值 | 说明 |
|------|------|
| official_pdf | 官方 PDF |
| scanned_pdf | 扫描件 PDF |
| internal_processed_copy | 内部处理副本 |
| authorized_source | 授权来源 |

### 14.4 status

生命周期状态。

| 值 | 说明 | 使用场景 |
|------|------|----------|
| draft | 草稿 | 正在建设中，未发布 |
| active | 激活 | 已发布，可被系统使用 |
| deprecated | 废弃 | 已被新版本替代，不再推荐使用 |
| archived | 归档 | 历史版本，仅供追溯 |

### 14.5 结构化状态字段

所有结构化状态字段统一使用以下状态值：

| 值 | 说明 |
|------|------|
| not_started | 未开始 |
| in_progress | 进行中 |
| completed | 已完成 |
| validated | 已校验 |

## 15. 元数据建立流程

```text
收集教材原始文件
    ↓
识别学科/学段/年级/册别
    ↓
确认出版社和版本信息
    ↓
生成教材主标识与版本标识
    ↓
登记来源与文件引用
    ↓
初始化结构化状态字段
    ↓
进入结构化流程
    ↓
按进度更新状态
```

### 15.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 收集原始文件 | 获取教材 PDF 或扫描件 | 原始文件 |
| 识别基本信息 | 学科、学段、年级、册别 | 基础信息清单 |
| 确认版本信息 | 出版社、版本名称、年份 | 版本信息 |
| 生成标识 | 生成 textbook_id 和 curriculum_version_id | 标识符 |
| 登记来源 | 记录来源和文件引用 | 来源记录 |
| 初始化状态 | 所有结构化状态设为 not_started | 初始状态 |
| 结构化流程 | 章节树、知识点等逐步建设 | 结构化成果 |
| 更新状态 | 按进度更新各状态字段 | 最新状态 |

## 16. 与章节树表的关系

教材元数据表是章节树表的上层入口。

规则：

| 规则 | 说明 |
|------|------|
| 一册教材元数据可关联一套章节树 | 一对多关系 |
| 章节树必须引用教材版本标识 | curriculum_version_id 必填 |
| 没有元数据的章节树应视为不合规结构 | 元数据是前置条件 |

---

## 17. 与知识点表、能力点表的关系

知识点表和能力点表均应挂载到教材版本之下。

这样系统才能明确：

| 明确项 | 说明 |
|--------|------|
| 哪些知识点属于哪本教材 | 知识归属 |
| 哪些能力点在哪套教材中定义和使用 | 能力定义范围 |
| 哪些对象可以一起参与事件映射和状态计算 | 可计算范围 |

---

## 18. 与教材锚点表的关系

锚点表必须引用教材元数据中的：

| 字段 | 说明 |
|------|------|
| curriculum_version_id | 教材版本标识 |
| volume_no | 册别 |
| source_ref / file_storage_ref（间接） | 来源和文件引用 |

教材元数据表决定锚点定位的对象空间是否合法。

---

## 19. 与学生绑定的关系

学生初始化时绑定的，不应是模糊教材名，而应是：

| 绑定项 | 说明 |
|--------|------|
| 地区 | 学生所在地区 |
| 学科 | 学生所学学科 |
| curriculum_version_id | 教材版本标识 |

教材元数据表因此是学生知识底座绑定的入口之一。

## 20. 质量要求

### 20.1 唯一性

| 要求 | 说明 |
|------|------|
| 教材主标识必须唯一 | textbook_id 不能重复 |
| 版本标识必须唯一 | curriculum_version_id 不能重复 |

### 20.2 完整性

| 要求 | 说明 |
|------|------|
| 学科、年级、版本、册别、来源至少要完整 | 核心字段不能缺失 |

### 20.3 可追溯性

| 要求 | 说明 |
|------|------|
| 来源和文件引用必须明确 | source_ref 和 file_storage_ref 必填 |

### 20.4 状态可维护

| 要求 | 说明 |
|------|------|
| 结构化状态字段必须真实反映建设进度 | 不能形同虚设 |

### 20.5 可扩展性

| 要求 | 说明 |
|------|------|
| 支持后续多地区、多学科、多版本增量进入 | 结构支持扩展 |

## 21. 常见错误

### 21.1 只存文件名

| 错误 | 后果 |
|------|------|
| 用文件名代替元数据体系 | 文件一改名系统就乱套 |

### 21.2 版本信息不完整

| 错误 | 后果 |
|------|------|
| 缺少出版社、年份、册别等信息 | 后面做版本映射和地区绑定会直接乱套 |

### 21.3 状态字段形同虚设

| 错误 | 后果 |
|------|------|
| 看起来写了状态，实际上没人更新 | 会把人骗得很惨 |

### 21.4 一个教材多个近义标识

| 错误 | 后果 |
|------|------|
| 同一本教材有多个相似但不一致的标识 | 后续所有引用都会开始漂 |

### 21.5 来源信息模糊

| 错误 | 后果 |
|------|------|
| source_ref 写"网上找的" | 无法追溯，无法校验 |

## 22. MVP 建议范围

MVP 阶段建议优先支持：

### 22.1 学科范围

| 范围 | 说明 |
|------|------|
| 高中物理 | 首建学科 |

### 22.2 内容范围

| 内容 | 说明 |
|------|------|
| 明确的出版社与册别 | 人教版必修第一册等 |
| 来源和文件引用完整 | source_ref 和 file_storage_ref 必填 |
| 结构化进度状态可追踪 | 各状态字段真实更新 |
| 能支撑章节树、知识点、锚点挂载 | curriculum_version_id 可被引用 |

### 22.3 优先级

| 优先级 | 教材 | 说明 |
|--------|------|------|
| P0 | 高中物理必修一 | 首建教材 |
| P1 | 高中物理必修二 | 第二批 |
| P2 | 高中其他学科核心教材 | 后续扩展 |

## 25. 教材元数据维护责任

### 25.1 角色定义

## 26. 风险与边界

### 26.1 不能忽视唯一性

| 风险 | 说明 |
|------|------|
| 教材标识不唯一 | 后续所有引用都会混乱 |

### 26.2 不能忽视版本信息

| 风险 | 说明 |
|------|------|
| 版本信息不完整 | 版本映射和地区绑定会乱套 |

### 26.3 不能忽视状态维护

| 风险 | 说明 |
|------|------|
| 状态字段不更新 | 无法真实反映建设进度 |

### 26.4 不能忽视来源追溯

| 风险 | 说明 |
|------|------|
| 来源信息模糊 | 无法校验和追溯 |

### 26.5 不能忽视扩展性

| 风险 | 说明 |
|------|------|
| 结构不支持多版本扩展 | 后续新增教材会困难 |

## 27. 结论

教材元数据表的核心意义，是让教材在系统里从"一个文件"变成"一个可识别、可管理、可绑定、可迭代的底座对象"。

它必须坚持：

| 原则 | 说明 |
|------|------|
| **身份清楚** | textbook_id 和 curriculum_version_id 唯一明确 |
| **版本清楚** | 出版社、年份、册别等信息完整 |
| **来源清楚** | source_ref 和 file_storage_ref 可追溯 |
| **状态清楚** | 结构化状态真实反映建设进度 |
| **可被全系统引用** | curriculum_version_id 可被各标准表引用 |

否则后面标准表越多，系统越容易陷入一种经典混乱：大家都在说同一本教材，但实际上没人确定说的是不是同一本。
