## 1. 文档目的

本文档用于定义教育孪生项目中的"能力点表"标准，包括能力点的定义、分层方式、字段结构、命名规则、与知识点及题型的映射关系。

本文档重点回答以下问题：

1. 什么叫能力点
2. 能力点和知识点有什么区别
3. 能力点应该如何抽取和命名
4. 能力点如何服务学生状态建模、风险识别和推演
5. 如何避免把能力点做成空泛口号表

---

## 2. 能力点定义

能力点，是指学生在学习过程中需要体现出的、可观察、可训练、可解释的能力要求单元。

它不是知识内容本身，而是学生处理知识、应用知识、表达知识、迁移知识时体现出来的能力维度。

例如：

- 理解概念
- 分析条件
- 建立模型
- 推导公式
- 解释现象
- 实验设计
- 图像判断
- 综合应用

能力点是"学生会不会做"的结构表达，不是"教材讲了什么"的结构表达。

---

## 3. 能力点表的目标

### 3.1 支撑学生状态表达
让 StudentTwinAgent 不只停留在知识点掌握层，还能表达学生能力结构。

### 3.2 支撑题目与错因解释
很多问题不是"不知道知识点"，而是"能力用不出来"。

### 3.3 支撑风险识别
识别知识掌握正常但能力表达薄弱的情况。

### 3.4 支撑干预设计
能力点可直接指向不同的练习和教学动作。

### 3.5 支撑推演层
后台推演需要区分"知识缺口"和"能力短板"。

---

## 4. 能力点与知识点的区别

| 维度 | 知识点 | 能力点 |
|------|--------|--------|
| 核心问题 | 学了什么 | 会怎么用 |
| 来源 | 教材内容结构 | 教学要求与表现要求 |
| 表达对象 | 内容单元 | 能力要求单元 |
| 映射对象 | 章节、锚点、题型 | 知识点、题型、错误类型 |
| 典型示例 | 牛顿第二定律及其公式关系 | 运用公式建立受力与加速度关系 |

知识点偏"内容"，能力点偏"使用方式"。

---

## 5. 设计原则

### 5.1 可观察
能力点应尽量能通过作业、考试、反馈、练习等被观察到。

### 5.2 可训练
能力点应能对应明确训练方向，而不是抽象大词。

### 5.3 可映射
能力点应能映射到知识点、题型和错因。

### 5.4 粒度适中
能力点不能粗到只剩"思维能力"，也不能细到"第 3 步写箭头"。

### 5.5 学科适配
不同学科能力点既有共性，也有学科特性，不能强行一刀切。

---

## 6. 能力点分层建议

建议能力点至少分为三层：

1. 通用能力层
2. 学科能力层
3. 知识应用能力层

---

## 7. 通用能力层

### 7.1 定义
跨学科普遍适用的基础能力要求。

### 7.2 示例
- 阅读理解
- 信息提取
- 条件分析
- 逻辑表达
- 过程复盘

### 7.3 用途
适合作为通用框架，但不应成为唯一层级。

---

## 8. 学科能力层

### 8.1 定义
某一学科中常见的核心能力要求。

### 8.2 物理示例
- 建模能力
- 受力分析能力
- 图像理解能力
- 实验分析能力
- 公式推导与应用能力

### 8.3 数学示例
- 运算能力
- 证明能力
- 函数建模能力
- 图形转换能力

---

## 9. 知识应用能力层

### 9.1 定义
能力点与具体知识点结合后形成的应用能力要求。

### 9.2 示例
- 运用牛顿第二定律建立受力与运动关系
- 根据匀变速公式解决多阶段运动问题
- 根据实验数据判断误差来源

### 9.3 用途
最适合用于学生状态建模、题目解释和干预建议。

---

## 10. 标准字段定义

能力点表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| ability_id | string | 是 | 能力点唯一标识 |
| subject | string | 是 | 学科 |
| curriculum_version_id | string/null | 否 | 教材版本（如需绑定） |
| ability_name | string | 是 | 能力点名称 |
| ability_definition | string | 是 | 能力点定义 |
| ability_level | string | 是 | 能力层级 |
| ability_type | string | 是 | 能力类型 |
| knowledge_refs | array[string] | 否 | 关联知识点 |
| chapter_refs | array[string] | 否 | 关联章节 |
| exercise_type_refs | array[string] | 否 | 关联题型 |
| misconception_refs | array[string] | 否 | 关联误区 |
| observable_signals | array[string] | 否 | 可观察信号 |
| importance_level | string | 是 | 重要程度 |
| status | string | 是 | active/deprecated |
| source_ref | string/null | 否 | 来源引用 |
| version | string | 是 | 版本号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

---

## 11. 字段说明

### 11.1 ability_id
能力点唯一主键。

建议格式：
`PHY_AP_001`

### 11.2 ability_level
建议值：
- general
- subject
- applied

### 11.3 ability_type
建议值可包括：
- comprehension
- analysis
- modeling
- reasoning
- calculation
- representation
- experiment
- application

### 11.4 observable_signals
用于说明系统如何观察该能力点。

示例：
- 相关题型重复错误
- 图像判断不稳定
- 步骤表达缺失
- 教师反馈提到分析过程薄弱

---

## 12. 能力点命名规范

### 12.1 命名原则
- 简洁
- 明确
- 可教学使用
- 可与知识点配合

### 12.2 推荐写法
- 受力分析能力
- 图像理解与转换能力
- 基于实验现象进行解释的能力
- 建立数量关系模型的能力

### 12.3 不推荐写法
- 理科思维
- 综合素养
- 解题感觉
- 高阶能力 A 类

---

## 13. 能力点抽取流程

建议流程如下：

```text
阅读教材和课程要求
    ↓
识别学科关键能力要求
    ↓
结合典型题型与错因分析
    ↓
抽取候选能力点
    ↓
去重与归类
    ↓
粒度校正
    ↓
标准命名
    ↓
补充定义与映射关系
    ↓
质量校验
```

---

## 14. 与知识点的关系

### 14.1 关系类型

知识点与能力点通常为多对多关系。

### 14.2 原则

- 一个知识点可关联多个能力点
- 一个能力点可关联多个知识点
- 能力点不直接替代知识点
- 两者应共同服务状态建模

---

## 15. 与题型映射表的关系

能力点与题型关系非常关键，建议至少支持：

- 某能力点常出现在哪些题型
- 某题型失败更可能反映哪些能力问题
- 某能力点需要哪些训练题型支持

---

## 16. 与误区表的关系

能力点应尽量关联常见误区，尤其是：

- 条件识别错误
- 表达错误
- 建模错误
- 图像理解错误
- 实验推断错误

这能帮助系统解释"问题不是不知道，而是不会用"。

---

## 17. 与学生状态模型的关系

能力点表将直接服务以下状态：

- 学科能力分布
- 能力短板识别
- 题型失败原因解释
- 风险来源分析
- 干预动作设计

StudentTwinAgent 后续不应只记录知识点掌握，还应逐步能记录关键能力状态。

---

## 18. 质量要求

### 18.1 一致性

命名、粒度、层级必须统一。

### 18.2 可观察性

能力点应能被事件、题型或反馈支撑。

### 18.3 可解释性

定义必须能为老师和系统理解使用。

### 18.4 可映射性

必须能和知识点、题型、误区连接。

### 18.5 可扩展性

支持先做单学科，再扩到多学科。

---

## 19. 常见错误

### 19.1 过于抽象

例如"核心素养""综合思维"这类词，听起来高级，落地时经常一脸空白。

### 19.2 过于细碎

把每一步解题动作都拆成能力点，会导致表爆炸且难以使用。

### 19.3 与知识点混淆

能力点不是知识点换个名字。

### 19.4 与题型混淆

能力点不是题型名录。

---

## 20. MVP 建议范围

MVP 阶段建议先支持：

- 高中物理
- 单一教材版本下的核心能力点
- 与知识点和题型的基础映射
- 与常见误区的基础映射

优先关注：

- 受力分析能力
- 图像理解能力
- 公式应用能力
- 实验分析能力
- 综合建模能力

## 23. 结论

能力点表的核心价值，在于让系统从"学生学了什么"走向"学生会不会用、用得稳不稳"。

它必须坚持：

- 不空
- 不碎
- 可观察
- 可训练
- 可解释

否则能力点表很容易沦为一份看上去很高端、实际上谁都用不起来的教育词库。
