---

## 1. 文档目的

本文档用于定义教育孪生项目中"章节树表"的标准结构、字段规范、层级规则、编码方式与使用方式。

章节树表是教材结构化的骨架表，用于回答以下问题：

1. 一本教材是如何分层组织的
2. 某个知识点位于哪一章、哪一节、哪一单元
3. 教材内容如何与学生学习事件、知识点表、锚点表建立上层结构关系
4. 不同教材版本之间如何做章节层映射

---

## 2. 章节树表定义

章节树表，是对教材目录层级结构的结构化表达。

它不是教材原文，不是知识点表，也不是锚点表，而是教材的"骨架导航层"。

章节树表通常描述以下层级：

- 册
- 单元
- 章
- 节
- 子节
- 主题模块（如需要）

章节树表的核心作用，是为知识点、锚点、事件映射、进度判断和报告展示提供统一的结构定位。

---

## 3. 使用场景

### 3.1 教材结构导航
用于表示某学科某册教材的完整层级结构。

### 3.2 知识点挂载
知识点表中的每个知识点应能挂载到至少一个章节节点。

### 3.3 学习事件映射
学习事件在映射时，可先映射到章节，再进一步映射到知识点。

### 3.4 学习进度判断
系统判断"当前学到哪"时，章节树表是核心参照。

### 3.5 产品展示
家长端、老师端、学校端展示"当前学到第几章第几节"时，依赖章节树表。

---

## 4. 设计原则

### 4.1 层级稳定
章节树应优先反映教材原始结构，不应为了系统方便随意改教材层级。

### 4.2 编码唯一
每个章节节点必须拥有唯一编码，且同一版本内不重复。

### 4.3 父子清晰
每个节点必须能明确找到父节点，顶层节点除外。

### 4.4 结构与内容分离
章节树表只描述结构，不承载完整知识内容解释。

### 4.5 兼容版本映射
不同教材版本章节结构不同，章节树表要支持跨版本映射。

---

## 5. 章节层级规范

建议统一采用以下层级模型：

| 层级 | 英文名 | 说明 |
|------|--------|------|
| L0 | volume | 册/本 |
| L1 | unit | 单元 |
| L2 | chapter | 章 |
| L3 | section | 节 |
| L4 | subsection | 子节 |
| L5 | topic | 主题模块（可选） |

说明：
- 不是所有教材都必须包含全部层级
- 若教材没有"单元"，可直接从册到章
- 若教材存在专题页、实验页、阅读页，可在 topic 层扩展

---

## 6. 标准字段定义

章节树表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| chapter_node_id | string | 是 | 章节节点唯一标识 |
| curriculum_version_id | string | 是 | 教材版本标识 |
| subject | string | 是 | 学科 |
| grade_level | string | 是 | 年级 |
| volume_no | string | 是 | 册别 |
| node_level | string | 是 | 节点层级 |
| node_code | string | 是 | 节点编码 |
| node_name | string | 是 | 节点名称 |
| parent_node_id | string/null | 否 | 父节点标识 |
| root_node_id | string | 是 | 所属根节点标识 |
| order_index | int | 是 | 同层排序序号 |
| full_path_name | string | 是 | 完整路径名称 |
| full_path_code | string | 是 | 完整路径编码 |
| start_page | int/null | 否 | 起始页码 |
| end_page | int/null | 否 | 结束页码 |
| node_type | string | 否 | 普通章节/实验/阅读/复习等 |
| status | string | 是 | 状态：active/deprecated |
| source_ref | string/null | 否 | 原始教材来源引用 |
| version | string | 是 | 结构版本 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

---

## 7. 字段解释

### 7.1 chapter_node_id
章节节点主键，建议全局唯一。

示例：
`PHY_PEP_G1_V1_CH_01_02`

### 7.2 node_level
节点层级，建议枚举值：
- volume
- unit
- chapter
- section
- subsection
- topic

### 7.3 node_code
用于章节内部编码，强调层内可读性。

示例：
- `01`
- `01.02`
- `01.02.03`

### 7.4 full_path_name
完整路径名称，便于展示和检索。

示例：
`必修一 > 第一章 运动的描述 > 第 2 节 时间和位移`

### 7.5 node_type
建议值：
- normal
- experiment
- review
- reading
- appendix
- activity

---

## 8. 编码规则建议

### 8.1 chapter_node_id 编码格式

建议结构：

`学科_教材版本_年级_册别_层级_顺序`

示例：
`PHY_PEP_G1_V1_CH_01`
`PHY_PEP_G1_V1_SEC_01_02`

### 8.2 root_node_id
用于标识同一本教材树的根节点，便于整体聚合。

示例：
`PHY_PEP_G1_V1_ROOT`

### 8.3 full_path_code
用于表达层级路径。

示例：
`V1/U01/CH01/SEC02`

---

## 9. 父子关系规则

### 9.1 顶层节点
顶层 volume 节点的 `parent_node_id` 为空。

### 9.2 非顶层节点
必须有且仅有一个父节点。

### 9.3 同层排序
同一父节点下，`order_index` 不可重复。

### 9.4 禁止循环引用
任意节点不得形成祖先循环。

---

## 10. 与知识点表的关系

章节树表与知识点表的关系应遵循以下规则：

1. 一个知识点至少挂载到一个章节节点
2. 一个章节节点可挂载多个知识点
3. 一个知识点允许挂载到多个章节节点，但必须指定主挂载节点
4. 章节树负责"结构定位"，知识点表负责"内容抽象"

---

## 11. 与教材锚点表的关系

章节树表与教材锚点表的关系如下：

- 章节节点可映射多个锚点
- 锚点表记录更细粒度的位置（页码、段落、图例、实验块等）
- 章节树用于粗定位，锚点表用于精定位

---

## 12. 与学习事件的关系

学习事件映射章节时应优先遵循以下路径：

原始输入  
→ 学科识别  
→ 教材版本识别  
→ 章节节点识别  
→ 知识点进一步映射

说明：
- 章节映射失败不一定导致事件无效
- 但高价值事件应尽量补齐章节映射

---

## 13. 数据质量要求

章节树表至少满足以下质量标准：

### 13.1 完整性
教材目录结构不能大面积缺失。

### 13.2 一致性
同一教材版本内，命名方式、层级口径、排序逻辑必须统一。

### 13.3 可追溯性
每个节点最好能映射回教材原始目录或页码来源。

### 13.4 可映射性
必须能支持知识点、锚点、事件和进度引用。

---

## 14. 常见错误

### 14.1 把章节解释写进章节树
章节树不是知识点说明表，不要塞长段解释。

### 14.2 把目录层级改成产品层级
教材原始结构和产品展示结构可以不同，但章节树应优先忠实教材原始结构。

### 14.3 同一节点多套编码
编码一旦定下，不应随意改动，否则后续映射全乱。

### 14.4 把专题页误当知识点节点
专题、阅读、实验页可以作为 node_type 特殊节点，但不要和知识点节点混淆。

---

## 15. MVP 最小标准

MVP 阶段章节树表至少应支持：

- 高中物理
- 单一地区
- 单一教材版本
- volume / chapter / section 三层结构
- 起止页码
- 与知识点表的主挂载关系

## 18. 结论

章节树表是教材结构化的骨架层。

如果没有章节树表：

- 教材内容无法形成稳定导航结构
- 知识点挂载会失去上层位置
- 学习事件难以粗定位
- 学习进度判断会失去统一口径

因此章节树表必须优先建立，并保持：

- 结构清晰
- 编码统一
- 层级稳定
- 可追溯
