# 教材版本映射规则

> 文档编号：STD-012  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：教材标准表负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中的"教材版本映射规则"，包括版本映射对象、映射层级、映射类型、映射强度、使用场景与质量要求。

本文档重点回答以下问题：

1. 为什么不同教材版本之间必须建立映射规则
2. 映射到底映什么，是章节、知识点还是能力点
3. 一对一、一对多、多对一场景如何处理
4. 版本映射如何服务学生迁移、地区适配、推演解释和跨版本比较
5. 如何避免把不同版本教材硬凑成一张假装精确的对照表

---

## 2. 教材版本映射定义

教材版本映射，是指不同教材版本之间在章节、知识点、能力点、题型或教学内容层面的对应关系表达。

它的任务是回答：

- 版本 A 中的某一章节，在版本 B 里大致对应什么
- 某知识点在不同版本中是否存在等价表达
- 哪些内容是部分重合，哪些内容无法直接对应
- 学生从一种版本切换到另一种版本时，系统应如何理解其历史状态

版本映射不是要求不同教材版本完全一致，而是提供一个"尽可能可靠的对照桥"。

---

## 3. 建设目标

### 3.1 支撑地区和学校差异
不同地区、学校或年级使用不同教材版本时，系统需要知道它们之间如何连接。

### 3.2 支撑学生迁移
学生转学、换地区、换教材版本时，历史学习状态不能全部失忆。

### 3.3 支撑推演与解释
系统在引用跨版本数据时，需要知道哪些内容可比、哪些不可比。

### 3.4 支撑版本迭代
同一出版社不同年份版本更新时，需要保留新旧版本衔接关系。

### 3.5 控制误映射风险
比"完全没映射"更危险的是"映射得很自信但其实不对"。

---

## 4. 设计原则

### 4.1 可映射的不强行等同
映射表示相对对应，不表示完全一致。

### 4.2 粒度分层
章节映射、知识点映射、能力点映射应分层处理。

### 4.3 强度区分
不同映射关系必须标明是强对应、部分对应还是弱对应。

### 4.4 不可映射要允许为空
某些内容就是没有稳定对应，不要为了整齐硬连线。

### 4.5 说明优先
关键映射关系尽量带说明，解释为何成立。

---

## 5. 映射对象层级建议

建议至少支持以下三层映射：

| 层级 | 说明 | MVP 优先级 |
|------|------|----------|
| 章节层映射 | 版本间章节对应关系 | P0 |
| 知识点层映射 | 版本间知识点对应关系 | P0 |
| 能力点层映射 | 版本间能力点对应关系 | P1 |

---

## 6. 章节层映射

### 6.1 定义
版本 A 中的某章节，与版本 B 中的某章节存在整体内容对应关系。

### 6.2 用途
- 粗粒度迁移
- 报告解释
- 教材切换初始适配

### 6.3 特点
最易建立，但精度相对有限。

### 6.4 示例
| 版本 A | 版本 B | 映射类型 |
|--------|--------|----------|
| 第一章 运动的描述 | 第一章 机械运动 | equivalent |
| 第二章 相互作用 | 第三章 力 | partial_overlap |

---

## 7. 知识点层映射

### 7.1 定义
版本 A 中的某个知识点，与版本 B 中的某个知识点存在内容层面的对应关系。

### 7.2 用途
- 学生状态迁移
- 事件映射兼容
- 风险解释与知识路径比较

### 7.3 特点
价值最高，也最需要谨慎。

### 7.4 示例
| 版本 A 知识点 | 版本 B 知识点 | 映射类型 | 强度 |
|--------------|--------------|----------|------|
| 受力分析基本方法 | 力的分析方法 | equivalent | strong |
| 牛顿第二定律 | 牛顿运动定律（二） | equivalent | strong |
| 匀变速直线运动公式 | 运动学公式 | partial_overlap | medium |

---

## 8. 能力点层映射

### 8.1 定义
不同版本下围绕类似知识内容形成的能力要求之间的对应关系。

### 8.2 用途
- 学生能力状态延续
- 干预动作迁移
- 推演解释

### 8.3 特点
能力点层映射通常依赖知识点映射，不建议第一阶段强推太细。

---

## 9. 映射关系类型建议

建议至少定义以下映射类型：

| 映射类型 | 编码 | 含义 |
|----------|------|------|
| equivalent | equivalent | 基本等价，内容结构和教学意义高度接近 |
| partial_overlap | partial_overlap | 部分重合，有共同内容但不完全一致 |
| broader_than | broader_than | 源对象范围比目标对象更宽 |
| narrower_than | narrower_than | 源对象范围比目标对象更窄 |
| split_to | split_to | 在目标版本中被拆分成多个对象 |
| merged_from | merged_from | 在目标版本中由多个源对象合并 |
| no_direct_match | no_direct_match | 无直接对应内容 |

---

## 10. 映射强度建议

建议使用以下强度字段：

| 强度 | 编码 | 说明 |
|------|------|------|
| 强 | strong | 内容结构和教学意义都高度接近 |
| 中 | medium | 大体对应，但细节有差异 |
| 弱 | weak | 仅在较粗层级可参考，不应直接作精细迁移 |

---

## 11. 标准字段定义

教材版本映射规则表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| mapping_rule_id | string | 是 | 映射规则唯一标识 |
| source_curriculum_version_id | string | 是 | 源版本标识 |
| target_curriculum_version_id | string | 是 | 目标版本标识 |
| source_object_id | string | 是 | 源对象 ID |
| source_object_type | string | 是 | 源对象类型 |
| target_object_id | string/null | 否 | 目标对象 ID |
| target_object_type | string/null | 否 | 目标对象类型 |
| mapping_type | string | 是 | 映射类型 |
| mapping_strength | string | 是 | 映射强度 |
| description | string/null | 否 | 说明 |
| valid_from | date/null | 否 | 生效时间 |
| valid_to | date/null | 否 | 失效时间 |
| source_ref | string/null | 否 | 来源引用 |
| status | string | 是 | active/deprecated |
| version | string | 是 | 版本号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

---

## 12. 字段说明

### 12.1 mapping_rule_id
映射规则唯一主键。

建议格式：`MAP_PHY_V1_TO_V2_001`

编码结构：`MAP_学科_源版本_目标版本_顺序号`

### 12.2 source_curriculum_version_id / target_curriculum_version_id
源版本和目标版本的教材版本标识。

建议格式：`PHY_PEP_G1_V1`

### 12.3 source_object_type / target_object_type
对象类型。

建议值：
- chapter（章节）
- knowledge_point（知识点）
- ability_point（能力点）

### 12.4 mapping_type
映射关系类型。

| 值 | 说明 | 使用场景 |
|------|------|----------|
| equivalent | 基本等价 | 内容高度一致 |
| partial_overlap | 部分重合 | 有共同内容但不完全一致 |
| broader_than | 范围更宽 | 源对象包含目标对象 |
| narrower_than | 范围更窄 | 源对象被目标对象包含 |
| split_to | 拆分 | 源对象在目标版本中拆分成多个 |
| merged_from | 合并 | 源对象由目标版本中多个对象合并 |
| no_direct_match | 无直接对应 | 没有稳定对应内容 |

### 12.5 mapping_strength
映射强度。

| 值 | 说明 |
|------|------|
| strong | 强对应，可安全迁移 |
| medium | 中等对应，需谨慎迁移 |
| weak | 弱对应，仅供参考 |

### 12.6 description
用于说明对应成立或不成立的原因，尤其在 partial、split、merged 等场景下非常重要。

示例：
- "两版本中该知识点的核心定义与教学作用基本一致，仅命名表述略有不同"
- "版本 A 中此知识点涵盖内容更广，版本 B 中拆分为两个独立知识点"

## 13. 映射方向规则

建议统一方向为：

```
source_version -> target_version
```

即：从源版本对象映射到目标版本对象。

这样可支持：
- 学生从旧版本迁移到新版本
- 某地区从标准版本对照到本地版本
- 推演层做明确方向的转换逻辑

**原则**：不允许一会儿正向一会儿反向，否则迁移逻辑会混乱。

---

## 14. 一对一映射

### 14.1 定义
源对象与目标对象基本一一对应。

### 14.2 适用场景
- 章节名称变化不大
- 知识点表达基本一致
- 只存在轻微表述差异

### 14.3 处理方式
可用：
- mapping_type = equivalent
- mapping_strength = strong

### 14.4 示例
```json
{
  "mapping_rule_id": "MAP_PHY_V1_TO_V2_001",
  "source_curriculum_version_id": "PHY_PEP_G1_V1",
  "target_curriculum_version_id": "PHY_PEP_G1_V2",
  "source_object_id": "PHY_PEP_G1_V1_KP_002",
  "source_object_type": "knowledge_point",
  "target_object_id": "PHY_PEP_G1_V2_KP_002",
  "target_object_type": "knowledge_point",
  "mapping_type": "equivalent",
  "mapping_strength": "strong",
  "description": "两版本中该知识点的核心定义与教学作用基本一致，仅命名表述略有不同。"
}
```

---

## 15. 一对多映射

### 15.1 定义
源对象在目标版本中被拆成多个对象。

### 15.2 适用场景
- 一个章节拆成多个小节
- 一个综合知识点被细分成多个知识点

### 15.3 处理方式
- 使用 `mapping_type = split_to`
- 一条源对象可对应多条目标记录

### 15.4 示例
```json
{
  "mapping_rule_id": "MAP_PHY_V1_TO_V2_015",
  "source_curriculum_version_id": "PHY_PEP_G1_V1",
  "target_curriculum_version_id": "PHY_PEP_G1_V2",
  "source_object_id": "PHY_PEP_G1_V1_KP_008",
  "source_object_type": "knowledge_point",
  "target_object_id": "PHY_PEP_G1_V2_KP_015",
  "target_object_type": "knowledge_point",
  "mapping_type": "split_to",
  "mapping_strength": "medium",
  "description": "版本 V1 中此知识点在 V2 中被拆分为两个独立知识点，此为第一部分对应。"
}
```

---

## 16. 多对一映射

### 16.1 定义
源版本多个对象在目标版本中被合并成一个更大对象。

### 16.2 适用场景
- 若干小知识点被合并描述
- 两个章节被整合成一个主题单元

### 16.3 处理方式
- 使用 `mapping_type = merged_from`
- 建议保留说明字段，写清范围

### 16.4 示例
```json
{
  "mapping_rule_id": "MAP_PHY_V1_TO_V2_020",
  "source_curriculum_version_id": "PHY_PEP_G1_V1",
  "target_curriculum_version_id": "PHY_PEP_G1_V2",
  "source_object_id": "PHY_PEP_G1_V1_KP_012",
  "source_object_type": "knowledge_point",
  "target_object_id": "PHY_PEP_G1_V2_KP_010",
  "target_object_type": "knowledge_point",
  "mapping_type": "merged_from",
  "mapping_strength": "medium",
  "description": "版本 V1 中此知识点与另一知识点在 V2 中被合并为一个综合知识点。"
}
```

---

## 17. 无直接映射场景

### 17.1 定义
某对象在目标版本中没有稳定对应对象。

### 17.2 处理原则
- 允许 `target_object_id = null`
- `mapping_type = no_direct_match`
- 必须补充说明

### 17.3 原则
没有直接对应并不可耻，强行对应才更可怕。

### 17.4 示例
```json
{
  "mapping_rule_id": "MAP_PHY_V1_TO_V2_030",
  "source_curriculum_version_id": "PHY_PEP_G1_V1",
  "target_curriculum_version_id": "PHY_PEP_G1_V2",
  "source_object_id": "PHY_PEP_G1_V1_KP_025",
  "source_object_type": "knowledge_point",
  "target_object_id": null,
  "target_object_type": null,
  "mapping_type": "no_direct_match",
  "mapping_strength": "weak",
  "description": "该知识点为 V1 版本特有内容，V2 版本中无直接对应，相关内容分散在其他章节。"
}
```

## 18. 映射建立流程建议

```text
确定源版本与目标版本
    ↓
先做章节层对齐
    ↓
再做知识点层对齐
    ↓
必要时补能力点层对齐
    ↓
标记映射类型与强度
    ↓
补充说明与边界
    ↓
质量校验
```

### 18.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 确定版本 | 明确源版本和目标版本 | 版本标识 |
| 章节层对齐 | 建立章节间对应关系 | 章节映射表 |
| 知识点层对齐 | 建立知识点间对应关系 | 知识点映射表 |
| 能力点层对齐 | 必要时建立能力点对应 | 能力点映射表 |
| 标记类型强度 | 标注 mapping_type 和 mapping_strength | 映射标注 |
| 补充说明 | 补充 description 字段 | 说明文本 |
| 质量校验 | 按质量要求校验 | 校验报告 |

## 19. 与学生迁移的关系

学生从版本 A 迁到版本 B 时，系统可参考该表：

| 迁移项 | 说明 |
|--------|------|
| 迁移已掌握知识点 | equivalent 映射可直接迁移 |
| 标记部分对应内容 | partial_overlap 需重新确认 |
| 识别需补齐或重新确认的内容 | no_direct_match 需特别处理 |
| 避免历史学习记录完全断裂 | 保持状态连续性 |

**但迁移不能机械复制。**  
映射表是桥，不是魔法复制器。

---

## 20. 与地区教材适配表的关系

| 表 | 回答的问题 |
|------|----------|
| 地区教材适配表 | "某地区当前该用哪个版本" |
| 教材版本映射规则 | "这个版本和那个版本之间怎么对应" |

两者联动后，系统才能支持：

| 场景 | 说明 |
|------|------|
| 地区切换 | 学生转学到不同教材版本地区 |
| 学校转入 | 转入学校使用不同教材版本 |
| 版本替换 | 教材版本更新换代 |
| 跨版本解释 | 解释时引用不同版本内容 |

---

## 21. 与学生状态模型的关系

教材版本映射规则将影响：

| 影响项 | 说明 |
|--------|------|
| 知识点状态迁移 | 版本切换后知识点状态如何延续 |
| 能力点状态延续 | 能力状态如何跨版本保持 |
| 风险解释的连续性 | 风险解释不因版本切换而断裂 |
| 推演边界的重建 | 推演边界在新版本中重新建立 |

没有映射规则，版本一换，学生历史状态很容易被系统当成失忆。

---

## 22. 质量要求

### 22.1 方向一致

| 要求 | 说明 |
|------|------|
| 全表必须统一源到目标方向 | 不允许混用方向 |

### 22.2 关系清楚

| 要求 | 说明 |
|------|------|
| 映射类型不能模糊 | 不能一律 equivalent |
| 类型选择需有依据 | 不能随意标注 |

### 22.3 强度稳定

| 要求 | 说明 |
|------|------|
| strong / medium / weak 要有相对统一判断 | 强度标准一致 |
| 强度标注需有教学依据 | 不能主观随意 |

### 22.4 允许不映射

| 要求 | 说明 |
|------|------|
| 不要为了完整率造假精度 | no_direct_match 是合法选项 |
| 不确定时可标注 weak 或留空 | 谨慎优于错误 |

### 22.5 可解释

| 要求 | 说明 |
|------|------|
| 重点映射应能说明原因 | description 字段充分利用 |
| 特殊映射需补充说明 | split/merged 场景必须说明 |

---

## 23. 常见错误

### 23.1 强行一一对应

| 错误 | 后果 |
|------|------|
| 实际上很多版本内容不是一一对应 | 映射关系失真，误导状态迁移 |

### 23.2 不区分章节映射和知识点映射

| 错误 | 后果 |
|------|------|
| 粗粒度映射被误拿去做细粒度状态迁移 | 迁移精度错误 |

### 23.3 把部分对应写成完全等价

| 错误 | 后果 |
|------|------|
| partial_overlap 写成 equivalent | 后续会直接污染状态和解释 |

### 23.4 没有方向

| 错误 | 后果 |
|------|------|
| 源版本和目标版本混用 | 没有稳定迁移逻辑 |

### 23.5 说明字段空置

| 错误 | 后果 |
|------|------|
| description 字段不填写 | 后续无法理解映射依据 |

---

## 24. MVP 建议范围

MVP 阶段建议优先支持：

### 24.1 版本范围

| 范围 | 说明 |
|------|------|
| 核心试点地区相关教材版本 | 如人教版 V1 到 V2 |
| 同一出版社不同年份版本 | 版本迭代场景 |

### 24.2 内容范围

| 内容 | 说明 |
|------|------|
| 章节层映射 | 优先完成 |
| 核心知识点层映射 | 高价值知识点优先 |
| `equivalent / partial_overlap / no_direct_match` 三类最小表达 | 先跑通基础类型 |

### 24.3 后置能力

| 能力 | 说明 |
|------|------|
| 能力点全量映射 | 后续扩展 |
| 多版本复杂链式映射 | 多版本对照场景 |
| 自动映射辅助 | 工具辅助映射建立 |

## 25. 推荐样例

### 25.1 知识点等价映射

```json
{
  "mapping_rule_id": "MAP_PHY_PEP_V1_TO_V2_001",
  "source_curriculum_version_id": "PHY_PEP_G1_V1",
  "target_curriculum_version_id": "PHY_PEP_G1_V2",
  "source_object_id": "PHY_PEP_G1_V1_KP_002",
  "source_object_type": "knowledge_point",
  "target_object_id": "PHY_PEP_G1_V2_KP_002",
  "target_object_type": "knowledge_point",
```

## 28. 风险与边界

### 28.1 不能忽视映射方向

| 风险 | 说明 |
|------|------|
| 源版本和目标版本混用 | 迁移逻辑混乱，状态迁移错误 |

### 28.2 不能忽视映射类型准确性

| 风险 | 说明 |
|------|------|
| 把部分对应写成完全等价 | 污染状态和解释，误导决策 |

### 28.3 不能忽视强度区分

| 风险 | 说明 |
|------|------|
| 强度标注随意 | 无法判断迁移可靠性 |

### 28.4 不能忽视说明字段

| 风险 | 说明 |
|------|------|
| description 字段空置 | 后续无法理解映射依据 |

### 28.5 不能强行映射

| 风险 | 说明 |
|------|------|
| 没有对应内容强行建立映射 | 比不映射更危险 |

## 29. 结论

教材版本映射规则的核心意义，是让系统在面对现实世界中"不止一套教材版本"的情况下，仍然能够保持状态连续、解释连续和结构连续。

它必须坚持：

| 原则 | 说明 |
|------|------|
| **分层映射** | 章节层、知识点层、能力点层分开处理 |
| **方向明确** | 统一源版本到目标版本方向 |
| **强度明确** | strong/medium/weak 有判断依据 |
| **允许不映射** | no_direct_match 是合法选项 |
| **说明清楚** | description 字段充分利用 |

否则系统只要一遇到跨版本场景，就会从"教育孪生"瞬间退化成"教育失忆"。
