# 知识能力映射表标准

> 文档编号：STD-006  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：教材标准表负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中的"知识能力映射表"标准，包括知识点与能力点之间的映射方式、字段结构、映射强度、使用场景与质量要求。

本文档重点回答以下问题：

1. 为什么知识点表和能力点表之间还需要单独一张映射表
2. 知识点与能力点之间应如何建立关系
3. 映射强度和关系类型应如何表达
4. 这张表如何服务学生状态、题型解释、干预动作和推演层
5. 如何避免把知识和能力生硬捆绑成一团

---

## 2. 知识能力映射定义

知识能力映射，是指"某个知识点通常支撑哪些能力点"以及"某个能力点通常依赖哪些知识点"的结构化关系表达。

这张表的意义在于回答：

- 学会这个知识点，通常是在训练什么能力
- 某个能力短板，可能对应哪些知识基础
- 一道题错了，到底更像是知识问题、能力问题，还是两者共同作用
- 干预应先补知识还是先补能力表现

知识能力映射不是依赖关系表的替代品。  
依赖关系更强调前后承接，映射表更强调"支撑关系"和"表现关系"。

---

## 3. 建设目标

### 3.1 连接知识层与能力层
让知识点表和能力点表之间形成可计算、可解释的桥梁。

### 3.2 支撑状态建模
学生状态不能只停留在"知识点掌握"，还要逐步表达"能力表现"。

### 3.3 支撑题目解释
帮助系统区分：
- 是不会知识点
- 还是会知识点但能力没形成
- 还是知识和能力一起薄弱

### 3.4 支撑干预设计
系统可据此决定：
- 先补知识
- 先练能力
- 还是同步做低强度双路径补强

### 3.5 支撑推演层
后台推演在比较动作效果时，需要知道动作影响的是知识层、能力层，还是两者联动。

---

## 4. 设计原则

### 4.1 多对多
一个知识点可映射多个能力点，一个能力点也可映射多个知识点。

### 4.2 支撑优先
映射关系重点表达"支撑"和"体现"，而不是泛泛关联。

### 4.3 强度可区分
不是所有知识点对某能力点的作用都一样强。

### 4.4 可解释
映射关系应能说清楚为什么成立。

### 4.5 可用于教学
老师看得懂，系统算得动，干预能落地。

---

## 5. 典型关系场景

### 5.1 知识点支撑能力点
例如：
- "受力分析基本方法"支撑"受力分析能力"
- "匀变速公式理解"支撑"数量关系建模能力"

### 5.2 能力点依赖多个知识点
例如：
- "实验分析能力"可能依赖实验原理知识、数据读取知识、误差理解知识

### 5.3 一个知识点对多个能力点有不同强度影响
例如：
- 牛顿第二定律既支撑公式应用能力，也支撑建模能力，但强度可能不同

---

## 6. 映射关系类型建议

建议映射表至少支持以下关系类型：

| 关系类型 | 含义 |
|----------|------|
| supports | 知识点支撑能力点 |
| strongly_supports | 强支撑 |
| partially_supports | 部分支撑 |
| manifests_in | 某能力通常体现于该知识点应用 |
| requires_knowledge_of | 某能力依赖相关知识基础 |

MVP 阶段建议先统一使用：
- `supports`
再通过强度字段表达差异。

---

## 7. 标准字段定义

知识能力映射表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| mapping_id | string | 是 | 映射唯一标识 |
| subject | string | 是 | 学科 |
| curriculum_version_id | string/null | 否 | 教材版本 |
| knowledge_id | string | 是 | 知识点 ID |
| ability_id | string | 是 | 能力点 ID |
| mapping_type | string | 是 | 映射类型 |
| mapping_strength | string | 是 | 映射强度 |
| description | string/null | 否 | 映射说明 |
| observable_signals | array[string] | 否 | 可观察信号 |
| exercise_type_refs | array[string] | 否 | 相关题型 |
| misconception_refs | array[string] | 否 | 相关误区 |
| source_ref | string/null | 否 | 来源引用 |
| status | string | 是 | active/deprecated |
| version | string | 是 | 版本号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

---

## 8. 字段说明

### 8.1 mapping_id
映射唯一主键。

建议格式：`PHY_KA_001`

### 8.2 mapping_type
映射关系类型。

建议值：
- supports（支撑）
- manifests_in（体现于）
- requires_knowledge_of（依赖知识）

### 8.3 mapping_strength
映射强度等级。

建议值：
- core（核心支撑）
- major（主要支撑）
- partial（部分支撑）

### 8.4 observable_signals
用于说明系统如何观察这条映射是否出现问题。

示例：
- 该知识点相关题型反复错误
- 题目过程正确但表达和建模不稳定
- 教师反馈提到不会从条件建立关系

---

## 9. 映射方向规则

建议默认方向统一为：

`knowledge_id -> ability_id`

即：某知识点对某能力点形成支撑。

这样做的好处是：
- 更符合教材结构向能力要求展开的表达方式
- 更利于从知识底座出发组织结构
- 方向统一便于查询和推演

若需要反向查询，可由系统查询层处理，不建议在表里一会儿正向一会儿反向。

---

## 10. 映射强度设计

### 10.1 core（核心支撑）

| 特征 | 说明 |
|------|------|
| 影响程度 | 该知识点是该能力点形成的核心支撑之一 |
| 判断标准 | 缺失该知识点时，能力很难稳定建立 |

### 10.2 major（主要支撑）

| 特征 | 说明 |
|------|------|
| 影响程度 | 该知识点对能力形成有重要支持作用 |
| 判断标准 | 缺失时会明显影响能力表现，但不一定完全阻断 |

### 10.3 partial（部分支撑）

| 特征 | 说明 |
|------|------|
| 影响程度 | 该知识点对能力形成有辅助作用 |
| 判断标准 | 可增强能力，但不是主要决定因素 |

---

## 11. 抽取流程建议

```text
确定知识点清单
    ↓
确定能力点清单
    ↓
逐项识别知识点对能力点的支撑关系
    ↓
判断映射类型
    ↓
判断映射强度
    ↓
补充说明和可观察信号
    ↓
关联题型与误区
    ↓
质量校验
```

---

## 12. 与题型映射表的关系

知识能力映射和题型映射常形成链路：

```
知识点 → 支撑能力点 → 在某类题型中体现 → 产生某类错误或误区
```

因此题型映射表可作为知识能力映射的观测层补充。

---

## 13. 与误区表的关系

知识能力映射表应尽量补充相关误区引用，帮助系统解释：

- 是知识理解错了
- 还是能力运用错了
- 还是两者共同导致错误

例如：
- "受力分析基本方法" → "受力分析能力"
- 可关联误区：漏画力、方向判断错误、研究对象切换错误

---

## 14. 与学生状态模型的关系

该表后续将直接支撑以下状态表达：

| 状态类型 | 说明 |
|----------|------|
| 知识点已接触但能力未稳定形成 | 知识有但能力弱 |
| 某能力点波动可能来自哪组知识基础 | 追溯知识根源 |
| 某题型反复失误更像知识缺口还是能力短板 | 错误归因 |
| 学科状态中的"能力型风险"解释 | 风险解释 |

---

## 15. 与干预动作库的关系

干预动作库可借助该表实现更精细的动作推荐：

| 场景 | 干预策略 |
|------|----------|
| 某知识点不稳，但能力暂未明显短板 | 优先知识复盘 |
| 某知识点已接触，但能力仍不稳定 | 增加题型练习或过程训练 |
| 多个知识点共同映射到同一能力点 | 可优先围绕能力组织训练 |

---

## 16. 质量要求

### 16.1 映射逻辑清楚

| 要求 | 说明 |
|------|------|
| 不能为了凑表强行映射 | 避免把所有知识点都绑几个能力点 |

### 16.2 强度稳定

| 要求 | 说明 |
|------|------|
| 不同映射强度需要相对稳定的判断标准 | 不能 core 满天飞 |

### 16.3 可观察

| 要求 | 说明 |
|------|------|
| 尽量能够通过题目、事件和反馈观察出来 | 有observable_signals 支撑 |

### 16.4 不过度泛化

| 要求 | 说明 |
|------|------|
| 避免出现"几乎所有知识点都支撑几乎所有能力点" | 保持区分度 |

### 16.5 学科适配

| 要求 | 说明 |
|------|------|
| 不同学科映射方式不同 | 不建议强行用一套模板全学科硬推 |

---

## 17. 常见错误

### 17.1 映射过满

| 错误 | 后果 |
|------|------|
| 把每个知识点都映射给很多能力点 | 后续没有区分度 |

### 17.2 映射过空

| 错误 | 后果 |
|------|------|
| 只映射最 obvious 的几条 | 系统连接弱 |

### 17.3 强度乱标

| 错误 | 后果 |
|------|------|
| 没有标准，最后 core 满天飞 | 强度失去意义 |

### 17.4 把题型关系误当知识能力关系

| 错误 | 后果 |
|------|------|
| 题型是表现层，不能直接替代映射逻辑 | 映射逻辑混乱 |

---

## 18. MVP 建议范围

MVP 阶段建议优先支持：

### 18.1 学科范围

| 范围 | 说明 |
|------|------|
| 高中物理 | 首建学科 |

### 18.2 内容范围

| 内容 | 说明 |
|------|------|
| 核心知识点到核心能力点映射 | 优先覆盖高价值映射 |
| 与题型和误区的基础联动 | 支撑事件解释 |
| `supports + core/major/partial` | 最小表达集 |

### 18.3 建议优先覆盖

| 映射 | 说明 |
|------|------|
| 受力分析相关映射 | 受力分析基本方法 → 受力分析能力 |
| 运动学公式应用相关映射 | 匀变速公式 → 数量关系建模能力 |
| 图像理解相关映射 | 图像知识点 → 图像理解能力 |
| 实验分析相关映射 | 实验知识 → 实验分析能力 |

## 22. 风险与边界

### 22.1 不能忽视映射逻辑

| 风险 | 说明 |
|------|------|
| 强行映射导致后续状态解释混乱 | 映射必须有教学依据 |

### 22.2 不能忽视强度区分

| 风险 | 说明 |
|------|------|
| 强度不分导致干预优先级判断失效 | core/major/partial 要有依据 |

### 22.3 不能忽视学科差异

| 风险 | 说明 |
|------|------|
| 不同学科映射特点不同 | 不能强行统一模板 |

### 22.4 不能忽视可观察性

| 风险 | 说明 |
|------|------|
| 无法通过事件和反馈观察的映射没有实际价值 | 必须有 observable_signals |

---

## 23. 结论

知识能力映射表的核心价值，是把"学了什么"和"能不能用出来"这两层真正连接起来。

它必须坚持：

- **映射清楚** - 知识点与能力点关系明确
- **强度清楚** - core/major/partial 有判断依据
- **能观察** - 通过题目、事件、反馈可识别
- **能解释** - 支撑学生状态和错误归因
- **能服务状态与干预** - 指导干预动作设计

否则系统就会长期卡在一个很典型的半成品阶段：知道学生错了，但说不清到底是知识没到位，还是能力没长出来。
