## 1. 文档目的

本文档用于定义教育孪生项目中的"依赖关系表"标准，包括依赖关系类型、字段结构、关系方向、强弱等级、使用场景与质量规则。

本文档重点回答以下问题：

1. 什么叫依赖关系
2. 知识点、能力点、章节之间的依赖关系如何表达
3. 系统为什么必须有依赖关系表
4. 依赖关系如何服务 StudentTwinAgent、推演层和图谱层
5. 如何避免把关系随便连成一张教育蜘蛛网

---

## 2. 依赖关系定义

依赖关系，是指两个结构化对象之间存在"前置、支撑、影响、包含、承接"这类有方向性的结构关系。

在教育孪生项目中，依赖关系主要用来表达：

- 某知识点学习前需要哪些基础
- 某能力点建立依赖哪些知识或能力
- 某章节推进依赖哪些前置内容
- 某问题为何会影响后续学习路径

依赖关系不是普通"关联"。  
它强调方向、强弱和作用意义。

---

## 3. 依赖关系表的目标

### 3.1 支撑学习路径理解
系统需要知道"为什么这里出问题，会影响后面哪里"。

### 3.2 支撑风险识别
识别前置知识缺口如何造成后续风险。

### 3.3 支撑干预设计
帮助系统知道先补什么，再补什么。

### 3.4 支撑推演层
后台推演需要依赖关系来比较学习路径和动作效果。

### 3.5 支撑图谱建模
依赖关系是图谱中的核心边类型之一。

---

## 4. 设计原则

### 4.1 方向明确
依赖关系必须明确从谁指向谁。

### 4.2 关系有类型
不能所有关系都叫"related_to"，那不叫建模，那叫偷懒。

### 4.3 关系有强弱
不是所有前置关系同等重要。

### 4.4 关系有边界
只记录对学习和推演真正重要的依赖，不乱连。

### 4.5 关系可解释
每条关键依赖关系最好能说清为什么存在。

---

## 5. 关系对象范围

依赖关系表建议至少支持以下对象之间的关系：

1. 知识点 -> 知识点
2. 能力点 -> 能力点
3. 知识点 -> 能力点
4. 章节 -> 章节
5. 章节 -> 知识点（可选）
6. 能力点 -> 知识点（可选）

MVP 阶段建议优先做：
- 知识点到知识点
- 知识点到能力点

---

## 6. 关系类型建议

建议至少定义以下关系类型：

| 关系类型 | 含义 | 示例 |
|----------|------|------|
| prerequisite_of | 前置关系 | 受力分析是牛顿定律应用的前置 |
| supports | 支撑关系 | 基础公式理解支撑综合应用 |
| depends_on | 依赖关系 | 综合应用依赖基础公式 |
| leads_to | 承接到后续 | 基础概念学习导向进阶应用 |
| constrains | 对后续形成限制 | 前置掌握程度限制后续进度 |
| maps_to | 映射关系（谨慎使用） | 知识点映射到能力点 |

说明：
- `prerequisite_of` 更强调先学后学
- `supports` 更强调能力或理解支撑
- `depends_on` 可用于较强结构依赖

---

## 7. 标准字段定义

依赖关系表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| relation_id | string | 是 | 关系唯一标识 |
| source_ref_id | string | 是 | 起点对象 ID |
| source_ref_type | string | 是 | 起点对象类型 |
| target_ref_id | string | 是 | 终点对象 ID |
| target_ref_type | string | 是 | 终点对象类型 |
| relation_type | string | 是 | 关系类型 |
| dependency_strength | string | 是 | 依赖强度 |
| description | string/null | 否 | 关系说明 |
| subject | string | 是 | 学科 |
| curriculum_version_id | string/null | 否 | 教材版本 |
| status | string | 是 | active/deprecated |
| source_ref | string/null | 否 | 来源引用 |
| version | string | 是 | 版本号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

---

## 8. 字段说明

### 8.1 relation_id
关系主键，建议唯一。

示例：
`PHY_REL_0001`

### 8.2 source_ref_id / target_ref_id
表示关系的方向。

示例：
`knowledge_A depends_on knowledge_B`

则：
- source_ref_id = knowledge_A
- target_ref_id = knowledge_B

### 8.3 dependency_strength
建议值：
- strong（强依赖）
- medium（中等依赖）
- weak（弱依赖）

### 8.4 description
用于说明依赖成立的原因。

---

## 9. 关系方向规则

### 9.1 方向必须固定
同一种关系在全系统中方向必须统一。

例如：
- `depends_on`：当前对象依赖目标对象
- `prerequisite_of`：当前对象是目标对象的前置

### 9.2 不允许一会儿正向一会儿反向
否则后续推演和图谱查询会像在迷宫里转圈。

---

## 10. 知识点依赖关系

### 10.1 定义
表示某知识点是否依赖其他知识点作为前置基础。

### 10.2 示例
- 牛顿第二定律应用 depends_on 受力分析基本方法
- 匀变速直线运动综合应用 depends_on 基本运动公式理解

### 10.3 用途
- 学生状态解释
- 风险追溯
- 学习路径调整
- 干预优先级判断

---

## 11. 能力点依赖关系

### 11.1 定义
表示某能力点的形成是否依赖其他能力点。

### 11.2 示例
- 综合建模能力 depends_on 条件分析能力
- 实验解释能力 depends_on 信息提取能力

### 11.3 用途
- 分析能力短板链条
- 支持干预动作设计
- 丰富能力状态建模

---

## 12. 知识点到能力点关系

### 12.1 定义
表示某知识点学习通常会支撑哪些能力点，或某能力点依赖哪些知识内容。

### 12.2 示例
- 受力分析基本方法 supports 受力分析能力
- 匀变速公式应用 supports 数量关系建模能力

### 12.3 用途
- 将知识状态和能力状态连接起来
- 支持解释"明明学了，但能力没起来"的情况

---

## 13. 章节依赖关系

### 13.1 定义
表示章节之间的推进关系和承接关系。

### 13.2 用途
- 粗粒度进度判断
- 报告表达
- 教学路径推演

### 13.3 说明
章节依赖通常较粗，不能替代知识点依赖。

---

## 14. 依赖强度设计

### 14.1 strong（强依赖）

| 特征 | 说明 |
|------|------|
| 影响程度 | 目标缺失会明显影响当前对象的学习或表现 |
| 判断标准 | 前置知识不掌握，后续基本无法进行 |

### 14.2 medium（中等依赖）

| 特征 | 说明 |
|------|------|
| 影响程度 | 目标能明显帮助当前对象，但不是唯一前提 |
| 判断标准 | 前置知识有缺口会影响效率，但可通过其他方式弥补 |

### 14.3 weak（弱依赖）

| 特征 | 说明 |
|------|------|
| 影响程度 | 目标有帮助，但影响相对有限 |
| 判断标准 | 前置知识有缺口影响较小 |

### 14.4 原则
强弱不能凭感觉写，至少要能给出相对稳定的判断依据。

---

## 15. 抽取流程建议

建议流程如下：

```text
识别章节与知识点结构
    ↓
识别前置关系
    ↓
识别支撑关系
    ↓
确认关系方向
    ↓
判断强弱等级
    ↓
补充说明
    ↓
质量校验
```

## 16. 与学生状态模型的关系

依赖关系表将直接服务以下能力：

| 能力 | 说明 |
|------|------|
| 解释某知识点为何反复不稳 | 追溯前置知识缺口 |
| 识别前置缺口 | 定位问题根源 |
| 判断学习路径阻塞点 | 识别阻塞环节 |
| 生成"先补什么"的建议 | 干预优先级判断 |
| 支撑推演准备状态 | 薄弱链路识别 |

---

## 17. 与干预动作库的关系

干预动作库需要依赖关系表来决定：

| 决策 | 说明 |
|------|------|
| 哪些动作先做 | 前置优先 |
| 哪些知识点不适合直接补后续 | 应回到前置 |
| 哪些能力点薄弱可能来自更底层支撑不足 | 追溯根源 |

---

## 18. 与图谱记忆层的关系

依赖关系表中的关系，将成为图谱中的核心边类型之一。

示例：

| 关系 | 图谱表示 |
|------|----------|
| 知识点依赖 | KnowledgePoint -> depends_on -> KnowledgePoint |
| 知识支撑能力 | KnowledgePoint -> supports -> AbilityPoint |
| 能力依赖 | AbilityPoint -> depends_on -> AbilityPoint |

图谱能让这些依赖关系被路径查询和解释调用。

---

## 19. 质量要求

### 19.1 关系方向一致

| 要求 | 说明 |
|------|------|
| 全表同类关系方向必须一致 | 不能今天 A 依赖 B，明天 B 依赖 A |

### 19.2 关系类型清楚

| 要求 | 说明 |
|------|------|
| 不要把所有关系都做成一个泛泛的 related | 明确关系类型 |

### 19.3 避免滥连

| 要求 | 说明 |
|------|------|
| 不是所有看起来有点关系的内容都值得连边 | 只连真正重要的依赖 |

### 19.4 避免循环污染

| 要求 | 说明 |
|------|------|
| 如 A depends_on B，B 又 depends_on A，这种情况要特别警惕 | 防止循环依赖 |

### 19.5 可解释

| 要求 | 说明 |
|------|------|
| 核心关系必须能说明为什么 | 有依据支撑 |

---

## 20. 常见错误

### 20.1 关系方向混乱

| 错误 | 后果 |
|------|------|
| 今天写 A 依赖 B，明天又写 B 依赖 A | 推演和查询混乱 |

### 20.2 关系类型偷懒

| 错误 | 后果 |
|------|------|
| 全部写 related_to | 等于没建模 |

### 20.3 强弱乱标

| 错误 | 后果 |
|------|------|
| 没有标准，最后 strong 到处都是 | 强弱失去意义 |

### 20.4 过度连边

| 错误 | 后果 |
|------|------|
| 把系统搞成一个谁都依赖谁的教育银河 | 关系网无法使用 |

## 21. MVP 建议范围

MVP 阶段建议优先支持：

### 21.1 学科范围
- 高中物理

### 21.2 关系类型
- 知识点到知识点依赖
- 知识点到能力点支撑

### 21.3 强度分级
- strong / medium 两档先跑通

### 21.4 优先场景
- 受力分析 -> 牛顿定律应用
- 基础运动公式 -> 综合运动问题
- 实验基础理解 -> 实验分析能力

## 25. 风险与边界

### 25.1 不能忽视关系方向

关系方向混乱会导致推演和查询完全失效。

### 25.2 不能忽视关系类型

所有关系都写成 related_to 等于放弃建模。

### 25.3 不能忽视强弱区分

强弱不分会导致干预优先级判断失效。

### 25.4 不能忽视循环检测

循环依赖会导致系统逻辑混乱。

### 25.5 不能忽视关系可解释性

无法解释的关系不应进入主链路。

---

## 26. 结论

依赖关系表的核心意义，不是给教材内容之间随便画箭头，而是让系统真正理解：

- 什么是前提
- 什么是支撑
- 什么会影响后续
- 遇到问题时该先补哪里

它必须坚持：

- **方向清楚** - 全系统统一方向
- **类型清楚** - 明确关系语义
- **强弱清楚** - 有判断依据
- **边界清楚** - 不乱连滥用

否则关系表会从"学习路径地图"变成"结构化胡思乱想"。
