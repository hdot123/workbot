## 1. 文档目的

本文档用于定义 KB 模块首轮正式开发的首发切片范围，明确本轮固定做什么、先做到什么粒度、哪些能力暂不进入本轮。

本文档重点回答以下问题：

1. 首轮到底先做哪个地区、哪个学段、哪个年级、哪个学科、哪套教材
2. 本轮必须完成哪些结构化入库对象
3. 哪些内容虽然重要，但不进入本轮正式开发范围
4. 本轮开发完成后，KB 模块对 INGEST 至少提供什么稳定支撑
5. 什么条件下，KB 可以被视为具备首轮放行基础

本文档是 KB 模块从“设计框架”进入“正式开发”的范围冻结文档。字段级标准由 `STD-001~012` 承接，教材入库细则由 `KB-004` 承接。

## 2. 切片定义

### 2.1 什么叫首发切片

知识底座首发切片，是指在不追求全国、多学段、多学科全面铺开的前提下，先选择一个真实可落地、可映射、可验收的最小知识底座范围，支撑首轮主链路开发。

### 2.2 本轮切片目标

本轮切片目标不是“把知识底座做大”，而是先做出一个可供系统主链路稳定使用的最小闭环，至少支撑以下动作：

- INGEST 将输入内容映射到章节、知识点、锚点
- TWIN 将学生事件挂载到稳定知识节点
- GRAPH 将学习事件与知识结构建立基础关联
- OBS 在解释层能回到教材出处和知识位置
- 后续地区扩展和版本扩展有统一骨架可承接

## 3. 首发切片原则

### 3.1 单地区优先
本轮只冻结一个地区规则环境，避免地区差异和教材差异同时扩散。

### 3.2 单学段单年级优先
本轮只覆盖高中学段中的高一年级，不向初中、小学和高二高三扩散。

### 3.3 单学科优先
本轮只覆盖高中物理，不并行展开数学、语文、英语等第二学科。

### 3.4 单教材版本优先
本轮只冻结一套可明确绑定地区和年级的教材版本，不做跨版本对照。

### 3.5 可映射优先
优先入库能够直接支撑 `chapter_refs / knowledge_refs / anchor_refs / ability_refs` 的对象，不优先扩写增强型教研对象。

### 3.6 可验收优先
本轮必须让结构完整性、一致性、可追溯性、基础可映射性可以被明确验收，而不是停留在“有一堆文档”。

## 4. 冻结值表

### 4.1 首发冻结值

| 维度 | 冻结值 | 说明 |
|------|--------|------|
| 国家 | 中国 | 编码口径以国内 K12 体系为准 |
| 地区 | 广东省深圳市 | 与现有地区规则示例保持一致，便于首轮规则绑定 |
| 地区标识 | `CN_GD_SZ` | 首轮地区主标识 |
| 学段 | 高中 | 与 `STR-002`、`KB-004` 首轮优先范围一致 |
| 年级 | 高一 | 与现有地区层与学生绑定示例口径一致 |
| 年级标识 | `G10` | 首轮年级标识 |
| 学科 | 物理 | 与知识底座首学科优先顺序一致 |
| 学科标识 | `PHY` | 首轮学科标识 |
| 教材版本 | 人教版高中物理必修一 | 与 `KB-004` 首轮优先教材一致 |
| 教材版本标识 | `PHY_PEP_G1_V1` | 与现有标准表示例口径对齐 |
| 试点学校类型 | 深圳市单校试点普通高中 | 用于试点约束，不在本轮冻结具体学校名称 |
| 知识底座版本号 | `KB_CN_GD_SZ_PHY_PEP_G1_V1_1.0.0` | 本轮首发底座正式版本号 |
| 首发目标 | 支撑 KB + INGEST 首轮骨架开发 | 不以全量运营上线为目标 |

### 4.2 冻结值解释

1. 本轮冻结到“深圳市”层，不继续拆到区县层差异。
2. 本轮冻结到“高一物理必修一”，不支持高二高三和跨册跳转。
3. 本轮冻结 `PHY_PEP_G1_V1` 这一个教材版本，不处理地区增补教材、校本教材和多出版社并存场景。
4. 试点学校保留“单校普通高中”口径，但具体学校名称在实施阶段冻结，不作为 KB 首轮开发的阻塞条件。

## 5. 首发范围内必须入库的表清单

### 5.1 必须入库对象

本轮必须完成以下对象的结构化入库，且达到可引用、可校验、可追溯状态。

| 序号 | 表/对象 | 对应标准 | 本轮状态 | 说明 |
|------|---------|----------|----------|------|
| 1 | 地区基础对象 | `KB-002` / `RULE-001` | 必做 | 至少完成深圳市地区对象与高中制度绑定 |
| 2 | 教材元数据表 | `STD-010` | 必做 | 明确教材版本、出版社、学科、册别、适用年级 |
| 3 | 地区教材适配表 | `STD-011` | 必做 | 将深圳高中高一物理绑定到 `PHY_PEP_G1_V1` |
| 4 | 章节树表 | `STD-001` | 必做 | 形成完整章节骨架 |
| 5 | 知识点表 | `STD-002` | 必做 | 形成知识点主表并挂到章节 |
| 6 | 能力点表 | `STD-003` | 必做 | 覆盖本册物理首轮核心能力点 |
| 7 | 教材锚点表 | `STD-004` | 必做 | 为首轮知识点建立教材页码/位置锚点 |
| 8 | 依赖关系表 | `STD-005` | 必做 | 至少覆盖核心强依赖和建议前置关系 |
| 9 | 知识能力映射表 | `STD-006` | 必做 | 支撑知识点到能力点映射 |
| 10 | 底座版本元数据 | `KB-007` | 必做 | 记录版本号、发布日期、适用范围、审核状态 |

### 5.2 必做表的最小完成标准

| 表/对象 | 最小完成标准 |
|---------|--------------|
| 地区基础对象 | 能完成 `region_id -> stage -> grade -> curriculum_version_id` 绑定 |
| 教材元数据表 | 能唯一标识本轮教材版本并供下游引用 |
| 地区教材适配表 | 能回答“深圳市高一物理使用哪套教材” |
| 章节树表 | 覆盖整册章/节/小节，不缺主骨架 |
| 知识点表 | 核心知识点覆盖率达到首轮可用标准 |
| 能力点表 | 至少覆盖理解、受力分析、建模、方程建立、图像理解等核心能力 |
| 教材锚点表 | 所有首轮核心知识点至少有一个可追溯锚点 |
| 依赖关系表 | 能支撑“前置知识”解释，不要求一开始穷尽所有弱关系 |
| 知识能力映射表 | 每个首轮核心知识点至少映射一个能力点 |
| 版本元数据 | 能支持发布、审计、回滚说明 |

## 6. 首发范围边界

### 6.1 本轮明确做的内容

| 范围 | 说明 |
|------|------|
| 深圳高中高一物理知识底座 | 作为唯一首发地区/学段/年级/学科切片 |
| 人教版高中物理必修一结构化入库 | 作为唯一教材版本切片 |
| 章节、知识点、能力点、锚点、依赖、版本元数据 | 作为本轮最小可用知识对象集合 |
| KB 对 INGEST 的映射支撑能力 | 支撑事件映射到章节/知识点/锚点 |
| 首轮质量校验与版本发布 | 至少形成一个正式可引用版本 |

### 6.2 本轮明确不做的内容

| 不做项 | 原因 |
|--------|------|
| 全国多地区并行入库 | 会把地区规则差异和教材差异同时放大 |
| 高二、高三内容扩展 | 与首轮试点链路无关，先不扩张 |
| 第二学科并行建设 | 避免同时放大知识结构差异 |
| 多出版社版本并行 | 版本映射成本过高，不适合作为首轮前置 |
| 题型映射表全量建设 | 属于增强层，可后置到下一轮 |
| 常见误区表全量建设 | 属于增强层，可后置 |
| 实验活动表全量建设 | 属于增强层，可后置 |
| 校本资料、讲义、自编卷底座化 | 先保证教材主干，后做扩展材料 |
| 面向推演层的复杂规则建模 | 不作为 KB 首轮开发放行前置 |
| 细粒度区县/学校差异覆盖 | 先冻结到市级，避免结构过早复杂化 |
| 历史版本全量归档治理 | 先做好首版发布与单步回滚说明 |

### 6.3 可延后但需预留接口的内容

| 延后项 | 本轮要求 |
|--------|----------|
| 常见误区 | 预留对象编码空间，不要求首轮完整入库 |
| 实验活动 | 预留锚点类型与对象扩展位 |
| 跨版本映射 | 预留版本映射入口，不作为首轮必做表 |
| 地区扩展 | 保留 `region_id` 与适配规则扩展能力 |
| 多学科扩展 | 保留 `subject` 与编码体系统一口径 |

## 7. 首发切片交付物

### 7.1 本轮必须交付

1. 深圳高中高一物理的人教版必修一教材元数据主记录  
2. 整册章节树  
3. 首轮知识点主表  
4. 首轮能力点主表  
5. 知识点-能力点映射  
6. 知识点-教材锚点映射  
7. 核心依赖关系  
8. 底座版本元数据与发布说明  
9. 可供 INGEST 调用的引用依据

### 7.2 本轮交付完成后的模块能力

交付完成后，KB 至少应能回答以下问题：

- 当前试点学生属于哪套地区规则和教材版本
- 某个学习事件涉及哪个章节、哪个知识点、哪个教材位置
- 某个知识点属于哪个章节、依赖哪些前置知识、关联哪些能力点
- 当前引用的是哪个知识底座版本
- 若映射错误，回退时应回退到哪个版本

## 8. 放行说明

### 8.1 放行前提

KB 首发切片达到以下条件后，可视为满足首轮正式开发放行基础：

- 首发冻结值不再漂移
- 必做表清单已冻结
- 首发范围与不做范围边界明确
- 本轮版本号和引用口径明确
- 能支撑 INGEST 的基础知识引用与映射
- 可通过最小验收矩阵进行检查

### 8.2 放行边界

本文件的放行含义是：

- 允许 KB 模块进入首轮骨架开发
- 允许围绕首发切片做字段实现、入库实现、映射实现和质量校验实现
- 不等于允许直接展开多地区、多学科、多教材版本的全量开发
- 不等于允许跳过 `STD-001~012`、`KB-004`、验收矩阵和样例集

### 8.3 当前放行结论

基于现有文档体系，本切片定义发布后，KB 模块可按以下口径进入首轮正式开发：

> 以“深圳市 / 高中 / 高一 / 物理 / `PHY_PEP_G1_V1`”为唯一首发切片，完成地区适配、章节、知识点、能力点、锚点、依赖与版本元数据的最小闭环建设。

## 10. 一句话冻结结论

KB 首发切片冻结为：深圳市高中高一物理，人教版必修一；本轮只做支撑 KB + INGEST 主链路的最小知识底座闭环，不做全国扩展、多学科扩展和增强型知识对象全量建设。
