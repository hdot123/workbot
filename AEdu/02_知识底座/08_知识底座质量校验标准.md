# 知识底座质量校验标准

> 文档编号：KB-008  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：知识底座负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中"固定知识底座"的质量校验标准，包括校验维度、校验流程、问题分级、准入门槛与持续校验机制。

本文档重点回答以下问题：

1. 什么样的知识底座才算"可用"
2. 章节树、知识点、能力点、锚点、依赖关系等内容应如何校验
3. 哪些问题可以带着进入系统，哪些问题必须阻断发布
4. 如何保证知识底座不是看上去完整、实际上漏洞百出
5. 如何在后续迭代中持续保持底座质量

---

## 2. 知识底座质量定义

知识底座质量，是指固定知识底座在结构完整性、命名一致性、映射准确性、可追溯性、可解释性和可维护性等方面达到的可用水平。

一个"看起来很多文档、很多表"的底座，不等于高质量底座。  
真正高质量的知识底座，应满足：

- 能支撑学生绑定
- 能支撑学习事件映射
- 能支撑 StudentTwinAgent 状态表达
- 能支撑前台解释与后台推演
- 能被持续维护而不失控

---

## 3. 校验目标

### 3.1 保证底座可用
确保底座不是停留在文档层，而是真的能进系统主链路。

### 3.2 保证底座一致
让章节、知识点、能力点、锚点和依赖关系能互相对得上。

### 3.3 保证底座可追溯
关键结构必须能回到原始教材与规则来源。

### 3.4 保证底座可扩展
为后续多学科、多版本、多地区扩展打下稳定基础。

### 3.5 控制维护成本
避免"今天能用，明天一改全乱"的伪稳定底座。

---

## 4. 设计原则

### 4.1 先结构，后规模
结构不稳时，内容越多，后面返工越痛。

### 4.2 主链路优先
优先保证会影响事件映射、状态建模、报告解释和推演的部分质量。

### 4.3 一致性优先
命名和编码一致，比堆更多内容更重要。

### 4.4 可追溯优先
关键对象必须能回到原始教材或规则依据。

### 4.5 持续校验
知识底座不是一次校验完就永远万事大吉。

---

## 5. 校验对象范围

知识底座质量校验至少覆盖以下对象：

| 对象 | 说明 |
|------|------|
| 地区层与教材适配 | 地区规则与教材版本绑定 |
| 章节树表 | 章节层级结构 |
| 知识点表 | 知识点定义与映射 |
| 能力点表 | 能力点定义与映射 |
| 教材锚点表 | 教材位置定位 |
| 依赖关系表 | 知识点/能力点依赖关系 |
| 题型映射表 | 题型与知识点映射 |
| 元数据与版本信息 | 版本记录与变更追踪 |

---

## 6. 校验维度

建议知识底座至少按以下八个维度校验：

| 维度 | 说明 |
|------|------|
| 完整性 | 关键结构是否齐全 |
| 一致性 | 不同表之间口径是否一致 |
| 唯一性 | 主键、编码是否有冲突 |
| 映射性 | 对象之间能否形成有效映射网络 |
| 可追溯性 | 关键对象能否回溯到原始资料 |
| 逻辑合理性 | 底座内部逻辑是否通顺 |
| 可解释性 | 能否支撑前台解释和老师理解 |
| 可维护性 | 是否可以长期维护而不失控 |

---

## 7. 完整性校验

### 7.1 目标
确认底座中关键结构是否齐全。

### 7.2 检查项

| 检查项 | 说明 |
|--------|------|
| 是否有地区和教材绑定 | 学生绑定基础 |
| 是否有章节树 | 结构骨架 |
| 是否有知识点表 | 内容核心 |
| 是否有能力点表 | 能力表达 |
| 是否有锚点表 | 可追溯性 |
| 是否有依赖关系表 | 学习路径 |
| 是否有题型映射表 | 事件映射 |
| 是否有版本信息和来源信息 | 可维护性 |

### 7.3 原则
MVP 阶段可以接受"部分不全"，但不能缺主骨架。

---

## 8. 一致性校验

### 8.1 目标
确认不同表、不同对象之间口径一致。

### 8.2 检查项

| 检查项 | 说明 |
|--------|------|
| 学科命名是否一致 | 物理/数学等命名统一 |
| 教材版本命名是否一致 | PEP_G1_V1 等统一 |
| 章节路径是否一致 | 章节树路径统一 |
| 知识点命名风格是否一致 | 命名规范统一 |
| 能力点粒度是否一致 | 能力点粒度统一 |
| 关系类型和方向是否一致 | 依赖关系方向统一 |
| 页码与锚点引用是否一致 | 页码引用统一 |

### 8.3 常见不一致问题

| 问题 | 后果 |
|------|------|
| 同一学科多种命名 | 映射失败 |
| 同一版本多种编码 | 版本混乱 |
| 关系方向不统一 | 推演逻辑错误 |

---

## 9. 唯一性校验

### 9.1 目标
确认主键、编码和关键对象没有重复和冲突。

### 9.2 检查项

| 检查项 | 说明 |
|--------|------|
| 章节节点编码是否唯一 | 不能重复 |
| 知识点 ID 是否唯一 | 不能重复 |
| 能力点 ID 是否唯一 | 不能重复 |
| 锚点 ID 是否唯一 | 不能重复 |
| 关系 ID 是否唯一 | 不能重复 |
| 同一对象是否被重复表达 | 避免近义重复 |

### 9.3 原则
重复对象如果不处理，后面状态映射和图谱建模会直接分裂世界线。

---

## 10. 映射性校验

### 10.1 目标
确认底座对象之间能形成有效映射网络。

### 10.2 检查项

| 检查项 | 说明 |
|--------|------|
| 知识点是否挂载章节 | 结构归属 |
| 知识点是否有能力点映射 | 能力表达 |
| 知识点是否有锚点引用 | 可追溯性 |
| 题型是否映射知识点与能力点 | 事件映射 |
| 依赖关系是否指向有效对象 | 关系有效 |
| 学生绑定规则是否能落到底座结构 | 可用性 |

### 10.3 原则
底座不是一组孤立表，它必须是一张能走通的地图。

---

## 11. 可追溯性校验

### 11.1 目标
确认关键对象能回溯到原始资料。

### 11.2 检查项

| 检查项 | 说明 |
|--------|------|
| 章节是否对应原始教材目录 | 结构来源 |
| 知识点是否有教材出处 | 内容来源 |
| 锚点是否有页码与来源引用 | 定位来源 |
| 规则是否有来源文件 | 规则来源 |
| 版本是否有变更记录 | 变更追踪 |

### 11.3 原则
没有可追溯性，系统说"依据教材/依据规则"，就会像空口白牙。

---

## 12. 逻辑合理性校验

### 12.1 目标
确认底座内部逻辑通顺，不出现明显结构性错误。

### 12.2 检查项

| 检查项 | 说明 |
|--------|------|
| 章节层级是否循环引用 | 不能有环 |
| 依赖关系是否有明显环路 | 不能有环 |
| 前置知识是否晚于后置知识出现 | 顺序合理 |
| 知识点挂载章节是否合理 | 归属合理 |
| 能力点与题型关系是否明显错位 | 映射合理 |
| 锚点页码是否超出教材页范围 | 页码有效 |

### 12.3 说明
很多底座错误不是"字段缺失"，而是逻辑拧巴。后者更难发现，也更容易害人。

---

## 13. 可解释性校验

### 13.1 目标
确认底座能支撑前台解释和老师理解。

### 13.2 检查项

| 检查项 | 说明 |
|--------|------|
| 知识点定义是否清楚 | 可理解 |
| 能力点定义是否可教学使用 | 可教学 |
| 题型名称是否可理解 | 可理解 |
| 关系说明是否能解释为何依赖 | 可解释 |
| 锚点是否能支撑报告引用 | 可引用 |

### 13.3 原则
系统内部对象如果老师都看不懂，后面解释层会很痛苦。

---

## 14. 可维护性校验

### 14.1 目标
确认底座不是一次性作品，而是可以长期维护的结构。

### 14.2 检查项

| 检查项 | 说明 |
|--------|------|
| 编码规则是否统一 | 编码规范 |
| 版本信息是否完整 | 版本追踪 |
| 变更记录是否可追踪 | 变更追踪 |
| 命名规范是否统一 | 命名规范 |
| 结构是否支持增量更新 | 可扩展 |

### 14.3 原则
越是底座，越不能靠"记得住的人"维护。必须靠规则维护。

## 15. 分对象校验建议

### 15.1 章节树表

| 重点检查 | 说明 |
|----------|------|
| 层级完整性 | 章 - 节 - 小节完整 |
| 父子关系 | 父子关系正确 |
| 排序正确性 | 同层排序正确 |
| 路径一致性 | 路径编码一致 |

### 15.2 知识点表

| 重点检查 | 说明 |
|----------|------|
| 粒度 | 不过粗也不过细 |
| 命名 | 符合命名规范 |
| 章节挂载 | 归属正确 |
| 定义清晰度 | 定义清楚 |

### 15.3 能力点表

| 重点检查 | 说明 |
|----------|------|
| 粒度 | 能力点粒度适中 |
| 可观察性 | 能被事件支撑 |
| 与知识点和题型映射 | 映射完整 |

### 15.4 锚点表

| 重点检查 | 说明 |
|----------|------|
| 页码准确性 | 页码正确 |
| 区块定位可用性 | 区块编码有效 |
| 与知识点和章节的映射 | 映射完整 |

### 15.5 依赖关系表

| 重点检查 | 说明 |
|----------|------|
| 方向一致 | 全表方向统一 |
| 类型清楚 | 关系类型明确 |
| 强弱合理 | 强弱有依据 |
| 无明显错误环路 | 无循环依赖 |

### 15.6 题型映射表

| 重点检查 | 说明 |
|----------|------|
| 题型粒度 | 粒度适中 |
| 知识点映射 | 映射完整 |
| 能力点映射 | 映射完整 |
| 误区映射 | 映射完整 |

## 16. 问题分级建议

### 16.1 P0：阻断发布

必须阻断入库，修复后才能发布。

| 问题 | 说明 |
|------|------|
| 主表缺失 | 章节树/知识点表缺失 |
| 编码大面积冲突 | ID 重复严重 |
| 章节结构错误严重 | 层级混乱 |
| 知识点无法映射 | 知识点孤立 |
| 锚点大面积失效 | 无法追溯 |

### 16.2 P1：高优先级修复

可暂缓发布，但需在下一版本前修复。

| 问题 | 说明 |
|------|------|
| 部分关系方向错误 | 依赖关系方向不统一 |
| 能力点粒度明显失衡 | 过粗或过细 |
| 关键知识点缺锚点 | 无法追溯 |
| 重要题型未映射 | 事件映射受影响 |

### 16.3 P2：可后续修复

可带问题发布，后续迭代修复。

| 问题 | 说明 |
|------|------|
| 文案表达不统一 | 部分定义表述不一致 |
| 次要字段缺失 | 非核心字段缺失 |
| 少量说明文本不完整 | 说明文本可补充 |

## 17. 校验流程建议

```text
完成底座初版
    ↓
执行结构校验（完整性、唯一性）
    ↓
执行对象间映射校验（映射性、一致性）
    ↓
执行逻辑与来源校验（逻辑合理性、可追溯性）
    ↓
输出问题清单（按 P0/P1/P2 分级）
    ↓
修复高优先级问题（P0 必须修复）
    ↓
复检（确认 P0 已修复）
    ↓
达到发布门槛后冻结版本
```

### 17.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 完成底座初版 | 完成所有章节表、知识点表等 | 底座初版 |
| 结构校验 | 检查完整性、唯一性 | 结构校验报告 |
| 映射校验 | 检查对象间映射关系 | 映射校验报告 |
| 逻辑与来源校验 | 检查逻辑和可追溯性 | 逻辑校验报告 |
| 输出问题清单 | 按 P0/P1/P2 分级 | 问题清单 |
| 修复高优先级问题 | 修复 P0 问题 | 修复记录 |
| 复检 | 确认 P0 已修复 | 复检报告 |
| 冻结版本 | 达到发布门槛 | 发布版本 |

## 18. 发布门槛建议

一个底座版本建议满足以下门槛后才可进入系统主链路：

### 18.1 必须满足（P0 级别）

| 门槛 | 说明 |
|------|------|
| 主对象齐全 | 章节树、知识点表等核心表完整 |
| 关键对象编码无冲突 | 主键、编码无重复 |
| 知识点与章节映射基本完成 | 知识点有章节归属 |
| 高优先级逻辑错误已修复 | 无 P0 问题 |

### 18.2 建议满足（P1 级别）

| 门槛 | 说明 |
|------|------|
| 知识点与能力点、题型、锚点有基础映射 | 映射基本完整 |
| 来源与版本信息完整 | 可追溯 |
| 关键链路可支撑事件映射和状态解释 | 可用性 |

### 18.3 可后续完善（P2 级别）

| 门槛 | 说明 |
|------|------|
| 次要字段完整 | 非核心字段 |
| 文案表达统一 | 表述一致性 |
| 全量对象级锚点 | 精细定位 |

## 19. 持续校验机制

知识底座不是发布一次就结束，建议建立：

### 19.1 版本发布前校验

| 内容 | 说明 |
|------|------|
| 每次发版前统一检查 | 确保新版本质量 |
| 按八个维度全面校验 | 完整性、一致性等 |

### 19.2 增量更新校验

| 内容 | 说明 |
|------|------|
| 每次新增章节、知识点、题型或依赖时局部检查 | 增量校验 |
| 重点检查新增部分与现有结构的一致性 | 一致性检查 |

### 19.3 周期性复检

| 内容 | 说明 |
|------|------|
| 按月或按阶段抽查高价值对象 | 定期复检 |
| 重点检查核心知识点和依赖关系 | 核心对象检查 |

### 19.4 试点反馈回流校验

| 内容 | 说明 |
|------|------|
| 老师或运营指出底座问题后，进入复核和修正流程 | 反馈驱动 |
| 问题修复后更新版本记录 | 版本追踪 |

### 20.2 内容范围

| 内容 | 说明 |

| 章节树 | 结构骨架 |

## 24. 风险与边界

### 24.1 不能忽视结构完整性

| 风险 | 说明 |
|------|------|
| 主表缺失导致系统无法运行 | 章节树、知识点表是核心 |

### 24.2 不能忽视编码一致性

| 风险 | 说明 |
|------|------|
| 编码冲突导致映射失败 | ID 重复会导致系统混乱 |

### 24.3 不能忽视可追溯性

| 风险 | 说明 |
|------|------|
| 无法追溯导致解释层无法引用教材 | 锚点和来源信息很重要 |

### 24.4 不能忽视逻辑合理性

| 风险 | 说明 |
|------|------|
| 逻辑错误导致推演层得出错误结论 | 依赖关系方向很重要 |

### 24.5 不能忽视持续校验

| 风险 | 说明 |
|------|------|
| 一次校验后不再检查导致质量下降 | 需要持续校验机制 |

## 25. 结论

知识底座质量校验标准的核心，不是多一道检查流程，而是保证底座真正能够被系统依赖，而不是只在文档里显得很完整。

它必须坚持：

- **校验结构** - 完整性、唯一性
- **校验映射** - 对象间能有效连接
- **校验逻辑** - 内部逻辑通顺
- **校验追溯** - 能回到原始资料
- **校验维护性** - 可以长期维护

否则底座就会变成一种非常昂贵的错觉：看起来什么都有，真正用起来却到处漏风。
