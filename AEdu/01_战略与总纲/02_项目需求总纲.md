## 一、项目背景与愿景

### 1.1 项目背景

当前教育场景中存在以下核心痛点：

| 痛点 | 描述 |
|------|------|
| 信息碎片化 | 学生数据散落在微信聊天、学校系统、纸质文档中，无法形成完整画像 |
| 缺乏连续性 | 学习记录无法跨学期、跨年级持续追踪 |
| 决策无依据 | 家长和老师做教育决策时缺乏数据支撑 |
| 干预滞后 | 问题发现时往往已经积累严重，错过最佳干预时机 |

### 1.2 项目愿景

**构建每个学生的数字孪生体，实现教育过程的可观察、可推演、可优化。**

### 1.3 核心一句话定位

> 课课 —— 用数字孪生技术，让每个孩子的成长轨迹清晰可见。

---

## 二、核心需求概述

### 2.1 总体目标

1. **知识底座**：建立覆盖 K12 的固定知识底座，包含教材、知识点、能力点的标准映射
2. **学生孪生**：为每个学生创建 StudentTwinAgent，持续追踪学习状态与成长轨迹
3. **数据接入**：支持微信、钉钉、学校扫描仪等多渠道数据输入
4. **图谱记忆**：基于 GraphRAG 构建学生时序记忆与知识图谱
5. **观察推演**：前台可观察、后台可推演，支持干预建议与决策辅助

### 2.2 核心用户与场景

| 用户角色 | 核心场景 | 关键需求 |
|----------|----------|----------|
| 家长 | 查看孩子学习状态、接收建议 | 简洁清晰的周报/月报、可操作的干预建议 |
| 老师 | 班级管理、学情分析 | 班级看板、风险学生预警、群体分析 |
| 学校管理者 | 教学质量监控 | 多维度数据报表、趋势分析 |
| 学生（间接） | 学习过程被记录与分析 | 无感采集、隐私保护 |

## 三、功能模块需求

### 3.1 知识底座模块

**需求编号：REQ-KB-001**

- 支持教材结构化入库（章节树、知识点、能力点）
- 支持地区教材版本适配与映射
- 支持知识点依赖关系定义
- 支持知识底座版本管理与迭代

### 3.2 学生数字孪生模块

**需求编号：REQ-ST-001**

- 每个学生创建独立 StudentTwinAgent
- 支持学生状态模型定义与更新
- 支持时间演化层记录学习轨迹
- 支持关系上下文层（班级、学校、家庭）
- 支持 Agent 生命周期管理

### 3.3 数据接入模块

**需求编号：REQ-IN-001**

- 微信端输入：家长聊天窗口、微信小程序
- 钉钉端输入：学校组织接入
- 硬件扫描：授权扫描仪 OCR 识别
- 批量导入：学校批量录入接口
- 事件流处理：学习事件生成与消息队列

### 3.4 记忆与图谱引擎模块

**需求编号：REQ-MEM-001**

- 图谱记忆层：事实 - 关系 - 事件三元组存储
- 学生时序记忆：时间线式学习记录
- GraphRAG 检索：基于图谱的上下文检索
- 增量更新机制：记忆动态追加与修正
- 版本回滚：支持记忆状态回溯

### 3.5 观察层模块（前台）

**需求编号：REQ-OBS-001**

- 家长端展示规则与界面
- 老师端班级看板
- 学校机构端管理视图
- 周报/月报自动生成
- 对比维度与观察口径定义

### 3.6 推演与决策模块（后台）

**需求编号：REQ-SIM-001**

- InterventionSimulator：干预动作推演
- 风险预测模型：学习风险提前预警
- 影子推演机制：后台持续模拟不同干预路径
- 志愿填报推演：高考决策辅助
- 推演结果校准与验证

## 四、技术架构需求

### 4.1 系统分层

```
┌─────────────────────────────────────┐
│         前台观察层（产品层）          │
│   家长端 / 老师端 / 学校端 / 小程序    │
├─────────────────────────────────────┤
│         后台推演层（决策层）          │
│   干预推演 / 风险预测 / 决策 Agent     │
├─────────────────────────────────────┤
│         智能中台层                    │
│   StudentTwinAgent / 记忆引擎 / 图谱  │
├─────────────────────────────────────┤
│         数据接入层                    │
│   微信 / 钉钉 / OCR / 批量导入         │
├─────────────────────────────────────┤
│         知识底座层                    │
│   教材库 / 知识点 / 能力点 / 规则库    │
└─────────────────────────────────────┘
```

### 4.2 核心技术选型要求

| 技术领域 | 要求 |
|----------|------|
| 图数据库 | 支持时序图谱、支持 GraphRAG（Graphiti/Zep 评估） |
| 向量检索 | 支持嵌入检索、支持混合检索 |
| Agent 框架 | 支持多 Agent 协作、支持状态管理 |
| 消息队列 | 支持事件流处理、支持异步任务 |
| 多租户 | 支持学校隔离、支持数据权限 |

### 4.3 非功能性需求

| 需求类型 | 指标 |
|----------|------|
| 性能 | 单次图谱检索 < 500ms，Agent 更新 < 2s |
| 可用性 | 系统可用性 > 99.5% |
| 扩展性 | 支持万级学生并发、支持多地区部署 |
| 安全性 | 数据加密存储、隐私合规、权限隔离 |
| 可维护性 | 完整日志监控、告警体系、灰度发布 |

## 五、数据需求

### 5.1 核心数据实体

| 实体类型 | 描述 | 存储方式 |
|----------|------|----------|
| 教材数据 | 章节树、知识点、能力点 | 关系数据库 + 图谱 |
| 学生数据 | 基础信息、状态、轨迹 | 图谱 + 向量库 |
| 事件数据 | 学习事件、行为记录 | 消息队列 + 时序库 |
| 规则数据 | 地区规则、高考政策 | 关系数据库 |
| 推演数据 | 干预记录、预测结果 | 图谱 + 关系库 |

### 5.2 数据质量标准

- 教材入库准确率 > 99%
- 事件生成准确率 > 95%
- 图谱实体关系一致性校验
- 支持数据溯源与审计

## 八、风险与假设

### 8.1 关键假设

- 学校愿意配合数据接入与硬件部署
- 家长愿意授权学生数据用于分析
- 教材版权可合法使用

### 8.2 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据隐私合规 | 高 | 早期引入法律顾问、设计隐私保护机制 |
| 学校接入意愿 | 中 | 聚焦私立学校试点、建立标杆案例 |
| 技术实现难度 | 中 | 分阶段交付、核心功能优先 |
