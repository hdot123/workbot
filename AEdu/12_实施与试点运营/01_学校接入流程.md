---

## 1. 文档目的

本文档用于定义教育孪生项目中试点学校的接入流程，包括接入阶段、角色分工、前置准备、系统开通、数据准备、培训上线与反馈闭环。

本文档重点回答以下问题：

1. 一所学校如何正式接入系统
2. 接入前需要准备什么
3. 不同角色分别负责什么
4. 接入后如何进入试点运行
5. 如何避免学校接入变成"装了系统但没人真用"的形式主义工程

---

## 2. 学校接入定义

学校接入，是指学校从"意向合作或试点确认"进入"账号开通、基础配置、数据准备、角色培训、试运行、正式使用"的全过程。

它不是单纯开几个账号，也不是把系统链接发过去就算完成，而是一个包含：

- 组织准备
- 规则绑定
- 数据准备
- 角色培训
- 上线验证
- 反馈闭环

的系统化实施流程。

---

## 3. 接入目标

### 3.1 快速完成基础可用闭环
让学校在较短周期内能真正用起来，而不是只停留在演示阶段。

### 3.2 保证角色协同
学校管理员、老师、家长、运营团队要知道各自该做什么。

### 3.3 保证规则和数据初始化正确
地区规则、教材版本、班级结构、学生主体绑定必须正确。

### 3.4 保证试点可复盘
学校接入过程必须可追踪、可检查、可复盘。

---

## 4. 适用范围

本流程适用于：

- 试点学校首次接入
- 新年级/新班级扩容接入
- 新学段场景接入（后续）

MVP 阶段优先面向：
- 单学校
- 单年级或少量班级
- 有明确负责人和试点意愿的场景

---

## 5. 接入流程总览

建议学校接入分为七个阶段：

```
接入确认 → 前置准备 → 系统配置 → 数据初始化 → 角色培训 → 试运行上线 → 跟踪与复盘
```

---

## 6. 第一阶段：接入确认

### 6.1 目标
明确学校是否正式进入试点接入阶段。

### 6.2 关键动作
- 明确试点学校和范围
- 明确学校负责人
- 明确平台对接人
- 确定试点班级/年级
- 明确接入时间表

### 6.3 输出物
- 接入确认单
- 试点范围清单
- 学校角色联系人表
- 初步时间计划

### 6.4 检查清单
- [ ] 学校名称和基本信息确认
- [ ] 试点年级/班级范围确认
- [ ] 学校负责人及联系方式确认
- [ ] 平台运营对接人确认
- [ ] 初步时间表确认

---

## 7. 第二阶段：前置准备

### 7.1 目标
完成接入所需的基础信息准备。

### 7.2 需要准备的内容

| 信息类别 | 具体内容 | 负责方 |
|----------|----------|--------|
| 学校基础信息 | 学校名称、地址、类型 | 学校 |
| 年级与班级结构 | 年级、班级名称和数量 | 学校 |
| 学生名单 | 学生姓名、学号、所属班级 | 学校 |
| 教师名单 | 教师姓名、工号、所教班级/学科 | 学校 |
| 教材版本信息 | 各学科使用教材版本 | 学校 |
| 地区绑定信息 | 所属地区、教育政策 | 学校+平台 |
| 数据输入方式确认 | 微信/钉钉/扫描/手工录入 | 学校+平台 |
| 硬件接入方式确认 | 扫描仪等设备（如适用） | 学校+平台 |

### 7.3 原则
前置准备越清楚，后面上线越少翻车。

### 7.4 输出物
- 学校基础信息表
- 年级班级结构表
- 学生名单（脱敏）
- 教师名单
- 教材版本确认表
- 数据输入方式确认单

---

## 8. 第三阶段：系统配置

### 8.1 目标
完成学校在系统中的组织和权限配置。

### 8.2 配置内容

| 配置项 | 说明 | 负责方 |
|--------|------|--------|
| 租户开通 | 创建学校租户 | 平台 |
| 学校组织结构配置 | 年级、班级结构 | 平台 |
| 角色配置 | 管理员/老师/家长角色 | 平台 |
| 权限开通 | 各角色权限设置 | 平台 |
| 地区规则绑定 | 绑定地区规则 | 平台 |
| 教材版本绑定 | 绑定教材版本 | 平台 |
| 功能范围开关配置 | MVP 功能范围 | 平台 |

### 8.3 输出物
- 学校租户配置完成
- 基础组织结构可用
- 角色账号可登录
- 配置确认单

### 8.4 检查清单
- [ ] 租户创建成功
- [ ] 年级/班级结构配置正确
- [ ] 角色权限配置正确
- [ ] 地区规则绑定正确
- [ ] 教材版本绑定正确
- [ ] 测试账号可正常登录

---

## 9. 第四阶段：数据初始化

### 9.1 目标
完成学生和教学环境的初始化绑定。

### 9.2 初始化内容

| 内容 | 说明 | 负责方 |
|------|------|--------|
| 学生基础档案导入 | 学生 ID、姓名、班级 | 平台 |
| 班级与老师绑定 | 班主任、学科老师 | 学校+平台 |
| 学生与班级绑定 | 学生归属班级 | 学校+平台 |
| 学生地区绑定 | 地区规则绑定 | 平台 |
| 学生教材版本绑定 | 教材版本绑定 | 平台 |
| 初始 StudentTwinAgent 创建 | 学生孪生体初始化 | 平台 |
| 基础数据校验 | 数据准确性校验 | 平台 |

### 9.3 原则
初始化是后续一切的起点，不能糊弄。学生主体一旦绑错，后面所有状态都会长歪。

### 9.4 输出物
- 学生档案导入完成确认
- 班级 - 老师绑定完成确认
- 学生 - 班级绑定完成确认
- 数据初始化校验报告

### 9.5 检查清单
- [ ] 学生名单导入完成
- [ ] 学生班级绑定正确
- [ ] 教师班级绑定正确
- [ ] 地区规则绑定正确
- [ ] 教材版本绑定正确
- [ ] StudentTwinAgent 初始化完成
- [ ] 数据校验无重大错误

---

## 10. 第五阶段：角色培训

### 10.1 目标
让学校端相关角色会用、敢用、知道为什么用。

### 10.2 培训对象

| 角色 | 培训内容 | 培训方式 |
|------|----------|----------|
| 学校管理员 | 系统管理、数据查看、问题反馈 | 现场/线上培训 |
| 班主任 | 班级查看、学生详情、反馈提交 | 现场/线上培训 |
| 学科老师 | 学科查看、知识点分析、反馈提交 | 现场/线上培训 |
| 运营支持人员 | 系统操作、问题处理、培训协助 | 现场培训 |
| 家长代表（如适用） | 家长端使用、反馈入口 | 线上培训/操作手册 |

### 10.3 培训内容
- 系统目标和边界（能做什么、不能做什么）
- 家长端/老师端/学校端使用方式
- 数据输入方式（微信/钉钉/扫描/手工）
- 反馈入口使用方式
- 重点查看项（首页、报告、看板）
- 常见误区说明

### 10.4 原则
培训不是念 PPT，而是让实际使用场景跑一遍。

### 10.5 输出物
- 培训记录
- 培训签到表
- 培训材料（PPT、操作手册）
- 培训反馈表

### 10.6 检查清单
- [ ] 学校管理员培训完成
- [ ] 班主任培训完成
- [ ] 学科老师培训完成
- [ ] 运营支持人员培训完成
- [ ] 家长操作手册发放
- [ ] 培训反馈收集完成

---

## 11. 第六阶段：试运行上线

### 11.1 目标
在真实场景中小范围跑通系统主链路。

### 11.2 上线动作

| 动作 | 说明 | 负责方 |
|------|------|--------|
| 打开试点班级/年级功能 | 激活试点范围功能 | 平台 |
| 开始收集输入数据 | 家长/老师开始输入 | 学校 |
| 开始生成学习事件 | 系统处理输入 | 平台 |
| 开始更新 StudentTwinAgent | 状态更新 | 平台 |
| 家长端/老师端开始查看 | 用户使用 | 学校 |
| 观察看板开始出数 | 数据展示 | 平台 |

### 11.3 重点关注
- 数据是否正常进入
- 状态是否正常生成
- 页面是否可用
- 角色是否真正使用
- 是否出现高频错误

### 11.4 输出物
- 试运行启动确认
- 试运行问题记录
- 用户使用反馈

### 11.5 检查清单
- [ ] 试点功能已激活
- [ ] 数据输入链路畅通
- [ ] 学习事件正常生成
- [ ] StudentTwinAgent 正常更新
- [ ] 家长端可正常查看
- [ ] 老师端可正常查看
- [ ] 学校看板可正常查看
- [ ] 反馈入口可用

---

## 12. 第七阶段：跟踪与复盘

### 12.1 目标
保证试点不只是上线，而是可持续运行和迭代优化。

### 12.2 跟踪内容

| 跟踪项 | 说明 | 跟踪频率 |
|--------|------|----------|
| 使用活跃度 | 日活/周活用户数 | 周 |
| 数据覆盖度 | 学生数据覆盖率 | 周 |
| 事件生成质量 | 事件解析准确率 | 周 |
| 反馈提交情况 | 反馈数量和质量 | 周 |
| 老师端使用情况 | 老师查看和反馈频率 | 周 |
| 家长端查看率 | 家长查看频率 | 周 |
| 关键问题记录 | 高频问题记录 | 持续 |

### 12.3 复盘输出
- 接入复盘表
- 问题清单
- 优化建议
- 下一阶段扩展建议

### 12.4 复盘会议
- 参与方：学校负责人、平台运营、技术支持
- 频率：上线后 2 周、1 个月
- 内容：使用情况、问题分析、优化建议

## 16. 常见问题与处理原则

### 16.1 学生绑定错误

**现象**：学生班级绑定错误、地区绑定错误

**处理**：
- 暂停相关输出
- 修正绑定
- 重算相关状态
- 记录问题原因

### 16.2 教材版本不一致

**现象**：学校实际使用教材与系统绑定不一致

**处理**：
- 优先校正版本绑定
- 暂缓依赖该版本的解释输出
- 重新生成相关状态

### 16.3 数据输入不足

**现象**：家长/老师输入频率低，数据覆盖不足

**处理**：
- 明确告知当前观察边界
- 优先推动基础输入流程跑通
- 加强培训和引导

### 16.4 老师使用率低

**现象**：老师登录少、反馈少

**处理**：
- 重新做场景化培训
- 降低使用门槛
- 明确看板和反馈价值
- 了解使用障碍

### 16.5 家长反馈率低

**现象**：家长查看少、反馈少

**处理**：
- 调整报告和提醒口径
- 简化反馈入口
- 选择更合适触达时机
- 了解家长使用障碍

### 16.6 系统技术问题

**现象**：页面加载慢、功能报错

**处理**：
- 技术团队快速响应
- 记录问题日志
- 及时修复和通知
- 必要时回滚版本

## 17. 接入节奏建议

### 17.1 MVP 阶段建议

| 阶段 | 建议 |
|------|------|
| 学校选择 | 一次先接入 1 所学校 |
| 范围控制 | 先接入 1 个年级或少量班级 |
| 链路验证 | 先跑通一条完整链路 |
| 范围扩展 | 验证稳定后再扩大范围 |

### 17.2 不推荐做法

- [ ] 一开始全校铺开
- [ ] 角色还没培训就正式上线
- [ ] 数据没准备好就逼着跑报告
- [ ] 权限边界没理清就开放给所有人

这类操作通常会把试点变成事故演练。

### 17.3 建议时间表（参考）

| 阶段 | 建议时长 |
|------|----------|
| 接入确认 | 1 周 |
| 前置准备 | 1-2 周 |
| 系统配置 | 3-5 天 |
| 数据初始化 | 3-5 天 |
| 角色培训 | 1 周 |
| 试运行上线 | 2-4 周 |
| 跟踪复盘 | 持续 |

## 21. 风险与边界

### 21.1 不能把接入当终点
接入完成只是开始，不是项目成功。

### 21.2 不能忽视培训质量
培训不到位，系统再好也没人用。

### 21.3 不能忽视数据质量
数据初始化错误，后面所有输出都会错。

### 21.4 不能忽视反馈闭环
没有反馈闭环，问题会越积越多。

### 21.5 不能忽视角色协同
学校、平台、家长、老师需要协同，不是单方面推进。

---

## 22. 结论

学校接入流程的核心，不是把系统送进学校，而是把"规则、数据、角色、培训、使用、反馈"这一整套机制真正带进学校。

它必须坚持：

- **先确认范围** - 明确试点范围和责任方
- **再做好准备** - 前置准备充分
- **再开系统** - 系统配置正确
- **再训角色** - 培训到位
- **再小范围上线** - 试运行验证
- **再持续复盘** - 跟踪优化

否则所谓接入，很可能只是把一个链接放进微信群，然后大家一起假装项目已经开始了。
