## 1. 文档目的

本文档用于定义教育孪生项目在试点和运营阶段的指标体系与复盘机制，包括指标分类、采集方式、判断口径、复盘节奏与问题闭环方式。

本文档重点回答以下问题：

1. 试点和运营阶段到底看哪些指标
2. 指标如何分类，哪些是核心指标，哪些是辅助指标
3. 如何通过指标判断系统是不是跑起来了
4. 复盘应该怎么做，不至于流于形式
5. 如何把复盘真正转成产品、技术、运营改进项

---

## 2. 指标与复盘定义

### 2.1 运营指标
运营指标，是用于衡量系统接入、使用、数据质量、产出质量、角色活跃和试点价值感知的结构化指标集合。

### 2.2 复盘机制
复盘机制，是对运营过程中的数据、问题、反馈、产出效果进行周期性回顾，并形成可执行改进动作的机制。

指标负责"看见现状"，复盘负责"推动变化"。  
只有指标没有复盘，系统会变成很会统计自己问题的机器。

---

## 3. 指标体系目标

### 3.1 判断系统是否跑通
看主链路是否真的运转。

### 3.2 判断角色是否使用
看家长、老师、学校是否真的在用，而不是账号开了就当活跃。

### 3.3 判断输出是否有价值
看前台观察、报告、看板、建议是否被理解和使用。

### 3.4 判断问题在哪里
帮助识别数据问题、产品问题、流程问题、角色问题。

### 3.5 为扩展决策提供依据
决定试点是否扩容，功能是否可推广。

---

## 4. 指标分类

建议将运营指标分为六类：

| 类别 | 说明 |
|------|------|
| 接入指标 | 衡量学校和班级是否完成系统进入 |
| 数据指标 | 衡量输入、事件和状态的基础质量 |
| 使用指标 | 衡量各角色是否真实使用系统 |
| 输出指标 | 衡量系统产出是否稳定、是否被消费 |
| 反馈指标 | 衡量反馈闭环是否形成 |
| 结果指标 | 衡量系统是否开始产生真实使用价值 |

---

## 5. 接入指标

### 5.1 目标
衡量学校和班级是否完成系统进入。

### 5.2 建议指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| 已接入学校数 | 完成系统接入的学校数量 | 计数 |
| 已接入年级数 | 完成接入的年级数量 | 计数 |
| 已接入班级数 | 完成接入的班级数量 | 计数 |
| 已初始化学生数 | 完成档案初始化的学生数 | 计数 |
| 已开通老师账号数 | 已开通的老师账号数 | 计数 |
| 已开通家长账号数 | 已开通的家长账号数 | 计数 |
| 接入完成率 | 实际接入数/计划接入数 | 百分比 |

### 5.3 用途
判断实施动作是否真正落地。

---

## 6. 数据指标

### 6.1 目标
衡量输入、事件和状态的基础质量。

### 6.2 建议指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| 原始输入量 | 接收到的原始输入总数 | 计数（日/周） |
| 有效输入占比 | 有效输入数/总输入数 | 百分比 |
| 学习事件生成成功率 | 成功生成事件数/输入数 | 百分比 |
| 事件校验通过率 | 校验通过事件数/生成事件数 | 百分比 |
| OCR 成功率 | OCR 识别成功数/识别请求数 | 百分比 |
| StudentTwinAgent 更新成功率 | 更新成功数/更新请求数 | 百分比 |
| 数据完整性覆盖率 | 数据完整学生数/总学生数 | 百分比 |
| 多来源输入覆盖率 | 有多来源输入学生数/总学生数 | 百分比 |

### 6.3 用途
判断系统是不是"看上去在运行，实际上数据在漏"。

---

## 7. 使用指标

### 7.1 目标
衡量各角色是否真实使用系统。

### 7.2 建议指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| 家长端日活跃率 | 日活跃家长数/总家长数 | 百分比 |
| 家长端周活跃率 | 周活跃家长数/总家长数 | 百分比 |
| 老师端日活跃率 | 日活跃老师数/总老师数 | 百分比 |
| 老师端周活跃率 | 周活跃老师数/总老师数 | 百分比 |
| 学校端看板访问率 | 访问看板的学校管理员数/总数 | 百分比 |
| 人均查看次数 | 总查看次数/活跃用户数 | 次数 |
| 周报查看率 | 查看周报的家长数/总家长数 | 百分比 |
| 月报查看率 | 查看月报的家长数/总家长数 | 百分比 |
| 反馈入口使用率 | 使用反馈入口的用户数/活跃用户数 | 百分比 |

### 7.3 原则
"登录过一次"不等于"系统被使用"。别拿尸体当活人统计。

## 8. 输出指标

### 8.1 目标
衡量系统产出是否稳定、是否被消费。

### 8.2 建议指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| 周报生成成功率 | 生成成功数/应生成数 | 百分比 |
| 月报生成成功率 | 生成成功数/应生成数 | 百分比 |
| 报告打开率 | 打开报告数/收到报告数 | 百分比 |
| 家长端重点提示点击率 | 点击提示数/看到提示数 | 百分比 |
| 老师端学生详情查看率 | 查看详情的老师数/活跃老师数 | 百分比 |
| 班级看板查看率 | 查看看板的老师数/总老师数 | 百分比 |
| 推演任务完成率（后台） | 完成推演数/推演请求数 | 百分比 |

### 8.3 用途
判断输出链路是否有效。

## 9. 反馈指标

### 9.1 目标
衡量反馈闭环是否形成。

### 9.2 建议指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| 家长反馈提交率 | 提交反馈的家长数/活跃家长数 | 百分比 |
| 老师反馈提交率 | 提交反馈的老师数/活跃老师数 | 百分比 |
| 状态纠正反馈量 | 状态纠正类反馈数量 | 计数 |
| 反馈处理完成率 | 已处理反馈数/总反馈数 | 百分比 |
| 问题关闭率 | 已关闭问题数/总问题数 | 百分比 |
| 干预执行回填率 | 有回填记录的干预数/执行干预数 | 百分比 |

### 9.3 用途
判断系统是不是单向广播，而不是闭环系统。

## 10. 结果指标

### 10.1 目标
衡量系统是否开始产生真实使用价值。

### 10.2 建议指标

| 指标 | 说明 | 采集方式 |
|------|------|----------|
| 家长端理解度评价 | 家长对报告理解程度的评分 | 问卷/访谈 |
| 老师端实用度评价 | 老师对系统实用性的评分 | 问卷/访谈 |
| 学校端管理参考度评价 | 学校对管理参考价值的评分 | 问卷/访谈 |
| 重点问题识别准确性感知 | 用户对问题识别准确性的感知 | 问卷/访谈 |
| 报告建议可执行性感知 | 用户对建议可执行性的感知 | 问卷/访谈 |
| 试点续用意愿 | 学校/老师/家长继续使用的意愿 | 问卷/访谈 |

### 10.3 原则
这类指标很多带主观性，但很重要。没有"感知价值"，系统再完整也可能没人用。

## 11. 指标分层建议

建议将指标分为三层：

### 11.1 核心指标
用于判断试点是否基本成立。

| 指标 | 说明 |
|------|------|
| 学习事件生成成功率 | 主链路是否畅通 |
| StudentTwinAgent 更新成功率 | 状态更新是否正常 |
| 家长周报查看率 | 家长是否使用 |
| 老师端周活跃率 | 老师是否使用 |
| 反馈提交率 | 闭环是否形成 |

### 11.2 诊断指标
用于定位问题出在哪。

| 指标 | 说明 |
|------|------|
| OCR 成功率 | 识别质量 |
| 数据完整性覆盖率 | 数据覆盖情况 |
| 报告打开率 | 报告吸引力 |
| 推演任务完成率 | 后台任务状态 |

### 11.3 观察指标
用于长期趋势跟踪。

| 指标 | 说明 |
|------|------|
| 家校反馈变化趋势 | 反馈积极性变化 |
| 班级看板活跃趋势 | 看板使用情况 |
| 试点扩展意愿 | 学校续用/扩展意愿 |

## 12. 指标口径规范

每个指标建议至少明确以下口径：

| 口径项 | 说明 | 示例 |
|--------|------|------|
| 时间窗口 | 日/周/月/阶段 | 周活跃率 |
| 统计范围 | 学校/年级/班级/试点整体 | 某学校 |
| 分母口径 | 全部用户/已开通用户/活跃用户/有效对象 | 已开通家长数 |
| 统计方式 | 次数/占比/趋势/平均值 | 百分比 |

不把口径写清楚，后面每个人都能统计出自己喜欢的真相。

## 14. 复盘机制目标

复盘机制建议完成三件事：

1. **发现问题** - 识别当前存在的问题
2. **确认原因** - 分析问题产生的原因
3. **形成改进动作** - 制定可执行的改进计划

如果复盘只停在"大家都很努力"，那不是复盘，是团建结束陈词。

| 使用活跃度观察 | 用户使用情况 |

### 15.2 月复盘
适用于：

| 场景 | 说明 |
|------|------|
| 阶段趋势 | 月度趋势分析 |
| 家长/老师使用情况 | 角色使用深度分析 |
| 报告与看板效果 | 输出效果评估 |
| 功能优化方向 | 产品优化规划 |

### 15.3 阶段复盘
适用于：

| 场景 | 说明 |
|------|------|
| 一个试点周期结束 | 试点总结 |
| 扩容决策前 | 扩展评估 |
| 关键版本发布后 | 版本效果评估 |

## 18. 阶段复盘内容建议

阶段复盘建议至少包含：

1. **试点范围回顾** - 试点范围是否达成
2. **核心指标回顾** - 核心指标达成情况
3. **关键成功点** - 做得好的地方
4. **关键问题点** - 遇到的主要问题
5. **高优先级改进项** - 需要优先改进的项
6. **下一阶段建议** - 后续发展建议

## 20. 问题闭环机制

### 20.1 问题分类
建议问题分为：

| 类别 | 说明 |
|------|------|
| 数据问题 | 数据质量、输入问题 |
| 产品理解问题 | 用户对功能理解偏差 |
| 角色使用问题 | 使用频率、使用方式问题 |
| 技术稳定性问题 | 系统稳定性、性能问题 |
| 规则口径问题 | 规则解释、口径一致性问题 |

### 20.2 闭环动作
每个问题建议记录：

| 记录项 | 说明 |
|--------|------|
| 问题描述 | 问题的具体描述 |
| 影响范围 | 影响的用户/功能范围 |
| 优先级 | 高/中/低 |
| 责任人 | 负责处理的人员 |
| 计划动作 | 计划采取的处理动作 |
| 完成状态 | 未开始/进行中/已完成 |
| 复盘结果 | 复盘时的处理结果 |

## 21. 指标与复盘的关系

指标告诉你"哪里不对劲"，复盘告诉你"为什么不对劲、接下来谁去改"。

所以：

| 关系 | 说明 |
|------|------|
| 指标不等于结论 | 指标只是数据，需要分析解读 |
| 复盘不等于抱怨 | 复盘是为了改进，不是为了吐槽 |
| 只有把指标和复盘连起来 | 系统才会成长 |

## 22. MVP 建议核心指标集

MVP 阶段建议优先盯住以下指标：

### 22.1 数据链路

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 原始输入量 | 日输入量 | 持续增长 |
| 学习事件生成成功率 | 事件生成质量 | ≥85% |
| StudentTwinAgent 更新成功率 | 状态更新质量 | ≥90% |

### 22.2 使用链路

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 家长周报查看率 | 家长使用情况 | ≥50% |
| 老师端周活跃率 | 老师使用情况 | ≥60% |
| 看板查看率 | 学校端使用情况 | ≥40% |

### 22.3 闭环链路

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 家长反馈率 | 家长反馈积极性 | ≥20% |
| 老师反馈率 | 老师反馈积极性 | ≥30% |
| 问题关闭率 | 问题处理效率 | ≥80% |

### 22.4 价值感知

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 家长理解度 | 家长对报告的理解 | ≥4/5 分 |
| 老师实用度 | 老师对实用性的评价 | ≥4/5 分 |
| 学校续用意愿 | 学校继续使用意愿 | ≥80% |

## 23. 推荐样例

### 23.1 周复盘摘要示例

```
试点周复盘（第 3 周）

一、核心指标
- 学习事件生成成功率：86%
- StudentTwinAgent 更新成功率：92%
- 家长周报查看率：61%
- 老师端周活跃率：74%

二、主要问题
- OCR 质量在扫描件场景下不稳定
- 家长反馈率偏低（仅 15%）
- 某班级老师端使用频率明显不足

三、下周动作
- 优化扫描上传提示文案
- 简化家长反馈入口
- 对该班级做一次定向场景培训
- 责任人：张三、李四
- 截止时间：下周五
```

### 23.2 月复盘摘要示例

```
试点月复盘（第 1 个月）

一、核心指标趋势
- 家长周查看率：45% → 58% → 61%（上升）
- 老师周活跃率：55% → 68% → 74%（上升）
- 反馈提交率：12% → 18% → 22%（上升）

二、关键成功点
- 周报内容得到家长普遍认可
- 老师端重点学生视图使用率高
- 问题响应速度快

三、关键问题点
- 数据输入来源仍较单一
- 学校端看板使用率低
- 部分家长反映专业术语过多

四、下月优化重点
- 增加更多输入渠道引导
- 优化学校端看板内容
- 简化家长端表达
```

## 25. 风险与边界

### 25.1 不能只看指标不看原因

指标异常需要分析原因，不能只看数字。

### 25.2 不能忽视定性反馈

问卷和访谈的定性反馈同样重要。

### 25.3 不能让复盘流于形式

复盘必须有改进动作，不是开会聊天。

### 25.4 不能让问题没有闭环

每个问题都应该有处理结果和关闭记录。

### 25.5 不能忽视指标口径一致性

不同人统计的指标应该可以对比。

## 26. 运营看板建议

建议建立运营看板，包含以下模块：

| 模块 | 内容 | 更新频率 |
|------|------|----------|

| 使用趋势 | 各角色使用趋势图 | 周更 |
| 输出质量 | 报告生成和查看情况 | 周更 |
| 反馈闭环 | 反馈处理进度 | 实时更新 |

## 27. 结论

运营指标与复盘机制的核心，不是做更多表，而是让系统知道：

- 现在跑得怎么样
- 问题在哪
- 谁去改
- 改完有没有变好

它必须坚持：

- **指标少而关键** - 聚焦核心指标
- **复盘真而可执行** - 复盘要真实、改进要可执行
- **闭环明确** - 每个问题有闭环
- **节奏稳定** - 定期复盘不中断

否则项目很容易进入一种很熟悉的状态：大家都在开会复盘，但系统一点没变好。
