## 1. 文档目的

本文档用于冻结 KB + INGEST 首轮试点的执行范围、责任边界、放行条件与阻断条件，避免试点阶段范围漂移、职责重叠和“边做边改边扩大”。

本文档重点回答以下问题：

1. 本轮试点到底做什么
2. 本轮试点明确不做什么
3. 范围冻结值是什么
4. KB、INGEST、架构、测试、业务各自负责什么、不负责什么
5. 在什么条件下可以升级范围，什么情况下必须阻断推进

## 3. 试点目标

### 3.1 首轮目标

在单地区、单学校、单年级、单学科、有限输入源范围内，跑通以下最小闭环：

```text
原始输入
  ↓
输入清洗
  ↓
学习事件生成
  ↓
知识底座引用
  ↓
标准事件输出
  ↓
人工复核回流
```

### 3.2 本轮要验证的核心问题

- KB 是否能提供稳定、可引用、可追溯的知识参照
- INGEST 是否能把有限输入源稳定转成标准学习事件
- KB 与 INGEST 之间的引用关系是否稳定可用
- 异常输入是否能被有效拦截并进入人工兜底
- 首轮试点是否具备后续扩大范围的基础

---

## 4. 范围原则

### 4.1 小而完整
本轮不追求“全量覆盖”，只追求“主链路跑通”。

### 4.2 先静态底座，后多源扩张
先把可引用的固定知识底座建稳，再增加更多输入源和复杂场景。

### 4.3 先高价值、低复杂度输入
优先覆盖家长文本、老师反馈、扫描件/OCR 三类输入，不同时开启所有渠道。

### 4.4 先单版本稳定，再讨论多版本映射
首轮只做单教材版本，不做跨版本切换和迁移。

### 4.5 人工兜底有边界
人工兜底只处理高价值异常，不替代系统日常主链路。

## 5. 范围冻结值

### 5.1 试点冻结值表

| 维度 | 冻结值 | 说明 |
|------|--------|------|
| 试点地区 | 安徽省 | 采用现有地区规则示例口径 |
| 地区规则包 | `AH_RULE_V1` | 首轮规则绑定包标识 |
| 学校范围 | 1 所安徽省普通高中试点校 | 单校试点，学校具体名称在接入确认单冻结 |
| 年级范围 | 高一 | 不扩到高二、高三 |
| 班级范围 | 1-2 个班 | 首轮建议不超过 2 个班 |
| 学生规模 | 不超过 120 人 | 控制人工复核和问题定位成本 |
| 学科范围 | 高中物理 | 首轮只做物理 |
| 教材版本 | `PHY_PEP_G1_V1` | 人教版高中物理必修一 |
| 知识底座版本 | `KB_CN_AH_PHY_PEP_G1_V1_1.0.0` | 首轮底座冻结版本号 |
| 输入源范围 | 家长文本、老师反馈文本、作业/试卷扫描件 | 三类输入源足以覆盖主链路 |
| 接入渠道 | 微信家长输入、老师后台录入、文件上传/OCR | 不启用钉钉、硬件、批量导入 |
| 事件范围 | 作业事件、教师反馈事件、进度事件、考试事件 | 干预事件与复杂任务事件暂不纳入首轮 |
| 人工复核池 | 启用 | 作为异常输入兜底机制 |
| 运行模式 | 首轮骨架开发 + 小范围试运行 | 不进入全量联调与全量推广 |

### 5.2 学校冻结说明

学校具体名称、班级名单、学生名单在 `OPS-002`《试点学校实施方案》的“试点确认阶段”冻结。  
在 KB + INGEST 模块级放行前，学校维度只冻结到“单校普通高中试点”即可，不阻塞本轮骨架开发。

## 6. 本轮试点做什么

### 6.1 KB 本轮做什么

- 建立安徽省高中场景下的地区绑定入口
- 建立高中物理、人教版必修一的教材版本对象
- 建立章节树、知识点、能力点、教材锚点、依赖关系的首轮可用底座
- 建立首轮底座版本号、发布状态和追溯信息
- 提供 INGEST 可引用的章节、知识点、锚点查询依据

### 6.2 INGEST 本轮做什么

- 接收并处理家长文本输入
- 接收并处理老师反馈文本输入
- 接收并处理作业/试卷扫描件输入
- 生成标准学习事件
- 对事件进行知识引用和质量分级
- 对低置信度、失败、歧义输入生成人工复核任务
- 输出标准事件供后续模块消费

### 6.3 试点执行本轮做什么

- 锁定 1 所试点校和 1 个年级
- 锁定教材版本与地区规则
- 锁定首批输入渠道
- 跟踪异常处理与人工复核负荷
- 产出首轮问题清单和升级建议

---

## 7. 本轮试点不做什么

### 7.1 KB 本轮不做

- 不做多地区并行底座
- 不做多学科并行入库
- 不做跨教材版本映射
- 不做误区表、实验活动表、题型映射表的全量建设
- 不做全国性规则覆盖

### 7.2 INGEST 本轮不做

- 不接入钉钉输入
- 不接入学校批量导入
- 不接入授权硬件采集
- 不处理全量历史数据回溯
- 不做多事件复杂切分优化到极致
- 不把所有弱信号都自动写入主链路

### 7.3 试点执行本轮不做

- 不全校铺开
- 不多学科并行开放
- 不把家长端、老师端、后台所有输出都同时拉满
- 不以“演示效果”替代“主链路稳定性验证”
- 不在未冻结范围前开启全量联调

## 8. 首轮交付边界

### 8.1 KB 首轮最低交付物

| 交付物 | 是否必须 | 说明 |
|--------|----------|------|
| 地区绑定信息 | 是 | 至少支持安徽省高中规则绑定 |
| 教材版本对象 | 是 | 明确到 `PHY_PEP_G1_V1` |
| 章节树 | 是 | 作为所有知识引用骨架 |
| 知识点表 | 是 | 作为主引用对象 |
| 能力点表 | 是 | 作为能力维度映射依据 |
| 教材锚点表 | 是 | 支撑可追溯性 |
| 依赖关系表 | 是 | 支撑最小解释能力 |
| 版本发布记录 | 是 | 支撑回滚和追溯 |
| 题型映射表 | 否 | 本轮可后置 |
| 误区表 | 否 | 本轮可后置 |
| 实验活动表 | 否 | 本轮可后置 |

### 8.2 INGEST 首轮最低交付物

| 交付物 | 是否必须 | 说明 |
|--------|----------|------|
| 输入接收能力 | 是 | 覆盖首批 3 类输入 |
| 清洗与规范化 | 是 | 能做格式、对象、时间、内容清洗 |
| 标准事件生成 | 是 | 输出符合 `INGEST-007` 口径 |
| 知识引用能力 | 是 | 能引用章节、知识点、锚点 |
| 质量分级能力 | 是 | 至少区分 `valid` / `weak_valid` / `invalid` |
| 人工复核任务生成 | 是 | 异常输入不能丢 |
| 异常回流 | 是 | 人工结果能回流处理 |
| 批量回溯 | 否 | 本轮可后置 |
| 硬件接入链路 | 否 | 本轮不做 |
| 钉钉链路 | 否 | 本轮不做 |

## 9. 五类责任边界

### 9.1 KB 责任边界

#### 9.1.1 KB 负责

- 固定知识底座首轮对象建设
- 地区、教材、章节、知识点、能力点、锚点、依赖关系的可引用交付
- 底座版本冻结、发布、回滚入口
- 底座追溯信息维护
- 向 INGEST 提供权威知识引用依据

#### 9.1.2 KB 不负责

- 原始输入接收
- OCR 识别
- 学习事件生成
- 学生状态更新
- 前台展示和报告组织

### 9.2 INGEST 责任边界

#### 9.2.1 INGEST 负责

- 原始输入清洗与标准化
- 学习事件抽取与生成
- 事件质量分级
- 事件与 KB 的知识引用建立
- 异常输入识别
- 人工复核任务生成与回流处理

#### 9.2.2 INGEST 不负责

- 定义知识点本体
- 维护教材版本
- 修改 KB 底座内容
- StudentTwin 当前态计算
- 图谱写入后的推理结果解释

### 9.3 架构责任边界

#### 9.3.1 架构负责

- 定义 KB 与 INGEST 的模块边界
- 定义引用契约和状态流转口径
- 定义同步/异步处理边界
- 定义版本归属、回滚边界和依赖关系
- 定义最小可运行链路的系统承载方式

#### 9.3.2 架构不负责

- 代替 KB 写入知识内容
- 代替 INGEST 写清洗规则
- 代替测试做验收判断
- 代替业务决定试点学校名单

### 9.4 测试责任边界

#### 9.4.1 测试负责

- 建立 KB + INGEST 最小验收矩阵
- 准备样例集和失败样例
- 验证正确性、一致性、完整性、可追溯性
- 验证异常输入是否被拦截
- 验证人工复核是否可回流

#### 9.4.2 测试不负责

- 决定试点范围
- 定义业务规则优先级
- 修改底座内容
- 替开发补实现缺口

### 9.5 业务责任边界

#### 9.5.1 业务负责

- 冻结试点范围
- 冻结地区、学段、年级、学科、教材版本口径
- 确认输入源优先级
- 确认人工复核的业务处理顺序
- 确认试点学校、老师、班级、家长触达方式
- 决定试点是否升级或维持当前范围

#### 9.5.2 业务不负责

- 字段级技术设计
- 代码实现
- 替架构决定服务边界
- 绕过测试直接放行高风险链路

## 10. 协作交接规则

### 10.1 KB → INGEST

KB 必须提供：

- 当前有效底座版本
- 当前有效教材版本
- 可引用章节集合
- 可引用知识点集合
- 可引用锚点集合
- 弱引用和失配时的判定口径

### 10.2 INGEST → KB

INGEST 不得直接修改 KB 内容。  
当发现以下问题时，只能发起问题回流：

- 知识点缺失
- 锚点缺失
- 章节不一致
- 教材版本不匹配
- 引用歧义无法收敛

### 10.3 测试 → 业务
测试只给出是否达标和问题清单，不替业务扩大试点范围。

### 10.4 业务 → 开发
业务冻结范围后，开发只能在冻结范围内实现，不得自行扩大学科、地区、输入源。

## 11. 升级条件

满足以下条件后，才允许从“首轮骨架开发 + 小范围试运行”升级到下一阶段：

- KB 首轮底座版本已发布并冻结
- 首批输入源至少连续一周稳定运行
- 标准学习事件生成链路已跑通
- 人工复核池可控，未出现持续堆积
- 最小验收矩阵通过
- 试点问题有明确分类和处理闭环

---

## 12. 阻断条件

出现以下任一情况，应阻断扩容、阻断全量联调，必要时回退到当前范围：

### 12.1 KB 阻断条件

- 章节树与知识点引用大面积不一致
- 锚点覆盖率明显不足，无法支撑可追溯
- 教材版本绑定不稳定
- 回滚路径不可用

### 12.2 INGEST 阻断条件

- 大量输入无法形成有效事件
- 低置信度事件误入主链路
- 人工复核池持续堆积且无法消化
- 事件与知识引用错配率过高

### 12.3 协作阻断条件

- KB 与 INGEST 对同一引用口径理解不一致
- 架构边界不清导致职责冲突
- 测试样例和开发范围脱节
- 业务冻结范围被频繁打破

### 12.4 试点阻断条件

- 试点学校、教材版本或输入源未冻结
- 试点对象超出单校/单年级/单学科范围
- 关键责任人未明确
- 主链路未跑通却试图开启全量推广

## 13. 本轮执行结论

### 13.1 当前允许的推进方式

- 允许进入 KB + INGEST 首轮骨架开发
- 允许进入小范围试运行准备
- 允许在冻结范围内并行补齐接口契约、验收矩阵和样例集

### 13.2 当前不允许的推进方式

- 不允许进入多地区并行开发
- 不允许进入多学科并行联调
- 不允许在未冻结学校和输入源前启动全量试点
- 不允许把异常输入处理压力转移给线下人工长期兜底

### 13.3 管理口径

本轮试点以“范围冻结、主链路跑通、异常可控、责任清楚”为放行前提。凡超出本文档冻结边界的需求，一律视为下一轮事项，不纳入本轮正式开发放行范围。
