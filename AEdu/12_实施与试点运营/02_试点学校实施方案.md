## 1. 文档目的

本文档用于定义教育孪生项目在试点学校中的实施方案，包括试点目标、试点范围、实施阶段、角色分工、关键动作、风险控制和评估方式。

本文档重点回答以下问题：

1. 试点学校实施到底怎么推进
2. 试点不应该一下铺多大
3. 试点阶段每个角色做什么
4. 如何判断试点是否跑通
5. 如何让试点产出真正可复盘的结果

---

## 2. 试点学校实施定义

试点学校实施，是指在选定学校和明确范围内，将教育孪生项目从系统接入推进到真实教学和家校场景使用的全过程。

它不仅包括技术开通，还包括：

- 试点范围确定
- 学校组织协调
- 数据与规则准备
- 角色培训
- 试运行
- 使用跟踪
- 问题收集
- 复盘与扩展建议

试点不是"给学校演示一下"，而是验证系统能不能在现实世界活下来。

---

## 3. 试点目标

### 3.1 验证主链路是否可跑通
至少验证以下链路：

```
输入 → 学习事件 → StudentTwinAgent → 前台观察 → 报告输出 → 反馈回流
```

### 3.2 验证角色是否愿意使用
老师、家长、学校管理员是否真的用起来，比页面漂不漂亮更重要。

### 3.3 验证数据质量是否足够支撑观察
试点阶段必须验证：输入是否足够、事件是否靠谱、状态是否有解释力。

### 3.4 验证产品输出是否有价值
输出是否对家长、老师、学校真的有帮助，而不是只有项目团队自我感动。

### 3.5 沉淀问题与后续扩展依据
试点不是终点，而是下一阶段扩展的依据。

---

## 4. 试点范围建议

### 4.1 范围原则
试点范围应"小而完整"，而不是"大而空"。

### 4.2 MVP 建议范围

| 维度 | 建议 |
|------|------|
| 学校数量 | 1 所学校 |
| 年级范围 | 1 个年级或少量班级 |
| 学科范围 | 1-2 个学科优先 |
| 负责人 | 明确的学校负责人 |
| 数据输入 | 明确的数据输入方式 |

### 4.3 不推荐做法

- [ ] 一上来全校铺开
- [ ] 学科、年级、角色同时全量开放
- [ ] 数据准备没完成就开始跑报告

这类做法通常能快速制造一种叫"集体迷茫"的高级氛围。

---

## 5. 试点对象选择标准

建议优先选择具备以下条件的学校：

| 条件 | 说明 |
|------|------|
| 有明确试点负责人 | 学校方有明确对接人 |
| 有老师愿意配合使用 | 老师有使用意愿 |
| 组织协调成本可控 | 沟通成本低 |
| 教材版本与地区规则明确 | 规则清晰 |
| 家长端可触达 | 家长群体可联系 |
| 学校对反馈和复盘有意愿 | 愿意配合复盘 |

## 6. 实施阶段划分

建议试点学校实施分为六个阶段：

```
试点确认 → 准备 → 初始化 → 试运行 → 稳定运行 → 复盘扩展
```

---

## 7. 第一阶段：试点确认

### 7.1 目标
确认试点范围和基本共识。

### 7.2 关键动作

| 动作 | 说明 |
|------|------|
| 确认试点学校 | 明确学校名称和基本信息 |
| 确认试点年级/班级 | 明确参与试点的具体范围 |
| 确认学科范围 | 明确参与试点的学科 |
| 明确学校负责人 | 明确学校方对接人 |
| 明确平台项目负责人 | 明确平台方对接人 |
| 明确实施时间表 | 明确各阶段时间节点 |

### 7.3 输出物

- 试点确认单
- 试点范围表
- 角色联系人清单
- 实施排期表

---

## 8. 第二阶段：准备阶段

### 8.1 目标
完成实施前所需的规则、数据和组织准备。

### 8.2 关键动作

| 动作 | 负责方 |
|------|--------|
| 绑定地区规则 | 平台 |
| 确认教材版本 | 学校 + 平台 |
| 准备学生名单 | 学校 |
| 准备老师名单 | 学校 |
| 准备班级结构 | 学校 |
| 确认数据输入方式 | 学校 + 平台 |
| 准备系统配置 | 平台 |

### 8.3 输出物

- 初始化数据清单
- 教材与地区绑定清单
- 数据输入方案确认单

---

## 9. 第三阶段：初始化阶段

### 9.1 目标
完成系统初始化和账号开通。

### 9.2 关键动作

| 动作 | 负责方 |
|------|--------|
| 学校租户开通 | 平台 |
| 角色和权限配置 | 平台 |
| 学生档案初始化 | 平台 |
| StudentTwinAgent 创建 | 平台 |
| 家长端/老师端可登录 | 平台 |
| 基础测试通过 | 平台 + 学校 |

### 9.3 验收点

- [ ] 学生主体能查到
- [ ] 班级关系正确
- [ ] 教材版本绑定正确
- [ ] 基础页面可访问

### 9.4 输出物

- 系统初始化完成确认单
- 账号开通清单
- 基础测试报告

---

## 10. 第四阶段：试运行阶段

### 10.1 目标
在真实场景中小范围跑主链路。

### 10.2 关键动作

| 动作 | 负责方 |
|------|--------|
| 开始收集家长/老师/扫描输入 | 学校 |
| 开始生成学习事件 | 平台 |
| 开始更新 StudentTwinAgent | 平台 |
| 家长端开始看观察结果 | 学校 |
| 老师端开始查看与反馈 | 学校 |
| 报告开始生成 | 平台 |

### 10.3 重点观察

| 观察项 | 说明 |
|--------|------|
| 数据是否进得来 | 输入链路是否畅通 |
| 事件是否结构化 | 事件生成质量 |
| 状态是否合理 | 状态输出是否可信 |
| 报告是否可用 | 报告是否可理解 |
| 老师家长是否愿意看 | 使用意愿 |

### 10.4 输出物

- 试运行启动确认
- 试运行问题记录
- 用户使用反馈

---

## 11. 第五阶段：稳定运行阶段

### 11.1 目标
让试点从"能跑"变成"可持续用"。

### 11.2 关键动作

| 动作 | 说明 |
|------|------|
| 固定运行节奏 | 形成稳定的输入和查看习惯 |
| 跟踪数据覆盖度 | 监控数据完整性 |
| 跟踪家长端查看率 | 监控家长使用情况 |
| 跟踪老师反馈率 | 监控老师使用情况 |
| 跟踪看板使用情况 | 监控学校端使用 |
| 处理高频问题 | 及时响应问题 |

### 11.3 输出物

- 周运行简报
- 问题清单
- 优化建议

---

## 12. 第六阶段：复盘扩展阶段

### 12.1 目标
输出试点结论和下一阶段建议。

### 12.2 关键动作

| 动作 | 说明 |
|------|------|
| 复盘实施过程 | 回顾实施各阶段情况 |
| 复盘数据质量 | 评估数据完整性 |
| 复盘角色使用情况 | 评估各角色使用意愿 |
| 复盘输出价值感知 | 评估输出价值 |
| 形成下一阶段扩展建议 | 提出扩展建议 |

### 12.3 输出物

- 试点复盘报告
- 扩展建议
- 问题优先级列表

```
试点确认
    ↓

    ↓

    ↓

    ↓

    ↓

    ↓
小范围试运行
    ↓

    ↓
稳定运行
    ↓
试点复盘
```

## 16. 试点成功判断标准

建议从以下五个维度判断试点是否成功：

### 16.1 技术维度

| 标准 | 说明 |
|------|------|
| 主链路能跑通 | 输入→事件→状态→输出畅通 |
| 页面可访问 | 各端页面可正常访问 |
| 报告可生成 | 周报/月报可正常生成 |
| 反馈可回流 | 反馈入口可用 |

### 16.2 数据维度

| 标准 | 说明 |
|------|------|
| 输入覆盖度达到最低要求 | 基础输入数据充足 |
| 事件生成质量可接受 | 事件解析准确率达标 |
| 状态输出具备基本解释性 | 状态输出可理解 |

### 16.3 使用维度

| 标准 | 说明 |
|------|------|
| 家长端有查看 | 家长周查看率达标 |
| 老师端有反馈 | 老师周反馈率达标 |
| 学校端有看板使用 | 学校看板有访问 |

### 16.4 价值维度

| 标准 | 说明 |
|------|------|
| 家长能看懂 | 家长理解度评价达标 |
| 老师觉得有辅助价值 | 老师实用度评价达标 |
| 学校认为有管理参考价值 | 学校管理参考度评价达标 |

### 16.5 运营维度

| 标准 | 说明 |
|------|------|
| 有问题清单 | 问题记录完整 |
| 有迭代闭环 | 问题有处理和关闭 |
| 有扩展建议 | 有下一阶段建议 |

## 17. 试点风险与控制

### 17.1 数据不足风险

| 现象 | 输入量少、数据覆盖不足 |
|------|----------------------|
| 处理 | 先聚焦最基础输入链路，明确当前输出边界 |

### 17.2 老师不使用风险

| 现象 | 老师登录少、反馈少 |
|------|-------------------|
| 处理 | 降低使用门槛，强化班级总览和重点学生视图价值，通过培训跑真实场景 |

### 17.3 家长理解成本高风险

| 现象 | 家长反馈看不懂、使用率低 |
|------|------------------------|
| 处理 | 简化家长端表达，优先周报和重点提示，收集典型误解样本迭代 |

### 17.4 学校期望过高风险

| 现象 | 学校期望系统能解决所有问题 |
|------|-------------------------|
| 处理 | 明确当前阶段目标，先验证观察链路，不夸大推演和决策能力 |

### 17.5 权限和角色边界混乱风险

| 现象 | 不同角色看到不该看的内容 |
|------|----------------------|
| 处理 | 上线前做角色检查，明确学校/老师/家长视图差异 |

## 21. 风险与边界

### 21.1 不能把试点当终点

试点完成只是开始，不是项目成功。

### 21.2 不能忽视培训质量

培训不到位，系统再好也没人用。

### 21.3 不能忽视数据质量

数据初始化错误，后面所有输出都会错。

### 21.4 不能忽视反馈闭环

没有反馈闭环，问题会越积越多。

### 21.5 不能忽视角色协同

学校、平台、家长、老师需要协同，不是单方面推进。

---

## 22. 结论

试点学校实施方案的核心，不是把项目"推进进学校"，而是让系统在学校场景中真的跑起来、有人用起来、问题暴露出来、价值被验证出来。

它必须坚持：

- **范围小而完整** - 不贪大求全
- **角色清楚** - 各方职责明确
- **节奏稳定** - 按阶段推进
- **反馈真实** - 收集真实使用反馈
- **复盘可用** - 产出可执行改进项

否则试点就会变成一种高成本的现场参观活动。
