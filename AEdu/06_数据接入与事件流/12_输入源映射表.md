---

## 1. 文档目的

本文档用于定义教育孪生项目首轮骨架开发阶段中，各类输入源进入 INGEST 模块后的标准映射口径，明确：

1. 输入源进入系统后优先生成什么类型的学习事件
2. 每类输入至少需要满足哪些字段要求
3. 正常、失败、低置信度、人工复核分别怎么处理
4. 哪些输入属于首轮优先接入，哪些输入本轮不做
5. 如何避免不同渠道输入在事件生成阶段口径漂移

本文档是 `INGEST-007`《学习事件生成标准》、`INGEST-009`《输入数据清洗规则》、`INGEST-010`《异常输入与人工兜底流程》的首轮落地映射补充文档。

---

## 2. 使用边界

### 2.1 本文档适用范围

- 首轮骨架开发阶段的输入接入与事件生成
- 单学校试点、单年级、单学科优先场景
- 家长文本输入、老师反馈输入、扫描/OCR 输入

### 2.2 本文档不覆盖内容

- 不定义 API URL、请求参数、返回值字段
- 不定义 OCR 模型、NLP 模型的算法实现
- 不定义下游 StudentTwinAgent 的状态更新逻辑
- 不定义多学校、多地区同时并行接入的复杂编排
- 不定义音频、视频、长文档多模态解析的实现细节

## 3. 首轮接入原则

### 3.1 小而完整
首轮只接入最能跑通试点主链路的输入源，不追求一次性覆盖所有渠道。

### 3.2 可追溯优先
所有输入必须保留 `raw_input_id` 和原始来源引用，不允许只留下清洗结果。

### 3.3 高风险不直入主链路
对象不明、时间冲突、内容歧义大、OCR 质量差的输入，不得直接生成主链路事件。

### 3.4 先事件，后复杂推断
首轮优先保证“输入可进来、事件可生成、知识可引用、异常可兜住”，不追求复杂语义推断。

### 3.5 单条输入单次处理口径明确
同一条原始输入在首轮阶段应有明确的主处理路径，避免同一输入被多个规则重复生成冲突事件。

## 4. 首轮优先接入源

### 4.1 P0 优先接入源

| 优先级 | 输入源 | 典型场景 | 首轮目标 |
|--------|--------|----------|----------|
| P0 | 家长聊天文本输入 | 家长汇报作业完成、错题情况 | 跑通文本到事件主链路 |
| P0 | 老师反馈文本输入 | 老师评语、课堂反馈、阶段评价 | 跑通反馈类事件主链路 |
| P0 | 单页图片/扫描件输入 | 作业照片、试卷照片、练习册单页 | 跑通 OCR 到事件主链路 |

### 4.2 P1 条件接入源

| 优先级 | 输入源 | 条件 |
|--------|--------|------|
| P1 | 批量成绩表导入 | 仅在首轮三类 P0 输入稳定后再开启 |
| P1 | 系统任务日志 | 仅在主链路稳定后作为补充来源启用 |
| P1 | 单学生 PDF 附件 | 仅限页数少、结构清晰、与单学生明确绑定的 PDF |
| P1 | 老师手工后台录入 | 仅限作为异常兜底补录入口，不作为主来源 |

## 5. 本轮不做源

| 输入源 | 本轮不做原因 | 处理口径 |
|--------|--------------|----------|
| 钉钉组织链路输入 | 首轮不扩第二接入渠道 | 不进入首轮范围 |
| 授权硬件直连接入 | 接入、绑定、运维成本高 | 不进入首轮范围 |
| 音频输入 | 识别链路长、噪声高、验证成本高 | 直接拦截 |
| 视频输入 | 成本高、对象识别复杂 | 不进入首轮范围 |
| 多页混合 PDF 档案包 | 单次输入可能包含多学生、多事件 | 仅保留原始文件，不做自动主链路 |
| 无明确学生绑定的群聊截图 | 对象识别风险高 | 进入异常池，不自动生成事件 |
| 外部规则更新通知 | 不属于学习事件 | 转规则侧，不进 INGEST 主链路 |
| 大模型自由总结文本 | 可读但不可审计 | 不作为标准学习事件直接来源 |

## 6. 输入源映射主表

| 输入源 | source_type / source_role | 典型输入 | 首选事件类型 | 最小字段要求 | 正常处理方式 | 失败处理方式 | 低置信度处理 | 人工复核触发 |
|--------|---------------------------|----------|--------------|--------------|--------------|--------------|--------------|--------------|
| 家长聊天文本输入 | `text_input` / `parent` | “今天物理作业做完了，错了两题” | `homework` / `parent_feedback` | `tenant_id`、`student_id`、`source_text_ref`、`summary` | 文本清洗后抽取作业完成、错题、学科、时间，生成单事件或主事件+附加反馈 | 无法识别学生或核心学习内容时不生成事件，进入异常池 | 仅识别出学科或仅识别出“完成了”但无法稳定抽题目/知识点时，生成 `review_needed` 候选结果 | 学生不明确、时间冲突、文本同时包含多个互斥事件、内容与学生绑定不匹配 |
| 老师反馈文本输入 | `text_input` / `teacher` | “受力分析掌握不稳，课堂参与正常” | `teacher_feedback` / `progress` | `tenant_id`、`student_id/class_id`、`operator_id`、`summary`、`source_text_ref` | 结构化为反馈事件，不直接改写事实状态；可附 `knowledge_refs` 候选 | 若对象无法确定到学生且不能安全落班级级别，不生成事件 | 对知识点候选和课堂行为候选分别打置信度，允许生成弱结果 | 同名学生冲突、反馈对象不明、文本过短、评价语义高度歧义 |
| 单页作业照片 | `image_scan` / `parent` 或 `teacher` | 作业拍照上传 | `homework` / `error` / `progress` | `tenant_id`、`student_id`、`source_file_ref`、图片质量达标、OCR 可读结果 | 先 OCR，再抽取题目、完成状态、错题数、章节/知识点候选，生成作业或错题事件 | OCR 不可读、图片模糊、无学生绑定时不生成事件 | OCR 可读但题目切分或知识映射不稳定时，进入 `review_needed` | 模糊、遮挡、页码错乱、题号缺失、分数识别冲突、无学生信息 |
| 单页试卷照片 | `image_scan` / `teacher` 或 `school_admin` | 单张月考试卷、随堂测试照片 | `exam` / `error` | `tenant_id`、`student_id`、`source_file_ref`、考试名候选、分数候选或题目结果候选 | OCR 后提取考试名称、分数、题目表现，优先生成考试事件，必要时附加错题事件 | 分数、对象、考试名称三者都不稳定时不生成正式考试事件 | 允许只生成 `review_needed`，不直接落 `exam` 主事件 | 分数冲突、同页多人、时间不明、总分不可确认 |
| 老师后台手工补录 | `manual_entry` / `teacher` 或 `operator` | 异常后人工确认补录 | `homework` / `exam` / `teacher_feedback` / `progress` | `tenant_id`、`student_id`、`operator_id`、人工确认标记、事件摘要 | 作为异常兜底入口，按人工确认结果生成正式事件 | 必填字段不全则禁止提交 | 不存在低置信自动路径，人工补录必须是高确认输入 | 补录内容与原始输入不一致、缺少复核依据、无原始引用 |

## 7. 输入源到事件类型的首轮收敛规则

### 7.1 家长文本输入
首轮只允许优先收敛到以下三类：

- `homework`
- `parent_feedback`
- `progress`

不在首轮自动生成：

- 复杂干预事件
- 多事件串联事件包
- 带强推断的知识掌握结论

### 7.2 老师反馈输入
首轮优先收敛到：

- `teacher_feedback`
- `progress`

不在首轮自动生成：

- 直接改写学生状态的事实类事件
- 复杂课堂行为画像

### 7.3 图片/OCR 输入
首轮优先收敛到：

- `homework`
- `exam`
- `error`

不在首轮自动生成：

- 多页综合诊断报告
- 一次 OCR 生成多学生全量事件包

## 8. 字段要求分级

### 8.1 A 级必填字段
以下字段缺失时，不得进入标准学习事件主链路：

| 字段 | 说明 |
|------|------|
| tenant_id | 必须明确租户归属 |
| student_id | 必须明确学生主体 |
| source_type | 必须明确来源类型 |
| source_role | 必须明确来源角色 |
| event_type | 必须明确一级事件类型 |
| summary | 必须有最小可读事件摘要 |
| raw_input_id 或 source_file_ref/source_text_ref | 必须可追溯到原始输入 |
| validation_status | 必须有校验结果 |

### 8.2 B 级强建议字段
以下字段允许缺失，但缺失时必须进入弱事件或待补态：

| 字段 | 缺失后的处理口径 |
|------|------------------|
| event_time | 可使用弱代理时间，但必须标注来源 |
| subject | 可先为空，但必须尝试补映射 |
| chapter_refs | 可为空，但须进入映射待补态 |
| knowledge_refs | 可为空，但高价值事件应优先补齐 |
| curriculum_version_id | 可由 KB 侧根据学生绑定后补齐 |

### 8.3 C 级增强字段
以下字段不作为首轮主链路阻断条件：

| 字段 | 说明 |
|------|------|
| ability_refs | 首轮允许延后补齐 |
| anchor_refs | 对文本事件可延后补齐，对 OCR 事件优先补 |
| reviewed_by / reviewed_at | 仅人工复核后填写 |

## 9. 异常处理边界表

| 场景 | 异常类型 | 是否允许自动生成正式事件 | 是否允许生成弱结果 | 是否必须人工复核 | 处理结果 |
|------|----------|--------------------------|--------------------|------------------|----------|
| 学生对象明确，内容明确，时间可解析 | 无 | 是 | 否 | 否 | 直接入主链路 |
| 学生对象明确，内容明确，但知识映射为空 | content / mapping | 是 | 否 | 否 | 生成正式事件，标记 `mapping_pending` |
| 学生对象明确，内容大体明确，但时间有多个候选 | time | 否 | 是 | 是 | 进入待复核，禁止直写主状态 |
| 学生对象不明确 | object | 否 | 否 | 是 | 进入异常池 |
| OCR 质量差但仍能识别部分内容 | file_ocr | 否 | 是 | 是 | 生成候选解析结果，不生成正式事件 |
| OCR 基本不可读 | file_ocr | 否 | 否 | 视价值而定 | 拒绝自动处理，提示重传 |
| 来源可信但文本极短，如“已完成” | content | 否 | 是 | 是 | 仅保留候选，不入正式事件 |
| 同一输入可能拆成多个事件 | content | 否 | 是 | 是 | 先入待复核，不自动多拆 |
| 输入内容与学生当前绑定学科明显不匹配 | object / content | 否 | 否 | 是 | 阻断并进入异常池 |

---

## 10. 人工复核最小触发条件

以下任一条件满足时，必须生成人工复核任务：

1. `student_id` 无法稳定确认
2. `event_time` 存在两个及以上互斥候选
3. OCR 结果中关键字段缺失超过 2 项
4. 文本可能对应多个独立事件，自动切分不稳定
5. `knowledge_refs` 候选冲突且影响事件解释
6. 输入内容与当前班级、教材版本、地区规则明显不一致
7. 人工补录与原始输入不一致且无补充说明

## 11. 首轮处理状态建议

首轮骨架开发阶段，输入处理状态建议统一为以下六类：

| 状态 | 说明 | 是否进入主链路 |
|------|------|----------------|
| `raw_received` | 原始输入已接收 | 否 |
| `cleaned` | 已完成基础清洗 | 否 |
| `parsed` | 已完成事件候选抽取 | 否 |
| `accepted` | 已生成正式学习事件 | 是 |
| `review_needed` | 需人工复核 | 否 |
| `rejected` | 明确拒绝进入主链路 | 否 |

说明：

- 首轮不再扩展更多细粒度状态，避免骨架阶段状态机过度复杂。
- `accepted` 是唯一允许进入主链路的状态。
- `review_needed` 必须有异常原因编码和原始输入引用。

## 12. 首轮放行约束

### 12.1 允许进入骨架开发的输入范围

- 家长文本输入
- 老师反馈文本输入
- 单页图片/扫描件

### 12.2 不允许在首轮放行前直接开发的输入范围

- 钉钉链路
- 授权硬件链路
- 音频、视频、多页复杂 PDF
- 无学生绑定的群聊截图
- 外部规则类通知
- 无法审计来源的大模型自由总结输入

### 12.3 放行判定口径

只有当以下条件同时满足，首轮输入映射方案才视为可进入正式开发：

- 已覆盖首轮优先接入源
- 已覆盖正常、失败、低置信度、人工复核四类路径
- 已定义异常边界和阻断口径
- 已定义主链路最小状态流转
- 已明确哪些输入本轮不做

## 14. 结论

首轮骨架开发阶段，INGEST 模块不再追求“所有输入都能自动变成事件”，而是先把最小闭环跑通：

- 有明确优先接入源
- 有统一事件映射口径
- 有清晰的失败和低置信度边界
- 有人工复核兜底
- 有明确的本轮不做范围

只要坚持这个边界，INGEST 就能从“概念上能接很多输入”推进到“工程上能稳定接住首轮输入”。
