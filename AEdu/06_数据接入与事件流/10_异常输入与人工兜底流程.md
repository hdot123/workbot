# 异常输入与人工兜底流程

> 文档编号：INGEST-010  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：数据接入与事件流负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中异常输入的识别标准、分级机制、人工兜底流程、处理角色与回流规则。

本文档重点回答以下问题：

1. 什么样的输入应被视为异常
2. 异常输入应该如何分级处理
3. 哪些问题可以自动降级，哪些必须人工介入
4. 人工兜底的最小流程是什么
5. 如何避免异常输入悄悄进入主链路，或人工兜底变成无底洞

---

## 2. 异常输入定义

异常输入，是指在对象识别、时间识别、内容理解、格式解析、来源可信度、结构一致性等方面无法达到自动处理要求，因而不适合直接进入学习事件主链路的输入。

异常不等于无效。  
很多异常输入本身有价值，只是不能直接自动处理。

例如：

- 学生对象不明确
- OCR 结果严重混乱
- 同一条记录出现多个互相冲突时间
- 内容歧义过大
- 文件格式损坏
- 输入内容明显与当前绑定学生不匹配

---

## 3. 建设目标

### 3.1 防止坏输入污染主链路
不能让高风险输入直接更新 StudentTwinAgent。

### 3.2 保留高价值但难自动处理的信息
异常不一定丢弃，应先兜住。

### 3.3 降低人工工作量
通过异常分类和优先级控制，避免人工兜底变成纯人海战术。

### 3.4 建立问题回流机制
人工处理结果应反哺清洗规则和自动识别能力。

### 3.5 保证全链路可追溯
每一条异常输入都应知道为何异常、谁处理了、最后结果如何。

---

## 4. 设计原则

### 4.1 自动优先，人工兜底
能自动处理的不走人工；不能稳定自动处理的必须兜住。

### 4.2 高风险输入不强入主链路
一旦对象、时间或内容核心字段不清，就不应直接入主链路。

### 4.3 人工处理有边界
人工兜底是解决高价值异常，不是手工替代整个系统。

### 4.4 分类清楚
不同异常类型要有不同处理策略。

### 4.5 处理结果回流
人工兜底不能只解决个案，还应推动规则优化。

---

## 5. 异常输入分类建议

建议至少分为以下六类：

| 异常类型 | 编码 | 说明 |
|----------|------|------|
| 对象异常 | object | 系统无法确认输入到底属于哪个学生、老师、班级或学校 |
| 时间异常 | time | 系统无法确认事件发生时间，或存在明显冲突 |
| 内容异常 | content | 输入内容无法稳定切分、解析或理解 |
| 文件与 OCR 异常 | file_ocr | 上传文件本身或 OCR 结果质量不足 |
| 结构异常 | structure | 批量导入或结构化数据格式不符合要求 |
| 来源异常 | source | 输入来源不明、来源不可信或来源与声明不符 |

---

## 6. 对象异常

### 6.1 定义
系统无法确认输入到底属于哪个学生、老师、班级或学校。

### 6.2 典型场景

| 场景 | 说明 |
|------|------|
| 同名学生冲突 | 同一班级有多个同名学生，输入未明确区分 |
| 家长昵称无法稳定识别 | 微信昵称无法对应真实家长 |
| 批量导入缺少学号/班级标识 | 表格中缺少关键身份字段 |
| 扫描件无学生信息 | 上传图片中无学生姓名或学号 |

### 6.3 风险
对象一旦绑错，后面全链路都会错。

### 6.4 处理策略

| 策略 | 说明 |
|------|------|
| 优先尝试自动匹配 | 通过手机号、班级等辅助信息匹配 |
| 匹配失败进入异常池 | 标记为 object_uncertain |
| 由运营人员联系学校确认 | 人工确认学生身份 |

---

## 7. 时间异常

### 7.1 定义
系统无法确认事件发生时间，或存在明显冲突。

### 7.2 典型场景

| 场景 | 说明 |
|------|------|
| 相对时间上下文不完整 | "昨天""上周"但无参考基准 |
| OCR 识别日期残缺 | 扫描件中日期部分识别失败 |
| 表格日期列格式错乱 | Excel 中日期格式不统一 |
| 同一输入中多个可能时间冲突 | 文本中出现多个矛盾时间 |

### 7.3 风险
时间错了，趋势、报告和推演都会被带歪。

### 7.4 处理策略

| 策略 | 说明 |
|------|------|
| 可推断则自动补全 | 根据上下文推断时间 |
| 无法推断则标记待确认 | time_uncertain 标签 |
| 使用录入时间作为弱代理时需标注 | 明确标注时间来源 |

---

## 8. 内容异常

### 8.1 定义
输入内容无法稳定切分、解析或理解。

### 8.2 典型场景

| 场景 | 说明 |
|------|------|
| 长文本混杂多事件 | 一条文本包含多个独立事件 |
| 学习内容与生活内容严重混杂 | 难以分离学习相关信息 |
| 表述歧义大 | 同一表述有多种理解 |
| 内容过短无法判断 | 信息量不足以做判断 |
| OCR 文本碎裂严重 | 识别结果无法连贯理解 |

### 8.3 处理策略

| 策略 | 说明 |
|------|------|
| 尝试自动切分多事件 | 按语义切分 |
| 切分失败则整条待复核 | content_noisy 标签 |
| 保留原始文本供人工查看 | 不丢弃原始内容 |

---

## 9. 文件与 OCR 异常

### 9.1 定义
上传文件本身或 OCR 结果质量不足，影响后续处理。

### 9.2 典型场景

| 场景 | 说明 |
|------|------|
| 图片模糊 | 分辨率过低或对焦失败 |
| 多页错序 | 多页文档顺序混乱 |
| 页面倾斜严重 | 拍摄角度导致倾斜 |
| 遮挡严重 | 关键内容被遮挡 |
| PDF 损坏 | 文件无法打开 |
| OCR 输出乱码占比高 | 识别错误过多 |

### 9.3 处理策略

| 策略 | 说明 |
|------|------|
| 自动检测图片质量 | 模糊度、倾斜度检测 |
| 质量差则标记 ocr_bad | 不强行识别 |
| 提示用户重新上传 | 给出明确提示 |

---

## 10. 结构异常

### 10.1 定义
批量导入或结构化数据格式不符合要求。

### 10.2 典型场景

| 场景 | 说明 |
|------|------|
| 列头缺失 | 缺少必填列名 |
| 字段顺序错乱 | 列顺序与模板不符 |
| 数值字段混入文本 | 分数列中出现文字 |
| 必填列大面积空缺 | 关键数据缺失 |
| 同一表中口径不统一 | 同一列多种格式 |

### 10.3 处理策略

| 策略 | 说明 |
|------|------|
| 自动检测列映射 | 尝试匹配列名 |
| 映射失败则整表待复核 | structure_error 标签 |
| 提供错误行明细 | 便于人工定位 |

---

## 11. 来源异常

### 11.1 定义
输入来源不明、来源不可信或来源与声明不符。

### 11.2 典型场景

| 场景 | 说明 |
|------|------|
| 无法确定是谁上传的 | 用户身份验证失败 |
| 文件被转发多次，原始来源不明 | 来源链断裂 |
| 批量表来源不清 | 无来源学校标识 |
| 第三方整理数据未经确认 | 非官方数据来源 |

### 11.3 处理策略

| 策略 | 说明 |
|------|------|
| 验证用户身份 | 登录态校验 |
| 来源不明则标记 source_unverified | 降低可信度 |
| 关键数据需官方确认 | 学校或老师确认 |

---

## 12. 异常分级建议

建议将异常分为三级：

### 12.1 P0：阻断型异常

必须阻断进入主链路。

| 异常 | 说明 |
|------|------|
| 学生对象不明 | 无法确认归属学生 |
| 时间冲突严重 | 多个矛盾时间无法解析 |
| OCR 完全不可读 | 识别失败率过高 |
| 批量导入关键字段缺失 | 必填字段大面积缺失 |

**处理方式**：必须人工确认后才能进入主链路。

### 12.2 P1：高优先级异常

可暂存，需尽快人工确认。

| 异常 | 说明 |
|------|------|
| 学科识别不稳定 | 学科字段置信度低 |
| 题目与知识点映射歧义大 | 映射关系不确定 |
| 同一输入疑似多学生混杂 | 可能涉及多个学生 |

**处理方式**：进入异常池，按优先级排序处理。

### 12.3 P2：一般异常

可先降级入库或附注进入后续处理。

| 异常 | 说明 |
|------|------|
| 文案噪声较多 | 内容含较多无关信息 |
| 缺少部分辅助字段 | 非必填字段缺失 |
| OCR 少量局部模糊 | 局部识别质量一般 |

**处理方式**：可带标签进入主链路，后续可修正。

## 13. 兜底处理结果类型

异常输入建议最终输出以下结果之一：

| 结果类型 | 编码 | 说明 |
|----------|------|------|
| 已处理并进入主链路 | resolved_and_ingested | 异常已解决，可正常进入事件生成层 |
| 已处理但带限制标签进入 | resolved_with_limits | 已处理但需带质量标签进入 |
| 等待更多信息 | pending_more_info | 需等待用户补充信息 |
| 拒绝进入主链路 | rejected | 不符合进入主链路要求 |
| 仅归档参考 | archived_for_reference | 仅保存参考，不进入计算链路 |

## 15. 人工兜底流程建议

```text
识别异常输入
    ↓
打异常标签与优先级
    ↓
进入异常池
    ↓
按类型路由到对应处理角色
    ↓
人工确认/修正/补充信息
    ↓
输出处理结果
    ↓
决定入主链路/降级入库/拒绝
    ↓
记录处理痕迹
    ↓
回流规则优化
```

### 15.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 识别异常输入 | 清洗层检测异常 | exception_id |
| 打异常标签与优先级 | 标注异常类型和优先级 | exception_type, priority_level |
| 进入异常池 | 存入异常队列 | 异常池记录 |
| 按类型路由 | 分配到对应处理角色 | assigned_to |
| 人工确认/修正/补充 | 人工处理异常 | resolution_notes |
| 输出处理结果 | 确定处理结果类型 | resolution_result |
| 决定入库方式 | 决定进入主链路或归档 | 入库决策 |
| 记录处理痕迹 | 记录处理全过程 | 完整处理日志 |
| 回流规则优化 | 推动规则优化 | 规则优化建议 |

## 16. 异常池设计建议

建议建立统一异常池，至少包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| exception_id | string | 异常记录唯一标识 |
| raw_input_id | string | 原始输入引用 |
| exception_type | string | 异常类型 |
| priority_level | string | 优先级 (P0/P1/P2) |
| detected_at | datetime | 异常发现时间 |
| assigned_to | string/null | 分配处理人 |
| current_status | string | 当前状态 |
| resolution_result | string/null | 处理结果 |
| resolution_notes | string/null | 处理备注 |
| resolved_at | datetime/null | 处理完成时间 |

## 17. 人工确认最小动作建议

为控制人工成本，建议人工处理以"最小必要确认"为原则：

### 17.1 确认优先级

| 优先级 | 确认项 | 说明 |
|--------|--------|------|
| 1 | 对象确认 | 先确认学生/老师身份 |
| 2 | 时间确认 | 再确认事件时间 |
| 3 | 事件类型确认 | 再确认事件类型 |
| 4 | 知识映射确认 | 最后做知识点等细处理 |

### 17.2 原则

不要一上来把每条异常都当成论文答辩。

## 18. 与输入清洗规则的关系

异常输入流程建立在 `INGEST-009 输入数据清洗规则` 基础上。

关系如下：

```text
原始输入
    ↓
清洗与打标
    ↓
无法稳定处理
    ↓
异常池
    ↓
人工兜底
    ↓
回流主链路或归档
```

---

## 19. 与学习事件生成标准的关系

人工兜底的目标之一，是把足够高价值的异常输入转换为：

- 可进入事件生成层的中间结构
- 或带限制标签的半结构化对象

不应让异常输入绕过事件生成标准直接写学生状态。

---

## 20. 与运营复盘机制的关系

异常输入及其处理结果应进入 `OPS-007 运营指标与复盘机制`，至少统计：

| 统计项 | 说明 |
|--------|------|
| 异常输入量 | 周期内异常输入总数 |
| 异常类型分布 | 各类型异常占比 |
| 人工处理时长 | 平均处理时间 |
| 处理成功率 | 成功解决比例 |
| 高频异常原因 | TOP 异常原因分析 |
| 可自动化优化项 | 可转为自动处理的项 |

---

## 21. 质量要求

### 21.1 可追溯

| 要求 | 说明 |
|------|------|
| 每条异常应知道原始输入 | raw_input_id 必须保存 |
| 每条异常应知道异常原因 | exception_type 必须明确 |
| 每条异常应知道处理人和结果 | assigned_to 和 resolution_result 必须记录 |

### 21.2 可分流

| 要求 | 说明 |
|------|------|
| 异常类型应能清晰路由 | 不混成一个"大问题池" |
| 优先级应能指导处理顺序 | P0 优先处理 |

### 21.3 可统计

| 要求 | 说明 |
|------|------|
| 异常不只是单次处理对象 | 也是系统优化线索 |
| 应支持按类型/时间/处理人统计 | 支持多维度统计 |

### 21.4 可回流

| 要求 | 说明 |
|------|------|
| 高频异常应进入规则或模型优化计划 | 推动系统改进 |
| 处理经验应沉淀为规则或案例 | 知识沉淀 |

### 21.5 可控成本

| 要求 | 说明 |
|------|------|
| 人工兜底必须控制在合理范围 | 不能成为隐性主流程 |
| P0/P1 异常占比应逐步下降 | 自动化能力提升 |

## 26. 风险与边界

### 26.1 不能忽视高风险输入拦截

| 风险 | 说明 |
|------|------|
| P0 异常进入主链路 | 污染学生状态数据 |

### 26.2 不能忽视异常分级

| 风险 | 说明 |
|------|------|
| 所有异常同等处理 | 人工资源浪费，高优先级异常处理延迟 |

### 26.3 不能忽视人工边界

| 风险 | 说明 |
|------|------|
| 人工兜底变成主流程 | 系统无法规模化 |

### 26.4 不能忽视处理留痕

| 风险 | 说明 |
|------|------|
| 处理过程无记录 | 无法追溯和复盘 |

### 26.5 不能忽视结果回流

| 风险 | 说明 |
|------|------|
| 异常处理不推动规则优化 | 同类异常反复出现 |

## 27. 结论

异常输入与人工兜底流程的核心价值，不是承认系统不完美，而是让系统在不完美输入面前仍然保持可控、可追溯、可改进。

它必须坚持：

| 原则 | 说明 |
|------|------|
| **高风险不硬进** | P0 异常必须阻断 |
| **异常要分级** | P0/P1/P2 分级处理 |
| **人工要有边界** | 不能成为隐性主流程 |
| **处理要留痕** | 全链路可追溯 |
| **结果要回流** | 推动规则优化 |

否则系统前面辛辛苦苦建立的规则和状态链路，会被一堆脏输入慢慢挖空。
