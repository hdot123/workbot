## 1. 文档目的

本文档用于定义教育孪生项目中的"学习事件"标准，明确学习事件的定义、分类、结构、生成流程、质量要求与下游使用方式。

本文件重点回答以下问题：

1. 什么叫学习事件
2. 哪些输入可以生成学习事件
3. 学习事件的标准字段是什么
4. 学习事件如何从原始输入中被提取、清洗和结构化
5. 学习事件如何驱动 StudentTwinAgent、图谱记忆与观察输出

---

## 2. 学习事件定义

学习事件，是指与学生学习过程、学习结果、学习反馈、学习环境变化相关，并且可以被系统记录、解释、追踪和用于状态更新的最小业务单元。

学习事件不是原始输入本身，而是系统对原始输入完成清洗、识别、结构化后的结果。

例如：

- 家长上传一张作业照片，不是学习事件
- 系统识别出"2024-04-20，学生完成了物理作业《受力分析练习 1》，完成度 90%，涉及知识点 PHY_KP_002"，这才是学习事件

---

## 3. 学习事件的作用

### 3.1 作为学生状态更新输入
学习事件是 StudentTwinAgent 更新的直接驱动力。

### 3.2 作为图谱记忆写入来源
学习事件可沉淀为事实节点、事件节点和关系边。

### 3.3 作为观察与报告依据
家长周报、老师看板、学校统计都应尽量基于事件生成解释，而不是凭空编故事。

### 3.4 作为推演与干预基础
后台推演需要事件序列，而不是只有几个孤零零的分数。

---

## 4. 学习事件的生成来源

学习事件可来自以下输入源：

| 来源类型 | 典型输入 | 是否可直接成事件 |
|----------|----------|------------------|
| 家长文本输入 | "今天物理作业做完了，错了两题" | 否，需解析 |
| 老师文本输入 | "受力分析掌握不稳" | 否，需结构化 |
| 学校扫描件 | 试卷、作业、练习册 | 否，需 OCR/解析 |
| 机构批量录入 | 成绩表、任务表 | 部分可直接映射 |
| 系统任务日志 | 打卡、任务完成 | 可较直接生成 |
| 教师评语/反馈 | 课堂评价、周评语 | 否，需抽取 |
| 外部规则更新 | 教学计划调整 | 不是学习事件，属于规则事件 |

## 5. 学习事件分类体系

建议学习事件分为八类：

### 5.1 作业事件
描述作业布置、提交、完成、延迟、纠错等。

### 5.2 考试事件
描述考试参加、成绩、排名、知识点表现等。

### 5.3 错题事件
描述错题识别、重复错误、纠错情况等。

### 5.4 学习任务事件
描述计划任务、打卡任务、复习任务、执行情况等。

### 5.5 教师反馈事件
描述老师评语、课堂反馈、阶段评价等。

### 5.6 家长反馈事件
描述家长观察、补充说明、家庭配合情况等。

### 5.7 进度事件
描述教材推进、章节完成、知识点接触等。

### 5.8 干预事件
描述已执行的提醒、辅导、复习安排、家校配合动作等。

## 6. 学习事件标准结构

每个学习事件建议包含以下九类字段：

1. 标识字段
2. 主体字段
3. 时间字段
4. 来源字段
5. 事件类型字段
6. 内容结构字段
7. 映射字段
8. 质量字段
9. 审计字段

---

## 7. 标识字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| event_id | string | 学习事件唯一标识 |
| raw_input_id | string/null | 原始输入引用 |
| event_batch_id | string/null | 批处理批次标识 |

---

## 8. 主体字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| tenant_id | string | 租户标识 |
| student_id | string | 学生标识 |
| class_id | string/null | 班级标识 |
| school_id | string/null | 学校标识 |
| operator_id | string/null | 输入/操作人标识 |

---

## 9. 时间字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| event_time | datetime | 事件实际发生时间 |
| event_time_precision | enum | 时间精度：日/时/分/不确定 |
| ingested_at | datetime | 系统入库时间 |
| processed_at | datetime/null | 完成结构化时间 |

说明：  
事件发生时间和入库时间必须分开。别把"今天上传了上周试卷"误写成"今天参加了上周考试"，那会把时间线搅成粥。

## 12. 内容结构字段

### 12.1 作用
用于表达事件本身的业务内容。

### 12.2 通用字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| title | string/null | 事件标题 |
| summary | string | 事件摘要 |
| detail_payload | object | 事件详情结构 |
| metric_payload | object/null | 指标结构 |

### 12.3 示例：作业事件

```json
{
  "title": "物理作业提交",
  "summary": "完成受力分析练习 1，错 2 题",
  "detail_payload": {
    "homework_name": "受力分析练习 1",
    "completion_status": "completed",
    "question_count": 10,
    "wrong_count": 2,
    "submit_status": "on_time"
  },
  "metric_payload": {
    "completion_rate": 1.0,
    "accuracy_rate": 0.8
  }
}
```

### 12.4 示例：考试事件

```json
{
  "title": "物理期中考试成绩录入",
  "summary": "期中考试物理 76 分",
  "detail_payload": {
    "exam_name": "高一物理期中考试",
    "score": 76,
    "full_score": 100,
    "class_rank": 18
  },
  "metric_payload": {
    "score_rate": 0.76
  }
}
```

## 13. 映射字段

### 13.1 作用

将事件映射到知识底座和学生模型可用的结构。

### 13.2 字段定义

| 字段名 | 类型 | 说明 |
|--------|------|------|
| subject | string/null | 学科 |
| chapter_refs | array[string] | 章节引用 |
| knowledge_refs | array[string] | 知识点引用 |
| ability_refs | array[string] | 能力点引用 |
| anchor_refs | array[string] | 教材锚点引用 |
| curriculum_version_id | string/null | 教材版本引用 |

### 13.3 规则

- 映射字段允许为空，但必须标记原因
- 映射失败事件应进入待处理池
- 高价值事件应优先补齐映射

## 16. 学习事件生成流程

### 16.1 标准流程

```text
原始输入进入
    ↓
输入鉴权与主体识别
    ↓
OCR / 文档理解 / 文本解析
    ↓
事件抽取
    ↓
字段清洗
    ↓
事件类型判定
    ↓
知识底座映射
    ↓
质量评分
    ↓
生成标准学习事件
    ↓
写入事件库
    ↓
推送 StudentTwinAgent 更新
    ↓
同步图谱记忆系统
```

## 17. 各来源的生成规则

### 17.1 家长文本输入

输入示例：

> 今天物理作业写完了，受力分析那页错了两题。

处理结果：

- 识别学生主体
- 提取学科：物理
- 提取事件类型：作业事件
- 提取完成状态：已完成
- 提取错误数量：2
- 尝试映射知识点：受力分析

### 17.2 扫描件/OCR 输入

输入示例：

- 练习册照片
- 试卷 PDF

处理重点：

- OCR 识别
- 题目切分
- 分数识别
- 错误标记识别
- 知识点映射
- 低置信内容进入人工复核

### 17.3 教师评语输入

输入示例：

> 受力分析掌握不稳，课堂参与正常。

处理结果：

- 事件类型：teacher_feedback
- 子类型：knowledge_feedback
- 提取状态信号：掌握不稳
- 提取行为信号：课堂参与正常
- 生成反馈事件，不直接改写事实状态

## 18. 学习事件与 StudentTwinAgent 的关系

### 18.1 直接驱动字段更新

以下事件通常会直接更新 Agent：

- 考试事件
- 作业完成事件
- 错题事件
- 进度事件

### 18.2 间接影响状态推断

以下事件通常用于辅助推断：

- 家长反馈事件
- 教师评语事件
- 干预事件

### 18.3 不应直接覆盖事实

教师说"掌握不稳"应生成反馈事件，再影响状态判断，不应直接把某知识点事实改成"差"。系统要学会尊重层次，不然就会一脚把事实层踹翻。

---

## 19. 学习事件与图谱记忆的关系

学习事件写入图谱时，建议至少形成：

- Student -> has_event -> LearningEvent
- LearningEvent -> refers_to -> KnowledgePoint
- LearningEvent -> occurred_at -> TimeNode
- LearningEvent -> sourced_from -> Source
- LearningEvent -> implies -> RiskSignal（如适用）

---

## 20. 异常与人工兜底规则

### 20.1 需要人工复核的典型情况

- 学生主体不明确
- 时间不明确
- 学科无法识别
- 知识点映射冲突
- OCR 置信度过低
- 原始文本含强歧义

### 20.2 人工兜底处理动作

- 补主体
- 补时间
- 修正事件类型
- 修正知识点映射
- 标记无效事件
- 回写修正日志

## 21. MVP 最小学习事件标准

MVP 阶段建议优先支持以下事件：

### 21.1 必须支持

- 作业提交事件
- 考试成绩事件
- 错题识别事件
- 教师反馈事件
- 家长反馈事件
- 教材进度事件

### 21.2 可以后置

- 复杂课堂行为事件
- 细粒度情绪事件
- 多人协作学习事件
- 自动推导型深层认知事件

## 24. 结论

学习事件标准是整个教育孪生系统的输入地基。

如果没有统一事件标准：

- 输入无法合并
- 状态无法稳定更新
- 图谱无法持续沉淀
- 报告无法追溯依据
- 推演只能建立在空气上

因此，学习事件设计必须坚持：

- 先统一结构
- 再扩展来源
- 先保证可追溯
- 再提升自动化
- 先服务学生状态更新
- 再服务更复杂的智能能力
