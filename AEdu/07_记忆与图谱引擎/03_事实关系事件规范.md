# 事实关系事件规范

> 文档编号：GRAPH-003  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：记忆与图谱引擎负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中图谱与记忆系统的三类核心对象规范：事实（Fact）、关系（Relation）与事件（Event），明确其边界、字段结构、写入原则与使用场景。

本文档重点回答以下问题：

1. 什么是事实、关系、事件
2. 三者之间如何区分
3. 图谱和记忆系统为什么必须严格区分这三类对象
4. 它们如何共同支撑 StudentTwinAgent、观察层、推演层和审计层
5. 如何避免把系统记忆写成一锅"看起来都相关"的稀泥

---

## 2. 核心定义

### 2.1 事实（Fact）

事实是指在某一时间范围内被确认成立、可相对稳定引用的对象性或状态性信息。

**特征**：
- 当前已确认
- 来源清楚
- 可作为系统当前判断依据
- 在一定时间窗口内可相对稳定使用

**例如**：
- 学生当前年级为高一
- 学生绑定教材版本为某版本
- 学生已参加某次考试
- 某知识点当前状态为"掌握不稳定"（若该状态已被系统确认并入当前态）

### 2.2 关系（Relation）

关系是指两个或多个对象之间相对稳定的结构连接。

**特征**：
- 结构性连接
- 相对稳定
- 不强调时间性
- 表达对象间的语义关联

**例如**：
- 学生 belongs_to 班级
- 知识点 depends_on 知识点
- 知识点 supports 能力点
- 学生 has_risk_signal 某风险对象

### 2.3 事件（Event）

事件是指在某个时间点或时间窗口发生的、带有触发性和过程性的记录。

**特征**：
- 时间点/时间窗口明确
- 带有触发动作
- 有输入来源
- 有过程记录
- 有变化痕迹

**例如**：
- 某次作业提交
- 某次考试完成
- 某次老师反馈
- 某次家长上传扫描件
- 某次状态修正
- 某次干预动作执行

## 3. 建设目标

### 3.1 清晰划分记忆对象边界

避免系统把所有东西都写成"节点 + 文本描述"。

### 3.2 支撑 StudentTwinAgent 当前态与长期态协同

- 事实服务当前态
- 事件服务变化轨迹
- 关系服务上下文结构

### 3.3 支撑解释与审计

系统必须能解释：
- "现在为什么这样"（事实 + 关系）
- "它是怎么变成这样的"（事件链）

### 3.4 支撑 GraphRAG

检索时需要知道：
- 哪些内容是稳定上下文（事实/关系）
- 哪些是时序线索（事件）

### 3.5 支撑推演层

推演层依赖事实、关系和事件共同构成状态输入和变化路径。

---

## 4. 设计原则

### 4.1 事实、关系、事件分离

三类对象边界不清，系统后面就会进入解释混乱状态。

### 4.2 事件优先记录变化

变化先写事件，再决定是否更新事实。

### 4.3 关系优先表达结构

不要把结构性关系藏在自然语言备注里。

### 4.4 事实强调当前有效性

事实不是永恒真理，要允许版本和时效。

### 4.5 全链路可追溯

事实和关系最好能追溯到事件来源。

---

## 5. 为什么必须区分三类对象

如果不区分，系统会出现以下问题：

| 问题 | 后果 |
|------|------|
| 同一个内容既像状态又像日志 | 无法判断是当前状态还是历史记录 |
| 无法区分当前有效信息和历史过程 | 检索时把旧事件当当前事实 |
| 图谱中无法做稳定结构查询 | 关系被事件污染 |
| GraphRAG 上下文容易把旧事件当当前事实 | 检索结果失真 |
| 推演层拿不到稳定输入 | 只能吃一堆语义碎片 |

**简单说**，不区分三类对象，系统就会失去"现在、过去、结构"这三个维度。

## 6. 事实规范

### 6.1 事实定义边界

事实应满足至少以下条件之一：

| 条件 | 说明 |
|------|------|
| 当前已确认 | 已被系统确认的状态 |
| 来源清楚 | 可追溯到具体事件或输入 |
| 可作为系统当前判断依据 | 可用于状态表达 |
| 在一定时间窗口内可相对稳定使用 | 不是瞬间变化的 |

### 6.2 事实类型建议

| 类型 | 说明 | 示例 |
|------|------|------|
| 基础身份事实 | 学生基础身份信息 | 年级、班级、学校 |
| 绑定事实 | 学生与外部对象绑定 | 教材版本、地区 |
| 当前状态事实 | 当前确认的状态 | 风险等级、知识点状态 |
| 规则适配事实 | 规则适配信息 | 选科组合、考试类型 |
| 聚合确认事实 | 经聚合确认的事实 | 学期总结状态 |

### 6.3 事实典型示例

| 示例 | 说明 |
|------|------|
| Student.current_grade = 高一 | 当前年级事实 |
| Student.bound_curriculum_version = PHY_PEP_G1_V1 | 教材版本绑定事实 |
| Student.current_risk_level = medium | 当前风险等级事实 |
| Student.current_focus_knowledge = KP_002 | 当前重点关注知识点 |

## 7. 事实字段建议

事实对象建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| fact_id | string | 是 | 事实唯一标识 |
| subject_id | string | 是 | 主体对象 ID（如学生 ID） |
| fact_type | string | 是 | 事实类型 |
| fact_key | string | 是 | 事实键（如 current_risk_level） |
| fact_value | string/object | 是 | 事实值 |
| confidence | float | 否 | 置信度 (0-1) |
| valid_from | datetime | 是 | 生效时间 |
| valid_to | datetime/null | 否 | 失效时间（null 表示长期有效） |
| source_event_refs | array[string] | 否 | 来源事件引用 |
| status | string | 是 | 状态 (active/inactive/superseded) |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

### 7.1 字段说明

**fact_type** 建议值：
- identity（身份事实）
- binding（绑定事实）
- current_state（当前状态）
- rule_adaptation（规则适配）
- aggregated（聚合确认）

**status** 建议值：
- active（当前有效）
- inactive（已失效）
- superseded（被新版本替代）

## 8. 关系规范

### 8.1 关系定义边界

关系用于表达对象之间相对稳定的连接，而不是一次性的时序动作。

| 特征 | 说明 |
|------|------|
| 结构性 | 表达对象间的结构连接 |
| 稳定性 | 相对稳定，不频繁变化 |
| 非时序性 | 不强调时间点 |

### 8.2 关系类型建议

| 关系类型 | 编码 | 含义 | 示例 |
|----------|------|------|------|
| 隶属关系 | belongs_to | 属于某组织/群体 | 学生→班级 |
| 依赖关系 | depends_on | 前置依赖 | 知识点→知识点 |
| 支撑关系 | supports | 支撑某能力/目标 | 知识点→能力点 |
| 关联关系 | related_to | 一般性关联 | 实验→知识点 |
| 风险关系 | has_risk_signal | 具有某风险信号 | 学生→风险信号 |
| 引用关系 | refers_to | 引用某对象 | 事件→知识点 |
| 使用关系 | uses | 使用某资源 | 学生→教材版本 |
| 链接关系 | linked_with | 与其他对象链接 | 事实→事件 |

### 8.3 关系典型示例

| 示例 | 说明 |
|------|------|
| Student belongs_to Class | 学生属于班级 |
| KnowledgePoint depends_on KnowledgePoint | 知识点依赖前置知识点 |
| KnowledgePoint supports AbilityPoint | 知识点支撑能力点 |
| Experiment refers_to Anchor | 实验引用教材锚点 |

---

## 9. 关系字段建议

关系对象建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| relation_id | string | 是 | 关系唯一标识 |
| source_id | string | 是 | 起点对象 ID |
| source_type | string | 是 | 起点对象类型 |
| relation_type | string | 是 | 关系类型 |
| target_id | string | 是 | 终点对象 ID |
| target_type | string | 是 | 终点对象类型 |
| strength | string | 否 | 关系强度 (strong/medium/weak) |
| description | string/null | 否 | 关系说明 |
| valid_from | datetime | 否 | 生效时间 |
| valid_to | datetime/null | 否 | 失效时间 |
| source_ref | string/null | 否 | 来源引用 |
| status | string | 是 | 状态 (active/inactive/deprecated) |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

### 9.1 字段说明

**relation_type** 必须从预定义关系类型中选择，不允许随意创建新类型。

**strength** 建议值：
- strong（强关系）
- medium（中等关系）
- weak（弱关系）

**status** 建议值：
- active（当前有效）
- inactive（已失效）
- deprecated（已废弃）

## 10. 事件规范

### 10.1 事件定义边界

事件是发生过的事情，强调：

| 特征 | 说明 |
|------|------|
| 时间点/时间窗口 | 有明确的发生时间 |
| 触发动作 | 由某个动作触发 |
| 输入来源 | 有明确的输入来源 |
| 过程记录 | 记录发生了什么 |
| 变化痕迹 | 可能导致状态变化 |

### 10.2 事件类型建议

| 事件类型 | 编码 | 说明 |
|----------|------|------|
| 作业提交 | homework_submitted | 学生完成并提交作业 |
| 考试完成 | exam_completed | 学生参加考试 |
| 反馈接收 | feedback_received | 收到老师/家长反馈 |
| 文件上传 | file_uploaded | 上传扫描件/图片 |
| 风险检测 | risk_detected | 系统检测到风险信号 |
| 状态更新 | state_updated | 学生状态更新 |
| 干预执行 | intervention_executed | 执行干预动作 |
| 报告生成 | report_generated | 生成周报/月报 |

### 10.3 事件典型示例

| 示例 | 说明 |
|------|------|
| 家长上传作业图片 | 文件上传事件 |
| 老师提交课堂反馈 | 反馈接收事件 |
| 学生完成某次考试 | 考试完成事件 |
| 系统完成一次状态重算 | 状态更新事件 |
| 系统生成周报 | 报告生成事件 |

---

## 11. 事件字段建议

事件对象建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| event_id | string | 是 | 事件唯一标识 |
| event_type | string | 是 | 事件类型 |
| actor_id | string | 否 | 行为主体/输入主体 ID |
| actor_type | string | 否 | 行为主体类型 |
| target_id | string | 是 | 主要影响对象 ID |
| target_type | string | 是 | 主要影响对象类型 |
| event_time | datetime | 是 | 事件发生时间 |
| event_source | string | 是 | 来源类型 (wechat/scan/batch 等) |
| raw_input_ref | string/null | 否 | 原始输入引用 |
| payload | object | 是 | 结构化事件内容 |
| derived_fact_refs | array[string] | 否 | 产生的事实引用 |
| derived_relation_refs | array[string] | 否 | 产生的关系引用 |
| status | string | 是 | 状态 (active/invalid/retracted) |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

### 11.1 字段说明

**event_source** 建议值：
- wechat（微信输入）
- dingtalk（钉钉输入）
- scan（扫描件）
- batch_import（批量导入）
- manual_entry（手工录入）
- system_generated（系统生成）

**status** 建议值：
- active（有效）
- invalid（无效）
- retracted（已撤销）

**payload** 是事件的核心内容，应包含事件类型相关的结构化字段。

## 12. 三类对象之间的关系

建议遵循以下主链路：

```text
原始输入
    ↓
学习事件 Event
    ↓
更新事实 Fact
    ↓
建立/更新关系 Relation
    ↓
进入 StudentTwinAgent 当前态与图谱上下文
```

**说明**：

| 对象 | 作用 |
|------|------|
| 事件 | 变化入口 |
| 事实 | 当前可引用结果 |
| 关系 | 结构骨架 |

### 12.1 事件驱动事实更新

事件发生后，可能产生新的事实或更新现有事实。

### 12.2 事件可能建立关系

某些事件可能建立新的关系连接。

### 12.3 事实和关系可追溯事件

每个事实和关系应能追溯到产生的事件来源。

## 13. 写入原则

### 13.1 先事件，后事实

系统应优先记录"发生了什么"，再更新"现在是什么"。

| 原则 | 说明 |
|------|------|
| 事件是入口 | 所有变化先写事件 |
| 事实是结果 | 事实由事件推导产生 |
| 不跳过事件直接改事实 | 保证可追溯性 |

### 13.2 关系按结构写入

只有相对稳定的结构性联系才进入关系层。

| 原则 | 说明 |
|------|------|
| 不是一次性动作 | 关系表达稳定连接 |
| 不藏在自然语言里 | 关系要结构化表达 |
| 有明确的起点和终点 | 关系方向清晰 |

### 13.3 不把自然语言总结直接当事实

总结文本可以是说明块，但事实应结构化。

| 原则 | 说明 |
|------|------|
| 事实要有 fact_key 和 fact_value | 结构化表达 |
| 文本总结可作为 description | 补充说明 |
| 不混用 | 事实和文本分开存储 |

### 13.4 同一内容不要重复写成三种对象

要分清它当前属于事件、事实还是关系。

| 内容 | 应属于 | 不应属于 |
|------|--------|----------|
| 学生当前年级 | 事实 | 事件/关系 |
| 学生属于班级 | 关系 | 事件/事实 |
| 学生提交作业 | 事件 | 事实/关系 |

## 14. 失效与替换规则

### 14.1 事实失效

当新状态覆盖旧状态时：

| 动作 | 说明 |
|------|------|
| 旧 fact 不删除 | 保留历史记录 |
| 标记为 superseded 或 inactive | 标记失效状态 |
| 新 fact 生效 | 新事实 valid_from 为当前时间 |
| 保留来源事件引用 | 保持可追溯性 |

### 14.2 关系失效

当结构关系不再成立时：

| 动作 | 说明 |
|------|------|
| 结束 valid_to | 设置失效时间 |
| 保留历史痕迹 | 不删除历史记录 |
| 标记为 inactive 或 deprecated | 标记状态 |

### 14.3 事件撤销

若原始输入被判定无效：

| 动作 | 说明 |
|------|------|
| 事件标记 invalid/retracted | 标记事件状态 |
| 相关事实应可回溯重算 | 事实可能需要更新 |
| 相关关系应可回溯调整 | 关系可能需要调整 |

## 15. 与 StudentTwinAgent 的关系

### 15.1 Fact → StudentTwinAgent

事实为 StudentTwinAgent 提供当前态基础。

| 事实类型 | 支撑的学生态字段 |
|----------|------------------|
| 基础身份事实 | 学生基础信息 |
| 绑定事实 | 教材版本、地区绑定 |
| 当前状态事实 | 风险等级、知识点状态 |

### 15.2 Relation → StudentTwinAgent

关系为 StudentTwinAgent 提供结构上下文。

| 关系类型 | 支撑的学生态字段 |
|----------|------------------|
| belongs_to | 班级、学校归属 |
| has_risk_signal | 风险信号关联 |
| depends_on | 知识依赖链路 |

### 15.3 Event → StudentTwinAgent

事件为 StudentTwinAgent 提供更新依据和时序轨迹。

| 事件类型 | 支撑的学生态更新 |
|----------|------------------|
| homework_submitted | 作业完成状态更新 |
| exam_completed | 考试记录更新 |
| state_updated | 状态变化轨迹 |

**StudentTwinAgent 不是图谱本体，但它必须建立在这三类对象的有序协同之上。**

---

## 16. 与 GraphRAG 的关系

GraphRAG 检索时建议采用以下优先级：

```text
1. 优先取当前有效事实
   ↓
2. 再取关键关系网络
   ↓
3. 再按时间窗口补充相关事件
```

### 16.1 为什么这个顺序

| 层级 | 作用 | 原因 |
|------|------|------|
| 事实 | 当前状态 | 回答"现在是什么" |
| 关系 | 结构上下文 | 回答"与什么相关" |
| 事件 | 变化线索 | 回答"怎么变成这样" |

### 16.2 避免的问题

| 问题 | 后果 |
|------|------|
| 只拿历史旧事件 | 把过去当现在 |
| 忽视当前事实 | 状态判断失真 |
| 把结构上下文丢掉 | 检索结果缺乏关联 |

---

## 17. 与审计和解释层的关系

### 17.1 解释层

解释层回答"为什么这样"，通常需要：

| 对象 | 作用 |
|------|------|
| 当前事实 | 当前状态是什么 |
| 支撑关系 | 与哪些对象相关 |
| 关键事件链 | 关键变化节点 |

### 17.2 审计层

审计层回答"是怎么变过来的"，则更偏：

| 对象 | 作用 |
|------|------|
| 事件序列 | 变化发生的时间线 |
| 事实替换链 | 状态变化的历史版本 |
| 关系变化记录 | 结构变化的历史 |

## 18. 质量要求

### 18.1 边界清晰

| 要求 | 说明 |
|------|------|
| 事实、关系、事件不能混写 | 每类对象有明确归属 |
| 类型字段必须准确 | event_type/fact_type/relation_type 准确 |
| 不跨类型复用 ID | 每类对象有独立 ID 空间 |

### 18.2 来源可追溯

| 要求 | 说明 |
|------|------|
| 事实应能追到事件 | source_event_refs |
| 关系应能追到来源 | source_ref |
| 事件应能追到原始输入 | raw_input_ref |

### 18.3 时效明确

| 要求 | 说明 |
|------|------|
| 当前态和历史态必须能区分 | valid_from/valid_to |
| 事件时间必须准确 | event_time |
| 状态变更有时间记录 | created_at/updated_at |

### 18.4 结构可查询

| 要求 | 说明 |
|------|------|
| 关系和事实必须支持系统查询 | 字段结构化 |
| 支持按类型/时间/对象检索 | 索引设计 |
| 支持批量操作 | 批量查询/更新 |

### 18.5 历史可保留

| 要求 | 说明 |
|------|------|
| 不能为了"当前简洁"把历史变化全抹掉 | 保留历史版本 |
| 失效对象标记状态而非删除 | status 字段 |
| 支持历史状态查询 | 时间范围查询 |

## 19. 常见错误

### 19.1 把事件写成事实

| 错误示例 | 正确做法 |
|----------|----------|
| "提交了一次作业"写成事实 | 应写成事件 (homework_submitted) |
| 事件没有 event_time | 事件必须有时间 |

### 19.2 把事实写成备注文本

| 错误示例 | 正确做法 |
|----------|----------|
| "学生当前高一"藏在说明文本里 | 应写成结构化事实 (fact_key=current_grade) |
| 事实没有 fact_key/fact_value | 事实必须结构化 |

### 19.3 把关系写成事件

| 错误示例 | 正确做法 |
|----------|----------|
| "belongs_to 班级"写成事件 | 应写成关系 (belongs_to) |
| 一次性动作写成关系 | 应写成事件 |

### 19.4 一条原始输入直接改多个事实但不留事件

| 错误 | 后果 |
|------|------|
| 没有事件记录 | 后面无法解释变化从哪来 |
| 事实更新无来源 | 无法追溯和审计 |

### 19.5 关系类型随意创建

| 错误 | 后果 |
|------|------|
| 关系类型不统一 | 查询困难 |
| 同一关系多种命名 | 结构混乱 |

## 24. 风险与边界

### 24.1 不能忽视边界清晰

| 风险 | 说明 |
|------|------|
| 事实、关系、事件混写 | 系统解释混乱，检索失真 |

### 24.2 不能忽视来源追溯

| 风险 | 说明 |
|------|------|
| 事实和关系无事件来源 | 无法追溯变化原因 |

### 24.3 不能忽视时效管理

| 风险 | 说明 |
|------|------|
| 不标记 valid_from/valid_to | 无法区分当前态和历史态 |

### 24.4 不能忽视结构查询支持

| 风险 | 说明 |
|------|------|
| 对象不支持结构化查询 | 图谱检索失效 |

### 24.5 不能忽视历史保留

| 风险 | 说明 |
|------|------|
| 为了"当前简洁"删除历史 | 无法审计和回溯 |

## 25. 结论

事实、关系、事件规范的核心意义，是让图谱和记忆系统真正具备"现在、结构、变化"三个维度，而不是把所有信息都堆成一层难以区分的记忆泥浆。

它必须坚持：

| 原则 | 说明 |
|------|------|
| **事件记录变化** | 所有变化先写事件 |
| **事实承载当前** | 当前可引用状态 |
| **关系表达结构** | 对象间稳定连接 |
| **三者边界清楚** | 不混写、不跨类型复用 |
| **全链路可追溯** | 事实→事件→原始输入 |

否则系统后面越做越复杂时，最先塌掉的通常就是"我现在到底记住了什么"。
