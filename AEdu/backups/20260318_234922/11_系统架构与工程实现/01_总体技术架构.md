## 一、文档目的

本文档用于定义教育孪生项目的总体技术架构，说明系统的核心模块、数据流、服务边界、部署思路与阶段性落地方案。

本文档重点回答以下问题：

1. 系统整体分为哪些层
2. 各层之间如何协同
3. StudentTwinAgent、事件流、图谱、产品层如何连接
4. 技术上如何支持多学校、多地区、多角色场景
5. MVP 阶段与后续扩展阶段分别如何落地

---

## 二、架构目标

### 2.1 支撑教育孪生主链路

系统必须能完整支撑以下链路：

```
固定知识底座 → 多源输入接入 → 学习事件生成 → StudentTwinAgent 状态更新 → 图谱记忆沉淀 → 观察层输出 → 后台推演与决策支持
```

### 2.2 支撑多角色输出

同一底层状态系统，应支持：
- 家长端
- 老师端
- 学校机构端
- 运营后台
- 推演后台

### 2.3 支撑多地区、多教材扩展

技术架构不能绑定单地区、单教材、单学校场景。

### 2.4 支撑可追溯与可回滚

关键事件、状态更新、图谱写入、规则版本都要能追溯和回滚。

### 2.5 先跑通 MVP，再逐步拆分

第一阶段优先建立闭环，不追求一开始就微服务宇宙大爆炸。

## 三、总体架构原则

| 原则 | 说明 |
|------|------|
| 分层清晰 | 知识、输入、状态、记忆、输出、推演、工程运维应有清晰边界 |
| 事件驱动 | 系统以学习事件为主要驱动单元，推动状态更新和图谱写入 |
| 状态中心化 | StudentTwinAgent 是个体状态中心，不同模块围绕其输入和消费 |
| 检索增强 | 图谱与记忆系统应为观察层和推演层提供上下文增强能力 |
| 多租户隔离 | 学校、机构、地区数据必须逻辑隔离，必要时物理隔离 |
| 渐进式架构 | 先单体/模块化，后服务化；先闭环，后规模；先能跑，后优雅 |

## 四、总体分层架构

系统分为七层：

```
┌─────────────────────────────────────────────────────────┐
│                    应用输出层                            │
│    家长端 / 老师端 / 学校端 / 运营后台 / 推演后台          │
├─────────────────────────────────────────────────────────┤
│                   记忆与图谱层                           │
│    事实记忆 / 关系记忆 / 事件记忆 / GraphRAG 检索         │
├─────────────────────────────────────────────────────────┤
│                  学生孪生状态层                          │
│    StudentTwinAgent / 状态更新 / 风险计算 / 审计日志      │
├─────────────────────────────────────────────────────────┤
│                   知识底座层                             │
│    地区规则 / 教材库 / 知识点 / 能力点 / 依赖关系          │
├─────────────────────────────────────────────────────────┤

│    OCR / 文档理解 / 事件清洗 / 事件标准化 / 消息队列      │

│    认证鉴权 / 租户管理 / 日志监控 / 任务调度 / 文件存储    │
└─────────────────────────────────────────────────────────┘
```

## 五、接入层

### 5.1 职责

负责接收外部输入，并完成基础鉴权、上传、路由与预处理。

### 5.2 输入来源

| 来源 | 说明 |
|------|------|
| 微信家长端 | 文本输入、图片上传、小程序交互 |
| 钉钉老师端 | 班级数据、作业发布、评语输入 |
| 学校扫描仪 | 试卷扫描、作业拍照、OCR 触发 |
| 批量文件导入 | Excel 成绩导入、历史数据迁移 |
| 人工录入后台 | 运营人员手工补录 |
| 外部规则数据 | 地区政策、考试规则导入 |

### 5.3 核心能力

- 用户身份识别
- 角色识别
- 租户识别
- 文件上传
- 文本接收
- OCR 请求触发
- 异步任务投递

### 5.4 技术建议

- API Gateway / BFF 层统一接入
- 文件存储走对象存储（OSS/S3）
- 大文件/OCR 任务走异步队列
- 接入层尽量轻，不承载复杂业务逻辑

## 六、事件处理层

### 6.1 职责

负责把原始输入转化为标准学习事件。

### 6.2 核心流程

```
原始输入 → OCR/文档理解 → 文本清洗 → 主体识别 → 时间识别 
→ 学科/教材映射 → 事件分类 → 标准事件生成 → 投递状态更新服务
```

### 6.3 标准学习事件字段

| 字段 | 类型 | 说明 |
|------|------|------|
| event_id | string | 事件唯一标识 |
| student_id | string | 学生 ID |
| tenant_id | string | 租户 ID |
| source_type | string | 来源类型（微信/钉钉/扫描/导入） |
| source_ref | string | 原始数据引用 |
| event_type | string | 事件类型（作业/考试/错题/反馈） |
| event_time | datetime | 事件发生时间 |
| subject | string | 学科 |
| chapter_ref | string | 章节引用 |
| knowledge_refs | string[] | 知识点引用列表 |
| confidence_score | float | 置信度（0-1） |
| raw_payload_ref | string | 原始数据引用 |
| parsed_payload | json | 解析后数据 |
| created_at | datetime | 创建时间 |

### 6.4 技术建议

- 队列系统用于解耦处理链路（Kafka/RabbitMQ）
- OCR、解析、映射、校验可拆分为独立 worker
- 异常事件进入人工兜底池
- 保留原始输入引用，避免后续追责时只剩空气

## 七、知识底座层

### 7.1 职责

提供系统的静态教育世界模型，包括地区规则、教材结构与知识图谱基础节点。

### 7.2 包含内容

| 内容 | 说明 |
|------|------|
| 地区规则库 | 省市区编码、考试制度、选科规则 |
| 教材元数据 | 出版社、版本、学科、学段、册别 |
| 章节树 | 教材章节层级结构 |
| 知识点表 | 知识点定义与编码 |
| 能力点表 | 能力要求定义 |
| 依赖关系表 | 知识点前置/后置依赖 |
| 题型映射 | 知识点与题型关联 |
| 锚点映射 | 知识点与教材位置映射 |
| 教材版本映射 | 不同版本间节点对应 |

### 7.3 技术建议

- 结构化表数据建议存放在关系型数据库（PostgreSQL/MySQL）
- 可检索说明文本可进入检索索引（Elasticsearch）
- 版本信息单独建表
- 与图谱实体定义保持一致

## 八、学生孪生状态层

### 8.1 职责

维护 StudentTwinAgent 的当前状态与局部历史状态。

### 8.2 核心能力

| 能力 | 说明 |
|------|------|
| 学生基础绑定 | 地区、学校、教材版本绑定 |
| 事实状态更新 | 考试、作业、错题等事实记录 |
| 派生状态计算 | 完成率、趋势、稳定性等计算 |
| 风险标签刷新 | 风险等级、预警标识 |
| 推断状态写入 | 模型推断结论（带置信度） |
| 状态快照生成 | 按时间窗口生成状态快照 |
| 状态审计记录 | 状态变更日志 |

### 8.3 关键服务建议

| 服务 | 职责 |
|------|------|
| Student Profile Service | 学生基础信息管理 |
| Student State Service | 状态 CRUD 与查询 |
| State Calculation Engine | 状态计算引擎 |
| Risk Evaluation Service | 风险评估服务 |
| Audit Log Service | 审计日志服务 |

### 8.4 技术建议

- 当前状态建议用关系型数据库或 KV/文档存储承载
- 状态快照建议按时间窗口留存
- 规则计算与模型推断解耦
- 状态写入需幂等设计，避免重复消费事件

## 九、记忆与图谱层

### 9.1 职责

负责长期事实、关系、事件和时序记忆的组织、检索与增强。

### 9.2 包含能力

| 能力 | 说明 |
|------|------|

| 关系记忆存储 | 实体间关系存储 |
| 事件记忆存储 | 学习事件时序记录 |

| GraphRAG 检索 | 图谱增强检索 |
| 上下文装配 | 多源上下文融合 |
| 版本回滚 | 图谱状态回滚 |

### 9.3 图谱对象建议

| 对象 | 说明 |
|------|------|
| Student | 学生实体 |
| Parent | 家长实体 |
| Teacher | 老师实体 |
| Class | 班级实体 |

| Subject | 学科实体 |

| RiskSignal | 风险信号实体 |

## 十二、核心数据流

### 12.1 主业务数据流

```
1. 家长/老师/设备提交输入
2. 接入层接收请求
3. 事件处理层完成清洗与标准化
4. 生成学习事件
5. Student State Service 消费事件
6. 更新 StudentTwinAgent 当前状态
7. 图谱服务写入事实/关系/事件
8. 报告服务或观察层读取状态与上下文
9. 输出给家长端、老师端、学校端
10. 推演层按需读取当前状态与历史记忆进行模拟
```

### 12.2 外部规则数据流

```
1. 导入地区规则/政策数据
2. 更新知识底座和规则库
3. 触发版本记录
4. 在学生状态解释和推演时被引用
```

### 12.3 反馈闭环数据流

```
1. 用户查看输出结果
2. 家长/老师提交纠正反馈
3. 反馈写入人工校准通道
4. 更新状态或映射规则
5. 回写审计记录
```

## 十四、存储架构建议

### 14.1 关系型数据库

用于存储：
- 学生基础信息
- 当前状态
- 教材标准表
- 租户与权限
- 规则配置
- 审计日志索引

**推荐**：PostgreSQL 14+

### 14.2 图谱数据库

用于存储：
- 实体关系
- 事实图谱
- 学习事件关系链
- 知识点依赖结构
- 干预关系路径

**推荐**：Neo4j / NebulaGraph

### 14.3 对象存储

用于存储：
- 原始教材 PDF
- 扫描件
- 图片
- OCR 原始文件
- 报告导出文件

**推荐**：阿里云 OSS / AWS S3

### 14.4 向量索引

用于存储：
- 文档语义片段
- 图谱检索辅助索引
- 报告增强上下文
- 政策说明与教材解释文本

**推荐**：Milvus / Weaviate

### 14.5 缓存层

用于承载：
- 高频学生状态读取
- 报告生成缓存
- 热点看板数据
- 会话类临时数据

**推荐**：Redis 7+

## 十五、多租户与权限架构

### 15.1 多租户目标

支持不同学校、试点机构、地区项目并行运行。

### 15.2 隔离维度

| 维度 | 说明 |
|------|------|
| 租户数据隔离 | 不同租户数据逻辑隔离 |
| 学校数据隔离 | 同租户内学校数据隔离 |
| 角色数据可见范围隔离 | 不同角色看到不同数据 |
| 环境隔离 | 开发/测试/生产环境隔离 |

### 15.3 角色建议

| 角色 | 说明 |
|------|------|
| 家长 | 查看自己孩子数据 |
| 老师 | 查看所教班级学生数据 |
| 班主任/年级负责人 | 查看班级管理数据 |
| 学校管理员 | 查看全校汇总数据 |
| 机构运营 | 查看运营指标数据 |
| 平台超级管理员 | 全平台管理权限 |

### 15.4 权限原则

- 最小权限原则
- 角色可见范围明确
- 高风险操作需审计
- 推演后台权限严格控制

## 十七、安全与合规要求

### 17.1 数据安全

| 要求 | 说明 |
|------|------|
| 敏感数据脱敏 | 学生姓名、身份证等脱敏处理 |
| 传输加密 | HTTPS/TLS 加密传输 |
| 存储加密 | 敏感字段加密存储 |
| 访问控制 | 严格权限控制 |
| 审计留痕 | 关键操作留痕 |

### 17.2 隐私边界

- 不同角色只能访问必要数据
- 家长、老师、学校端展示内容要区分
- 推断性信息不应无边界扩散

### 17.3 模型输出边界

- 高风险建议必须带说明和限制条件
- 不得把推断包装成绝对判断
- 必须支持人工复核链路

## 十八、MVP 阶段技术落地建议

### 18.1 MVP 目标

在一个试点学校/地区跑通最小闭环。

### 18.2 MVP 范围

| 范围 | 说明 |
|------|------|
| 高中物理知识底座 | 必修一章节树、知识点表 |

## 二十、结论

教育孪生项目的技术架构，不应被设计成一个"会说话的教育大模型外壳"，而应是一个真正围绕学生个体状态、教育知识底座和持续事件流搭建的系统工程。

其核心技术主线应当稳定为：

```
知识底座 → 学习事件 → StudentTwinAgent → 图谱记忆 → 观察输出 → 后台推演
```

技术实现上必须坚持：
- 先闭环，再扩展
- 先模块化，再服务化
- 先能追溯，再谈智能
- 先解决真实试点问题，再追求架构优雅
