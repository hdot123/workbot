# 核心服务拆分方案

---

## 1. 文档目的

本文档用于定义教育孪生项目中的核心服务拆分方案，包括服务拆分原则、服务边界、MVP 阶段服务形态与后续演进方式。

本文档重点回答以下问题：

1. 项目应该拆成哪些核心服务
2. 各服务分别负责什么
3. 哪些能力先做模块化单体，哪些后续适合独立服务
4. 服务之间如何协同
5. 如何避免过早微服务化或边界混乱

---

## 2. 服务拆分目标

### 2.1 支撑主业务链路
系统必须能稳定支撑：

```
知识底座 → 输入接入 → 学习事件 → StudentTwinAgent → 图谱记忆 → 前台观察 → 后台推演 → 报告输出
```

### 2.2 保持职责清晰
服务拆分的目标不是显得高级，而是让责任归属清楚。

### 2.3 控制耦合
不同服务应通过明确 API 和事件协同，而不是互相越权改内部状态。

### 2.4 支持渐进式演进
MVP 阶段先可运行，后续再根据规模和瓶颈拆得更细。

---

## 3. 服务拆分原则

### 3.1 业务边界优先
按业务能力和对象边界拆，不按"想象中的技术时髦感"拆。

### 3.2 当前态与长期态分离
StudentTwinAgent 当前态与图谱长期记忆应分属不同服务边界。

### 3.3 高耗时链路可独立
OCR、报告生成、推演任务等高耗时任务适合独立处理。

### 3.4 共性能力集中
认证、租户、文件、任务调度等基础能力统一集中。

### 3.5 MVP 不过度服务化
第一阶段宁可模块清楚的单体，也不要十几个互相吵架的小服务。

---

## 4. 服务总体建议

建议核心服务分为以下 11 类：

| 服务 | 英文 | 核心职责 |
|------|------|----------|
| 接入网关服务 | Access Gateway Service | 统一承接前台和后台请求 |
| 接入处理服务 | Ingest Processing Service | 把原始输入转成标准学习事件 |
| 知识底座服务 | Knowledge Base Service | 承载固定知识底座与教材标准表 |
| 学生孪生服务 | Student Twin Service | 维护 StudentTwinAgent 当前态 |
| 图谱记忆服务 | Graph Memory Service | 承载长期事实、关系、事件链 |
| 观察服务 | Observation Service | 组织前台观察页所需数据 |
| 报告服务 | Report Service | 负责周期报告和专项解释报告生成 |
| 推演服务 | Simulation Service | 负责后台推演、风险预测、干预模拟 |
| 规则与外部数据服务 | Rule & External Data Service | 承载外部规则与外部参考数据 |
| 管理与运营服务 | Admin & Ops Service | 服务试点实施、后台管理 |
| 平台基础服务 | Platform Foundation Services | 提供通用基础设施能力 |

## 5. 接入网关服务 (Access Gateway Service)

### 5.1 职责
统一承接前台和后台请求。

### 5.2 负责内容
- API 接入
- 鉴权入口
- 租户识别
- 请求路由
- 基础参数校验
- 角色化 BFF 转发

### 5.3 不负责内容
- 不做深业务处理
- 不直接更新 StudentTwinAgent
- 不直接跑 OCR、推演、报告

### 5.4 MVP 形态
可与主应用保持同进程模块化实现。

---

## 6. 接入处理服务 (Ingest Processing Service)

### 6.1 职责
把原始输入转成标准学习事件。

### 6.2 负责内容
- 原始输入接收
- OCR / 文档理解
- 文本解析
- 事件抽取
- 事件校验
- 知识映射
- 标准学习事件生成

### 6.3 输入
- 文本
- 图片
- PDF
- 批量文件
- 手工录入数据

### 6.4 输出
- 标准学习事件
- 解析状态
- 需人工复核任务

### 6.5 MVP 形态
建议独立为可异步运行的处理模块，必要时可先不独立部署。

---

## 7. 知识底座服务 (Knowledge Base Service)

### 7.1 职责
承载固定知识底座与教材标准表。

### 7.2 负责内容
- 地区规则
- 教材版本
- 章节树
- 知识点表
- 能力点表
- 依赖关系
- 题型映射
- 锚点映射

### 7.3 输出
- 查询接口
- 映射能力
- 规则引用能力

### 7.4 原则
知识底座服务不直接修改学生状态，只提供静态世界模型。

---

## 8. 学生孪生服务 (Student Twin Service)

### 8.1 职责
维护 StudentTwinAgent 当前态与状态更新逻辑。

### 8.2 负责内容
- 学生基础绑定
- 当前事实状态
- 当前表现状态
- 行为节奏状态
- 风险标签
- 推演准备字段
- 状态快照
- 状态审计

### 8.3 输入
- 标准学习事件
- 人工校准
- 知识底座规则
- 图谱返回上下文（必要时）

### 8.4 输出
- 当前态查询
- 状态变化记录
- 前台观察与后台推演消费数据

---

## 9. 图谱记忆服务 (Graph Memory Service)

### 9.1 职责
承载长期事实、关系、事件链和上下文检索。

### 9.2 负责内容
- 图谱写入
- 图谱查询
- 历史事件关系链
- GraphRAG 上下文装配
- 关系与演化轨迹管理

### 9.3 输入
- 标准学习事件
- 状态变化事件
- 干预执行记录
- 知识点依赖关系

### 9.4 输出
- 关系查询结果
- 历史上下文
- 解释依据链

---

## 10. 观察服务 (Observation Service)

### 10.1 职责
组织前台观察页所需数据与解释结果。

### 10.2 负责内容
- 家长端观察首页
- 老师端观察页
- 学校班级看板
- 当前状态摘要
- 解释块装配
- 低风险建议组装

### 10.3 输入
- StudentTwinAgent 当前态
- 图谱上下文
- 报告层部分结果
- 低风险推演结果

### 10.4 原则
Observation Service 只负责展示组织和解释装配，不直接修改底层状态。

---

## 11. 报告服务 (Report Service)

### 11.1 职责
负责周期报告和专项解释报告的生成与管理。

### 11.2 负责内容
- 周报生成
- 月报生成
- 阶段报告生成
- 专项解释报告生成
- 报告导出
- 报告版本管理

### 11.3 输入
- 当前态
- 周期事件摘要
- 图谱上下文
- 模板与口径规则

### 11.4 输出
- 报告内容
- 报告文件
- 报告任务状态

---

## 12. 推演服务 (Simulation Service)

### 12.1 职责
负责后台推演、风险预测、干预模拟与志愿推演等能力。

### 12.2 负责内容
- 风险预测
- InterventionSimulator
- 动作集比较
- 路径推演
- 志愿填报方案推演（后续）

### 12.3 输入
- StudentTwinAgent 当前态
- 图谱记忆上下文
- 干预动作库
- 外部规则
- 历史事件

### 12.4 输出
- 推演结果
- 风险结果
- 候选方案比较
- 后台专业视图数据

---

## 13. 规则与外部数据服务 (Rule & External Data Service)

### 13.1 职责
承载外部规则与外部参考数据。

### 13.2 负责内容
- 地区考试规则
- 政策与口径说明
- 高考历史数据
- 学校与专业数据
- 教材版本映射

### 13.3 原则
- 与知识底座层协同但不等同
- 更偏外部动态约束和参考数据

---

## 14. 管理与运营服务 (Admin & Ops Service)

### 14.1 职责
服务试点实施、后台管理、问题追踪和运营复盘。

### 14.2 负责内容
- 学校接入配置
- 用户与角色管理后台
- 试点指标看板
- 异常事件处理入口
- 问题库管理
- 反馈与工单流转

---

## 15. 平台基础服务 (Platform Foundation Services)

### 15.1 职责
提供通用基础设施能力。

### 15.2 包含内容
- Auth Service（认证与鉴权）
- Tenant Service（租户管理）
- File Service（文件存储）
- Task Scheduler（任务调度）
- MQ / Event Bus（消息队列）
- Notification Service（通知）
- Logging / Metrics / Trace（日志监控追踪）

### 15.3 原则
这些能力不应散落在业务服务中各自长一套。

---

## 16. 服务间协同主链路

建议主链路如下：

```
Access Gateway
    ↓
Ingest Processing
    ↓
Student Twin Service
    ├─→ Graph Memory Service
    ├─→ Observation Service
    ├─→ Report Service
    └─→ Simulation Service

Knowledge Base Service
    ├─→ Ingest Processing
    ├─→ Student Twin Service
    ├─→ Graph Memory Service
    └─→ Simulation Service

Rule & External Data Service
    ├─→ Simulation Service
    ├─→ Report Service
    └─→ Observation Service（有限使用）
```

## 17. 同步调用与异步调用建议

### 17.1 同步调用适用场景

| 场景 | 说明 |
|------|------|
| 查询当前态 | 学生当前状态查询 |
| 查询章节树与知识点 | 知识底座查询 |
| 获取观察页 | 家长端/老师端页面 |
| 获取看板 | 学校班级看板 |
| 获取报告详情 | 报告内容查询 |

### 17.2 异步调用适用场景

| 场景 | 说明 |
|------|------|
| OCR 解析 | 耗时且可重试 |
| 学习事件生成 | 链式处理流程 |
| 状态更新后联动写图谱 | 异步联动 |
| 报告生成 | 批量生成任务 |
| 推演任务 | 高耗时计算 |
| 大批量导入 | 批量处理任务 |

### 17.3 原则
高耗时、链式联动、可能失败重试的流程优先用异步。别把所有重活都塞进同步 API 里等用户盯着转圈。

## 18. MVP 阶段建议拆法

MVP 阶段建议采用：

### 18.1 一个主业务应用

包含：

| 模块 | 说明 |
|------|------|
| Access Gateway | 接入网关 |
| Knowledge Base | 知识底座 |
| Student Twin | 学生孪生 |
| Observation | 观察服务 |
| 基础 Admin | 基础管理 |

### 18.2 若干异步任务模块

包括：

| 模块 | 说明 |
|------|------|
| Ingest Processing | 接入处理 |
| Report Generation | 报告生成 |
| Simulation Task | 推演任务 |
| Graph Write Task | 图谱写入 |

### 18.3 基础平台组件

| 组件 | 说明 |
|------|------|
| 数据库 | PostgreSQL / MySQL |
| 对象存储 | OSS / S3 / MinIO |
| 队列 | Redis / RabbitMQ |
| 缓存 | Redis |
| 日志监控 | ELK / Prometheus |

**说明**：这阶段关键是"边界清晰"，不是"服务数量越多越显得专业"。

## 19. 第二阶段演进建议

建议在以下条件出现时再独立部署服务：

### 19.1 Ingest Processing 独立
当 OCR 和解析耗时明显拉长主系统响应时。

### 19.2 Graph Memory 独立
当图谱查询和写入复杂度明显上升时。

### 19.3 Simulation 独立
当推演任务明显耗时且有独立权限体系时。

### 19.4 Report 独立
当报告生成量和导出量上升时。

---

## 20. 不推荐的拆分方式

### 20.1 按数据库表拆服务
这会把服务拆成碎片，业务流断裂。

### 20.2 一开始就把每个目录都拆成一个服务
目录不是服务边界，别被文件夹骗了。

### 20.3 把 Observation 和 Report 混成同一坨"大输出服务"
观察是实时视图，报告是周期生成，它们口径相关但职责不同。

### 20.4 把 StudentTwin 和 GraphMemory 混成一个服务
当前态和长期态职责不同，混在一起后期会很难维护。

---

## 21. 服务边界判断标准

一个能力是否适合独立服务，建议看以下五项：

| 判断项 | 说明 |
|--------|------|
| 是否有清晰业务边界 | 职责是否明确 |
| 是否有独立读写模式 | 数据访问模式是否独立 |
| 是否有独立扩容需求 | 是否需要独立伸缩 |
| 是否有独立权限要求 | 是否有特殊权限控制 |
| 是否有明显异步任务特征 | 是否适合异步处理 |

满足越多，越适合独立。

## 24. 结论

核心服务拆分方案的目标，不是把系统拆得很散，而是把责任拆得很清楚。

它必须坚持：

- **先明确边界** - 职责清晰再考虑独立部署
- **再决定是否独立** - 根据实际压力和规模演进
- **先跑通主链路** - MVP 阶段优先保证功能完整
- **再按压力点拆分** - 根据瓶颈有针对性地拆分

否则系统很容易从一个可控的模块化工程，变成一群互相甩锅的小服务动物园。
