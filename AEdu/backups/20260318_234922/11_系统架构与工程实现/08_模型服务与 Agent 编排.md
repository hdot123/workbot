# 模型服务与 Agent 编排

> 版本：V1.0

> 最后更新：待定  
> 维护人：系统架构与工程实现负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中的模型服务与 Agent 编排方案，包括模型服务分层、Agent 类型、调用路径、状态依赖、工具依赖、输出控制与运行治理。

本文档重点回答以下问题：

1. 模型在哪些环节发挥作用
2. StudentTwinAgent、观察层 Agent、推演层 Agent 如何分工
3. 模型服务如何与知识底座、图谱、状态模型、事件流协同
4. 为什么不能把所有任务都塞进一个超级 Agent
5. 如何让模型输出可控、可解释、可审计

---

## 2. 定义

模型服务，是系统中承担理解、抽取、归纳、解释、生成和推演等能力的模型化服务集合。

Agent 编排，是将多个 Agent、模型服务、规则引擎、图谱检索和任务流组织成可执行业务流程的机制。

在本项目中，Agent 不是"会聊天的页面壳"，而是围绕特定任务定义的、受约束的执行单元。

---

## 3. 建设目标

### 3.1 明确模型服务边界
哪些任务交给模型，哪些交给规则，哪些交给图谱和主数据层。

### 3.2 明确 Agent 分工
不同 Agent 服务不同目标和风险级别。

### 3.3 支撑长链路编排
输入理解、状态解释、报告生成、推演模拟都需要有序编排。

### 3.4 控制输出风险
高风险任务必须受规则、边界和人工复核约束。

### 3.5 支撑后续扩展
允许从 MVP 的少量 Agent 演进到更复杂的多 Agent 协作。

---

## 4. 设计原则

### 4.1 规则能做的先别交给模型
例如绑定、校验、权限、路由、状态机。

### 4.2 Agent 任务化
每个 Agent 应有明确职责，不做无限膨胀的全能 Agent。

### 4.3 当前态优先
模型调用前先读取 StudentTwinAgent 当前态和 GraphRAG 上下文。

### 4.4 高风险任务多重约束
越接近推演和建议，越要加强规则和审计。

### 4.5 输出可追溯
重要输出应能回到输入、上下文和版本。

---

## 5. 模型服务在系统中的位置

模型服务主要分布在以下节点：

1. 输入理解层
2. 事件生成辅助层
3. 状态解释层
4. 报告生成层
5. 推演模拟层
6. 反馈归纳层

模型服务不是主数据源，也不是权限系统，而是围绕数据和规则工作的智能层。

---

## 6. 模型服务分类建议

建议至少分为以下五类模型服务：

1. 文档理解模型服务
2. 事件理解模型服务
3. 状态解释模型服务
4. 报告生成模型服务
5. 推演模拟模型服务

---

## 7. 文档理解模型服务

### 作用
处理 OCR 后文本块、版面结构、字段抽取和文档语义理解。

### 上下游
- 上游：OCR 与版面分析
- 下游：文档中间对象、事件候选

### 风险控制
只输出候选结构，不直接写主业务对象。

---

## 8. 事件理解模型服务

### 作用
辅助将清洗后的输入和文档理解结果转成标准学习事件。

### 典型任务
- 事件类型判断
- 学科判断
- 知识点候选挂载
- 事件摘要生成

### 原则
最终事件标准仍应受规则层约束。

---

## 9. 状态解释模型服务

### 作用
基于当前态、图谱关系和近期时序记忆，生成解释性文本或解释块。

### 典型任务
- 为什么当前值得关注
- 近期变化如何形成
- 某知识点为何是重点
- 解释链压缩与表达

### 原则
解释服务不创造状态，只解释状态。

---

## 10. 报告生成模型服务

### 作用
将结构化上下文包转成家长周报、月报、老师摘要等输出。

### 特点
- 强依赖上下文装配
- 强依赖角色口径
- 强依赖模板约束

### 原则
报告生成不是自由发挥，而是受控生成。

---

## 11. 推演模拟模型服务

### 作用
围绕干预动作、风险变化和决策场景做受限推演。

### 典型任务
- 读取当前态与历史态
- 读取规则约束
- 对比不同动作
- 输出方向性模拟结果

### 原则
高风险推演不应裸模型直出前台。

---

## 12. Agent 类型建议

建议至少定义以下四类 Agent：

1. StudentTwinAgent
2. ObservationAgent
3. ReportAgent
4. Decision / Simulation Agent

---

## 13. StudentTwinAgent

### 角色
系统核心个体 Agent。

### 职责
- 维护学生当前态
- 接收标准事件更新
- 读取图谱上下文
- 输出当前态给下游

### 边界
它是状态核心，不是万能对话器。

---

## 14. ObservationAgent

### 角色
观察层解释 Agent。

### 职责
- 消费当前态
- 调用 GraphRAG
- 装配角色化上下文
- 生成家长/老师/学校的观察摘要或详情解释

---

## 15. ReportAgent

### 角色
报告生成 Agent。

### 职责
- 消费角色化上下文包
- 调用模板
- 生成周报/月报/阶段摘要
- 写入报告对象

---

## 16. Decision / Simulation Agent

### 角色
推演与决策支持 Agent。

### 职责
- 读取当前态、时序记忆、规则和动作库
- 执行受控模拟
- 生成后台建议和边界说明

---

## 17. 为什么不能做"大一统超级 Agent"

如果把输入理解、状态更新、解释、报告和推演都塞给一个 Agent，会出现：

1. 职责不清
2. 输出不稳定
3. 调试困难
4. 风险难控
5. 复用困难
6. 审计困难

因此本项目更适合：

**多个职责清楚的 Agent + 图谱 + 规则 + 模型服务编排**

---

## 18. Agent 编排层建议

建议建立独立编排层，负责：

- 识别任务类型
- 选择调用哪个 Agent
- 注入上下文
- 注入规则与边界
- 管理工具调用
- 审计输出
- 路由后续任务

编排层负责"管流程"，Agent 负责"做任务"。

---

## 19. 编排触发来源建议

建议至少支持：

1. 输入事件触发
2. 用户查询触发
3. 定时任务触发
4. 报告生成触发
5. 推演任务触发
6. 人工复核触发

---

## 20. 输入事件触发编排

新学习事件进入后，主要触发：

- twin update orchestration
- graph update orchestration
- context refresh orchestration

输入事件主要驱动状态和记忆，不直接生成长文本输出。

---

## 21. 用户查询触发编排

用户查看页面时，流程建议为：

- 读取当前态
- 进行 GraphRAG 检索
- 上下文装配
- 选择 ObservationAgent
- 输出角色化解释

---

## 22. 定时任务触发编排

适用于：

- 周报
- 月报
- 阶段总结

流程建议：

- 生成目标清单
- 装配上下文
- 调用 ReportAgent
- 写入报告对象
- 通知可查看

---

## 23. 推演任务触发编排

适用于：

- 干预模拟
- 风险变化模拟
- 后台决策支持

流程建议：

- 校验推演权限
- 读取当前态、时序记忆、规则
- 加载干预动作集
- 调用 SimulationAgent
- 输出后台结果对象
- 必要时进入人工复核

---

## 24. Agent 调用统一输入建议

每个 Agent 执行前建议统一收到：

- `task_id`
- `target_id`
- `role_type`
- `scenario_type`
- `current_state_ref`
- `context_bundle_ref`
- `rule_constraints`
- `output_constraints`
- `version_context`

这样可以提升可控性和可审计性。

---

## 25. 工具依赖建议

Agent 可调用的工具建议分为：

### 只读工具
- 当前态查询
- 图谱查询
- GraphRAG 检索
- 规则查询
- 报告历史读取

### 写工具
- 反馈提交
- 状态更新请求
- 报告写入
- 推演任务写入

高风险 Agent 默认应只读优先。

---

## 26. 输出分层建议

Agent 输出建议分为三类：

1. 结构化输出
2. 文本输出
3. 审计输出

### 结构化输出
供系统继续消费，例如：
- 解释块
- 关注点
- 建议项

### 文本输出
供前台展示，例如：
- 周报文案
- 老师解释

### 审计输出
供内部追踪，例如：
- 使用的上下文版本
- 模型版本
- 规则版本

---

## 27. 风险控制规则建议

### 低风险任务
如观察摘要、文本整理，可较开放。

### 中风险任务
如重点关注项解释，应强依赖上下文和模板。

### 高风险任务
如推演、建议、志愿模拟，应增加：

- 规则检查
- 输出限制
- 人工复核
- 不确定性说明

---

## 28. 模型版本管理建议

建议记录：

- `model_service_name`
- `model_version`
- `prompt_template_version`
- `rule_pack_version`
- `context_bundle_version`
- `output_schema_version`

这样才能解释为何同类任务不同时间输出有差异。

---

## 29. 缓存策略建议

### 可缓存对象
- 上下文包
- 周报月报输出
- 低频学校端聚合解释
- 部分老师端摘要

### 不建议长期缓存对象
- 当前态高实时解释
- 高风险推演输出
- 强依赖新输入的短周期说明

---

## 30. 审计要求

关键 Agent 任务建议至少审计：

- 谁触发
- 用了哪个 Agent
- 读了哪些上下文版本
- 是否调用规则包
- 是否调用图谱检索
- 输出类型
- 是否进入人工复核
- 最终状态

---

## 31. MVP 建议范围

MVP 阶段建议优先编排以下四类任务：

1. 文档理解辅助编排
2. 学习事件 → StudentTwinAgent 更新编排
3. 家长周报生成编排
4. 老师学生详情解释编排

后置能力包括：

- 多方案推演编排
- 复杂多 Agent 协作
- 高阶自治 Agent

## 34. 结论

模型服务与 Agent 编排的核心价值，不是让系统里多出很多"智能体"名词，而是让模型能力在正确的位置、以正确的边界、按正确的流程工作。

它必须坚持：

- 任务化分工
- 编排层控流程
- 当前态和图谱先行
- 高风险任务强约束
- 输出可追溯

否则系统很容易从"多 Agent 协作"滑向"多个黑箱互相甩锅"。
