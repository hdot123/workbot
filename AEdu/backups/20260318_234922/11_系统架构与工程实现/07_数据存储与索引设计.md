## 1. 文档目的

本文档用于定义教育孪生项目中的数据存储与索引设计，包括数据分层存储策略、各类数据的存储归属、索引策略、查询模式、冷热分层与演进原则。

本文档重点回答以下问题：

1. 系统中的不同数据应该存在哪里
2. 哪些数据适合关系型存储，哪些适合图谱、对象存储、检索索引
3. 当前态、历史态、事件、原始文件如何分层
4. 如何为前台观察、后台推演、GraphRAG 提供高效索引
5. 如何避免"所有东西进一个库"的灾难

---

## 2. 设计目标

### 2.1 支撑主链路查询
必须支撑以下核心链路的存取：

```
知识底座 → 学习事件 → StudentTwinAgent 当前态 → 图谱记忆 → 观察查询 → 推演查询 → 报告生成
```

### 2.2 当前态与长期态分离
当前状态高频读写，长期记忆强调关系和历史，不能混为同一类存储。

### 2.3 原始输入与结构化结果分离
扫描件、图片、原始文本应与结构化事件、状态、图谱关系分层存储。

### 2.4 面向多查询模式
系统同时存在：
- 结构化查询
- 关系路径查询
- 文本检索
- 语义检索
- 聚合统计
- 报告生成读取

### 2.5 支持渐进演进
MVP 阶段先用简单可靠方案，后续再按规模演进。

## 3. 存储分层原则

建议系统按五类存储层设计：

| 存储层 | 用途 | 典型技术选型 |
|--------|------|-------------|
| 关系型存储层 | 结构化主业务数据 | PostgreSQL / MySQL |
| 图谱存储层 | 实体关系与长期记忆 | Neo4j / Nebula Graph |
| 对象存储层 | 原始文件与大文件 | OSS / S3 / MinIO |
| 检索与索引层 | 文本/语义检索 | Elasticsearch / 向量库 |
| 缓存层 | 高频读取加速 | Redis / Memcached |

---

## 4. 关系型存储层

### 4.1 适合存储的内容

| 数据类型 | 说明 |
|----------|------|
| 学生基础档案 | 学生 ID、学校、班级等基础信息 |
| StudentTwinAgent 当前态 | 当前状态摘要、风险等级、关注点 |
| 知识底座标准表 | 章节树、知识点表、能力点表等 |
| 学习事件主表 | 标准学习事件记录 |
| 报告主表 | 周报/月报/阶段报告主记录 |
| 推演任务主表 | 推演任务与结果主记录 |
| 干预动作库主表 | 干预动作定义与执行记录 |
| 权限与租户信息 | 用户、角色、权限、租户 |
| 审计索引 | 关键操作审计记录 |

### 4.2 适合原因
- 结构清晰
- 一致性要求高
- 支持事务
- 支持条件查询和聚合统计

### 4.3 不适合内容
- 原始大文件
- 复杂关系路径查询
- 纯语义检索载体

---

## 5. 图谱存储层

### 5.1 适合存储的内容

| 数据类型 | 说明 |
|----------|------|
| 实体关系 | Student、Teacher、KnowledgePoint 等实体间关系 |
| 学习事件关系链 | 事件之间的时序和因果关联 |
| 历史状态变化关系 | 状态变化的历史轨迹 |
| 干预动作与结果关系 | 动作执行与效果关联 |
| 知识点依赖结构 | 知识点前置/后置依赖 |
| 多实体上下文连接 | 跨实体的复杂关系网络 |

### 5.2 适合原因
- 适合路径查询
- 适合关系追踪
- 适合 GraphRAG 上下文构造

### 5.3 不适合内容
- StudentTwinAgent 当前态主对象
- 高频大批量无结构日志
- 原始文件本体

---

## 6. 对象存储层

### 6.1 适合存储的内容

| 数据类型 | 说明 |
|----------|------|
| 扫描件图片 | 作业、试卷照片 |
| PDF 文件 | 教材、政策文档 |
| 原始上传文件 | 家长/老师上传的原始材料 |
| 导出报告文件 | PDF/Excel 导出文件 |
| OCR 中间产物 | OCR 识别中间结果 |
| 附件 | 其他附件材料 |

### 6.2 原则
- 文件本体存对象存储
- 元数据和引用存关系库
- 不把大文件塞数据库字段里，别给未来自己挖坑

## 7. 检索与索引层

### 7.1 适合承载的能力

| 能力 | 说明 |
|------|------|
| 文本关键词检索 | 教材解释、知识点定义检索 |
| 语义向量检索 | 基于嵌入的语义相似度检索 |
| GraphRAG 辅助检索 | 图谱上下文增强检索 |
| 报告和解释内容检索 | 历史报告内容检索 |
| 政策与知识说明文段检索 | 政策文档、教学说明检索 |

### 7.2 索引对象

| 对象 | 说明 |
|------|------|
| 教材解释文本 | 章节说明、知识点定义 |
| 教师评语片段 | 老师反馈文本 |
| 家长反馈片段 | 家长输入文本 |
| 报告解释块 | 报告中的解释内容 |
| 图谱上下文片段 | 图谱检索辅助索引 |

## 8. 缓存层

### 8.1 适合缓存的内容

| 内容 | 说明 |
|------|------|
| 高频学生当前态摘要 | 家长端/老师端首页数据 |
| 家长端首页数据 | 学生状态摘要、关注点 |
| 老师端班级总览数据 | 班级分布、风险学生列表 |
| 周报月报最近结果 | 最近生成的报告 |
| 推演结果短期缓存 | 近期推演任务结果 |
| 知识底座热点查询结果 | 热门章节/知识点信息 |

### 8.2 原则
- 缓存加速，不做真相源
- 失效策略必须清晰
- 状态更新后关键缓存应主动刷新或失效

## 10. StudentTwinAgent 存储设计建议

### 10.1 当前态主表

**存储内容**：
- 基础绑定字段（twin_id, student_id, tenant_id 等）
- 当前事实摘要（latest_exam_summary, latest_homework_summary）
- 当前状态字段（knowledge_mastery_snapshot, subject_state_summary）
- 行为节奏核心字段（homework_completion_rate_14d 等）
- 推演准备字段（top_risk_factors, simulation_ready_score）
- 审计字段（version, created_at, updated_at）

### 10.2 状态快照表

**存储内容**：
- 周期性状态快照（按周/月）
- 风险等级快照
- 知识点掌握快照
- 学科状态快照

### 10.3 原则
- 当前态与快照分开
- 当前态服务实时读取
- 快照服务趋势与报告

## 16. 索引设计原则

### 16.1 高频查询优先建索引
根据核心查询路径建索引，而不是给所有字段乱建。

### 16.2 组合索引围绕业务场景

| 场景 | 建议索引 |
|------|----------|
| 学生事件查询 | (tenant_id, student_id, event_time) |
| 学科状态查询 | (student_id, subject, updated_at) |
| 班级看板查询 | (school_id, class_id, grade_level) |
| 知识点事件查询 | (knowledge_refs, event_time) |

### 16.3 低选择性字段谨慎建索引
例如单独给 `status` 建索引往往性价比不高，别为了"看起来专业"到处插旗。

## 18. 热冷数据分层建议

### 18.1 热数据（高频访问）

| 数据 | 说明 |
|------|------|
| 最近 30-90 天事件 | 近期学习事件 |
| 当前 StudentTwinAgent 状态 | 实时状态 |
| 最近周报月报 | 最近生成的报告 |
| 当前推演任务结果 | 近期推演结果 |

### 18.2 温数据（中频访问）

| 数据 | 说明 |
|------|------|
| 本学期历史事件 | 学期内历史事件 |
| 状态快照 | 周期性状态记录 |
| 干预执行记录 | 历史干预记录 |

### 18.3 冷数据（低频访问）

| 数据 | 说明 |
|------|------|
| 长期归档事件 | 历史学期事件 |
| 老版本报告 | 历史报告 |
| 历史推演结果 | 过期推演结果 |
| 老教材版本数据 | 旧版本知识底座 |

### 18.4 原则
冷热分层是为了成本和性能，不是为了把老数据藏起来。

## 19. 多租户与隔离设计

### 19.1 逻辑隔离
建议所有主表都带：
- tenant_id（租户标识）
- school_id（学校标识，如适用）

### 19.2 查询过滤
所有前台和后台查询必须受租户边界约束。

### 19.3 敏感数据隔离
报告、推演、反馈等敏感数据需更严格控制可见范围。

## 21. MVP 建议方案

MVP 阶段建议采用以下简化方案：

### 21.1 关系库（主业务存储）

**存储内容**：
- 学生档案
- StudentTwinAgent 当前态
- 学习事件主表
- 知识底座标准表
- 报告表
- 推演表

**推荐选型**：PostgreSQL / MySQL

### 21.2 对象存储

**存储内容**：
- 原始扫描件
- PDF 文件
- 导出文件

**推荐选型**：阿里云 OSS / AWS S3 / MinIO（自建）

### 21.3 轻量图谱/关系层

**策略**：
- 先支持核心实体与关系
- 可用关系库 + 邻接表模拟图谱
- 不急着搞成宇宙级图谱平台

### 21.4 基础检索索引

**优先支持**：
- 知识说明文本检索
- 反馈片段检索
- 报告解释内容检索

**推荐选型**：Elasticsearch / PostgreSQL 全文检索

### 21.5 缓存

**缓存内容**：
- 家长端首页数据
- 学生摘要
- 老师端班级总览

**推荐选型**：Redis

## 22. 演进建议

### 22.1 第二阶段

- 扩展图谱关系查询能力（引入专用图数据库）
- 增强向量检索与 GraphRAG（引入向量库）

## 24. 结论

数据存储与索引设计的本质，不是选几个听起来高级的数据库名字，而是把不同类型的数据放到适合它们的地方，并让系统能以合理成本完成高频读取、结构查询、关系检索和长期追溯。

它必须坚持：
- 当前态和长期态分离
- 原始数据和结构化结果分离
- 文件和主业务数据分离
- 关系查询和普通查询分离
- 高频读取和低频归档分层
