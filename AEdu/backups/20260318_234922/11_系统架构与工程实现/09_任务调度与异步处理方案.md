## 1. 文档目的

本文档用于定义教育孪生项目中的任务调度与异步处理方案，包括异步任务分类、调度机制、执行链路、重试策略、幂等控制、失败处理与监控要求。

本文档重点回答以下问题：

1. 哪些任务必须异步处理
2. 任务应该如何调度、分发和执行
3. 如何处理重试、超时、死信和幂等
4. 如何让输入处理、图谱更新、报告生成和推演任务稳定运行
5. 如何避免异步链路变成"没人知道它现在死没死"的黑箱

---

## 2. 任务调度与异步处理定义

任务调度与异步处理，是指系统将高耗时、可重试、可排队、可后台执行的任务从实时请求链路中拆出，由调度器、队列和工作节点进行有序执行、监控和恢复的机制。

在本项目中，异步不是附属优化，而是主链路稳定运行的必要条件。

原因很简单：

- OCR 不适合同步等
- 图谱更新不适合同步串满
- 报告生成不适合阻塞前台
- 推演任务更不适合让用户盯着页面转圈

---

## 3. 建设目标

### 3.1 保护主请求链路
前台查询和基础录入不应被重任务拖垮。

### 3.2 支撑长链路任务
输入处理、图谱更新、报告生成、推演计算必须可后台执行。

### 3.3 支撑失败恢复
任务失败后可重试、可定位、可重放。

### 3.4 支撑优先级调度
不同任务重要性不同，必须分层调度。

### 3.5 支撑可观测性
任务执行状态、耗时、失败原因必须可见。

---

## 4. 设计原则

### 4.1 同步只做轻链路
实时接口只负责轻量校验、入队和快速响应。

### 4.2 异步处理重逻辑
高耗时、链式依赖、可延后任务优先异步。

### 4.3 幂等优先
异步链路天然会重试，必须先设计幂等。

### 4.4 失败可恢复
失败不是例外，而是系统默认要准备面对的现实。

### 4.5 状态透明
每个任务必须知道：排队中、执行中、成功、失败、重试中、死信。

## 5. 异步任务分类建议

建议至少分为以下六类任务：

| 任务类型 | 说明 | 典型任务 |
|----------|------|----------|
| 输入处理任务 | 处理原始输入 | OCR 解析、文档理解、输入清洗、学习事件生成 |
| 图谱更新任务 | 更新图谱和记忆 | 事件写图谱、事实更新、关系更新、索引刷新 |
| 报告生成任务 | 生成报告和看板 | 周报、月报、班级看板预聚合、导出 PDF |
| 推演任务 | 执行推演计算 | 风险预测、干预模拟、多方案比较 |
| 周期调度任务 | 周期性执行任务 | 每日状态刷新、每周报告预计算、月度数据校验 |
| 修复与重放任务 | 修复和重放任务 | 事件重放、图谱回滚后重算、批量修复 |

## 6. 输入处理任务

### 6.1 典型任务
- OCR 解析
- 文档理解
- 输入清洗
- 学习事件生成
- 异常分流

### 6.2 特点
- 高耗时
- 高并发
- 易失败
- 需要重试与异常池衔接

### 6.3 处理策略
- 提交后立即返回受理结果
- 后台异步处理
- 处理完成后通知或轮询状态

---

## 7. 图谱更新任务

### 7.1 典型任务
- 事件写图谱
- 当前事实更新
- 关系更新
- GraphRAG 索引刷新
- 快照刷新

### 7.2 特点
- 链路长
- 对一致性要求高
- 适合拆成串行子任务或有向流水线

### 7.3 处理策略
- 按学生维度分片
- 支持批量处理
- 保证幂等性

---

## 8. 报告生成任务

### 8.1 典型任务
- 周报生成
- 月报生成
- 班级看板预聚合
- 导出 PDF/文件

### 8.2 特点
- 可延后
- 可批量
- 可缓存
- 适合定时触发

### 8.3 处理策略
- 周期性调度
- 支持预生成和缓存
- 支持按需刷新

---

## 9. 推演任务

### 9.1 典型任务
- 风险预测
- 干预模拟
- 多方案比较
- 志愿推演（后续）

### 9.2 特点
- 耗时不确定
- 资源消耗较高
- 常需排队
- 对超时和优先级要求高

### 9.3 处理策略
- 支持优先级调度
- 支持超时控制
- 支持取消和补偿

---

## 10. 周期调度任务

### 10.1 典型任务
- 每日状态刷新
- 每周报告预计算
- 每月数据校验
- 清理过期缓存
- 重建部分索引

### 10.2 特点
- 周期性
- 时间窗口明确
- 更适合调度器统一管理

### 10.3 处理策略
- 统一调度器管理
- 支持 cron 表达式
- 支持执行历史追踪

---

## 11. 修复与重放任务

### 11.1 典型任务
- 事件重放
- 图谱回滚后重算
- 批量修复错误对象
- 规则变更后的局部重建

### 11.2 特点
- 量大
- 风险高
- 必须有严格审计与幂等保护

### 11.3 处理策略
- 严格审批流程
- 支持断点续传
- 支持回滚

## 12. 核心组件建议

建议任务调度与异步处理至少包含以下组件：

| 组件 | 说明 |
|------|------|
| Task API / 提交入口 | 负责接收任务创建请求 |
| Scheduler / 调度器 | 管理定时任务和有节奏批处理任务 |
| MQ / 任务队列 | 缓冲任务，解耦提交和执行 |
| Worker / 执行节点 | 真正执行任务 |
| Retry Manager / 重试控制 | 管理重试策略和退避 |
| Dead Letter Queue / 死信队列 | 收纳反复失败或不可自动恢复的任务 |
| Task Monitor / 任务监控台 | 监控任务执行状态和指标 |

---

## 13. 提交入口

### 13.1 作用
负责接收任务创建请求。

### 13.2 职责
- 基础校验
- 生成 task_id
- 写任务主表
- 投递队列
- 立即返回任务受理结果

### 13.3 原则
提交入口不执行重任务，它只负责把任务送上轨道。

---

## 14. 调度器

### 14.1 作用
管理定时任务和有节奏批处理任务。

### 14.2 典型职责
- 定时生成周报任务
- 定时刷新状态
- 定时扫描异常重试
- 定时执行索引刷新

### 14.3 原则
调度器负责"什么时候做"，不是"具体怎么做"。

---

## 15. 队列

### 15.1 作用
缓冲任务，解耦提交和执行。

### 15.2 建议分队列

| 队列名 | 用途 |
|--------|------|
| ingest_queue | 输入处理任务 |
| graph_update_queue | 图谱更新任务 |
| report_queue | 报告生成任务 |
| simulation_queue | 推演任务 |
| repair_queue | 修复与重放任务 |

### 15.3 原则
不同任务尽量分队列，避免彼此抢资源。

---

## 16. Worker 执行节点

### 16.1 作用
真正执行任务。

### 16.2 建议职责
- 拉取任务
- 执行业务逻辑
- 更新任务状态
- 记录执行日志
- 触发下游任务

### 16.3 原则
Worker 要做业务，但不应承担全部调度策略。

---

## 17. 任务状态机建议

建议任务至少支持以下状态：

| 状态 | 说明 |
|------|------|
| created | 任务已创建 |
| queued | 已入队等待执行 |
| running | 执行中 |
| success | 执行成功 |
| failed | 执行失败 |
| retrying | 重试中 |
| dead_letter | 已进入死信队列 |
| cancelled | 已取消 |

这样后续监控、复盘和人工介入才有抓手。

## 19. 优先级设计建议

建议至少分为三档：

### 19.1 high（高优先级）

| 任务 | 说明 |
|------|------|
| 关键输入处理 | 影响主链路输入 |
| 当前态刷新 | 影响前台展示 |
| 高优先级人工触发修复 | 人工紧急修复 |

### 19.2 medium（中优先级）

| 任务 | 说明 |
|------|------|
| 普通报告生成 | 周报/月报 |
| 普通图谱更新 | 常规更新 |
| 班级聚合刷新 | 班级看板 |

### 19.3 low（低优先级）

| 任务 | 说明 |
|------|------|
| 批量重建 | 历史数据重建 |
| 历史补算 | 历史补算任务 |
| 冷数据整理 | 冷数据清理 |
| 低优先索引重算 | 索引优化 |

## 20. 链式任务设计

很多任务是链式的，建议拆为明确步骤：

### 20.1 示例链路

```text
原始输入
    ↓
清洗任务
    ↓
事件生成任务
    ↓
图谱更新任务
    ↓
当前态刷新任务
    ↓
索引刷新任务
```

### 20.2 原则
- 每一步单独可重试
- 每一步输出清晰
- 不建议全塞进一个大任务里执行到底

### 20.3 下游触发
- 上一步成功后触发下一步
- 支持失败时中断链路
- 支持补偿逻辑

---

## 21. 幂等控制规则

### 21.1 原则
任务重试时不应重复产生副作用。

### 21.2 建议做法
- 每类任务有 idempotency_key
- 执行前检查是否已成功处理
- 写入型操作使用 upsert / compare-and-swap 思路
- 下游任务触发要防重复派发

### 21.3 典型幂等键

| 幂等键 | 说明 |
|--------|------|
| `raw_input_id + task_type` | 输入处理任务幂等 |
| `student_id + report_type + period` | 报告生成任务幂等 |
| `simulation_id + scenario_version` | 推演任务幂等 |
| `student_id + fact_key + valid_from` | 事实更新幂等 |
| `relation_source + relation_type + relation_target` | 关系更新幂等 |

---

## 22. 重试策略建议

### 22.1 可重试错误

| 错误类型 | 说明 |
|----------|------|
| 网络抖动 | 临时网络问题 |
| 临时依赖不可用 | 依赖服务临时不可用 |
| OCR 服务短时失败 | OCR 服务临时故障 |
| 索引服务超时 | 索引服务响应超时 |

### 22.2 不可重试错误

| 错误类型 | 说明 |
|----------|------|
| 参数结构错误 | 任务参数错误 |
| 对象不存在 | 目标对象不存在 |
| 映射规则缺失 | 规则配置缺失 |
| 业务校验失败 | 业务逻辑校验失败 |

### 22.3 重试策略

| 策略 | 说明 |
|------|------|
| 指数退避 | 重试间隔指数增长 |
| 设置最大重试次数 | 避免无限重试 |
| 达到上限后进入死信队列 | 无法自动恢复时人工介入 |

### 22.4 建议配置

| 任务类型 | 最大重试次数 | 退避策略 |
|----------|-------------|----------|
| 输入处理 | 3 | 指数退避 (1s, 2s, 4s) |
| 图谱更新 | 3 | 指数退避 (1s, 2s, 4s) |
| 报告生成 | 2 | 固定间隔 (5s) |
| 推演任务 | 2 | 固定间隔 (10s) |

## 23. 死信队列设计

### 23.1 作用
收纳反复失败或不可自动恢复的任务。

### 23.2 典型来源

| 来源 | 说明 |
|------|------|
| OCR 持续失败 | 图片质量差导致持续失败 |
| 映射规则缺失 | 缺少必要配置 |
| 下游对象已失效 | 目标对象已失效 |
| 非幂等写入冲突 | 写入冲突无法解决 |

### 23.3 后续动作

| 动作 | 说明 |
|------|------|
| 人工排查 | 运营人员介入处理 |
| 补规则后重放 | 补充配置后重新执行 |
| 明确作废 | 确认无需处理 |
| 进入问题复盘 | 推动系统优化 |

### 23.4 死信任务字段

| 字段 | 说明 |
|------|------|
| dead_letter_id | 死信记录 ID |
| task_id | 原任务 ID |
| failure_reason | 失败原因 |
| retry_history | 重试历史 |
| created_at | 进入死信时间 |
| handled_by | 处理人 |
| handled_at | 处理时间 |
| resolution | 处理结果 |

## 24. 超时与取消规则

### 24.1 超时

| 任务类型 | 超时阈值 |
|----------|---------|
| 输入处理 | 60s |
| 图谱更新 | 120s |
| 报告生成 | 300s |
| 推演任务 | 600s |

### 24.2 取消

以下任务可支持取消：

| 任务类型 | 说明 |
|----------|------|
| 尚未开始的报告生成 | 未执行可取消 |
| 尚未开始的批量补算 | 未执行可取消 |
| 后台长时推演 | 支持中途取消 |

### 24.3 原则
一旦任务已产生关键副作用，取消必须谨慎处理，必要时走补偿逻辑。

## 25. 监控与告警要求

### 25.1 至少应监控

| 指标 | 说明 |
|------|------|
| 队列堆积长度 | 队列中等待任务数 |
| 任务平均等待时长 | 从入队到开始执行的时间 |
| 任务平均执行时长 | 任务执行耗时 |
| 失败率 | 失败任务占比 |
| 死信数量 | 死信队列中任务数 |
| 重试次数分布 | 各重试次数任务分布 |
| 不同任务类型的成功率 | 各类型任务成功率 |

### 25.2 告警建议覆盖

| 告警项 | 触发条件 |
|--------|----------|
| 队列积压异常 | 队列长度超过阈值 |
| 高优先级任务超时 | 高优先级任务超时 |
| 死信激增 | 死信数量异常增长 |
| 周期任务未按时触发 | 周期任务未执行 |

### 25.3 监控面板建议

| 面板 | 内容 |
|------|------|
| 任务总览 | 各状态任务数量、成功率 |
| 队列监控 | 各队列长度、等待时长 |
| 执行耗时 | 各类型任务执行耗时分布 |
| 失败分析 | 失败原因分布、TOP 错误 |
| 死信分析 | 死信来源、处理进度 |

## 26. 多租户与隔离要求

### 26.1 原则
任务应具备租户上下文，避免跨学校/跨机构串任务。

### 26.2 建议字段

| 字段 | 说明 |
|------|------|
| tenant_id | 租户 ID |
| school_id | 学校 ID（如适用） |

### 26.3 资源控制

| 控制项 | 说明 |
|--------|------|
| 租户级限流 | 防止单租户占用全部资源 |
| 队列隔离 | 高优先级租户独立队列 |
| Worker 隔离 | 关键任务独立 Worker 池 |

### 26.4 高耗时推演或批量任务不应挤占全部共享资源，必要时需做租户级限流或队列隔离。

## 27. 审计要求

每次任务执行建议记录：

| 审计项 | 说明 |
|--------|------|
| 谁提交的 | 任务提交人/系统 |
| 为谁执行 | 目标学生/学校 |
| 何时开始 | 开始执行时间 |
| 何时结束 | 结束时间 |
| 成功还是失败 | 执行结果 |
| 重试次数 | 重试次数记录 |
| 产生了哪些对象更新 | 产生的事实/关系/事件 |
| 是否触发下游任务 | 下游任务触发记录 |

## 33. 结论

任务调度与异步处理方案的核心价值，不是把任务丢到后台就完事，而是让系统中那些必须后台执行的重链路，在可观测、可重试、可恢复、可隔离的条件下稳定运转。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 轻请求同步 | 实时接口只做轻量校验和入队 |
| 重任务异步 | 高耗时任务走异步链路 |
| 幂等优先 | 任务重试不产生重复副作用 |
| 失败可恢复 | 失败后可重试、可定位、可重放 |
| 状态全透明 | 每个任务状态可见可追踪 |

否则系统表面上用了很多异步，实际上只是把问题从前台藏到了后台。
