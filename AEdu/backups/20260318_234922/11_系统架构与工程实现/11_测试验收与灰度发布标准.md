# 测试验收与灰度发布标准

> 文档编号：ARCH-011  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：系统架构与工程实现负责人

## 1. 文档目的

本文档用于定义教育孪生项目的测试验收标准与灰度发布流程，确保系统上线前质量可控、发布过程风险可控、问题可快速回滚。

本文档重点回答以下问题：

1. 系统上线前需要完成哪些测试
2. 验收标准是什么
3. 灰度发布如何分阶段进行
4. 出现问题时如何快速回滚
5. 如何保证发布过程可追溯

---

## 2. 测试分层体系

### 2.1 单元测试（Unit Test）

**目标**：验证核心函数、类、模块的逻辑正确性。

**覆盖范围**：
- 核心业务逻辑函数
- 数据转换与校验函数
- Agent 工具调用逻辑
- 状态更新规则

**要求**：
- 核心模块覆盖率 ≥ 70%
- 关键路径覆盖率 ≥ 90%

---

### 2.2 集成测试（Integration Test）

**目标**：验证模块间调用与数据流转正确性。

**覆盖范围**：
- Agent 调用链路
- 事件流完整链路（输入→清洗→事件→状态）
- OCR → 文档理解 → 事件生成链路
- 图谱写入与读取链路

**要求**：
- 核心链路成功率 ≥ 98%
- 异常场景正确处理

---

### 2.3 系统测试（System Test）

**目标**：验证端到端业务流程。

**覆盖范围**：
- 家长输入 → 报告生成全流程
- 老师反馈 → 状态更新全流程
- 批量导入 → 事件生成全流程
- 多角色并发使用场景

**要求**：
- 主流程成功率 ≥ 99%
- 响应时间符合 SLA

---

### 2.4 回归测试（Regression Test）

**目标**：确保新版本不破坏已有功能。

**覆盖范围**：
- 所有 P0/P1 级别用例
- 历史 Bug 修复验证
- 核心业务场景

**要求**：
- 每次发布前必须执行
- 通过率 100%

---

## 3. 核心测试场景

### 3.1 学生状态更新测试

- 单事件触发状态更新是否正确
- 多事件并发更新是否一致
- 状态回滚是否有效

### 3.2 事件生成测试

- 各类输入是否能正确生成事件
- 事件字段是否完整
- 事件幂等性是否保证

### 3.3 图谱写入测试

- 事实写入是否一致
- 关系建立是否正确
- 版本回滚是否有效

### 3.4 报告生成测试

- 周报/月报内容是否正确
- 上下文装配是否完整
- 角色化输出是否正确

### 3.5 推演隔离测试

- 推演结果不污染真实状态
- 影子推演与正式输出隔离
- 推演版本可追溯

---

## 4. 验收标准

### 4.1 功能验收

| 级别 | 要求 |
|------|------|
| P0 | 核心流程成功率 ≥ 99% |
| P1 | 无阻塞性 Bug |
| P2 | 体验问题有记录 |

### 4.2 性能验收

| 指标 | 要求 |
|------|------|
| API 响应时间 | P95 < 500ms |
| 报告生成时间 | < 30s |
| 批量导入处理 | 100 条/分钟 |

### 4.3 数据验收

- 数据一致性无异常
- 关键字段完整性 ≥ 95%
- 日志可追溯率 100%

### 4.4 安全验收

- 权限控制有效
- 敏感数据脱敏
- 审计日志完整

---

## 5. 灰度发布策略

### 5.1 分阶段发布

```
阶段 1：内部环境
    ↓ 验证通过
阶段 2：单学校试点
    ↓ 验证通过
阶段 3：小范围试点（3-5 校）
    ↓ 验证通过
阶段 4：全量发布
```

### 5.2 灰度控制维度

| 维度 | 说明 |
|------|------|
| 按学校 | 选择特定学校先上线 |
| 按班级 | 同校内选择特定班级 |
| 按功能开关 | 功能级别灰度控制 |
| 按用户角色 | 先内部角色后外部用户 |

### 5.3 灰度观察指标

- 错误率变化
- 响应时间变化
- 用户反馈
- 业务指标波动

---

## 6. 回滚机制

### 6.1 功能开关回滚

- 支持快速关闭新功能
- 不影响已有功能
- 配置即时生效

### 6.2 服务版本回滚

- 支持快速回退到上一版本
- 回滚时间 < 10 分钟
- 数据兼容性好

### 6.3 数据写入回滚

- 关键链路支持数据回滚
- 图谱写入支持版本回退
- 状态更新支持撤销

---

## 7. 监控指标

### 7.1 系统指标

| 指标 | 告警阈值 |
|------|----------|
| 接口成功率 | < 99% |
| P95 延迟 | > 1s |
| 错误率 | > 1% |
| CPU/内存 | > 80% |

### 7.2 业务指标

| 指标 | 告警阈值 |
|------|----------|
| 事件生成失败率 | > 5% |
| 报告生成失败率 | > 2% |
| 推演异常率 | > 5% |
| OCR 失败率 | > 10% |

### 7.3 日志要求

- 所有请求有 trace_id
- 关键操作有审计日志
- 错误日志包含堆栈

## 10. 结论

测试验收与灰度发布不是简单的上线流程，而是系统稳定性的第一道防线。

它必须坚持：

- 测试分层覆盖
- 验收标准明确
- 灰度分阶段
- 回滚快速有效
- 监控告警齐全

否则系统扩展之日，就是问题爆发之时。
