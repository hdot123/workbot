## 1. 文档目的

本文档用于定义教育孪生项目中的多租户与学校隔离方案，包括租户模型、学校隔离边界、数据隔离、任务隔离、索引隔离、权限隔离与运维隔离策略。

本文档重点回答以下问题：

1. 多租户在本项目中具体指什么
2. 学校之间的数据和运行边界如何隔离
3. 哪些层必须隔离，哪些层可以共享
4. 如何支持试点学校快速接入，同时避免后期串数据
5. 如何在共享平台和学校专属边界之间取得平衡

---

## 2. 多租户与学校隔离定义

多租户，是指在同一平台上服务多个学校、机构或项目主体，并通过统一机制确保它们的数据、配置、权限、任务和运行边界相互隔离的能力。

在本项目中，租户通常可理解为：

- 一个学校
- 一个教育机构
- 一个学校联盟项目
- 一个试点主体

学校隔离，是多租户能力在教育场景中的具体体现。  
因为教育数据高度敏感，学校之间默认必须隔离，不能靠"大家都会注意"的人类善意来实现。

---

## 3. 建设目标

### 3.1 保证学校间数据隔离
不同学校的数据默认不可互见、不可串写、不可串算。

### 3.2 支撑平台共享能力
在隔离前提下共享底层平台、公共能力和底座资源。

### 3.3 支撑按学校配置
不同学校可有不同配置、角色绑定、接入方式和实施状态。

### 3.4 支撑试点扩张
从单校试点逐步扩到多校时，不需推翻架构。

### 3.5 控制运维复杂度
隔离要够硬，但不能把平台运维成本推到失控。

---

## 4. 设计原则

### 4.1 默认隔离
默认按租户隔离，不做默认共享。

### 4.2 显式共享
必须共享的资源要明确标注共享范围，不做隐式共享。

### 4.3 数据隔离优先于展示隔离
UI 不显示不等于真的隔离，底层数据层必须先隔离。

### 4.4 查询必须带租户上下文
任何关键查询都不应在无租户上下文下执行。

### 4.5 分层隔离
数据、权限、索引、任务、缓存、日志都要考虑隔离。

## 5. 租户模型建议

建议至少定义以下对象：

| 对象 | 编码 | 说明 |
|------|------|------|
| 租户 | Tenant | 平台上的隔离主体对象 |
| 学校 | School | 学校业务主体对象 |
| 租户学校绑定 | TenantSchoolBinding | 租户与学校的绑定关系对象 |
| 租户配置 | TenantConfig | 租户级配置对象 |
| 租户范围策略 | TenantScopePolicy | 定义某租户可使用哪些共享能力 |

## 6. Tenant

### 6.1 定义
平台上的隔离主体对象。

### 6.2 说明
在简单试点阶段，Tenant 可直接等同于 School。  
在后续复杂场景下，一个租户也可能包含多校联合项目。

### 6.3 建议字段
| 字段 | 类型 | 说明 |
|------|------|------|
| tenant_id | string | 租户唯一标识 |
| tenant_name | string | 租户名称 |
| tenant_type | string | 租户类型 (school/district/consortium) |
| status | string | 租户状态 (active/inactive/suspended) |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

---

## 7. School

### 7.1 定义
学校业务主体对象。

### 7.2 说明
学校是业务组织对象，Tenant 是隔离对象。  
两者在初期可一对一，但模型上最好不要强绑死。

### 7.3 建议字段
| 字段 | 类型 | 说明 |
|------|------|------|
| school_id | string | 学校唯一标识 |
| school_name | string | 学校名称 |
| school_code | string | 学校编码 |
| region_id | string | 所属地区 |
| status | string | 学校状态 |
| created_at | datetime | 创建时间 |

## 10. TenantScopePolicy

### 10.1 定义
定义某租户可使用哪些共享能力、共享底座和共享索引范围。

### 10.2 说明
这能避免平台层某些公共资源被所有租户无限访问。

### 10.3 建议字段
| 字段 | 类型 | 说明 |
|------|------|------|
| policy_id | string | 策略唯一标识 |
| tenant_id | string | 租户 ID |
| resource_type | string | 资源类型 |
| access_level | string | 访问级别 (private/shared/restricted) |
| created_at | datetime | 创建时间 |

## 11. 隔离层级建议

建议至少从以下六层做隔离：

| 层级 | 说明 | 隔离对象 |
|------|------|----------|
| 数据隔离 | 所有业务主数据和事件数据必须带 tenant_id | 学生档案、学习事件、状态快照等 |
| 权限隔离 | 权限判断必须同时带租户上下文 | 用户角色、访问范围等 |
| 任务隔离 | 异步任务必须带租户上下文 | 输入处理、图谱更新、报告生成等 |
| 索引隔离 | 检索索引必须明确区分租户私有和平台共享 | 向量索引、全文索引等 |
| 缓存隔离 | 缓存 key 必须带租户上下文 | 当前态缓存、报告缓存等 |
| 日志与审计隔离 | 日志和审计对象必须带租户维度 | 访问日志、操作审计等 |

## 12. 数据隔离

### 12.1 核心原则
所有业务主数据和事件数据必须带 `tenant_id`。

### 12.2 适用对象
| 数据类型 | 说明 | 隔离要求 |
|----------|------|----------|
| 学生档案 | 学生基础信息和绑定关系 | 必须带 tenant_id |
| 家长/老师绑定 | 身份绑定关系 | 必须带 tenant_id |
| 学校配置 | 学校级配置信息 | 必须带 tenant_id |
| 学习事件 | 学习事件记录 | 必须带 tenant_id |
| 状态快照 | 学生状态快照 | 必须带 tenant_id |
| 报告产物 | 生成的报告 | 必须带 tenant_id |
| 图谱对象（租户相关部分） | 学生、反馈、状态等 | 必须带 tenant_id |

### 12.3 原则
查询必须默认带 `tenant_id`，而不是查询后再做过滤。

### 12.4 实现方式
```sql
-- 错误示例：查询后过滤
SELECT * FROM learning_events WHERE student_id = 'STU_001';

-- 正确示例：查询时强制带租户过滤
SELECT * FROM learning_events WHERE student_id = 'STU_001' AND tenant_id = 'TENANT_001';
```

## 13. 权限隔离

### 13.1 核心原则
权限判断必须同时带：

- user_id
- role
- tenant_id
- school_id（如适用）
- scope

### 13.2 说明
同一个老师姓名出现在不同学校，不应有任何天然跨校访问能力。

### 13.3 权限判断流程
```text
接收请求
    ↓
验证身份
    ↓
确认角色
    ↓
验证租户上下文 ← 关键步骤
    ↓
检查对象范围
    ↓
检查动作权限
    ↓
返回允许/拒绝
```

## 14. 任务隔离

### 14.1 核心原则
异步任务必须带租户上下文。

### 14.2 适用任务
| 任务类型 | 说明 | 隔离要求 |
|----------|------|----------|
| 输入处理 | 原始输入清洗和事件生成 | 必须带 tenant_id |
| 图谱更新 | 图谱对象写入和更新 | 必须带 tenant_id |
| 报告生成 | 周报/月报生成 | 必须带 tenant_id |
| 推演任务 | 风险预测和干预模拟 | 必须带 tenant_id |
| 批量修复 | 批量数据修复 | 必须带 tenant_id |

### 14.3 原则
任务队列可共享，但任务对象、执行上下文和幂等键应带租户标识。

### 14.4 任务幂等键示例
```
# 错误：不带租户上下文
幂等键 = event_id + task_type

# 正确：带租户上下文
幂等键 = tenant_id + event_id + task_type
```

## 15. 索引隔离

### 15.1 核心原则
检索索引必须明确区分：

- 租户私有索引
- 平台共享索引
- 可授权共享索引

### 15.2 说明
例如：
- 学生事件、报告、反馈应为租户私有
- 通用教材底座可为平台共享
- 某校私有补充材料不得误入共享索引

### 15.3 GraphRAG 隔离建议
GraphRAG 检索时建议明确三层过滤：

1. 先按租户过滤私有对象
2. 再按授权范围补充共享对象
3. 最后按角色边界裁剪结果

说明：
不能先全量检索再指望最后一步把风险消掉，那会很脆。

### 15.4 索引命名规范
```
# 租户私有索引
tenant_{tenant_id}_student_events
tenant_{tenant_id}_reports

# 平台共享索引
platform_shared_knowledge_base
platform_shared_curriculum

# 授权共享索引
shared_{policy_id}_resources
```

## 16. 缓存隔离

### 16.1 核心原则
缓存 key 必须带租户上下文。

### 16.2 典型对象
| 缓存对象 | 说明 | 隔离要求 |
|----------|------|----------|
| 当前态缓存 | 学生当前状态缓存 | 必须带 tenant_id |
| 报告缓存 | 生成的报告缓存 | 必须带 tenant_id |
| 看板缓存 | 学校/班级看板缓存 | 必须带 tenant_id |
| GraphRAG 上下文缓存 | 检索上下文缓存 | 必须带 tenant_id |

### 16.3 说明
缓存是最容易出"看起来偶发、实际上很严重"的串数据事故的地方。

### 16.4 缓存 Key 命名规范
```
# 错误：不带租户上下文
student_state:STU_000123
teacher_dashboard:USER_001:2024W16

# 正确：带租户上下文
student_state:TENANT_001:STU_000123
teacher_dashboard:TENANT_001:USER_001:2024W16
graphrag_context:TENANT_001:STU_000123:teacher_detail
```

## 17. 日志与审计隔离

### 17.1 核心原则
日志和审计对象必须带租户维度，便于：
- 定位问题范围
- 做租户级审计
- 控制导出与查看边界

### 17.2 原则
平台管理员可做全局看板，但学校级查看必须受租户边界约束。

### 17.3 日志字段规范
所有日志必须包含：
- tenant_id
- school_id（如适用）
- operator_id
- action
- timestamp

## 18. 共享与隔离边界建议

建议分为三类资源：

### 18.1 租户私有资源
| 资源类型 | 说明 |

| 家校反馈 | 家长/老师反馈记录 |
| 校内任务 | 学校内部任务 |
| 学校端配置 | 学校级配置信息 |
| 私有报告 | 生成的报告产物 |
| 私有索引 | 租户私有检索索引 |

### 18.2 平台共享资源
| 资源类型 | 说明 |
|----------|------|
| 通用系统服务 | 基础平台服务 |
| 公共教材底座 | 通用知识底座 |
| 通用模型服务 | 公共模型服务 |
| 公共规则引擎框架 | 规则引擎框架 |
| 公共代码与调度基础设施 | 基础设施层 |

### 18.3 授权共享资源
| 资源类型 | 说明 |
|----------|------|
| 经授权的模板 | 授权使用的报告模板等 |
| 平台公共知识底座的某些扩展包 | 特定扩展内容 |
| 经匿名脱敏的统计基准（后续） | 脱敏统计数据 |

## 19. 图谱隔离建议

### 19.1 图谱对象分层
建议将图谱对象分为：

| 层级 | 对象类型 | 隔离要求 |
|------|----------|----------|
| 平台共享图谱对象 | 知识点、能力点、教材公共底座 | 可共享 |
| 租户私有图谱对象 | 学生、反馈、状态、风险、报告 | 必须私有 |
| 引用共享但不复制的公共对象 | 学生引用知识点 | 只引用不复制 |

### 19.2 说明
例如：
- 知识点、能力点、教材公共底座可以平台共享
- 学生、反馈、状态、风险、报告相关对象必须租户私有
- 学生对象可引用共享知识点对象，但学生关系边不得跨租户串联

### 19.3 关系边隔离规则
```
# 允许：租户私有对象引用共享对象
Student(TENANT_001) --uses--> KnowledgePoint(SHARED)

# 禁止：跨租户关系边
Student(TENANT_001) --classmate--> Student(TENANT_002) ❌

# 禁止：私有对象跨租户串联
Feedback(TENANT_001) --related_to--> Feedback(TENANT_002) ❌
```

说明：
不能先全量检索再指望最后一步把风险消掉，那会很脆。

## 21. 数据库与存储隔离策略建议

建议按阶段采用分层策略：

### 21.1 MVP 阶段
| 策略 | 说明 |
|------|------|
| 逻辑隔离为主 | 所有核心表带 tenant_id |
| 查询强制 tenant filter | repository/service 层自动注入 |
| 关键索引包含 tenant_id | 索引设计包含租户字段 |

### 21.2 规模化阶段
| 策略 | 说明 |
|------|------|
| 热点租户可拆分物理库/物理索引 | 性能隔离 |
| 高敏感租户可做更强隔离 | 安全隔离 |
| 支持租户级资源配额与限流 | 资源隔离 |

## 22. 查询隔离规则

关键业务查询至少满足：

| 规则 | 说明 |
|------|------|
| 请求上下文中存在 tenant_id | 所有请求必须带租户上下文 |
| repository / service 层默认注入 tenant filter | 代码层自动过滤 |
| 禁止无条件全表扫描后业务层过滤 | 数据库层强制过滤 |
| 平台级跨租户查询仅允许在特定内部角色下执行，并强审计 | 特殊场景强管控 |

### 22.1 代码层实现示例
```python
# 错误示例：手动过滤
def get_student_events(student_id):
    events = db.query("SELECT * FROM events WHERE student_id = ?", student_id)
    return [e for e in events if e.tenant_id == current_tenant_id]

# 正确示例：自动注入租户过滤
def get_student_events(student_id):
    # repository 层自动注入 tenant_id 过滤
    events = db.query(
        "SELECT * FROM events WHERE student_id = ? AND tenant_id = ?", 
        student_id, 
        current_tenant_id
    )
    return events
```

## 23. 配置隔离规则

租户级配置至少应覆盖：

| 配置项 | 说明 |
|--------|------|
| 学科启用范围 | 哪些学科对该租户启用 |
| 学校组织结构 | 学校年级班级结构 |
| 报告模板 | 报告展示模板配置 |
| 角色开关 | 哪些角色对该租户启用 |
| 推演能力开关 | 推演能力是否启用 |
| 接入参数 | 接入渠道配置 |
| 数据保留策略 | 数据保留周期配置 |

说明：
不做配置隔离，最终会出现"一个学校开关一动，另一所学校行为变了"的幽默事故。

## 26. 常见错误

### 26.1 逻辑隔离只做在前端
后端查询仍可能串数据。

**正确做法**：数据层、服务层、查询层都要做隔离。

### 26.2 tenant_id 只存在部分表
最终一定会出现边界不完整问题。

**正确做法**：所有业务主表、事件表、任务表、日志表都必须带 tenant_id。

### 26.3 公共图谱和私有图谱边界不清
GraphRAG 很容易误召回私有对象。

**正确做法**：明确区分共享图谱对象和租户私有图谱对象，检索时分层过滤。

### 26.4 缓存 key 不带租户
这是非常经典、非常危险的坑。

**正确做法**：所有缓存 key 必须带 tenant_id 前缀。

### 26.5 后台管理角色默认跨租户全开
短期方便，长期一定出事。

**正确做法**：运营/管理员角色必须按授权租户范围严格控制，所有操作审计留痕。

## 31. 风险与边界

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 不能忽视数据隔离 | 会导致跨租户数据泄露 | 所有核心表强制带 tenant_id |
| 不能忽视缓存隔离 | 缓存串数据是经典高危问题 | 缓存 key 必须带租户前缀 |
| 不能忽视索引隔离 | 会导致检索结果串租户 | 索引设计明确租户边界 |
| 不能忽视任务隔离 | 会导致任务执行串租户 | 任务上下文带租户标识 |
| 不能忽视审计留痕 | 出问题无法追溯 | 跨租户操作强制审计 |

## 32. 结论

多租户与学校隔离方案的核心价值，不是为了让架构图更高级，而是为了让平台在服务多校、多机构时，仍然能保持边界稳定、责任清楚、数据不串。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 默认隔离 | 不做默认共享，共享必须显式 |
| 显式共享 | 明确标注共享范围 |
| 查询强带租户 | 所有查询必须带 tenant_id |
| 缓存和索引也隔离 | 缓存 key、索引命名都要带租户 |
| 高权限操作强审计 | 跨租户操作必须审计留痕 |

否则项目一旦从单校试点走向多校平台，最早爆雷的通常不是功能，而是边界。
