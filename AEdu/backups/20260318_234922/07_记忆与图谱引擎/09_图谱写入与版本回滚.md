# 图谱写入与版本回滚

> 文档编号：GRAPH-009  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：记忆与图谱引擎负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中图谱写入与版本回滚机制，包括写入对象、写入策略、版本粒度、回滚触发条件、回滚方式与审计要求。

本文档重点回答以下问题：

1. 图谱对象应该如何写入
2. 新旧版本如何共存
3. 哪些写入是追加，哪些写入是替换
4. 出现错误后如何回滚
5. 如何避免图谱被错误写入后越修越乱

---

## 2. 图谱写入与版本回滚定义

图谱写入，是指将事实、关系、事件、状态快照、引用对象等结构化内容按既定规则写入图谱存储系统的过程。

版本回滚，是指当写入逻辑、规则映射、事件处理或人工修正出现错误时，将受影响图谱对象恢复到某个先前稳定版本，或基于历史事件流重新生成正确版本的机制。

这套机制的核心目标不是"图谱能写进去"，而是"图谱写进去以后还能管得住"。

---

## 3. 建设目标

### 3.1 保证图谱写入稳定
不同类型对象按统一规则写入，不随实现人员心情波动。

### 3.2 保证版本可追踪
任一对象应能回答：
- 现在是什么版本
- 上一个版本是什么
- 为什么变了

### 3.3 保证错误可恢复
错误写入后应能局部回滚、批量回滚或按事件重放恢复。

### 3.4 保证当前态与历史态共存
当前有效对象与历史对象要能同时保留，不互相覆盖到失忆。

### 3.5 保证审计可用
所有关键写入与回滚动作应能留痕。

---

## 4. 设计原则

### 4.1 先事件，后写图谱
图谱写入的源头应尽量来自标准事件，而不是临时文本。

### 4.2 追加优先于删除
历史对象原则上不物理删除，而是通过状态与版本管理控制有效性。

### 4.3 当前态可替换，历史态不可抹
当前状态允许 supersede，历史事件链必须尽量保留。

### 4.4 回滚优先局部
能按对象、学生、批次局部回滚，就不应全库折腾。

### 4.5 回滚不等于悄悄改掉
回滚本身也是重要事件，必须进审计链。

## 5. 写入对象范围

建议图谱写入至少覆盖以下对象：

| 对象类型 | 说明 | 写入频率 |
|----------|------|----------|
| 实体对象 | Student, Class, School, KnowledgePoint 等 | 低频 |
| 关系对象 | depends_on, supports, belongs_to 等 | 低频 |
| 事实对象 | 当前态事实、历史事实 | 中频 |
| 事件对象 | 学习事件、状态更新事件 | 高频 |
| 状态快照对象 | 学生状态快照 | 中频 |
| GraphRAG 索引引用对象 | 检索上下文块 | 中频 |
| 解释引用对象 | 报告、反馈引用 | 低频 |

---

## 6. 写入类型建议

建议将图谱写入分为五类：

| 写入类型 | 编码 | 说明 | 适用对象 |
|----------|------|------|----------|
| create | create | 新建写入 | 首次出现的实体/关系 |
| append | append | 追加写入 | 事件、审计、反馈对象 |
| upsert | upsert | 更新或插入 | 缓存式聚合对象 |
| supersede | supersede | 替代当前版本 | 当前态事实、风险等级 |
| invalidate | invalidate | 失效标记 | 错误事件、废弃关系 |

---

## 7. create 新建写入

### 7.1 适用对象

| 对象 | 说明 |
|------|------|
| 首次出现的实体 | 新学生、新老师、新学校 |
| 首次出现的关系 | 新建立的依赖/支撑关系 |
| 新规则对象 | 新政策规则、新教材版本 |
| 新教材对象 | 新入库教材资产 |

### 7.2 特点
一旦成功创建，应获得稳定主标识。

### 7.3 写入规则
- 检查主标识唯一性
- 记录创建来源
- 初始化版本链

---

## 8. append 追加写入

### 8.1 适用对象

| 对象 | 说明 |
|------|------|
| 事件对象 | 学习事件、状态更新事件 |
| 审计对象 | 操作审计记录 |
| 反馈对象 | 家长/老师反馈记录 |
| 报告产物对象 | 生成的周报/月报 |

### 8.2 特点
追加写入通常不覆盖旧对象。

### 8.3 写入规则
- 保留原始事件引用
- 记录时间戳
- 支持按时间窗口查询

---

## 9. upsert 更新或插入

### 9.1 适用对象

| 对象 | 说明 |
|------|------|
| 可重复刷新但逻辑唯一的索引对象 | GraphRAG 索引块 |
| 某些缓存式聚合对象 | 聚合统计结果 |
| 某些中间态对象 | 临时计算结果 |

### 9.2 原则
只能用于明确允许幂等更新的对象，不能滥用到历史对象。

### 9.3 写入规则
- 检查幂等键
- 存在则更新，不存在则创建
- 记录更新原因

---

## 10. supersede 替代当前版本

### 10.1 适用对象

| 对象 | 说明 |
|------|------|
| 当前学生状态事实 | 当前风险等级、知识点状态 |
| 当前风险等级 | 学生当前风险水平 |
| 当前教材绑定 | 学生当前使用的教材版本 |
| 当前学科状态摘要 | 学科当前状态 |

### 10.2 规则
- 新对象写入为 active
- 旧对象转为 superseded / inactive
- 保留版本链引用
- 记录变化原因

### 10.3 版本链示例
```
FACT_V1 (superseded) → FACT_V2 (superseded) → FACT_V3 (active)
```

---

## 11. invalidate 失效标记

### 11.1 适用对象

| 对象 | 说明 |
|------|------|
| 错误写入的事件 | 无效或错误的事件记录 |
| 已废弃关系 | 不再适用的关系连接 |
| 规则失效对象 | 政策变更导致的失效对象 |
| 错误映射产生的派生对象 | 错误规则产生的对象 |

### 11.2 原则
失效不等于删除，尤其在审计敏感系统中更不能手一抖就删光。

### 11.3 写入规则
- 标记 status = invalid
- 记录失效原因
- 保留原始内容供审计
- 触发相关对象重算

## 12. 版本粒度建议

建议至少支持以下四个版本粒度：

| 粒度 | 说明 | 适用场景 |
|------|------|----------|
| 对象级版本 | 单个 Fact/Relation/Snapshot/Event 的版本变化 | 局部追踪与回滚 |
| 学生级版本 | 围绕单个学生当前态与相关图谱对象的更新快照 | 单学生恢复 |
| 批次任务级版本 | 某批输入导入、某次规则重算对应的一批写入结果 | 批量错误回滚 |
| 规则/底座级版本 | 底座或规则改变引发的图谱结构变化版本 | 规则切换后大范围重算 |

---

## 13. 对象级版本

### 13.1 定义
单个 Fact / Relation / Snapshot / Event 的版本变化。

### 13.2 用途
最细粒度追踪与局部回滚。

### 13.3 版本字段
- object_id
- object_version_id
- previous_version_id
- change_reason
- changed_by
- valid_from
- valid_to
- status

---

## 14. 学生级版本

### 14.1 定义
围绕单个学生当前态与相关图谱对象形成的一次更新快照。

### 14.2 用途
适合学生状态异常后做单学生恢复。

### 14.3 版本内容
- 学生当前态事实集合
- 相关关系集合
- 近期事件集合
- 快照引用

---

## 15. 批次任务级版本

### 15.1 定义
某批输入导入、某次规则重算、某次批量修复对应的一批写入结果版本。

### 15.2 用途
适合批量错误回滚。

### 15.3 版本内容
- batch_id
- task_type
- affected_object_ids
- write_plan_ref
- result_status

---

## 16. 规则/底座级版本

### 16.1 定义
底座或规则改变引发的图谱结构变化版本。

### 16.2 用途
适合规则切换后的大范围重算与回滚。

### 16.3 版本内容
- rule_version_id
- affected_scope
- migration_plan_ref
- rollback_plan_ref

---

## 17. 图谱写入主流程建议

```text
接收标准事件或变更任务
    ↓
识别写入对象类型
    ↓
生成 write_plan
    ↓
校验幂等键与版本上下文
    ↓
写入事件对象
    ↓
写入/更新事实对象
    ↓
写入/更新关系对象
    ↓
写入状态快照
    ↓
记录版本链与审计日志
    ↓
触发索引刷新
```

---

## 18. 写入前校验规则

写入前建议至少校验：

| 校验项 | 说明 |
|--------|------|
| 对象主标识是否合法 | 检查 ID 格式与唯一性 |
| 来源事件是否存在 | 检查 event_id 有效性 |
| 是否为重复写入 | 检查幂等键 |
| 当前版本上下文是否可用 | 检查 previous_version_id |
| 是否存在明显冲突对象 | 检查冲突事实/关系 |
| 当前租户/学校上下文是否正确 | 检查 tenant_id / school_id |

---

## 19. 幂等控制规则

### 19.1 原则
同一事件重复触发时，不应重复生成等价对象。

### 19.2 建议幂等键

| 幂等键 | 说明 |
|--------|------|
| `event_id + target_object_type` | 事件 + 对象类型幂等 |
| `student_id + fact_key + version_window` | 学生事实版本幂等 |
| `relation_source + relation_type + relation_target + active_window` | 关系幂等 |

### 19.3 说明
没有幂等控制，图谱会很快长成"同一件事被写了三遍"的教育苔藓层。

---

## 20. 版本链字段建议

建议关键对象至少包含以下版本字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| object_id | string | 对象主标识 |
| object_version_id | string | 本版本标识 |
| previous_version_id | string/null | 上一版本标识 |
| root_object_id | string | 根对象标识 |
| change_reason | string/null | 变化原因 |
| changed_by | string | 变化来源 (system/manual) |
| valid_from | datetime | 生效时间 |
| valid_to | datetime/null | 失效时间 |
| status | string | active/superseded/inactive |

## 21. 回滚触发场景

建议至少支持以下回滚触发：

| 场景 | 说明 | 回滚方式 |
|------|------|----------|
| 错误事件写入 | 事件内容或映射错误 | 对象级/批次级 |
| 错误对象映射 | 知识点/能力点映射错误 | 对象级/重放 |
| 错误规则版本生效 | 规则版本切换错误 | 批次级/重放 |
| 批量任务逻辑错误 | 批量处理逻辑错误 | 批次级 |
| 人工误修正 | 人工校准错误 | 对象级 |
| 异步任务重复写入造成污染 | 幂等失效导致重复 | 对象级/批次级 |

---

## 22. 回滚方式建议

建议至少支持以下三种方式：

| 回滚方式 | 适用场景 | 特点 |
|----------|----------|------|
| 对象级回滚 | 单个对象错误 | 最细粒度，影响最小 |
| 批次级回滚 | 批量任务错误 | 可回滚一批对象 |
| 事件重放式恢复 | 大面积污染/规则重构 | 最彻底，但成本最高 |

---

## 23. 对象级回滚规则

### 23.1 适用场景

| 场景 | 说明 |
|------|------|
| 某条当前事实写错 | 风险等级、知识点状态写错 |
| 某关系错误建立 | 依赖关系、支撑关系错误 |
| 某状态快照错误 | 快照内容错误 |

### 23.2 原则
- 当前版本失效
- 恢复上一版本为 active
- 写入一条 rollback_event 审计对象

### 23.3 回滚流程
```text
定位错误对象
    ↓
确认目标版本
    ↓
标记当前版本 invalid
    ↓
恢复目标版本 active
    ↓
写入回滚审计事件
    ↓
触发索引刷新
```

---

## 24. 批次级回滚规则

### 24.1 适用场景

| 场景 | 说明 |
|------|------|
| 某批规则变更写错 | 规则版本切换错误 |
| 某批导入任务写错 | 批量导入数据错误 |
| 某批修复任务有问题 | 批量修复逻辑错误 |

### 24.2 原则
- 按 batch_id 找到受影响对象
- 统一标记回滚
- 对依赖对象做一致性重算

### 24.3 回滚流程
```text
定位批次任务
    ↓
获取受影响对象列表
    ↓
批量标记回滚
    ↓
重算依赖对象
    ↓
写入回滚审计事件
    ↓
触发索引与缓存刷新
```

---

## 25. 事件重放式恢复规则

### 25.1 适用场景

| 场景 | 说明 |
|------|------|
| 图谱对象大面积污染 | 规则错误导致大面积错误 |
| 规则映射重构 | 映射规则重大变更 |
| 需要从干净状态重新生成当前态 | 当前态已不可信 |

### 25.2 流程
```text
确定回滚边界
    ↓
失效错误对象
    ↓
锁定目标学生/对象范围
    ↓
按事件时间顺序重放
    ↓
重新生成事实/关系/快照
    ↓
刷新索引与缓存
```

### 25.3 原则
重放是恢复手段，不是常态主流程。太频繁说明前面写入设计没管住。

## 26. 回滚影响控制

回滚时需至少评估：

| 影响项 | 说明 | 处理动作 |
|--------|------|----------|
| StudentTwinAgent 当前态 | 是否影响学生当前态 | 触发当前态刷新 |
| GraphRAG 索引 | 是否影响检索上下文 | 触发索引刷新 |
| 报告缓存 | 是否影响已生成报告 | 失效报告缓存 |
| 推演结果缓存 | 是否影响推演结果 | 失效推演缓存 |
| 前台已展示结果 | 是否影响前台展示 | 通知刷新或标记 |

---

## 27. 审计要求

每次关键写入与回滚至少记录：

| 字段 | 类型 | 说明 |
|------|------|------|
| operation_id | string | 操作唯一标识 |
| operator_type | string | system/manual |
| trigger_source | string | 触发来源 |
| affected_object_ids | array[string] | 受影响对象 ID 列表 |
| old_version_refs | array[string] | 旧版本引用 |
| new_version_refs | array[string] | 新版本引用 |
| batch_id | string/null | 批次 ID |
| rollback_reason | string/null | 回滚原因 |
| started_at | datetime | 开始时间 |
| finished_at | datetime | 结束时间 |
| result_status | string | success/failed/partial |

---

## 28. 失败处理规则

### 28.1 部分写入失败

必须能识别：

| 识别项 | 说明 |
|--------|------|
| 哪一步失败 | 定位失败环节 |
| 哪些对象已写入 | 已写入对象列表 |
| 是否需要补偿或回滚 | 判断恢复策略 |

### 28.2 原则
- 不允许"半写入但系统假装完成"
- 必须支持事务边界或补偿逻辑
- 失败状态必须可见

### 28.3 处理流程
```text
检测写入失败
    ↓
识别已写入对象
    ↓
评估一致性影响
    ↓
执行补偿或回滚
    ↓
记录失败审计日志
```

---

## 29. 与索引和缓存的联动规则

图谱对象写入或回滚后，应按对象类型决定是否联动：

| 联动项 | 触发条件 | 处理动作 |
|--------|----------|----------|
| GraphRAG 索引刷新 | 事实/关系/事件变化 | 增量更新索引 |
| 当前态缓存刷新 | 当前态事实变化 | 失效或更新缓存 |
| 报告缓存失效 | 相关事实/事件变化 | 失效旧报告缓存 |
| 推演结果缓存失效 | 推演输入变化 | 失效旧推演结果 |

说明：
图谱回滚了但索引还活在旧世界里，这类问题会让系统显得人格分裂。

## 30. MVP 建议范围

MVP 阶段建议优先支持：

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 事件 append | 事件追加写入 | P0 |
| 当前事实 supersede | 当前态事实替换 | P0 |
| 关系 invalidate / activate | 关系失效/激活 | P0 |
| 对象级回滚 | 单对象回滚 | P0 |
| 批次级回滚 | 批量回滚 | P1 |
| 单学生重放恢复 | 单学生事件重放 | P1 |
| 写入审计日志 | 写入操作审计 | P0 |
| 索引联动刷新 | 索引同步刷新 | P0 |

```json
  "operation_id": "GWRITE_20240420_001",

  "trigger_source_type": "learning_event",
  "tenant_id": "TENANT_001",
  "school_id": "SCH_001",
  "result_status": "success",
  "duration_ms": 245,
  "created_at": "2024-04-20T10:35:00Z"
}
```

```json
{
  "rollback_id": "GROLLBACK_20240420_001",
  "rollback_type": "object_level",
  "target_object_id": "FACT_STU_000123_RISK",

  "from_version": "FACT_STU_000123_RISK_V4",
```

## 34. 风险与边界

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 不能忽视幂等控制 | 重复写入会导致数据污染 | 建立幂等键机制 |
| 不能忽视版本链管理 | 版本链断裂会导致无法回滚 | 强制记录 previous_version_id |
| 不能忽视回滚影响评估 | 回滚可能影响多个下游系统 | 建立影响评估清单 |
| 不能忽视索引联动 | 图谱回滚后索引不同步会导致检索错误 | 建立索引联动机制 |
| 不能忽视审计留痕 | 回滚操作必须可追溯 | 强制审计日志记录 |

## 35. 结论

图谱写入与版本回滚的核心价值，不是让系统"能改"，而是让系统在持续更新、批量修复、规则变更和人工校准过程中，仍然能够保持结构稳定、历史可追、错误可退。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 写入有规则 | 不同类型对象按统一规则写入 |
| 版本有链路 | 任一对象能追溯版本变化历史 |
| 当前态可替换 | 当前状态允许 supersede |
| 历史态不乱删 | 历史对象通过状态管理控制有效性 |
| 错误可局部回退 | 支持对象级、批次级、重放式回滚 |

否则图谱一旦进入真实试点环境，迟早会遇到一种非常熟悉的局面：写是写进去了，但谁也不敢再动。
