> 文档编号：RULE-002  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：规则与外部数据负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中"高考历史数据"的接入规范，包括数据范围、字段结构、清洗规则、版本管理、质量校验与系统使用边界。

本文档重点回答以下问题：

1. 高考历史数据接入哪些内容
2. 数据如何标准化和结构化
3. 数据如何与地区规则、学校专业数据、志愿推演衔接
4. 如何控制年份差异、口径差异和数据污染
5. 如何避免把历史数据误用成未来确定结论

---

## 2. 高考历史数据定义

高考历史数据，是指与高考录取、分数、排名、批次、学校、专业等相关的历年结构化参考数据。

这类数据主要用于：

- 志愿填报推演
- 风险分层判断
- 方案比较
- 长期趋势参考

它不是未来录取结果的保证书，只是带时间边界的历史参考材料。

---

## 3. 接入目标

### 3.1 为志愿推演提供历史参照
帮助系统理解不同学校和专业在不同年份的录取区间和变化趋势。

### 3.2 为风险评估提供参考边界
支持"过冲 / 稳妥 / 保守"等结构判断。

### 3.3 为规则层提供历史背景
辅助理解政策变化对录取口径的影响。

### 3.4 为学校/专业数据提供时间维度
让学校和专业不是静态对象，而是带历史表现的数据对象。

---

## 4. 接入范围

建议高考历史数据优先接入以下内容：

| 数据类别 | 说明 |
|----------|------|
| 年份 | 数据所属年份 |
| 地区 | 数据所属地区 |
| 批次 | 录取批次 |
| 学校 | 学校基本信息与录取数据 |
| 专业 | 专业基本信息与录取数据 |
| 分数线/录取线 | 最低分、平均分等 |
| 位次/排名 | 最低位次、平均位次等 |
| 科类/选科要求口径 | 文理/新高考选科要求 |
| 特殊说明 | 政策变化、口径说明等 |

---

## 5. 数据对象定义

建议至少定义以下几类核心对象：

### 5.1 AdmissionRecord（录取记录）
某地区某年份某学校/专业的录取记录。

### 5.2 BatchRuleSnapshot（批次规则快照）
某地区某年份某批次规则快照。

### 5.3 SchoolAdmissionProfile（学校录取概况）
学校在历年中的录取概况摘要。

### 5.4 MajorAdmissionProfile（专业录取概况）
专业在历年中的录取概况摘要。

## 6. 核心字段建议

高考历史录取记录主表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| admission_record_id | string | 是 | 记录唯一标识 |
| region_id | string | 是 | 地区标识 |
| exam_year | int | 是 | 年份 |
| batch_code | string | 是 | 批次编码 |
| school_id | string | 是 | 学校标识 |
| school_name | string | 是 | 学校名称 |
| major_id | string/null | 否 | 专业标识 |
| major_name | string/null | 否 | 专业名称 |
| subject_mode | string | 是 | 文理/新高考选科口径 |
| required_subjects | array[string] | 否 | 选科要求（如适用） |
| min_score | number/null | 否 | 最低录取分 |
| avg_score | number/null | 否 | 平均录取分 |
| min_rank | int/null | 否 | 最低录取位次 |
| avg_rank | int/null | 否 | 平均录取位次 |
| enrollment_count | int/null | 否 | 招生人数 |
| source_ref | string | 是 | 来源引用 |
| source_date | date/null | 否 | 来源日期 |
| data_quality_flag | string | 是 | 数据质量标识 |
| notes | string/null | 否 | 特殊说明 |

| status | string | 是 | active/deprecated |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

## 7. 标准化原则

### 7.1 年份显式化
每条数据必须明确年份，不能混用"近几年"这种模糊表达。

**要求**：
- exam_year 字段必填
- 同一记录不能跨年份
- 多年数据应拆分为多条记录

### 7.2 地区显式化
不同地区规则不同，历史数据必须绑定地区。

**要求**：
- region_id 字段必填
- 地区编码与地区规则库一致
- 不支持"全国通用"这种模糊地区

### 7.3 批次显式化
必须保留批次信息，避免不同批次数据混合。

**要求**：
- batch_code 字段必填
- 批次编码与当年地区规则一致
- 不同批次数据不能合并

### 7.4 学校与专业拆分
学校层数据与专业层数据要能分开存，也要能建立关联。

**要求**：
- 学校层数据：major_id 可为空
- 专业层数据：major_id 必填
- 支持按学校聚合查询专业数据

### 7.5 口径说明显式化
文理分科、新高考选科、赋分模式等口径必须带说明。

**要求**：
- subject_mode 字段必填
- 政策变化年份需附加说明
- 口径不一致数据需标记

## 8. 数据来源要求

### 8.1 可接受来源

| 来源类型 | 说明 | 可信度 |
|----------|------|--------|
| 官方发布数据 | 教育考试院、招生办公布 | 高 |
| 权威教育机构公开资料 | 教育部、省级教育部门 | 高 |
| 学校官方公告 | 学校招生网站发布 | 中高 |
| 经校验的历史数据集 | 经过多源校验的数据 | 中 |

### 8.2 不建议直接采用来源

| 来源类型 | 说明 | 风险 |
|----------|------|------|
| 来源不明的整理表 | 无法追溯原始来源 | 高 |
| 无时间说明的二手截图 | 年份和批次不明 | 高 |
| 口径不清的自媒体表格 | 可能存在错误或误导 | 高 |
| 商业机构未经验证数据 | 可能存在商业目的篡改 | 中高 |

### 8.3 原则
来源必须可追溯，别把互联网流言伪装成历史数据。

## 9. 清洗与转换规则

建议标准流程如下：

```
收集原始数据
    ↓
识别年份与地区
    ↓
识别批次和口径
    ↓
标准化学校/专业名称
    ↓
补齐学校/专业 ID 映射
    ↓
清洗分数与位次字段
    ↓
补充来源和版本
    ↓
质量校验
    ↓
入库
```

### 9.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 收集原始数据 | 从各来源收集原始表格/文档 | 原始数据集 |
| 识别年份与地区 | 提取年份和地区信息 | 年份/地区标识 |
| 识别批次和口径 | 提取批次和科类信息 | 批次/科类标识 |
| 标准化名称 | 统一学校/专业命名 | 标准化名称 |
| 补齐 ID 映射 | 关联学校/专业标准库 | 学校/专业 ID |
| 清洗数值字段 | 校验分数/位次的有效性 | 清洗后数值 |
| 补充来源信息 | 记录来源和版本 | 来源记录 |
| 质量校验 | 执行质量检查规则 | 质量标识 |
| 入库 | 写入历史数据库 | 可用数据记录 |

## 10. 学校与专业标准化规则

### 10.1 学校标准化

| 处理项 | 说明 |
|--------|------|
| 名称统一 | 映射到学校标准库名称 |
| 合并历史别名 | 处理学校更名情况 |
| 明确校区/分校差异 | 区分主校区和分校区 |
| 处理合并/拆分 | 记录学校合并/拆分历史 |

**示例**：
- "北京大学" → school_id: SCH_001
- "北大" → 映射到 SCH_001
- "北京大学医学部" → 独立 school_id 或明确标注

### 10.2 专业标准化

| 处理项 | 说明 |
|--------|------|
| 名称映射 | 映射到专业标准库名称 |
| 处理专业更名 | 记录专业历史名称 |
| 处理专业合并/拆分 | 记录专业变迁历史 |
| 保留历史原名引用 | 便于追溯和解释 |

**示例**：
- "计算机科学与技术" → major_id: MAJ_001
- "计算机科学" → 映射到 MAJ_001（如确认是同一专业）
- "软件工程（含嵌入式方向）" → 映射到 MAJ_002 或独立处理

### 10.3 原则
标准化是为了可比较，不是为了把历史差异抹平。

## 11. 分数与位次处理规则

### 11.1 分数优先还是位次优先

| 场景 | 建议 |
|------|------|
| 分数可获得 | 同时保留分数和位次 |
| 仅有分数 | 保留分数，标记位次缺失 |
| 仅有位次 | 保留位次，标记分数缺失 |
| 新旧高考交替年份 | 优先位次，分数仅作参考 |

### 11.2 分数与位次的关系

| 特点 | 分数 | 位次 |
|------|------|------|
| 受年份难度影响 | 大 | 小 |
| 横向比较稳定性 | 低 | 高 |
| 政策变化敏感度 | 高 | 中 |
| 参考价值 | 需结合年份 | 更稳定 |

### 11.3 原则
- 分数受年份难度影响更大
- 位次在横向比较中通常更稳定
- 两者同时保留最佳
- 位次缺失时需明确标注

---

## 12. 批次与规则口径处理

### 12.1 必须绑定批次

同一学校不同批次差异可能很大。

**示例**：
- 同一学校在"本科一批"和"本科二批"录取分数可能相差很大
- 不能将不同批次数据混合

### 12.2 必须绑定规则年份

政策变化会改变批次结构和可比性。

**示例**：
- 2020 年某地区合并本科一批二批 → 前后数据不可直接横比
- 新高考改革年份 → 科目组合变化影响可比性

### 12.3 口径不一致年份要做标记

| 变化类型 | 处理要求 |
|----------|----------|
| 文理分科年份 | 标记为 traditional_science_arts |
| 新高考改革年份 | 标记为 new_gaokao |
| 批次改革年份 | 标记批次变化说明 |
| 赋分制变化 | 标记赋分规则变化 |

## 13. 数据质量要求

### 13.1 完整性

| 检查项 | 要求 | 等级 |
|--------|------|------|
| 核心字段完整 | region_id, exam_year, batch_code, school_id 必填 | 必须 |
| 分数或位次至少一项 | min_score 或 min_rank 至少一项必填 | 必须 |
| 来源信息完整 | source_ref 必填 | 必须 |
| 口径说明完整 | subject_mode 必填 | 必须 |

### 13.2 一致性

| 检查项 | 要求 | 等级 |
|--------|------|------|
| 学校/专业 ID 映射一致 | 与标准库一致 | 必须 |
| 批次与年份匹配 | 批次在当年存在 | 必须 |
| 地区编码一致 | 与地区规则库一致 | 必须 |
| 分数逻辑合理 | min_score ≤ avg_score | 建议 |

### 13.3 时效性

| 检查项 | 要求 | 等级 |
|--------|------|------|
| 年份清晰 | exam_year 明确 | 必须 |
| 来源日期清晰 | source_date 记录 | 建议 |
| 数据更新及时 | 新数据及时入库 | 建议 |

### 13.4 可追溯性

| 检查项 | 要求 | 等级 |
|--------|------|------|
| 来源文件可查 | source_ref 可追溯 | 必须 |
| 变更记录可查 | 版本更新有记录 | 建议 |

### 13.5 可解释性

| 检查项 | 要求 | 等级 |
|--------|------|------|
| 规则变化有说明 | notes 字段记录 | 建议 |
| 异常情况有标注 | data_quality_flag 标注 | 必须 |

---

## 14. 数据质量标识

建议 data_quality_flag 包含以下值：

| 标识 | 说明 | 使用建议 |
|------|------|----------|
| verified | 已验证，多源一致 | 可放心使用 |
| single_source | 单一来源，未交叉验证 | 谨慎使用 |
| incomplete | 核心字段有缺失 | 仅作部分参考 |
| inconsistent | 与其他来源存在冲突 | 需人工复核 |
| deprecated | 已废弃，被新数据替代 | 不应使用 |

## 15. 版本管理要求

### 15.1 版本更新场景

| 场景 | 版本类型 | 说明 |
|------|----------|------|
| 新增年份数据 | 次版本 | V1.1.0 → V1.2.0 |
| 补充缺失字段 | 修订版本 | V1.0.0 → V1.0.1 |
| 修正错误映射 | 修订版本 | V1.0.0 → V1.0.1 |
| 政策口径补充说明 | 修订版本 | V1.0.0 → V1.0.1 |
| 数据结构变化 | 主版本 | V1.0.0 → V2.0.0 |

### 15.2 每次更新必须记录

| 记录项 | 说明 |
|--------|------|
| 更新时间 | 版本更新时间 |
| 更新人 | 更新责任人 |
| 更新原因 | 为什么更新 |
| 影响范围 | 影响哪些模块和功能 |
| 是否影响既有推演结果 | 是否需要重新评估历史推演 |

### 15.3 回滚要求

如数据接入错误，必须支持：
- 回退到上一稳定版本
- 记录回滚原因和时间
- 通知受影响的相关方

## 16. 历史数据使用边界

### 16.1 只能作为参考

| 原则 | 说明 |
|------|------|
| 不是未来承诺 | 历史数据不保证未来录取结果 |
| 需结合当年政策 | 每年政策可能有变化 |
| 需考虑数据完整性 | 部分数据可能存在缺失 |

### 16.2 必须受规则解释约束

| 原则 | 说明 |
|------|------|
| 不同年份规则不同 | 不能机械横比 |
| 不同地区规则不同 | 不能跨区域机械比较 |
| 政策变化需说明 | 重大政策变化需标注 |

### 16.3 必须输出边界说明

在志愿推演中使用时，应明确：

- 数据年份范围
- 地区范围
- 规则变化说明
- 数据完整性情况
- 置信度说明

## 17. 与志愿推演的关系

### 17.1 高考历史数据主要用于

| 用途 | 说明 |
|------|------|
| 学校/专业方案的历史参考 | 了解历史录取区间 |
| 冲稳保分层 | 判断冲刺/稳定/保底 |
| 录取风险提示 | 识别过冲/扎堆风险 |
| 结构平衡判断 | 评估方案结构合理性 |

### 17.2 不应用于

| 禁止用途 | 说明 |
|----------|------|
| 直接生成"必录取"结论 | 历史数据不是保证 |
| 脱离地区规则做跨区域机械比较 | 地区规则差异大 |
| 忽略政策变化的简单外推 | 政策会变 |

---

## 18. 与地区规则的关系

### 18.1 高考历史数据必须与地区规则一起使用

| 关系 | 说明 |
|------|------|
| 规则定义制度边界 | 地区规则定义录取规则 |
| 历史数据提供历史表现参考 | 历史录取情况参考 |

**两者缺一不可。**

只看数据不看规则，会错；只看规则不看数据，会空。

### 18.2 典型协同场景

| 场景 | 规则作用 | 数据作用 |
|------|----------|----------|
| 志愿填报推演 | 定义批次、选科要求 | 提供历史录取参考 |
| 风险评估 | 定义制度边界 | 提供历史波动参考 |
| 方案比较 | 定义有效方案范围 | 提供历史表现对比 |

---

## 19. MVP 建议范围

MVP 阶段建议优先接入：

### 19.1 必须支持

| 内容 | 说明 |
|------|------|
| 一个试点地区 | 先聚焦一个地区 |
| 近 3-5 年核心数据 | 优先近期数据 |
| 核心学校数据 | 重点学校优先 |
| 批次、分数、位次核心字段 | 基础字段完整 |
| 基础规则变化说明 | 重大政策变化标注 |

### 19.2 可以后置

| 内容 | 说明 |
|------|------|
| 全国多地区全覆盖 | 后续逐步扩展 |
| 复杂专业历史迁移处理 | 专业变迁历史逐步补充 |
| 更深层学校画像 | 学校详细画像后续补充 |
| 长周期历史数据（10 年+） | 远期数据逐步补充 |

## 23. 风险与边界

### 23.1 不能把历史数据当未来承诺

历史数据只能作为参考，不能保证未来结果。

### 23.2 不能忽视政策变化

政策变化年份的数据需要特别标注和谨慎使用。

### 23.3 不能机械横比不同年份数据

不同年份难度、政策、考生人数都不同。

### 23.4 不能忽视数据质量问题

来源不明、口径不清的数据不应直接使用。

### 23.5 不能脱离地区规则使用数据

历史数据必须结合地区规则解释。

## 24. 结论

高考历史数据接入规范的核心，不是把历年录取表堆进数据库，而是把"带年份、带地区、带规则边界的历史参考"规范化接入系统。

它必须坚持：

- **年份清楚** - 每条数据明确年份
- **地区清楚** - 数据绑定具体地区
- **批次清楚** - 批次信息完整
- **来源清楚** - 数据来源可追溯
- **边界清楚** - 使用边界明确说明

否则历史数据会从参考材料变成误导材料，志愿推演会从辅助工具变成高风险意见制造机。
