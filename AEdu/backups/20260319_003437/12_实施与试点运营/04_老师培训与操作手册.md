## 1. 文档目的

本文档用于定义教育孪生项目在试点学校中的老师培训与操作手册，包括培训目标、培训对象、培训内容、操作流程、常见问题和使用边界。

本文档重点回答以下问题：

1. 老师为什么要参加培训
2. 老师培训到底要讲什么
3. 老师端应该怎么用
4. 老师最容易在哪些地方踩坑
5. 如何让培训不是"听完就散"，而是真能上手

---

## 2. 老师培训定义

老师培训，是指面向班主任、学科老师、年级负责人开展的系统目标说明、角色边界说明、操作演示和实际场景练习过程。

老师操作手册，是将老师在试点中最常用的查看、反馈、标记和协作动作整理成可执行说明的文档。

培训不是把产品介绍讲一遍，操作手册也不是把按钮位置抄一遍。  
它们真正的任务是：让老师知道这套系统怎么帮到自己、怎么最省力地用。

---

## 3. 培训目标

### 3.1 理解系统定位
让老师知道系统是辅助工具，不是替代教学判断的"电子班主任"。

### 3.2 理解角色边界
让老师明确老师端、家长端、学校端分别看什么、做什么。

### 3.3 学会核心操作
至少掌握查看、反馈、校正、标记、查看报告等基础动作。

### 3.4 降低使用阻力
让老师感受到"这东西不是来增加工作量，而是帮我少走弯路"。

### 3.5 建立反馈机制
让老师知道问题和建议怎么回流系统。

## 4. 培训对象

建议培训对象包括：

| 角色 | 关注重点 |
|------|----------|
| 班主任 | 班级整体和重点学生 |
| 学科老师 | 学科与知识点结构 |
| 年级负责人 | 年级趋势和班级差异 |
| 学校管理员 | 学校整体运行情况 |
| 项目校内联络人 | 协调和沟通 |

---

## 5. 培训原则

| 原则 | 说明 |
|------|------|
| 场景优先 | 培训应围绕老师真实使用场景，而不是功能清单背诵 |
| 简洁优先 | 老师时间紧，培训内容必须抓核心，不宜堆满系统细枝末节 |
| 演示优先 | 能现场演示的，不要只靠嘴说 |
| 操作优先 | 讲完必须让老师自己动一次，不然培训结束后系统会重新变陌生人 |
| 边界优先 | 一定要讲清楚系统能做什么、不能做什么 |

## 6. 培训内容结构建议

建议老师培训分为六部分：

1. 项目背景与目标
2. 角色定位与边界
3. 老师端核心页面介绍
4. 高频操作演示
5. 常见问题说明
6. 反馈与支持机制

## 7. 第一部分：项目背景与目标

### 7.1 教育孪生项目在做什么

- 为每个学生建立数字孪生体
- 持续追踪学生学习状态和成长轨迹
- 帮助老师和家长更早发现问题

### 7.2 为什么试点要从少量班级开始

- 验证系统主链路是否可跑通
- 收集真实使用反馈
- 逐步优化后再扩展

### 7.3 这套系统重点是帮助老师

- 更快定位问题学生
- 跟踪学生变化趋势
- 辅助教学判断，不是替代判断

### 7.4 当前阶段定位

- 重点是观察和辅助
- 不是全自动决策
- 不要求老师对所有模型结果负责

## 8. 第二部分：角色定位与边界

### 8.1 老师端的核心作用

| 功能 | 说明 |
|------|------|
| 查看班级和学生状态 | 了解整体和个体情况 |
| 看知识点和趋势 | 定位学科和知识点问题 |
| 提交反馈 | 校正系统判断、补充情境 |
| 记录干预 | 记录采取的教学动作 |
| 辅助教学判断 | 提供参考信息 |

### 8.2 老师端不负责什么

| 不负责 | 说明 |
|--------|------|
| 不替代学校管理系统 | 不是教务系统 |
| 不负责全部学生原始资料录入 | 原始资料由家长/学生/系统采集 |
| 不直接生成最终决策结论 | 结论需要老师判断 |
| 不要求老师对所有模型结果负责 | 系统是辅助工具 |

### 8.3 与家长的边界

| 老师端 | 家长端 |
|--------|--------|
| 看班级整体和个体详情 | 只看自己孩子 |
| 看学科知识点结构 | 看家庭可配合事项 |
| 提交教学反馈 | 提交家庭情况反馈 |
| 标记重点关注学生 | 查看关注点和建议 |

### 8.4 与学校端的边界

| 老师端 | 学校端 |
|--------|--------|
| 看班级和学生详情 | 看年级/学校聚合 |
| 提交个体反馈 | 看整体趋势和分布 |
| 记录干预动作 | 看干预执行率 |

## 10. 第四部分：高频操作演示

### 10.1 登录与进入班级页

```
1. 打开系统链接/APP
2. 输入账号密码登录
3. 选择所教班级
4. 进入班级总览页
```

### 10.2 查看重点学生

```
1. 在班级总览页找到"重点学生"模块
2. 点击学生姓名
3. 查看学生详情
```

### 10.3 查看某个学生的学科问题

```
1. 进入学生详情页
2. 切换到"学科"标签
3. 查看各学科掌握情况
4. 点击具体学科查看知识点详情
```

### 10.4 查看某条状态的解释依据

```
1. 在学生详情页找到状态项
2. 点击"查看依据"或"?"图标
3. 查看支撑该判断的事件和数据来源
```

### 10.5 提交老师反馈

```
1. 在学生详情页或反馈入口点击"提交反馈"
2. 选择反馈类型（状态校正/情况补充/建议反馈）
3. 填写简要说明
4. 选择关联学生/知识点
5. 提交
```

### 10.6 标记需要关注的学生

```
1. 在学生详情页找到"标记"功能
2. 选择标记类型（重点关注/需跟踪/已干预）
3. 可添加备注
4. 保存
```

### 10.7 查看报告

```
1. 进入报告页面
2. 选择周报/月报
3. 选择学生或班级
4. 查看报告内容
```

### 10.8 反馈报告是否准确

```
1. 在报告页面找到"反馈"按钮
2. 选择反馈类型（准确/部分准确/不准确）
3. 填写说明（如不准确）
4. 提交
```

## 11. 老师日常使用最小流程

建议给老师一个最小使用路径：

```
登录老师端
    ↓
查看班级总览
    ↓
查看重点学生列表
    ↓
进入学生详情页
    ↓
查看当前知识点/风险问题
    ↓
补充反馈或标记
    ↓
下次对比变化
```

**说明**：这条路径越简单，老师越可能真的用。

## 12. 老师反馈操作说明

### 12.1 可反馈内容

| 反馈类型 | 说明 |
|----------|------|
| 当前状态是否准确 | 系统判断与实际情况是否一致 |
| 某知识点映射是否合理 | 知识点归属是否正确 |
| 某提示是否过强或过弱 | 风险提示强度是否合适 |
| 学生真实课堂情况补充 | 系统看不到的课堂表现 |
| 干预动作是否已执行 | 记录采取的教学动作 |

### 12.2 反馈方式

```
1. 选择反馈类型
2. 填写简要说明（不必太长）
3. 选择关联学生/知识点
4. 提交反馈
```

### 12.3 原则

反馈入口必须短，不要把老师逼成写周记。

## 13. 老师标记与协作操作说明

### 13.1 标记重点关注学生

```
1. 进入学生详情页
2. 点击"标记"按钮
3. 选择"重点关注"
4. 可添加备注说明关注原因
5. 保存
```

### 13.2 标记当前教学重点知识点

```
1. 进入学科问题页
2. 找到目标知识点
3. 点击"标记为教学重点"
4. 保存
```

### 13.3 标记某条解释需要复核

```
1. 找到需要复核的解释项
2. 点击"标记复核"
3. 填写复核原因
4. 提交
```

### 13.4 标记建议暂不适用

```
1. 找到建议项
2. 点击"暂不适用"
3. 填写原因（可选）
4. 提交
```

### 13.5 留给班主任或其他老师的协作备注

```
1. 进入学生详情页
2. 点击"协作备注"
3. 填写备注内容
4. 选择可见范围（班主任/学科老师）
5. 保存
```

## 14. 常见使用场景示例

### 14.1 场景一：班主任看重点学生

**目标**：快速知道这周谁最需要重点关注。

**操作步骤**：
```
1. 登录系统
2. 进入班级总览页
3. 查看"重点学生"列表
4. 逐个查看学生详情
5. 标记已关注的学生
6. 下次查看变化
```

### 14.2 场景二：学科老师看知识点分布

**目标**：知道哪个知识点班级里问题最集中。

**操作步骤**：
```
1. 登录系统
2. 进入学科问题页
3. 查看知识点薄弱热区
4. 标记教学重点
5. 调整教学计划
```

### 14.3 场景三：老师修正系统判断

**目标**：避免系统把模糊推断当事实一直传下去。

**操作步骤**：
```
1. 发现系统判断与实际情况不符
2. 进入学生详情页
3. 点击"提交反馈"
4. 选择"状态校正"
5. 填写实际情况说明
6. 提交
```

### 14.4 场景四：老师看报告后补充说明

**目标**：让家长端输出更贴近真实情况。

**操作步骤**：
```
1. 查看生成的周报/月报
2. 发现需要补充的情况
3. 点击"反馈"
4. 填写补充说明
5. 提交后系统会更新报告
```

## 15. 常见问题说明

### 15.1 为什么有些学生没有明显结论

**原因**：

| 原因 | 说明 |
|------|------|
| 数据不足 | 输入数据量不够 |
| 事件不够 | 学习事件积累不足 |
| 当前阶段只做低强度判断 | 系统保守策略 |

**应对**：鼓励学生/家长增加数据输入，持续观察。

### 15.2 为什么系统提示和老师感觉不完全一样

**原因**：

| 原因 | 说明 |
|------|------|
| 系统主要基于当前数据和事件链 | 数据来源有限 |
| 老师掌握了更多现场情境 | 老师有更全面信息 |
| 这正是需要老师反馈的地方 | 反馈可帮助系统改进 |

**应对**：提交反馈，补充系统看不到的情境信息。

### 15.3 为什么有些知识点看起来"判断很保守"

**原因**：

| 原因 | 说明 |
|------|------|
| 系统在低数据完整性下会降低结论强度 | 防止胡说八道 |
| 这是故意的，不是它胆小 | 设计原则 |
| 在防胡说八道 | 控制风险 |

**应对**：理解这是保护机制，增加数据输入可提高判断强度。

### 15.4 为什么有些建议不直接给家长

**原因**：

| 原因 | 说明 |
|------|------|
| 有些建议需要老师或后台先判断 | 需要专业判断 |
| 家长端只展示低风险、可沟通内容 | 控制风险 |

**应对**：老师可查看完整建议，酌情与家长沟通。

### 15.5 系统会不会替代老师的判断

**回答**：不会。

| 说明 |
|------|
| 系统是辅助工具，不是决策主体 |
| 最终判断权在老师手中 |
| 系统提供的是参考信息，不是裁决 |

### 15.6 数据从哪里来

**来源**：

| 来源 | 说明 |
|------|------|
| 家长输入 | 作业、学习情况反馈 |
| 学校扫描 | 试卷、作业扫描件 |
| 老师录入 | 成绩、评语等 |
| 系统自动采集 | 学习行为记录（如适用） |

## 16. 使用边界说明

老师培训必须明确以下边界：

| 边界 | 说明 |
|------|------|
| 系统结果不是教学裁决 | 仅供参考 |
| 系统提示需要结合老师真实观察 | 不能盲从 |
| 老师反馈会影响系统后续判断 | 反馈有价值 |
| 高风险判断不会直接强推给家长 | 风险控制 |
| 当前阶段重点是观察和辅助 | 不是全自动决策 |

## 24. 结论

老师培训与操作手册的核心，不是让老师记住系统里有多少功能，而是让老师在最短时间内形成一句感觉：

**"这个东西我能用，而且它确实能帮我省一点判断成本。"**

做到这一点，试点才算真正开门。否则系统再完整，也只会在账号开通列表里活着。
