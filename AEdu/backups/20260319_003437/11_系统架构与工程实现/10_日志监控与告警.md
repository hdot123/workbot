---

## 1. 文档目的

本文档用于定义教育孪生项目中的日志、监控和告警体系，包括日志分层、关键指标、监控范围、告警规则、问题定位流程与运营协同要求。

本文档重点回答以下问题：

1. 系统要记录哪些日志
2. 监控到底要盯哪些指标
3. 什么情况下应该触发告警
4. 如何让输入、图谱、报告、推演这些链路真正可观测
5. 如何避免系统出问题时只能靠"猜哪里坏了"

---

## 2. 日志监控与告警定义

日志监控与告警，是指通过结构化日志、运行指标、链路追踪、状态看板和阈值触发机制，对系统运行情况进行持续观察、异常发现和快速响应的体系。

在本项目中，它的作用不是"留个记录"，而是为了回答：

- 现在系统是否健康
- 哪条链路变慢了
- 哪类任务失败了
- 哪些学校/租户受影响
- 应该谁来处理

---

## 3. 建设目标

### 3.1 让关键链路可见
输入、事件、图谱、报告、推演、前台查询等都要能看见状态。

### 3.2 让异常能被尽快发现
系统不应靠用户先报错，平台后知后觉。

### 3.3 让问题能快速定位
出了问题后要能快速收缩范围，而不是全面翻日志海。

### 3.4 让运营和技术形成闭环
日志和告警不只服务技术，也服务试点运营与复盘。

### 3.5 控制监控成本
监控要抓关键，不做"什么都记、什么都看不懂"。

---

## 4. 设计原则

### 4.1 关键链路优先
先监控核心业务链，再补边缘链路。

### 4.2 结构化优先
日志必须尽量结构化，少依赖自由文本。

### 4.3 指标与日志配合
指标发现异常，日志定位问题，追踪还原路径。

### 4.4 告警克制
告警过多会让团队学会忽略告警。

### 4.5 租户与链路可分维度观察
要能知道问题影响的是全局、某学校，还是某类任务。

## 5. 监控对象范围

建议至少覆盖以下对象：

| 对象 | 说明 |
|------|------|
| API 与前台请求 | 前台访问和接口调用 |
| 输入接入链路 | 微信/钉钉/扫描/批量导入 |
| 事件生成链路 | 学习事件生成和清洗 |
| StudentTwinAgent 更新链路 | 状态更新和快照刷新 |
| 图谱更新与检索链路 | 图谱写入和 GraphRAG 检索 |
| 报告生成链路 | 周报/月报生成 |
| 推演任务链路 | 风险预测和干预模拟 |
| 任务调度与队列 | 异步任务执行和队列状态 |
| 基础设施资源 | CPU、内存、磁盘、网络 |
| 租户与学校级运行情况 | 按租户/学校维度的运行指标 |

## 6. 日志分类建议

建议日志至少分为五类：

| 日志类型 | 说明 | 保留期限 |
|----------|------|----------|
| 访问日志 | API 和前台访问记录 | 30 天 |
| 业务日志 | 关键业务动作记录 | 90 天 |
| 任务日志 | 异步任务全生命周期 | 90 天 |
| 异常日志 | 失败、超时、重试、死信 | 180 天 |
| 审计日志 | 高价值变更和敏感操作 | 长期 |

## 7. 访问日志

### 7.1 作用
记录 API 和前台访问情况。

### 7.2 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| request_id | string | 请求唯一标识 |
| user_id | string/null | 用户 ID |
| role | string | 角色 (parent/teacher/school/admin) |
| tenant_id | string | 租户 ID |
| path | string | 请求路径 |
| method | string | HTTP 方法 |
| status_code | int | 响应状态码 |
| latency_ms | int | 响应耗时 (毫秒) |
| client_type | string | 客户端类型 |
| timestamp | datetime | 请求时间 |

### 7.3 用途
- 看请求量
- 看错误率
- 看接口耗时
- 看角色使用情况

---

## 8. 业务日志

### 8.1 作用
记录关键业务动作。

### 8.2 典型内容

| 业务动作 | 日志内容 |
|----------|----------|
| 输入接收成功 | raw_input_id, source_type, student_id |
| 事件生成完成 | event_id, event_type, student_id |
| 状态更新完成 | student_id, updated_fields, version |
| 图谱写入完成 | affected_objects, relation_count |
| 报告生成完成 | report_id, report_type, student_id |
| 推演任务结束 | task_id, result_summary, duration_ms |

### 8.3 原则
业务日志关注"业务链条发生了什么"。

---

## 9. 任务日志

### 9.1 作用
记录异步任务全生命周期。

### 9.2 典型字段

| 字段 | 类型 | 说明 |
|------|------|------|
| task_id | string | 任务唯一标识 |
| task_type | string | 任务类型 |
| status | string | 当前状态 |
| retry_count | int | 已重试次数 |
| queue_name | string | 队列名称 |
| worker_id | string | 执行节点 |
| started_at | datetime | 开始执行时间 |
| finished_at | datetime | 结束时间 |
| duration_ms | int | 执行耗时 |

### 9.3 用途
定位异步链路问题的首要入口。

---

## 10. 异常日志

### 10.1 作用
记录失败、超时、重试、死信和关键异常。

### 10.2 建议内容

| 字段 | 类型 | 说明 |
|------|------|------|
| error_code | string | 错误码 |
| error_message | string | 错误信息 |
| stack_trace | string | 堆栈跟踪（内部） |
| task_id | string/null | 任务 ID |
| request_id | string/null | 请求 ID |
| affected_object | string/null | 受影响对象 |
| retryable_flag | boolean | 是否可重试 |
| tenant_id | string | 租户 ID |
| timestamp | datetime | 异常时间 |

### 10.3 原则
异常日志必须携带足够上下文，否则只是高分贝抱怨。

---

## 11. 审计日志

### 11.1 作用
记录高价值变更和敏感操作。

### 11.2 典型场景

| 场景 | 说明 |
|------|------|
| 人工修正学生绑定 | 学生 - 班级/教材绑定变更 |
| 人工校准状态 | 学生状态人工修正 |
| 规则版本切换 | 教材版本或规则版本变更 |
| 回滚任务执行 | 图谱或状态回滚 |
| 批量导入或批量修复 | 批量数据处理操作 |

### 11.3 原则
审计日志强调"谁做了什么"，不是普通运行日志替代品。

## 21. 告警分级建议

建议告警至少分为三级：

### 21.1 P0 紧急告警

会影响主链路或多个租户核心使用。

| 示例 | 说明 |
|------|------|
| 输入主链路大面积失败 | 影响数据输入 |
| 图谱更新整体中断 | 影响记忆更新 |
| 推演服务不可用 | 影响后台推演 |
| 高优先级任务严重积压 | 影响核心链路 |

### 21.2 P1 高优先级告警

影响单条关键链路或部分学校/核心功能。

| 示例 | 说明 |
|------|------|
| OCR 成功率持续大幅下降 | 影响输入质量 |
| 周报生成失败率升高 | 影响报告输出 |
| GraphRAG 查询超时显著升高 | 影响检索性能 |

### 21.3 P2 一般告警

需要观察与排查，但不立即阻断业务。

| 示例 | 说明 |
|------|------|
| 某类低优先级任务积压 | 需要关注 |
| 某学校数据完整性异常下降 | 单学校问题 |
| 某些接口响应变慢 | 性能下降 |

## 22. 告警触发原则

### 22.1 阈值 + 持续时间
避免因为瞬时抖动频繁告警。

| 原则 | 说明 |
|------|------|
| 持续超过阈值时间 | 如持续 5 分钟超阈值 |
| 连续多次触发 | 如连续 3 次检测超阈值 |

### 22.2 绝对值 + 变化率结合
某些指标看绝对值，某些更要看突变。

| 指标类型 | 关注点 |
|----------|--------|
| 成功率类 | 绝对值 |
| 请求量类 | 变化率 |
| 耗时类 | 绝对值 + 变化率 |

### 22.3 链路优先
优先为核心链路设置稳定告警。

### 22.4 降噪
同类告警可聚合，避免刷屏。

## 24. 问题定位链路建议

建议按以下顺序定位问题：

```text
先看告警面板
    ↓
定位受影响链路
    ↓
看指标突变点
    ↓
查对应任务/请求日志
    ↓
缩小到服务/队列/对象
    ↓
确认影响范围
    ↓
执行修复或降级
```

---

## 25. 可观测性最小链路要求

对于每条关键业务链，建议至少具备：

| 要求 | 说明 |
|------|------|
| request_id / task_id | 链路追踪标识 |
| 关键阶段状态 | 各阶段执行状态 |
| 开始时间/结束时间 | 时间戳记录 |
| 成功或失败标记 | 执行结果标记 |
| 错误码 | 失败时的错误码 |
| 主要对象引用 | 学生/学校/任务等对象 |
| tenant_id / school_id | 租户和学校维度 |

没有这些，排障时就会进入经典状态：到处都有日志，但没有一条真正有用。

## 27. 多租户监控建议

### 27.1 目标
区分是全局问题，还是某学校局部问题。

### 27.2 建议维度

| 维度 | 说明 |
|------|------|
| tenant_id | 租户维度 |
| school_id | 学校维度 |
| region_id | 地区维度 |
| task_type | 任务类型维度 |
| role_type | 角色类型维度 |

### 27.3 原则
教育项目很容易出现"全局看没事，某所学校已经很痛苦"的情况，所以必须支持租户维度下钻。

## 28. 日志留存与成本控制建议

### 28.1 原则

| 原则 | 说明 |
|------|------|
| 核心日志保留更久 | 业务关键日志长期保留 |
| 高频明细日志做分层存储 | 热/温/冷分层 |
| 审计日志长期保留 | 审计合规要求 |
| 低价值噪声日志及时归档或清理 | 控制成本 |

### 28.2 分类建议

| 层级 | 保留期限 | 说明 |
|------|----------|------|
| 热日志 | 7-30 天 | 高频查询日志 |
| 温日志 | 30-180 天 | 阶段性排障与复盘 |
| 冷日志 | 180 天以上 | 长期归档与审计 |

## 29. 与运营复盘的关系

日志监控与告警不应只服务技术排障，还应反哺：

| 用途 | 说明 |
|------|------|
| 试点学校复盘 | 学校运行情况复盘 |
| 家长/老师活跃度分析 | 用户使用情况分析 |
| 高风险问题识别 | 风险问题早期发现 |
| 异常输入高发原因 | 输入质量问题根因分析 |
| 任务失败模式总结 | 任务失败模式归纳 |

| 风险 | 说明 |

## 35. 结论

日志监控与告警的核心价值，不是让系统留下更多痕迹，而是让团队在系统复杂度上升后，仍然能够看得见、找得到、修得动。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 关键链路先可观测 | 核心业务链优先监控 |
| 日志结构化 | 日志必须结构化 |
| 指标和日志联动 | 指标发现异常，日志定位问题 |
| 告警克制但有效 | 告警不宜过多但必须有效 |
| 支持租户和学校维度定位 | 支持多维度下钻 |

否则系统一旦进入试点真实场景，最先崩掉的通常不是服务，而是排障信心。
