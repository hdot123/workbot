# 政策更新与版本管理

> 文档编号：RULE-003  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：规则与外部数据负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中政策类规则的更新机制、版本管理方式、变更评估流程、发布规范与回滚策略。

本文档重点回答以下问题：

1. 政策和规则变化时系统如何更新
2. 更新后的规则如何做版本管理
3. 变更如何影响知识底座、推演层和前台解释口径
4. 如何避免新旧规则混用
5. 如何在政策更新出错时快速回滚

---

## 2. 政策更新与版本管理定义

政策更新与版本管理，是指针对地区考试规则、选科要求、升学制度、教材适配口径、高考录取口径等外部规则变化，建立标准化的变更识别、结构化更新、影响评估、版本发布和回滚控制机制。

它的核心任务不是"保存一堆旧文件"，而是确保系统在规则变化时：

- 能及时更新
- 能明确区分新旧版本
- 能知道影响范围
- 能保持前后台口径一致
- 能在出错时退回稳定版本

---

## 3. 管理目标

### 3.1 保持规则时效性
系统不能长期使用过时政策。

### 3.2 保持规则一致性
同一时间点内，不同模块应尽量使用同一有效版本。

### 3.3 保持可追溯性
每一条规则都应能回到来源、时间和版本。

### 3.4 控制变更风险
规则变更可能影响推演、解释、志愿评估和报告口径，必须先评估再发布。

### 3.5 支持回滚
当新规则接入错误或解释歧义过大时，应快速恢复。

---

## 4. 适用范围

本机制适用于以下对象：

1. 地区考试规则
2. 选科规则
3. 升学与志愿填报规则
4. 教材适配规则
5. 分值与排名口径规则
6. 政策解释与补充说明

---

## 5. 版本管理原则

### 5.1 一条规则必须有版本
没有版本号的规则，等于未来找不到责任人。

### 5.2 规则生效时间必须明确
不能只写"最新版"，必须写清何时生效。

### 5.3 旧版本不能直接删除
旧版本必须归档并可追溯。

### 5.4 变更必须带说明
更新不只是改字段，还要说明为什么改、影响什么。

### 5.5 发布前必须评估影响
先更新规则再去想会不会把系统搞乱，这种流程十分英勇，但不建议实施。

## 6. 版本对象定义

建议对每个规则版本至少定义以下元数据：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| rule_version_id | string | 规则版本唯一标识 |
| rule_id | string | 规则主标识 |
| region_id | string | 适用地区 |
| rule_type | string | 规则类型 |
| version | string | 版本号 |
| source_ref | string | 来源引用 |
| source_date | date/null | 来源日期 |
| valid_from | date | 生效开始日期 |
| valid_to | date/null | 生效结束日期 |
| status | string | draft/active/deprecated/archived |
| change_summary | string | 变更摘要 |
| impact_scope | object | 影响范围 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

## 7. 版本号规则建议

建议采用三段式版本号：

| 版本类型 | 说明 | 示例 |
|----------|------|------|
| 主版本 | 重大制度变化 | V1.0.0 → V2.0.0 |
| 次版本 | 规则补充或新增结构 | V1.0.0 → V1.1.0 |
| 修订版本 | 纠错与文案修正 | V1.0.0 → V1.0.1 |

示例：
- `V1.0.0`：首个正式版
- `V1.1.0`：新增专业选科要求结构
- `V1.1.1`：修正文案和来源引用错误

---

## 8. 政策更新类型

建议将政策更新分为三类：

### 8.1 主变更

适用于：
- 制度规则重大变化
- 批次结构变化
- 选科规则重大调整
- 志愿填报口径重大变化

**版本号变更**：主版本 +1

### 8.2 次变更

适用于：
- 新增补充说明
- 新增地区特例
- 数据结构扩展
- 非主逻辑变化

**版本号变更**：次版本 +1

### 8.3 修订变更

适用于：
- 来源补充
- 文字修正
- 字段纠错
- 不影响主判断逻辑的修正

**版本号变更**：修订版本 +1

## 9. 政策更新流程

建议标准流程如下：

```
发现政策变化
    ↓
收集原始政策来源
    ↓
确认地区与适用范围
    ↓
抽取结构化变化内容
    ↓
生成候选新版本
    ↓
评估影响范围
    ↓
人工审核
    ↓
发布新版本
    ↓
通知下游模块更新
    ↓
观察上线效果
```

### 9.1 各阶段说明

| 阶段 | 工作内容 | 输出 |
|------|----------|------|
| 发现政策变化 | 监控官方发布、学校通知等 | 政策变化通知 |
| 收集原始来源 | 收集政策文件、公告原文 | 原始政策文档 |
| 确认适用范围 | 明确适用地区、学段、学校类型 | 适用范围说明 |
| 抽取变化内容 | 对比新旧规则，提取变化点 | 变化内容清单 |
| 生成候选版本 | 创建新版本规则记录 | 候选版本记录 |
| 评估影响范围 | 评估对系统各模块的影响 | 影响评估报告 |
| 人工审核 | 规则负责人审核确认 | 审核通过记录 |
| 发布新版本 | 更新规则状态为 active | 新版本发布 |
| 通知下游模块 | 通知相关模块更新引用 | 通知记录 |
| 观察上线效果 | 监控规则使用情况 | 效果观察记录 |

## 10. 变更识别机制

### 10.1 变更来源

| 来源类型 | 说明 |
|----------|------|
| 官方政策文件 | 教育部门发布的正式文件 |
| 教育主管部门公告 | 官方公告、通知 |
| 学校/地区执行口径更新 | 学校或地区的具体执行要求 |
| 已接入规则的补充说明 | 对已有规则的补充和澄清 |

### 10.2 识别内容

| 识别项 | 说明 |
|--------|------|
| 规则条目新增 | 新增规则条款 |
| 规则条目删除 | 删除原有规则条款 |
| 适用条件变化 | 规则适用范围变化 |
| 时间边界变化 | 生效时间、截止时间变化 |
| 口径解释变化 | 规则解释口径变化 |

### 10.3 原则

只要影响系统判断口径，就算真正变更，不要把"只是补充说明"想得太轻松，有些补充说明专门用来改变命运。

---

## 11. 影响范围评估

规则更新前应至少评估以下影响范围：

| 模块 | 评估内容 |
|------|----------|
| 知识底座层 | 教材适配、章节结构是否受影响 |
| StudentTwinAgent | 初始化与绑定逻辑是否受影响 |
| 学习事件解释 | 事件映射和解释逻辑是否受影响 |
| 后台推演层 | 推演边界和约束是否受影响 |
| 志愿填报推演 | 志愿评估逻辑是否受影响 |
| 前台观察层 | 报告模板和解释口径是否受影响 |

---

## 12. 影响等级建议

建议将影响分为三档：

### 12.1 高影响

| 特征 | 说明 |
|------|------|
| 直接影响决策逻辑 | 推演结果可能改变 |
| 直接影响推演边界 | 推演约束条件变化 |
| 直接影响志愿评估结果 | 志愿填报建议可能改变 |

**处理方式**：必须人工审核，发布前需充分测试

### 12.2 中影响

| 特征 | 说明 |
|------|------|
| 影响解释口径 | 报告解释需要调整 |
| 影响部分规则判断 | 部分判断逻辑变化 |
| 影响地区适配关系 | 地区绑定关系变化 |

**处理方式**：需要审核，发布后需观察反馈

### 12.3 低影响

| 特征 | 说明 |
|------|------|
| 文字修订 | 文案修正 |
| 来源补充 | 来源信息补充 |
| 不影响判断主链路的元数据修正 | 元数据更新 |

**处理方式**：快速审核，可直接发布

## 13. 发布策略

### 13.1 Draft 阶段

| 特征 | 说明 |
|------|------|
| 候选规则版本 | 尚未正式生效 |
| 仅供内部审核与测试 | 不对外暴露 |
| 可随时修改 | 测试阶段可调整 |

### 13.2 Active 阶段

| 特征 | 说明 |
|------|------|
| 已正式生效 | 可供系统使用 |
| 下游模块按此版本运行 | 默认引用版本 |
| 对外可见 | 前台可使用 |

### 13.3 Deprecated 阶段

| 特征 | 说明 |
|------|------|
| 已被新版本替代 | 不再是默认版本 |
| 不再作为默认规则 | 新请求不使用 |
| 保留历史兼容查询 | 历史数据仍可引用 |

### 13.4 Archived 阶段

| 特征 | 说明 |
|------|------|
| 只做长期归档 | 永久保存 |
| 不参与当前业务逻辑 | 不再被引用 |
| 仅用于审计追溯 | 历史追溯使用 |

---

## 14. 发布与生效规则

### 14.1 原则

一个时间窗口内，同地区同规则类型应只有一个默认 active 版本。

### 14.2 生效方式

| 生效方式 | 说明 | 适用场景 |
|----------|------|----------|
| 立即生效 | 发布后立即使用 | 紧急修复、低影响变更 |
| 定时生效 | 设定未来生效时间 | 计划性政策变更 |
| 兼容过渡期生效 | 新旧版本并行一段时间 | 重大规则变化 |

### 14.3 兼容期

当规则变化较大时，可保留短期双版本并明确：

| 明确项 | 说明 |
|--------|------|
| 哪些模块仍使用旧版本 | 明确兼容范围 |
| 哪些模块切换到新版本 | 明确切换范围 |
| 切换截止日期 | 明确过渡期结束时间 |

## 15. 回滚机制

### 15.1 回滚触发场景

| 场景 | 说明 |
|------|------|
| 新规则结构错误 | 字段或结构定义错误 |
| 来源引用错误 | 来源文件或日期错误 |
| 下游推演出现异常 | 推演结果异常 |
| 解释口径明显冲突 | 前后台解释不一致 |
| 地区适用范围配置错误 | 适用地区配置错误 |

### 15.2 回滚原则

| 原则 | 说明 |
|------|------|
| 优先回退到上一稳定 active 版本 | 保证系统可用 |
| 保留错误版本记录 | 记录问题版本 |
| 记录回滚原因与影响 | 便于后续分析 |
| 通知相关模块和角色 | 确保各方知情 |

### 15.3 回滚后动作

| 动作 | 说明 |
|------|------|
| 重新评估新版本 | 分析问题原因 |
| 修正问题 | 修复错误 |
| 重新发布候选版本 | 再次尝试发布 |

## 16. 与系统模块的联动规则

### 16.1 知识底座层

若教材适配或地区规则变化，应同步更新相关映射。

**联动动作**：
- 检查教材版本映射
- 更新章节树适配
- 更新知识点映射

### 16.2 推演层

若规则影响推演边界，应重新标记相关推演结果适用性。

**联动动作**：
- 标记受影响推演结果
- 重新评估推演约束
- 更新推演规则引用

### 16.3 前台观察层

若规则影响解释口径，应同步更新说明文本和报告模板。

**联动动作**：
- 更新报告模板
- 更新解释文本
- 更新口径说明

### 16.4 学校接入与运营层

若规则影响试点学校使用方式，应通知运营和实施团队。

**联动动作**：
- 通知运营团队
- 更新学校接入文档
- 必要时重新培训

## 22. 风险与边界

### 22.1 不能忽视变更影响

规则变更可能影响多个模块，必须充分评估。

### 22.2 不能忽视版本兼容性

新旧版本切换时需注意兼容性问题。

### 22.3 不能忽视回滚准备

发布前必须准备好回滚方案。

### 22.4 不能忽视通知机制

相关模块和角色必须及时获知变更。

### 22.5 不能忽视审计记录

所有变更必须完整记录以便追溯。

## 23. 规则维护责任

### 23.1 规则负责人

| 角色 | 职责 |
|------|------|
| 规则维护人 | 负责规则的接入、更新、版本管理 |
| 规则审核人 | 负责规则的审核和发布审批 |
| 规则使用方 | 反馈规则使用中的问题 |

### 23.2 维护周期

| 周期 | 工作内容 |
|------|----------|
| 日常 | 规则更新、版本管理、问题修复 |
| 季度 | 规则完整性检查、版本审查 |
| 年度 | 规则全面审查、历史版本归档 |
| 政策发布后 | 及时跟进政策变化并更新 |

---

## 24. 结论

政策更新与版本管理的核心，不是让规则越来越多，而是让规则变化在系统里可控、可追踪、可回退。

它必须坚持：

- **先识别变更** - 明确政策变化内容
- **再评估影响** - 评估对各模块的影响
- **再发布版本** - 按流程发布新版本
- **必要时可回滚** - 有问题能快速恢复

否则系统会出现最讨厌的一种混乱：每个模块都觉得自己用的是"最新版"，但大家实际上活在不同世界线里。
