> 文档编号：GRAPH-008  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：记忆与图谱引擎负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中知识图谱的核心实体类型、关系类型、边界规则和建模约束，作为图谱建模、写入、查询和 GraphRAG 检索的基础规范。

本文档重点回答以下问题：

1. 图谱里到底有哪些实体
2. 实体之间允许建立哪些关系
3. 哪些对象适合做实体，哪些不适合
4. 如何防止图谱建模失控
5. 如何让图谱既能支持查询，又能支持解释和推演

---

## 2. 知识图谱定义

知识图谱，是教育孪生项目中用于表达对象、关系和长期上下文结构的核心数据层。

它的作用不是替代所有数据库，也不是把所有文本都图谱化，而是聚焦回答：

- 系统里有哪些关键对象
- 它们之间如何连接
- 哪些连接是稳定结构
- 这些结构如何支撑学生理解、解释和推演

图谱不是"什么都能连"的宇宙网，而是一张有边界、有重点的教育结构图。

---

## 3. 建设目标

### 3.1 明确对象边界
知道哪些对象是图谱实体，哪些只是属性或日志。

### 3.2 明确关系边界
知道哪些关系值得建边，哪些只应作为普通字段。

### 3.3 支撑多层场景
同时服务 StudentTwinAgent、GraphRAG、观察层和推演层。

### 3.4 控制建模复杂度
避免图谱一开始就长成不可维护的怪兽。

### 3.5 支撑长期演进
允许后续扩展学科、地区、学校、推演对象，而不推翻当前结构。

---

## 4. 设计原则

### 4.1 高价值对象优先
先建对解释、检索、推演有明显价值的实体。

### 4.2 关系应有业务意义
不是所有"有点关系"的对象都值得连边。

### 4.3 实体与属性分离
能作为对象独立存在的才考虑做实体，其余优先做属性。

### 4.4 结构稳定优先
优先建相对稳定的对象和关系。

### 4.5 渐进扩展
MVP 阶段先做最小实体集，不一口气上全宇宙。

---

## 5. 实体层级建议

建议实体按四层组织：

| 层级 | 说明 | 典型实体 |
|------|------|----------|
| 基础主体层 | 系统核心参与对象 | Student, Parent, Teacher, Class, School, Region |
| 知识底座层 | 教育知识结构对象 | CurriculumVersion, Chapter, KnowledgePoint, AbilityPoint |
| 事件与状态层 | 动态变化和状态对象 | LearningEvent, StateSnapshot, RiskSignal |
| 规则与资源层 | 规则和外部资源对象 | PolicyRule, AdmissionRule, Major |

---

## 6. 基础主体层实体

建议至少包含：

| 实体 | 编码 | 说明 |
|------|------|------|
| 学生 | Student | 学生主体，是学生数字孪生的核心对象 |
| 家长 | Parent | 家长主体，用于表达家校关系与输入来源 |
| 老师 | Teacher | 老师主体，用于反馈、干预和班级结构关系 |
| 班级 | Class | 班级主体，用于学生归属和班级视图 |
| 学校 | School | 学校主体，用于组织边界和多租户隔离 |
| 地区 | Region | 地区主体，用于规则绑定和教材适配 |

### 6.1 Student 核心属性

| 属性 | 类型 | 说明 |
|------|------|------|
| student_id | string | 学生唯一标识 |
| name | string | 姓名（可脱敏） |
| grade_level | string | 年级 |
| enrollment_date | date | 入学时间 |
| status | string | 状态 (active/graduated/transferred) |

## 10. 关系类型总体建议

建议至少分为以下六类：

| 关系类别 | 说明 | 典型关系 |
|----------|------|----------|
| 归属关系 | 表达从属和归属 | belongs_to, located_in, teaches |
| 支撑关系 | 表达支撑和承载 | supports, contains, reflects |
| 依赖关系 | 表达前置依赖 | depends_on, requires |
| 引用关系 | 表达引用和出处 | refers_to, cites |
| 状态关系 | 表达当前状态 | has_state, has_risk_signal, focuses_on |
| 行为与事件关系 | 表达行为和事件链路 | submitted, created, affects, targets |

---

## 11. 归属关系

典型关系包括：

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|
| 属于班级 | belongs_to | Student | Class | 学生属于班级 |
| 属于学校 | belongs_to | Class | School | 班级属于学校 |
| 位于地区 | located_in | Student | Region | 学生位于地区 |
| 教授班级 | teaches | Teacher | Class | 老师教授班级 |
| 适用地区 | applicable_to | CurriculumVersion | Region | 教材版本适用地区 |

这类关系较稳定，优先建边。

---

## 12. 支撑关系

典型关系包括：

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|
| 支撑能力 | supports | KnowledgePoint | AbilityPoint | 知识点支撑能力点 |
| 包含知识点 | contains | Chapter | KnowledgePoint | 章节包含知识点 |
| 支撑能力 | supports | ExperimentActivity | AbilityPoint | 实验活动支撑能力点 |
| 反映能力 | reflects | ExerciseType | AbilityPoint | 题型反映能力点 |

这类关系是解释层和推演层的重要支架。

---

## 13. 依赖关系

典型关系包括：

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|
| 依赖知识点 | depends_on | KnowledgePoint | KnowledgePoint | 知识点依赖前置知识点 |
| 依赖能力 | depends_on | AbilityPoint | AbilityPoint | 能力点依赖前置能力点 |
| 关联误区 | linked_to | KnowledgePoint | Misconception | 知识点关联误区 |
| 关联误区 | linked_to | ExerciseType | Misconception | 题型关联误区 |

依赖关系通常有方向和强度。

---

## 14. 引用关系

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|

## 15. 状态关系

典型关系包括：

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|
| 拥有状态 | has_state | Student | StateSnapshot | 学生拥有状态快照 |

## 16. 行为与事件关系

典型关系包括：

| 关系 | 编码 | 源实体 | 目标实体 | 说明 |
|------|------|--------|----------|------|
| 提交事件 | submitted | Parent | LearningEvent | 家长提交学习事件 |
| 创建反馈 | created | Teacher | FeedbackRecord | 老师创建反馈记录 |
| 影响学生 | affects | LearningEvent | Student | 学习事件影响学生 |
| 针对学生 | targets | InterventionAction | Student | 干预动作针对学生 |
| 关联知识点 | related_to | InterventionAction | KnowledgePoint | 干预动作关联知识点 |

这类关系更偏动态链路。

## 17. 实体属性与关系边界

### 17.1 适合作属性的内容

| 类型 | 示例 |
|------|------|
| 简单名称 | name, code |
| 编码 | id, code |
| 时间戳 | created_at, updated_at |
| 状态值 | status, state |
| 分数 | score, rating |
| 说明文字 | description, notes |

### 17.2 适合做实体的内容

| 条件 | 说明 |
|------|------|
| 能被独立引用 | 可被其他对象引用 |
| 能与多个对象相连 | 有多个关系连接 |
| 能跨场景复用 | 在多个场景中使用 |
| 有生命周期 | 有创建、更新、失效过程 |
| 有解释价值 | 对解释和推演有价值 |

### 17.3 原则

不要因为图谱很酷，就把所有字段都做成实体。

## 20. 命名规则建议

### 20.1 实体类型命名

建议统一使用 PascalCase：

| 正确 | 错误 |
|------|------|
| Student | student |
| KnowledgePoint | knowledge_point |
| AbilityPoint | ability_point |

### 20.2 关系类型命名

建议统一使用 snake_case 或固定动词短语：

| 正确 | 错误 |
|------|------|
| belongs_to | BelongsTo |
| depends_on | depend_on |
| supports | Supports |
| refers_to | refer_to |
| has_state | hasState |

### 20.3 原则

全系统统一，不要同类关系出现多个别名。

## 21. 建模边界建议

以下内容原则上不建议第一阶段直接图谱化：

| 内容 | 原因 |
|------|------|
| 高频低价值日志明细 | 适合日志系统，不适合图谱 |
| UI 展示临时状态 | 临时状态，无长期价值 |
| 大段未结构化全文 | 适合文档存储，不适合图谱 |
| 仅一次性使用的中间变量 | 无复用价值 |
| 没有解释价值的属性对象 | 增加复杂度无收益 |

---

## 22. 与 StudentTwinAgent 的关系

StudentTwinAgent 不是图谱本体，但会深度使用图谱：

| 使用方式 | 说明 |
|----------|------|
| 读取基础关系和知识结构 | 获取学生绑定的教材、班级等 |
| 写入事件与状态变化 | 学习事件写入图谱 |
| 获取解释上下文 | GraphRAG 检索上下文 |
| 支撑推演准备层 | 提供结构关系支持 |

图谱更像长期结构层，StudentTwinAgent 更像当前个体运行层。

---

## 23. 与 GraphRAG 的关系

GraphRAG 检索会优先基于图谱实体和关系展开：

| 检索对象 | 说明 |
|----------|------|
| 当前事实实体 | Student, StateSnapshot, RiskSignal |
| 相关知识与能力实体 | KnowledgePoint, AbilityPoint |
| 关键事件实体 | LearningEvent, FeedbackRecord |
| 结构性关系边 | depends_on, supports, has_state |

图谱定义越清楚，GraphRAG 就越稳定。

---

## 24. 质量要求

| 要求 | 说明 |
|------|------|
| 实体边界清晰 | 对象应可独立被理解和引用 |
| 关系边界清晰 | 不同关系类型不能语义重叠过大 |
| 命名统一 | 命名混乱会直接拖垮查询和维护 |
| 结构可扩展 | 允许后续加实体，但不推翻当前骨架 |
| 可查询 | 图谱不是摆设，设计必须服务查询和解释 |

---

## 25. 常见错误

| 错误 | 后果 |
|------|------|
| 全都做实体 | 最后图谱会变成字段博物馆 |
| 关系类型过多 | 刚开始就定义几十种关系，维护会很痛 |
| 关系语义重叠 | supports/linked_to/related_to 混用，后面无法稳定查询 |
| 动态状态全塞成静态关系 | 会导致历史变化难以表达 |
| 不区分实体和属性 | 图谱过度膨胀，查询性能下降 |

| 周期 | 工作内容 |
|------|----------|
| 日常 | 实体关系更新、问题修复 |
| 月度 | 实体关系完整性检查 |
| 季度 | 实体关系全面审查、历史版本归档 |
| 新实体类型出现时 | 新增实体类型定义 |

---

## 30. 风险与边界

| 风险 | 说明 |
|------|------|
| 不能忽视实体边界 | 全都做实体会导致图谱过度膨胀 |
| 不能忽视关系语义 | 关系类型混用会导致查询困难 |
| 不能忽视命名统一 | 命名混乱会拖垮查询和维护 |
| 不能忽视动态状态表达 | 静态关系无法表达历史变化 |
| 不能忽视可查询性 | 图谱设计必须服务查询和解释 |

---

## 31. 结论

知识图谱实体关系定义的核心价值，是让系统从"能存很多信息"进化为"能看清对象、结构和连接方式"。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 高价值对象优先 | 先建对解释、检索、推演有明显价值的实体 |
| 关系语义清楚 | 不同关系类型不能语义重叠 |
| 实体与属性分离 | 能作为对象独立存在的才考虑做实体 |
| 结构服务查询与解释 | 设计必须服务查询和解释 |
| 先小后大 | MVP 阶段先做最小实体集 |

否则图谱最终只会长成一张结构很壮观、但没人真正敢查也没人敢改的复杂网络。
