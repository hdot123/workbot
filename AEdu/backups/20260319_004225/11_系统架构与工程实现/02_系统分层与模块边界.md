---

## 1. 文档目的

本文档用于定义教育孪生项目的系统分层方式与模块边界，明确各模块职责、输入输出关系、耦合边界与演进原则。

本文档重点回答以下问题：

1. 系统应该分成哪些层
2. 每一层负责什么，不负责什么
3. 模块之间如何协同
4. 哪些能力应集中，哪些能力应隔离
5. 如何避免职责重叠和边界漂移

---

## 2. 分层设计目标

### 2.1 保持系统结构清晰
不同能力必须有明确归属，避免逻辑四处乱长。

### 2.2 支撑主业务链路
系统应稳定支撑以下主链路：

```
知识底座 → 输入接入 → 学习事件 → StudentTwinAgent → 图谱记忆 → 前台观察 → 后台推演
```

### 2.3 降低耦合
各层应通过明确接口协同，而不是互相直接篡改内部状态。

### 2.4 支持渐进演进
MVP 可先模块化单体，后续根据压力点和边界清晰度逐步服务化。

---

## 3. 系统总体分层

建议系统分为七层：

1. 接入层
2. 事件处理层
3. 知识底座层
4. 学生孪生状态层
5. 图谱记忆层
6. 应用输出层
7. 平台支撑层

---

## 4. 接入层

### 4.1 主要职责
负责承接所有外部输入与访问请求。

### 4.2 典型输入
- 微信输入
- 钉钉输入
- 后台管理录入
- 文件上传
- 批量导入
- 外部 API 请求

### 4.3 负责内容
- 身份识别
- 租户识别
- 基础参数校验
- 请求路由
- 文件接收
- 任务投递

### 4.4 不负责内容
- 不负责复杂业务规则计算
- 不负责 StudentTwinAgent 状态更新
- 不负责图谱写入
- 不负责推演计算

### 4.5 模块建议
- API Gateway
- BFF（按角色拆分）
- Upload Service
- Auth Entry

---

## 5. 事件处理层

### 5.1 主要职责
将原始输入转化为标准学习事件。

### 5.2 负责内容
- OCR / 文档理解
- 文本解析
- 事件抽取
- 字段清洗
- 事件分类
- 学科/章节/知识点映射
- 质量评分
- 标准学习事件生成

### 5.3 不负责内容
- 不直接维护 StudentTwinAgent 当前态
- 不直接生成前台展示页面
- 不直接做复杂推演

### 5.4 模块建议
- OCR Parser
- Text Parser
- Event Engine
- Mapping Engine
- Validation Engine
- Manual Review Queue

---

## 6. 知识底座层

### 6.1 主要职责
提供静态教育世界模型和结构标准。

### 6.2 负责内容
- 地区规则
- 教材版本
- 章节树
- 知识点表
- 能力点表
- 依赖关系
- 题型映射
- 锚点映射

### 6.3 不负责内容
- 不负责学生个体状态
- 不负责原始事件处理
- 不负责前台展示逻辑

### 6.4 模块建议
- Curriculum Service
- Knowledge Base Service
- Rule Base Service
- Version Control Service

---

## 7. 学生孪生状态层

### 7.1 主要职责
维护 StudentTwinAgent 当前态和局部状态快照。

### 7.2 负责内容
- 学生基础绑定
- 当前事实状态
- 当前表现状态
- 行为节奏状态
- 风险标签
- 推演准备字段
- 状态变更审计

### 7.3 不负责内容
- 不保存全部历史事件明细
- 不直接充当长期记忆库
- 不负责前台页面拼装
- 不负责原始输入解析

### 7.4 模块建议
- Student Profile Service
- Student State Service
- State Calculation Engine
- Risk Evaluation Service
- Audit Service

---

## 8. 图谱记忆层

### 8.1 主要职责
维护长期事实、关系、事件链和时序演化轨迹。

### 8.2 负责内容
- 图谱实体与关系管理
- 学习事件图谱写入
- 历史关系链沉淀
- GraphRAG 检索
- 状态变化溯源
- 干预与结果关系记录

### 8.3 不负责内容
- 不作为 StudentTwinAgent 当前态主存储
- 不直接承担前台展示模板
- 不直接处理原始输入

### 8.4 模块建议
- Graph Storage Service
- Graph Write Service
- Graph Query Service
- Memory Retrieval Service
- Context Assembly Service

---

## 9. 应用输出层

### 9.1 主要职责
面向不同角色组织产品能力和可视化输出。

### 9.2 负责内容
- 家长端观察页
- 老师端观察页
- 学校端看板
- 周报/月报生成
- 后台推演结果展示
- 反馈入口组织

### 9.3 不负责内容
- 不直接更新底层状态
- 不直接解析原始输入
- 不直接维护知识底座

### 9.4 模块建议
- Parent App Service
- Teacher App Service
- School Dashboard Service
- Report Service
- Simulation Console

## 10. 平台支撑层

### 10.1 主要职责
提供全系统通用基础设施能力。

### 10.2 负责内容
- 认证与鉴权
- 租户管理
- 权限控制
- 文件存储
- 日志与监控
- 配置中心

## 11. 关键模块边界

建议将系统中的核心模块边界明确如下：

| 模块 | 输入 | 输出 | 边界说明 |
|------|------|------|----------|
| Ingest Gateway | 外部请求 | 原始输入任务 | 只接入，不做深业务处理 |
| Event Engine | 原始输入 | 标准学习事件 | 只负责事件生成 |
| Knowledge Base | 教材/规则数据 | 结构化知识与规则 | 只负责静态世界模型 |
| Student Twin | 学习事件/规则 | 当前状态 | 只维护当前态与状态更新 |
| Graph Memory | 学习事件/状态变化 | 关系与长期记忆 | 只维护长期图谱与上下文 |
| Observation Service | 当前状态 + 图谱 | 前台观察结果 | 只负责观察视图和解释 |
| Simulation Service | 当前状态 + 图谱 + 规则 | 推演结果 | 只负责后台推演 |
| Report Service | 观察结果/状态 | 周报月报 | 只负责报告生成 |

## 12. 层间调用关系

建议优先采用以下调用关系：

```
接入层
  ↓
事件处理层
  ↓
学生孪生状态层
  ├─→ 图谱记忆层
  ├─→ 应用输出层
  └─→ 后台推演服务（属于应用输出层中的后台能力）

知识底座层
  ├─→ 事件处理层
  ├─→ 学生孪生状态层
  ├─→ 图谱记忆层
  └─→ 推演服务

平台支撑层
  → 为所有层提供通用能力
```

### 12.1 关键原则

- 下层不应依赖上层展示逻辑
- 图谱层不应直接篡改 StudentTwinAgent 当前态
- 观察层不应直接改底层状态
- 推演层不应绕开 StudentTwinAgent 直接捏造状态

---

## 13. 模块耦合控制原则

### 13.1 单向依赖优先

尽量让依赖方向清晰，避免双向纠缠。

### 13.2 状态更新集中

StudentTwinAgent 的当前态更新应集中在状态层完成。

### 13.3 事件标准统一

所有下游尽量消费统一学习事件，而不是各自解析原始输入。

### 13.4 图谱写入集中

图谱写入逻辑应集中，不应多个模块各写各的野生关系。

---

## 14. 当前态与长期态边界

### 14.1 当前态

由 StudentTwinAgent 维护，特点：

- 轻量
- 高频读取
- 当前可用
- 面向前台和推演

### 14.2 长期态

由图谱记忆层维护，特点：

- 结构化历史
- 强关系
- 强时序
- 面向溯源和 GraphRAG

### 14.3 原则

当前态不是长期记忆，长期记忆也不是当前态缓存。两者各有职责，别混。

---

## 15. 规则计算与模型推断边界

### 15.1 规则计算

适用于：

- 明确状态分类
- 事件归类
- 风险阈值判断
- 基础进度判断
- 规则约束检查

### 15.2 模型推断

适用于：

- 模糊文本理解
- 复杂趋势判断
- 弱信号整合
- 上下文增强解释
- 后台推演补充

### 15.3 原则

规则优先兜底，模型用于增强。不能把所有业务逻辑都交给模型自由发挥，那是给未来事故攒素材。

## 16. 模块演进建议

### 16.1 MVP 阶段

采用模块化单体 + 异步任务的方式：

- 减少分布式复杂度
- 保持边界清晰
- 优先跑通主链路

### 16.2 扩展阶段

根据压力和边界清晰度拆分：

- OCR/解析服务独立
- 图谱检索服务独立
- 推演服务独立
- 报告服务独立

### 16.3 原则

先把模块边界写清，再拆服务；不是先拆成十几个服务，再回头猜每个服务该干什么。

## 17. 错误边界与异常归属

### 17.1 接入错误

归接入层处理，如：

- 鉴权失败
- 参数缺失
- 文件上传失败

### 17.2 解析错误

归事件处理层处理，如：

- OCR 失败
- 文本无法解析
- 事件结构不完整

### 17.3 状态更新错误

归学生孪生状态层处理，如：

- 状态冲突
- 字段更新失败
- 状态快照异常

### 17.4 图谱写入错误

归图谱记忆层处理，如：

- 关系写入失败
- 实体重复冲突
- 版本回滚异常

### 17.5 展示错误

归应用输出层处理，如：

- 观察视图缺字段
- 报告生成失败
- 推演结果展示异常

## 18. MVP 模块最小集

MVP 阶段建议至少包含以下模块：

- Ingest Gateway
- Event Engine
- Knowledge Base Service
- Student State Service
- Graph Memory Service（轻量版）
- Parent Observation Service
- Teacher Observation Service（基础版）
- Report Service
- Auth / Tenant / File / Task 等基础支撑

## 20. 结论

系统分层与模块边界，不是为了让架构图看起来高级，而是为了防止系统长成一锅职责互相踩踏的粥。

教育孪生项目的系统边界必须坚持：

- 接入层只接入
- 事件层只产事件
- 知识层只管静态世界
- 状态层只管当前学生态
- 图谱层只管长期记忆
- 输出层只管角色化呈现
- 支撑层只做通用能力

只有边界清楚，后续扩展、调试、审计、协作和试点落地才不会陷入"哪里都能改、哪里都说不清"的混乱局面。
