---

## 1. 文档目的

本文档用于定义教育孪生项目中老师端的展示规则，包括展示目标、信息结构、展示粒度、解释方式、风险提示边界与老师可操作入口设计。

本文档重点回答以下问题：

1. 老师端应该展示什么，不应该展示什么
2. 老师端与家长端的展示差异在哪里
3. 老师端如何兼顾"快速看懂"和"可深入定位"
4. 老师端怎样支撑教学判断、课堂干预与家校协同
5. 如何避免老师端变成"指标堆叠页面"或"复杂但不好用的后台"

---

## 2. 老师端展示定义

老师端展示，是指系统面向老师角色输出学生个体状态、班级整体分布、知识点与题型问题结构、近期变化、风险提示与可跟进事项的展示规则集合。

老师端的目标不是把后台所有中间变量直接抛出来，也不是只做一个更花的成绩分析页。

老师端真正要解决的是：

- 哪些学生值得关注
- 哪些问题值得先处理
- 问题更像知识、能力、题型还是节奏问题
- 当前判断依据是什么
- 下一步教学上可以怎么跟进

---

## 3. 展示目标

### 3.1 支撑快速定位
让老师快速知道"谁需要关注、哪类问题最集中"。

### 3.2 支撑结构化理解
不仅看结果，还能看：
- 相关知识点
- 相关能力点
- 相关题型
- 相关误区
- 近期事件依据

### 3.3 支撑教学动作
帮助老师形成：
- 课堂提醒
- 练习安排
- 个别跟进
- 家校沟通建议

### 3.4 支撑班级视角
老师端不应只看单学生，也应支持班级级别的问题分布。

### 3.5 控制认知负担
老师时间很紧，展示必须分层，不要把系统复杂性直接转嫁给老师。

---

## 4. 设计原则

### 4.1 教学决策优先
展示设计首先服务老师日常教学，不先服务技术展示欲。

### 4.2 结构优先于漂亮指标
要先回答"问题是什么、在哪里、为什么"，再谈图表。

### 4.3 个体与班级双视角
老师既需要单学生深看，也需要班级面浅看。

### 4.4 依据可见
重要判断应可查看依据，不能只给结论。

### 4.5 可操作性优先
老师端展示最好能自然导向"下一步可做什么"。

---

## 5. 老师端展示范围建议

建议老师端重点展示以下六类内容：

1. 班级总览
2. 重点学生列表
3. 单学生详情
4. 知识点/能力点/题型问题结构
5. 近期变化与风险提示
6. 教学反馈与跟进入口

---

## 6. 班级总览展示规则

### 6.1 作用
帮助老师先看整体，再下钻个体。

### 6.2 建议展示内容
- 班级整体状态分布
- 当前重点学科问题聚集
- 高关注学生数量
- 知识点问题 Top 列表
- 题型问题 Top 列表
- 本周变化摘要

### 6.3 表达原则
- 先聚合
- 后列表
- 再支持下钻

### 6.4 示例
- 本周物理"受力分析"相关问题在班级中集中上升
- 班级中有 6 名学生在图像类题型上表现不稳定
- 近期作业节奏波动较大的学生共 4 人

---

## 7. 重点学生列表规则

### 7.1 作用
帮助老师从班级视图快速找到当前最值得看的学生。

### 7.2 建议展示字段
- 学生姓名/匿名名
- 当前关注等级
- 当前重点学科
- 当前重点问题
- 最近变化方向
- 最近老师是否已跟进

### 7.3 原则
重点学生列表不是给学生贴永久标签，而是阶段性教学关注列表。

### 7.4 关注等级建议
- 高关注：近期问题持续或上升，建议优先跟进
- 中关注：存在波动，建议持续观察
- 低关注：整体稳定，常规关注即可

---

## 8. 单学生详情规则

### 8.1 作用
为老师提供单学生的结构化理解页面。

### 8.2 建议展示模块
1. 当前状态摘要
2. 近期变化
3. 知识点结构
4. 题型与误区结构
5. 关键事件依据
6. 老师反馈区
7. 家校协同提示

### 8.3 原则
- 先给摘要
- 再给结构
- 最后给依据和操作入口

---

## 9. 当前状态摘要规则

### 9.1 建议展示内容
- 当前整体状态
- 当前重点学科
- 当前重点知识点
- 当前风险等级
- 当前数据完整性提示
- 当前建议关注方向

### 9.2 说明
老师端可以比家长端更结构化，但不应直接抛过多内部技术字段。

### 9.3 推荐表达
- 物理学科近期表现波动
- 受力分析相关知识点需重点关注
- 当前判断基于近 14 天有效事件

### 9.4 不推荐表达
- 置信度 0.73
- 模型推断结果
- 后台中间变量名

---

## 10. 知识点展示规则

### 10.1 目标
帮助老师看到学生在哪些知识点上出现问题。

### 10.2 建议展示内容
- 当前重点知识点
- 知识点状态
- 相关前置依赖
- 近期是否波动
- 是否存在对应误区

### 10.3 建议状态表达
- 稳定掌握
- 初步掌握
- 表现波动
- 重点关注
- 数据不足

### 10.4 原则
知识点展示应支持下钻，但不要默认展开成一整页知识森林。

### 10.5 示例
```
重点知识点：受力分析基本方法
状态：表现波动
前置依赖：力的概念、力的合成与分解
近期变化：近 7 天相关错误增加 3 次
关联误区：漏画或多画受力
```

---

## 11. 能力点展示规则

### 11.1 目标
帮助老师判断问题是否更偏能力层。

### 11.2 建议展示内容
- 当前相关能力点
- 哪类能力表现不稳定
- 该能力对应的典型题型
- 与知识点的支撑关系摘要

### 11.3 说明
老师端比家长端更适合展示能力点，但仍应使用清楚、教学语义明确的表达。

### 11.4 示例
```
相关能力点：受力分析能力
表现：步骤表达不稳定
典型题型：受力图分析题
支撑知识点：受力分析基本方法
```

---

## 12. 题型展示规则

### 12.1 目标
帮助老师快速理解"学生总在什么类型题上出问题"。

### 12.2 建议展示内容
- 高频问题题型
- 近期重复出现题型
- 与知识点/能力点的关系
- 该题型常见误区

### 12.3 示例
- 受力图分析题：近 2 周反复出错
- 图像判断题：错误增加，主要集中在斜率理解
- 多阶段运动综合题：列式链条不稳定

### 12.4 原则
题型展示应直接关联教学安排，帮助老师设计针对性练习。

---

## 13. 误区展示规则

### 13.1 目标
帮助老师更精准地区分"不会"和"总是以某种方式错"。

### 13.2 建议展示内容
- 当前高频误区
- 该误区关联知识点
- 该误区关联题型
- 近期出现频次趋势

### 13.3 原则
误区展示要具体，不要抽象成"理解不到位"。

### 13.4 示例
```
高频误区：漏画或多画受力
关联知识点：受力分析基本方法
关联题型：受力图分析题
近期趋势：近 7 天出现 4 次，较前期增加
```

---

## 14. 近期变化展示规则

### 14.1 建议展示内容
- 近 7 天变化
- 近 14 天变化
- 问题上升/平稳/下降
- 作业节奏变化
- 风险变化方向

### 14.2 原则
老师端趋势比家长端可以更具体，但仍要突出重点，不应画满整屏小折线。

### 14.3 推荐表达
- 近 7 天物理相关错误增加
- 作业完成节奏较前期更稳定
- 风险等级从"中"降至"低"

---

## 15. 风险提示展示规则

### 15.1 目标
帮助老师及时识别值得教学关注的问题，而不是制造额外恐慌。

### 15.2 建议展示内容
- 当前关注等级
- 风险来源摘要
- 风险关联知识点/题型
- 近期是否持续

### 15.3 原则
老师端可以比家长端更明确，但仍要区分：
- 已确认事实
- 派生判断
- 待观察信号

### 15.4 不建议
直接展示未经校准的高风险模型结论或概率值。

---

## 16. 依据展示规则

### 16.1 目标
让老师知道判断不是凭空来的。

### 16.2 建议展示内容
- 最近关键作业事件
- 最近考试或测评事件
- 老师过往反馈摘要
- 相关教材位置或知识定义
- 关键题型和误区说明

### 16.3 原则
老师端应支持"看依据"，但不应默认把所有底层证据全摊开。

### 16.4 依据层级
- 第一层：关键事件摘要（默认展示）
- 第二层：事件详情（点击展开）
- 第三层：原始输入（必要时查看）

---

## 17. 操作入口规则

老师端建议提供以下操作入口：

| 操作 | 说明 | 优先级 |
|------|------|--------|
| 提交老师反馈 | 记录教学观察和判断 | P0 |
| 标记重点学生 | 标记需要重点跟进的学生 | P0 |
| 标记已跟进 | 记录已采取的跟进动作 | P0 |
| 补充课堂观察 | 记录课堂表现观察 | P1 |
| 发起家校沟通提醒 | 提示与家长沟通 | P1 |
| 查看历史报告 | 查看学生历史报告记录 | P1 |
| 查看班级同类问题分布 | 查看班级层面问题聚集 | P1 |

### 原则
展示页面应自然连接操作，而不是只做被动看板。

---

## 18. 老师反馈展示规则

### 18.1 建议展示
- 最近老师反馈摘要
- 是否已有跟进
- 当前反馈状态
- 是否需要再次观察

### 18.2 说明
老师自己的反馈记录，应帮助其形成连续跟进，而不是一次次重新判断。

### 18.3 反馈类型建议
- 课堂观察反馈
- 作业情况反馈
- 家校沟通反馈
- 跟进措施反馈

---

## 19. 家校协同提示规则

### 19.1 目标
帮助老师判断哪些问题适合与家长协同。

### 19.2 建议展示内容
- 是否建议家长配合
- 建议配合方向
- 是否存在近期家庭反馈
- 是否适合保持观察而暂不打扰家长

### 19.3 原则
老师端看到的家校协同信息应更偏"协作提示"，不是家庭内容全透明。

### 19.4 示例
```
家校协同提示：
- 建议与家长沟通近期作业节奏问题
- 家长近期反馈孩子学习压力较大，建议先观察
- 家长已配合完成受力图复盘，可继续观察变化
```

---

## 20. 班级比较展示规则

### 20.1 建议展示内容
- 班级问题 Top
- 班级内知识点波动分布
- 班级内题型问题集中区
- 高关注学生数量变化
- 作业节奏整体变化

### 20.2 原则
比较应主要服务教学，而不是形成简单排名刺激。

### 20.3 不推荐
- 学生成绩排名
- 学生能力排名
- 简单的好坏对比

### 20.4 推荐
- 问题类型分布
- 变化趋势对比
- 关注学生数量变化

---

## 21. 老师端不宜直接展示的内容

原则上不建议在老师端默认展示：

| 内容 | 原因 |
|------|------|
| 复杂推演中间变量 | 老师不是模型调参员 |
| 未确认的高敏感家庭信息 | 隐私保护 |
| 系统内部置信度计算细节 | 增加认知负担 |
| 其他老师不应见的内部备注 | 权限边界 |
| 系统调试信息 | 非教学相关 |
| 不属于本老师授权范围的学生数据 | 权限控制 |

## 22. 页面结构建议

建议老师端至少包含以下页面或模块：

| 页面 | 核心内容 | 使用场景 |
|------|----------|----------|
| 班级总览页 | 班级整体状态、问题聚集、高关注学生数量 | 日常快速查看 |
| 重点学生页 | 重点关注学生列表、变化方向、跟进状态 | 确定跟进优先级 |
| 单学生详情页 | 学生个体结构化信息、依据、操作入口 | 深入理解与跟进 |
| 学科问题页 | 学科层面知识点/题型/误区分布 | 教学设计参考 |
| 反馈与跟进页 | 老师反馈记录、跟进历史、待办事项 | 跟进管理 |

## 23. 语言口径规则

老师端语言建议：

| 维度 | 建议 |
|------|------|
| 专业性 | 更专业，使用教学术语 |
| 结构化 | 更结构化，层次分明 |
| 明确性 | 更明确，减少模糊表达 |
| 情绪化 | 少情绪化，保持客观 |
| 依据感 | 多依据感，可追溯来源 |

### 推荐表达
- 表现波动
- 重点关注
- 高重复误区
- 相关题型反复出错
- 近期变化上升
- 当前判断主要基于近两周事件

### 不推荐表达
- 学生学习态度差
- 明显能力缺陷
- 问题严重
- 未来会拖后腿
- 必然会怎样

## 24. 图表与可视化规则

### 24.1 建议图表
- 班级分布图
- 问题 Top 列表
- 变化趋势简图
- 知识点/题型聚合图
- 跟进状态表

### 24.2 不建议
- 大量炫技图表
- 复杂三维或多层环图
- 没解释的分数雷达图
- 一屏塞太多并列指标卡

### 24.3 原则
图表应服务快速理解，不是展示技术能力。

## 25. 常见错误

### 25.1 只展示结果不展示结构
老师看到"波动"但不知道波动在哪里。

**正确做法**：同时展示波动相关的知识点/题型结构。

### 25.2 信息太全但没优先级
页面像仓库，不像工具。

**正确做法**：突出当前最值得关注的 1-3 个点。

### 25.3 只做单学生详情，不做班级总览
老师日常用起来会断层。

**正确做法**：班级总览与单学生详情并重。

### 25.4 展示过多技术变量
老师不是模型调参员。

**正确做法**：隐藏内部技术字段，展示教学语义信息。

### 25.5 没有反馈入口
老师只能看，不能顺手跟进，价值会大打折扣。

**正确做法**：展示页面自然连接操作入口。

## 27. 推荐样例

### 27.1 单学生详情摘要样例

```json
{
  "student_id": "STU_000123",
  "student_name": "张**",
  "overall_status": "近期物理表现波动",
  "focus_subject": "物理",
  "focus_knowledge": ["受力分析"],
  "focus_exercise_type": ["受力图分析题"],
  "focus_misconception": ["漏画或多画受力"],
  "recent_change": "近 7 天相关错误增加",
  "risk_level": "中",
  "teacher_action_hint": "建议优先关注受力图步骤表达",
  "followed_up": false
}
```

### 27.2 老师端文案样例

```text
该学生近期物理问题主要集中在"受力分析"相关内容。

近两周在受力图分析题中重复出现"漏画或多画受力"的情况，建议先从研究对象和受力图步骤表达入手跟进。

当前判断主要基于近 14 天有效事件记录。
```

### 27.3 班级总览样例

```json
{
  "class_id": "CLASS_101",
  "class_name": "高一 (1) 班",
  "overall_status": "整体稳定，物理学科需关注",
  "focus_subject": "物理",
  "focus_knowledge_top": [
    {"knowledge": "受力分析", "student_count": 6},
    {"knowledge": "图像理解", "student_count": 4},
    {"knowledge": "多阶段运动", "student_count": 3}
  ],
  "high_attention_students": 5,
  "recent_change": "本周物理相关错误较上周增加"
}
```

## 30. 风险与边界

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 不能忽视教学决策优先 | 展示设计应服务教学，不是技术展示 | 以教学场景驱动设计 |
| 不能忽视依据可见 | 重要判断应可查看依据 | 建立依据展示层级 |
| 不能忽视可操作性 | 展示应导向可执行动作 | 每个页面自然连接操作入口 |
| 不能忽视权限边界 | 老师只能看授权范围内的数据 | 权限体系强制控制 |
| 不能忽视认知负担 | 信息过载会让老师放弃使用 | 分层展示、突出优先级 |

## 31. 结论

老师端展示规则的核心价值，不是把系统的复杂度展示给老师看，而是把真正有助于教学判断和跟进的内容，按合理结构交给老师。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 先总览 | 先看班级整体 |
| 再定位 | 再找重点学生 |
| 再看依据 | 再看判断依据 |
| 最后可操作 | 最后可执行跟进 |

否则老师端很容易变成一个内容很多、图很多、但真正忙的时候没人愿意点开的后台页面。
