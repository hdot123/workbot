## 1. 文档目的

本文档用于定义教育孪生项目中的"干预动作库"设计，包括动作分类、动作字段、适用条件、风险边界、推演使用方式与执行记录结构。

本文档重点回答以下问题：

1. 什么叫干预动作
2. 干预动作如何分类和标准化
3. 干预动作如何与 StudentTwinAgent、学习事件、图谱记忆和后台推演衔接
4. 哪些动作可以自动建议，哪些必须人工判断
5. 干预动作执行后如何形成记录与反馈

---

## 2. 干预动作库定义

干预动作库，是后台推演层中对"可操作的干预手段"进行标准化沉淀的结构化动作集合。

它不是一句泛泛而谈的建议，也不是单纯的自然语言模板，而是一个包含：

- 动作类型
- 适用条件
- 作用对象
- 预期影响维度
- 风险说明
- 执行记录方式

的标准动作库。

---

## 3. 干预动作库的目标

### 3.1 让建议可结构化
把"应该做点什么"变成可比较、可选择、可记录的动作单元。

### 3.2 服务后台推演
推演层需要使用标准动作集作为变量，而不是每次临场现编动作。

### 3.3 服务前台低风险建议
观察层和报告层可消费一部分低风险动作建议。

### 3.4 服务干预效果追踪
动作执行后，应能记录结果并反馈到 StudentTwinAgent 和图谱系统中。

### 3.5 避免动作表达混乱
不同老师、不同报告、不同页面不应对同一类动作说出三套口径。

---

## 4. 设计原则

### 4.1 动作应可执行
动作必须具体，不应停留在空泛鼓励。

### 4.2 动作应可分类
同类动作应归于统一类别，便于推演和统计。

### 4.3 动作应带边界
每个动作都应说明适用条件和不适用条件。

### 4.4 动作应带风险说明
不是所有动作都适合直接给家长端输出。

### 4.5 动作与结果应能关联
动作不是发出去就蒸发，应支持执行记录和效果回看。

---

## 5. 干预动作定义

干预动作，是指系统、老师、家长或学校在某一时间窗口内，为影响学生学习状态、学习节奏、知识点掌握或风险变化而采取的可描述、可记录、可追踪的行为单元。

### 5.1 示例
- 增加某知识点专项练习
- 调整复习顺序
- 增加错题复盘频率
- 老师增加一次针对性反馈
- 家长每周增加两次作业复盘
- 暂缓推进后续章节

---

## 6. 干预动作分类体系

建议干预动作分为六大类：

1. 学习内容类
2. 学习路径类
3. 练习强化类
4. 反馈支持类
5. 节奏调整类
6. 协同配合类

---

## 7. 学习内容类动作

### 7.1 定义
围绕具体知识点、章节或学科内容展开的干预动作。

### 7.2 示例
- 补强"受力分析"专项内容
- 回看教材第 2 章第 1 节
- 针对某误区做复习说明

### 7.3 特点
- 与知识底座强关联
- 适合知识点状态不稳场景
- 易于和错题、章节、知识点挂钩

---

## 8. 学习路径类动作

### 8.1 定义
围绕学习顺序、推进路径和前后依赖关系的调整动作。

### 8.2 示例
- 先补前置知识再继续当前内容
- 调整章节复习顺序
- 先巩固基础题型再进入综合题型

### 8.3 特点
- 与知识点依赖图强关联
- 适合路径阻塞场景
- 常用于后台推演比较

---

## 9. 练习强化类动作

### 9.1 定义
通过练习频率、题型分布或复盘方式进行强化。

### 9.2 示例
- 每周增加 2 次专项练习
- 增加错题回顾频率
- 增加基础题型巩固
- 降低综合题比例，先稳基础

### 9.3 特点
- 易于执行
- 易量化
- 易于观察短期效果

---

## 10. 反馈支持类动作

### 10.1 定义
通过老师、家长、系统反馈增强学生理解和修正。

### 10.2 示例
- 老师增加一次针对性口头反馈
- 家长每周一次错题复盘沟通
- 系统追加一次知识点提醒
- 增加阶段性学习小结

### 10.3 特点
- 与关系上下文强关联
- 对行为和节奏状态影响较大
- 效果常带时间滞后

---

## 11. 节奏调整类动作

### 11.1 定义
围绕学习频率、强度和节奏稳定性进行调整。

### 11.2 示例
- 降低短期任务堆积
- 增加固定复习时段
- 延后下一章节推进
- 将任务拆分为更小单元

### 11.3 特点
- 适合节奏波动场景
- 常用于降低风险而非直接提分
- 应避免过度频繁调整

---

## 12. 协同配合类动作

### 12.1 定义
围绕家长、老师、学校间协作展开的干预动作。

### 12.2 示例
- 增加一次家校沟通
- 老师和家长共享重点关注项
- 家庭和教师统一复盘口径
- 学校安排阶段性辅导支持

### 12.3 特点
- 不是直接知识点干预
- 但对执行效果影响很大
- 应谨慎输出，避免过度归因

---

## 13. 干预动作标准字段

建议干预动作至少包含以下字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| action_id | string | 动作唯一标识 |
| action_name | string | 动作名称 |
| action_category | string | 动作类别 |
| action_subtype | string | 子类型 |
| description | string | 动作说明 |
| target_scope | string | 作用对象范围 |
| target_refs | array[string] | 作用知识点/章节/状态引用 |
| applicable_conditions | array[string] | 适用条件 |
| contraindications | array[string] | 不适用条件 |
| expected_effect_dimensions | array[string] | 预期影响维度 |
| risk_level | string | 动作风险等级 |
| execution_role | array[string] | 执行角色 |
| duration_suggestion | string/null | 建议持续周期 |
| feedback_required | boolean | 是否需反馈 |
| automation_level | string | 自动化等级 |
| status | string | active/deprecated |
| version | string | 版本号 |

---

## 14. 字段说明

### 14.1 action_category
建议值：
- content（学习内容）
- path（学习路径）
- practice（练习强化）
- feedback（反馈支持）
- rhythm（节奏调整）
- collaboration（协同配合）

### 14.2 target_scope
建议值：
- knowledge_point（知识点）
- chapter（章节）
- subject（学科）
- behavior（行为）
- risk_signal（风险信号）
- family_support（家庭支持）
- teacher_support（教师支持）

### 14.3 risk_level
建议值：
- low（低风险）
- medium（中风险）
- high（高风险）

### 14.4 automation_level
建议值：
- auto_recommendable（可自动推荐）
- manual_review_required（需人工复核）
- expert_only（仅专家使用）

---

## 15. 动作适用条件

每个动作应至少说明以下条件：

1. 适用学生状态
2. 适用知识点或学科范围
3. 适用时间窗口
4. 数据完整性要求
5. 执行角色要求

### 15.1 示例
动作：增加受力分析专项练习  
适用条件：
- 当前知识点状态为"掌握不稳定"
- 近两周重复错误率较高
- 数据完整性中等及以上
- 当前未处于严重任务堆积状态

---

## 16. 动作不适用条件

为避免系统乱给建议，每个动作应尽量定义不适用条件。

### 16.1 示例
动作：增加专项练习  
不适用条件：
- 当前任务负荷已过高
- 学生主要问题不在练习量，而在前置知识缺口
- 最近已连续执行同类动作且无改善
- 学生当前处于明显节奏中断状态

---

## 17. 动作风险等级设计

### 17.1 低风险动作
可较稳定用于前台建议或报告建议。

示例：
- 每周增加一次错题回顾
- 关注某章节复习
- 增加一次知识点复盘

### 17.2 中风险动作
建议老师/专业角色确认后使用。

示例：
- 调整当前学习路径
- 暂缓某章节推进
- 重新安排复习优先级

### 17.3 高风险动作
必须人工复核，不应自动输出。

示例：
- 涉及重大学习目标调整
- 涉及高敏感决策场景
- 涉及复杂长期路径改变

---

## 18. 动作与推演层的关系

后台推演层使用干预动作库时，建议流程如下：

```
读取当前学生状态
    ↓
选择候选动作集
    ↓
过滤不适用动作
    ↓
组合动作方案
    ↓
执行推演比较
    ↓
输出不同动作方案的潜在影响
```

说明：
推演层不应临时发明动作，而应优先从动作库中选取标准动作。

## 19. 动作执行记录

动作一旦执行，应形成标准记录，用于后续回看与图谱沉淀。

### 19.1 建议字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| action_execution_id | string | 执行记录标识 |
| action_id | string | 动作引用 |
| student_id | string | 学生标识 |
| executor_role | string | 执行角色 |
| executed_at | datetime | 执行时间 |
| execution_status | string | 执行状态 |
| execution_note | string/null | 执行说明 |
| feedback_summary | string/null | 执行反馈摘要 |
| observed_effect | object/null | 观察到的效果摘要 |

### 19.2 作用

- 回写 StudentTwinAgent
- 写入图谱记忆层
- 服务后续推演校准
- 服务报告生成

---

## 20. 动作与图谱记忆层的关系

干预动作进入图谱后，可形成以下关系：

- Student -> intervened_by -> InterventionAction
- InterventionAction -> refers_to -> KnowledgePoint
- InterventionAction -> executed_at -> TimeNode
- InterventionAction -> related_to -> RiskSignal
- InterventionAction -> followed_by -> LearningEvent / StateChange

这能帮助系统以后回答：
"这个动作过去做过吗？做完后有什么变化？"

## 21. 动作与前台的衔接规则

### 21.1 可直接进入前台的动作

- 低风险
- 可解释
- 可执行
- 不涉及高敏感决策

### 21.2 需老师确认后进入前台的动作

- 中风险
- 依赖更多上下文判断
- 容易引发误读

### 21.3 不应直接进入前台的动作

- 高风险动作
- 需要专业判断的动作
- 强依赖后台推演条件的动作

---

## 22. MVP 建议范围

MVP 阶段建议动作库先支持以下内容：

### 22.1 必须支持

- 知识点专项复习
- 错题回顾
- 增加低频专项练习
- 家长复盘支持
- 老师增加反馈
- 章节顺序微调建议

### 22.2 可以后置

- 群体干预动作
- 高复杂路径重构
- 长期选科相关动作
- 志愿填报相关动作

## 25. 结论

干预动作库是后台推演层从"看出问题"走向"比较可执行方案"的关键中间层。

如果没有动作库：

- 后台推演没有稳定变量
- 建议口径会四处漂
- 干预效果无法回看
- 报告层很难形成一致行动建议

因此动作库必须坚持：

- 动作标准化
- 条件明确
- 风险分级
- 可追踪执行
- 可反馈回流
