## 2. 服务定位

### 2.1 服务定义

**微信接入服务（WeChat Access Service）** 是 INGEST 模块的**微信渠道输入层**，负责接收和管理来自微信平台的各类输入消息，为后续的事件处理服务提供原始输入数据。

### 2.2 核心定位

| 定位维度 | 描述 |
|---------|------|
| **输入入口** | 微信渠道的统一消息接收入口 |
| **消息网关** | 微信消息的鉴权、校验、路由 |
| **原始输入层** | 保留原始消息内容，供后续处理 |
| **角色识别** | 识别发送者角色（家长、老师、学校） |

### 2.3 服务边界

**包含**：
- 微信消息接收
- 微信用户身份识别
- 消息类型识别
- 消息基础校验
- 消息路由转发
- 消息存储（原始消息）
- 异常消息处理

**不包含**：
- ❌ 消息内容解析（由事件处理服务负责）
- ❌ 学习事件生成（由事件处理服务负责）
- ❌ 学生状态更新（由 TWIN 模块负责）
- ❌ 微信 OAuth 认证（由平台基础服务负责）

## 3. 核心职责

### 3.1 核心职责

#### 3.1.1 消息接收
- 接收来自微信平台的各类消息（文本、图片、文件、事件通知）
- 支持微信个人号、企业微信、微信小程序多种消息来源
- 支持消息的批量接收和异步处理

#### 3.1.2 身份识别
- 识别微信用户的身份（OpenID、UnionID）
- 映射用户身份到系统角色（家长、老师、学校管理员）
- 验证用户与学生的绑定关系

#### 3.1.3 消息校验
- 验证消息签名（微信服务器签名）
- 校验消息格式合法性
- 过滤重复消息

#### 3.1.4 消息路由
- 根据消息类型路由到不同的处理队列
- 根据用户角色路由到不同的处理流程
- 支持消息的优先级调度

#### 3.1.5 消息存储
- 存储原始消息内容
- 存储消息元数据（发送时间、发送者、消息类型）
- 支持消息的追溯和审计

### 3.2 服务原则

| 原则 | 说明 |
|------|------|
| **原始消息保留** | 所有原始消息必须完整保留，不得修改或丢弃 |
| **异步处理** | 消息接收与消息处理解耦，采用异步处理方式 |
| **幂等性** | 支持消息的重复接收，处理逻辑必须幂等 |
| **可追溯** | 所有消息处理过程必须有日志记录 |

## 6. 数据流位置

### 6.1 数据流总览

```
微信用户 → 微信接入服务 → 消息队列 → 事件处理服务 → 学习事件
```

### 6.2 数据流详细流程

```
1. 微信用户发送消息
       ↓
2. 微信接入服务接收消息
       ↓
3. 验证消息签名和格式
       ↓
4. 识别用户身份和角色
       ↓
5. 存储原始消息
       ↓
6. 路由到消息队列
       ↓
7. 事件处理服务消费消息
       ↓
8. 生成学习事件
```

### 6.3 与其他服务的数据流

| 数据流 | 起点 | 终点 | 数据类型 |
|--------|------|------|---------|
| **消息接收流** | 微信平台 | 微信接入服务 | 原始消息 |
| **消息路由流** | 微信接入服务 | 消息队列 | 路由指令 |
| **身份查询流** | 微信接入服务 | 身份服务 | 用户 ID |
| **异常通知流** | 微信接入服务 | 管理后台 | 异常消息 |

## 7. 接口边界

### 7.1 接收接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **消息接收** | `receiveMessage` | 接收微信消息 |
| **事件接收** | `receiveEvent` | 接收微信事件通知 |
| **批量接收** | `receiveBatch` | 批量接收消息 |

### 7.2 查询接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **用户查询** | `queryUser` | 查询微信用户信息 |
| **消息查询** | `queryMessage` | 查询历史消息 |
| **状态查询** | `queryStatus` | 查询消息处理状态 |

### 7.3 管理接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **路由配置** | `configureRouting` | 配置消息路由规则 |
| **异常处理** | `handleException` | 处理异常消息 |
| **日志查询** | `queryLogs` | 查询处理日志 |

### 7.4 接口原则

- **只接收不解析**：微信接入服务只负责接收消息，不解析消息内容
- **异步响应**：消息接收后异步处理，立即返回接收成功
- **幂等性**：支持消息的重复接收
- **可追溯**：所有接口调用必须有日志记录

## 8. 与其他模块的交互关系

### 8.1 与事件处理服务的交互

| 交互类型 | 说明 |
|---------|------|
| **消息路由** | 微信接入服务将消息路由到事件处理服务 |
| **状态同步** | 事件处理服务返回消息处理状态 |

**数据流向**：微信接入服务 → 事件处理服务（单向）

### 8.2 与身份服务的交互

| 交互类型 | 说明 |
|---------|------|
| **身份查询** | 微信接入服务查询用户身份和角色 |
| **绑定验证** | 验证用户与学生的绑定关系 |

**数据流向**：微信接入服务 ↔ 身份服务（双向）

### 8.3 与消息队列的交互

| 交互类型 | 说明 |
|---------|------|
| **消息入队** | 微信接入服务将消息发送到消息队列 |
| **优先级调度** | 根据消息优先级调度到不同队列 |

**数据流向**：微信接入服务 → 消息队列（单向）

### 8.4 与管理后台的交互

| 交互类型 | 说明 |
|---------|------|
| **异常通知** | 微信接入服务通知管理后台异常消息 |
| **日志查询** | 管理后台查询微信接入服务的日志 |

**数据流向**：微信接入服务 ↔ 管理后台（双向）

### 8.5 交互总结

```
                微信接入服务
                     ↑
                微信平台
                     ↓
    ┌────────────────┼────────────────┐
    ↓                ↓                ↓
事件处理服务      身份服务        消息队列
                     ↓
                管理后台
```

**特点**：
- 微信接入服务是**输入层**，只负责接收和路由
- 所有交互都是**单向的**（除身份查询外）
- 微信接入服务是**无状态的**，不存储业务状态

### 10.3 开发建议

#### 10.3.1 优先级建议
- **P0（第一批）**：消息接收、身份识别、消息路由
- **P1（第二批）**：批量接收、异常处理、日志查询
- **P2（第三批）**：高级路由规则、性能优化

#### 10.3.2 技术实现建议
1. **使用微信官方 SDK**：优先使用微信官方提供的 SDK
2. **异步处理**：消息接收后异步处理，避免阻塞
3. **消息去重**：实现消息去重机制，避免重复处理
4. **日志完整**：实现完整的日志记录，便于追溯

#### 10.3.3 测试建议
1. **单元测试**：为所有接口编写单元测试
2. **集成测试**：测试与身份服务、消息队列的集成
3. **压力测试**：测试高并发消息接收能力
4. **异常测试**：测试异常消息的处理能力
