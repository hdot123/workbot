## 2. 服务定位

### 2.1 服务定义

**知识底座服务（Knowledge Base Service）** 是教育孪生系统的**静态知识提供层**，负责承载和管理固定知识底座，为系统的其他模块提供统一的知识参照系。

### 2.2 核心定位

| 定位维度 | 描述 |
|---------|------|
| **知识地图** | 提供学生所处的客观教学体系的结构化地图 |
| **参照系统** | 为学生状态、学习事件提供知识节点映射的基准 |
| **规则引擎** | 提供地区规则、考试口径、选科规则等政策约束 |
| **版本管理** | 支持教材版本更替、政策调整的版本化管理 |

### 2.3 服务边界

**包含**：
- 地区规则管理
- 教材版本管理
- 章节树结构
- 知识点体系
- 能力点体系
- 依赖关系
- 题型映射
- 锚点映射
- 版本管理
- 质量校验

**不包含**：
- ❌ 学生个体数据（由 TWIN 模块负责）
- ❌ 学习事件数据（由 INGEST 模块负责）
- ❌ 图谱记忆数据（由 GRAPH 模块负责）
- ❌ 实时状态数据（由 TWIN 模块负责）

## 4. 服务职责

### 3.1 核心职责

#### 3.1.1 知识地图提供
- 提供完整的章节树结构
- 提供知识点体系和能力点体系
- 提供知识点之间的依赖关系
- 支持多教材版本的并行管理

#### 3.1.2 地区规则管理
- 管理不同地区的教育制度规则
- 提供考试口径、选科规则查询
- 支持地区与教材的适配关系

#### 3.1.3 映射服务
- 提供学习事件到知识节点的映射
- 提供题目到知识点、能力点的映射
- 提供教材锚点的定位服务

#### 3.1.4 版本管理
- 支持教材版本的创建、发布、回滚
- 支持规则的版本化管理
- 提供版本对比和变更追踪

#### 3.1.5 质量校验
- 提供知识底座的质量校验接口
- 支持完整性、一致性、正确性检查
- 生成质量报告

### 3.2 服务原则

| 原则 | 说明 |
|------|------|
| **只读为主** | 知识底座服务主要提供查询能力，修改由专门的管理流程控制 |
| **不修改学生状态** | 知识底座服务不直接修改学生状态，只提供静态世界模型 |
| **版本隔离** | 不同版本的知识底座相互隔离，避免互相影响 |
| **可追溯** | 所有知识节点都能追溯到原始资料出处 |

## 7. 数据存储设计

### 6.1 存储架构

知识底座服务采用**混合存储架构**：

| 存储类型 | 存储引擎 | 用途 |
|---------|---------|------|
| **关系型数据库** | PostgreSQL | 结构化数据、版本管理、元数据 |
| **图数据库** | Neo4j | 知识图谱、依赖关系、层级关系 |
| **缓存** | Redis | 热点数据缓存、查询加速 |

### 6.2 数据分层

```
┌─────────────────────────────────────┐
│      应用层（查询服务）              │
├─────────────────────────────────────┤
│      缓存层（Redis）                 │
├─────────────────────────────────────┤
│  关系型层（PostgreSQL）  │  图谱层（Neo4j）  │
├─────────────────────────────────────┤
│      基础设施层（存储、备份）        │
└─────────────────────────────────────┘
```

### 6.3 数据结构概览

#### 6.3.1 PostgreSQL 数据表
- `textbooks`：教材元数据表
- `chapters`：章节树表
- `knowledge_points`：知识点表
- `ability_points`：能力点表
- `regions`：地区规则表
- `versions`：版本管理表

#### 6.3.2 Neo4j 图结构
- 节点：教材、章节、知识点、能力点
- 关系：包含、依赖、映射、属于

### 6.4 数据隔离

| 隔离维度 | 实现方式 |
|---------|---------|
| **版本隔离** | 每个版本有独立的版本号，查询时指定版本 |
| **地区隔离** | 通过地区编码过滤，不同地区看到不同规则 |
| **教材隔离** | 通过教材 ID 隔离，不同教材版本独立存储 |

## 8. 服务接口边界

### 7.1 查询接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **章节查询** | `getChapterTree` | 获取完整的章节树 |
| **知识点查询** | `getKnowledgePoints` | 获取知识点列表或详情 |
| **能力点查询** | `getAbilityPoints` | 获取能力点列表或详情 |
| **依赖关系查询** | `getDependencies` | 获取知识点的依赖关系 |
| **地区规则查询** | `getRegionRules` | 获取地区规则 |
| **版本查询** | `getVersionInfo` | 获取版本信息 |

### 7.2 映射接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **事件映射** | `mapEventToKnowledge` | 将学习事件映射到知识节点 |
| **题目映射** | `mapQuestionToPoints` | 将题目映射到知识点和能力点 |
| **教材定位** | `locateTextbookAnchor` | 定位教材中的锚点位置 |

### 7.3 管理接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **版本创建** | `createVersion` | 创建新的知识底座版本 |
| **版本发布** | `publishVersion` | 发布版本 |
| **版本回滚** | `rollbackVersion` | 回滚到指定版本 |
| **质量校验** | `validateQuality` | 执行质量校验 |
| **质量报告** | `getQualityReport` | 获取质量报告 |

### 7.4 接口原则

- **只读优先**：大部分接口是只读的，修改接口需要权限控制
- **版本参数**：所有查询接口都支持版本参数，默认使用最新版本
- **缓存友好**：查询接口设计考虑缓存策略，减少数据库压力
- **幂等性**：所有接口保证幂等性，重复调用不会产生副作用

                知识底座服务 (KB)

## 10. 质量校验与版本管理接口

### 9.1 质量校验接口

#### 9.1.1 完整性校验
- **章节完整性**：检查章节树是否完整，是否有孤立节点
- **知识点完整性**：检查知识点是否都有对应的章节
- **能力点完整性**：检查能力点是否都有对应的知识点

#### 9.1.2 一致性校验
- **依赖一致性**：检查依赖关系是否存在循环依赖
- **映射一致性**：检查题型映射是否合理
- **版本一致性**：检查同一版本内的数据是否一致

#### 9.1.3 正确性校验
- **锚点正确性**：检查教材锚点是否有效
- **规则正确性**：检查地区规则是否符合逻辑

### 9.2 版本管理接口

#### 9.2.1 版本创建
- **输入**：版本号、版本描述、变更内容
- **输出**：版本 ID、创建时间、状态
- **约束**：版本号必须唯一，不能重复

#### 9.2.2 版本发布
- **输入**：版本 ID、发布时间、发布范围
- **输出**：发布状态、发布日志
- **约束**：只有通过质量校验的版本才能发布

#### 9.2.3 版本回滚
- **输入**：目标版本 ID、回滚原因
- **输出**：回滚状态、回滚日志
- **约束**：回滚操作需要记录原因和操作人

#### 9.2.4 版本对比
- **输入**：版本 ID 1、版本 ID 2
- **输出**：差异报告、变更清单
- **约束**：只能对比已发布的版本

### 9.3 变更追踪

| 变更类型 | 追踪内容 |
|---------|---------|
| **新增** | 新增的知识节点、规则 |
| **修改** | 修改的内容、修改原因 |
| **删除** | 删除的节点、删除原因 |
| **关系变更** | 依赖关系、映射关系的变化 |
