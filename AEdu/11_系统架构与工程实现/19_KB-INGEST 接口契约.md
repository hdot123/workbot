# KB-INGEST 接口契约

> 文档编号：ARCH-019  
> 版本：V1.0  
> 创建日期：2026-03-21  
> 最后更新：2026-03-21  
> 维护人：系统架构与工程实现负责人  
> 状态：已冻结（首轮骨架开发基线）  
> 用途：支撑 KB + INGEST 首轮骨架开发  
> 变更方式：后续修改需通过新一轮准入评审

---

## 1. 文档定位

本文档是 KB 与 INGEST 之间的模块级契约文档，用于明确知识底座服务与事件处理服务之间的职责边界和引用规则。

本文档是：

- KB 与 INGEST 之间的模块级契约文档
- 正式开发放行前置文档
- 跨模块协作文档

本文档不是：

- API 详细协议文档
- 数据库字段级设计文档
- 算法实现文档

---

## 2. 文档目的与边界

本文档用于定义 KB 模块与 INGEST 模块之间的最小稳定接口契约，重点回答以下问题：

1. KB 到底向 INGEST 提供哪些能力
2. INGEST 可以如何引用 KB 的对象
3. `chapter_refs / knowledge_refs / anchor_refs / ability_refs / curriculum_version_id` 应如何对齐
4. 什么情况下算映射成功，什么情况下只能弱成功，什么情况下必须失败
5. 如何避免 INGEST 自己造一套“看起来像知识底座”的临时结构

本文档的边界是模块契约级，只定义：

- 能力边界
- 引用边界
- 一致性边界
- 状态边界

不定义：

- HTTP / RPC 协议细节
- 具体表结构字段
- 置信度计算公式
- 下游 TWIN / GRAPH 的消费逻辑

---

## 3. 契约目标

### 3.1 保证知识引用来源唯一
学习事件中的知识引用必须以 KB 为唯一权威来源，不能由 INGEST 自由生成或长期缓存自定义编码。

### 3.2 保证事件可映射、可校验、可追溯
INGEST 生成的标准学习事件，必须能说明：

- 引用了哪套教材版本
- 引用了哪些章节、知识点、能力点、锚点
- 这些引用是如何得出的
- 引用失败时为什么失败

### 3.3 保证模块职责不重叠

- KB 负责提供静态知识地图和映射依据
- INGEST 负责将原始输入转成标准学习事件
- INGEST 不维护知识底座主数据
- KB 不处理原始输入解析

### 3.4 保证首轮试点可收口
契约必须支持首轮范围内的最小闭环：

`输入 → 事件 → 知识引用 → 下游路由/人工复核`

---

## 4. 模块职责边界

### 4.1 KB 的职责

KB 是知识引用的权威提供方，负责：

- 提供 `curriculum_version_id` 的权威集合
- 提供章节树、知识点、能力点、教材锚点的权威集合
- 提供知识点与章节、能力点、锚点之间的静态映射关系
- 提供知识引用有效性校验
- 提供版本隔离与版本回滚能力
- 提供地区、学段、年级、学科、教材版本之间的适配关系

KB 不负责：

- 解析家长/老师原始文本
- OCR 识别
- 学习事件生成
- 人工复核任务编排
- 学生个体状态更新

### 4.2 INGEST 的职责

INGEST 是学习事件的生成方和引用方，负责：

- 接收原始输入
- 进行解析、清洗、分类、质量评分
- 基于 KB 提供的权威对象进行知识映射
- 在标准学习事件中写入知识引用结果
- 在无法稳定映射时进行降级、入池或发起人工复核

INGEST 不负责：

- 创建或修改 KB 主数据
- 绕过 KB 直接生成长期有效的知识编码
- 自行维护教材版本主表
- 自行定义章节、知识点、锚点的权威关系

### 4.3 契约原则

| 原则 | 说明 |
|------|------|
| 权威单源 | 所有知识引用以 KB 为准 |
| 引用先于写入 | 先完成知识引用校验，再决定是否写入主链路 |
| 版本隔离 | 同一事件中的知识引用不得跨知识底座版本混用 |
| 可降级 | 不能稳定映射时允许弱成功或失败，不允许强行伪造成功 |
| 可追溯 | 每个引用结果都应能回溯到 KB 中的对象和版本 |

## 5. KB 提供给 INGEST 的能力

| 能力类型 | 能力说明 | 作用 |
|------|------|------|
| 教材版本判定能力 | 根据地区、学段、年级、学科返回适用教材版本 | 生成 `curriculum_version_id` |
| 章节树查询能力 | 返回指定教材版本下的章节树 | 支撑 `chapter_refs` |
| 知识点查询能力 | 返回指定教材版本下的知识点集合 | 支撑 `knowledge_refs` |
| 能力点关联能力 | 返回知识点到能力点的映射 | 支撑 `ability_refs` |
| 教材锚点定位能力 | 返回知识点/章节关联的教材锚点 | 支撑 `anchor_refs` |
| 有效性校验能力 | 校验引用对象是否存在、是否属于当前版本 | 防止脏引用进入事件 |
| 映射候选返回能力 | 在无法唯一命中时返回候选集合或无结果 | 支撑弱成功和人工复核 |

## 6. INGEST 对 KB 的引用方式

### 6.1 先确定上下文，再做引用
INGEST 不能脱离上下文直接映射知识点。最小上下文应尽量包括：

- `tenant_id`
- `student_id`
- 地区
- 学段
- 年级
- 学科
- 教材版本候选

### 6.2 先定教材版本，再定知识引用
`curriculum_version_id` 是章节、知识点、锚点、能力点引用的上位约束。未确定教材版本时，不应直接把知识点引用写成主链路成功状态。

### 6.3 允许候选，但不允许伪造确定值
INGEST 可以得到“候选章节/候选知识点”，但候选不等于正式引用。正式写入事件的引用必须满足本契约中的成功或弱成功条件。

### 6.4 引用结果只写权威 ID，不写自定义别名
事件中可以保留原始文本中的自然语言描述，但进入 `*_refs` 的内容必须是 KB 权威 ID。

## 7. 核心引用对象对齐规则

### 7.1 `curriculum_version_id` 对齐规则

`curriculum_version_id` 表示当前事件所绑定的教材版本，是所有知识引用的根约束。

对齐要求：

- 必须由 KB 的地区适配和教材版本体系提供
- 应与学生当前绑定的地区、学段、年级、学科一致
- 同一事件只能有一个主 `curriculum_version_id`
- 未能确定教材版本时，事件只能进入弱成功或失败状态

禁止行为：

- INGEST 直接硬编码教材版本
- 同一事件同时引用多个教材版本
- 用“学科名”替代教材版本 ID

### 7.2 `chapter_refs` 对齐规则

`chapter_refs` 表示事件可稳定对应到的章节节点集合。

对齐要求：

- 必须属于同一个 `curriculum_version_id`
- 必须引用 KB 章节树中的合法章节 ID
- 可为空，但为空时应有原因
- 若存在 `knowledge_refs`，则 `chapter_refs` 应与对应知识点所属章节一致，不得冲突

### 7.3 `knowledge_refs` 对齐规则

`knowledge_refs` 表示事件稳定命中的知识点集合，是契约中的核心引用对象。

对齐要求：

- 必须属于当前 `curriculum_version_id`
- 必须是 KB 中有效且未废弃的知识点 ID
- 若事件能稳定命中知识点，应优先填充 `knowledge_refs`
- 同一事件可映射多个知识点，但应控制在业务合理范围内
- 若存在多个候选且无法稳定裁定，应转弱成功或失败

禁止行为：

- 用章节名代替知识点 ID
- 使用 KB 不存在的临时编码
- 通过纯猜测写入知识点

### 7.4 `ability_refs` 对齐规则

`ability_refs` 表示事件所关联的能力点集合。

对齐要求：

- 应优先从 `knowledge_refs` 通过 KB 的知识能力映射获得
- 可由老师反馈或系统任务显式触发，但仍需经 KB 校验
- 若 `knowledge_refs` 为空，`ability_refs` 不应无依据独立大规模填充
- 必须属于当前教材版本语境下的有效能力点集合

### 7.5 `anchor_refs` 对齐规则

`anchor_refs` 表示事件可回贴到教材原始位置的锚点集合。

对齐要求：

- 必须属于当前 `curriculum_version_id`
- 应与 `chapter_refs` 或 `knowledge_refs` 保持一致
- 主要用于 OCR / 扫描件 / 教材定位类场景
- 允许为空，不作为所有事件的必填项
- 不能只命中锚点却完全脱离章节或知识点语义长期存在

## 8. 引用规则表

| 引用对象 | 权威来源 | INGEST 可否直接写入 | 写入前置条件 | 一致性约束 | 失败处理 |
|------|------|------|------|------|------|
| `curriculum_version_id` | KB 教材版本与地区适配 | 可 | 地区/学段/年级/学科已确定 | 同一事件仅允许 1 个主版本 | 转弱成功或失败 |
| `chapter_refs` | KB 章节树 | 可 | 已确定教材版本，章节可稳定命中 | 必须属于同一版本；与知识点不冲突 | 留空并标因，必要时复核 |
| `knowledge_refs` | KB 知识点体系 | 可 | 已确定教材版本，知识点可稳定命中 | 必须属于同一版本；不得使用临时编码 | 转弱成功或失败 |
| `ability_refs` | KB 能力点体系与知识能力映射 | 条件可 | 已命中知识点，或能力表达经 KB 校验 | 应与知识点映射关系一致 | 留空或转弱成功 |
| `anchor_refs` | KB 教材锚点体系 | 条件可 | 已确定教材版本，且 OCR/教材定位可稳定命中 | 应与章节/知识点一致 | 留空或转人工复核 |

## 9. 契约状态定义

### 9.1 成功

表示该事件已经满足进入主链路的稳定条件。

判定条件：

- `curriculum_version_id` 已确定
- 主体对象、时间、事件类型已通过校验
- 至少命中 `chapter_refs` 或 `knowledge_refs` 之一
- 若写入 `ability_refs` / `anchor_refs`，其与主引用一致
- 不存在核心冲突字段
- 不需要人工复核

处理结果：

- 可写入标准学习事件主链路
- 可路由到下游 TWIN / GRAPH

### 9.2 弱成功

表示事件有价值，但尚未达到完全自动放行条件。

典型场景：

- `curriculum_version_id` 可确定，但知识点只命中候选
- 章节可命中，但知识点不稳定
- 能力点表达明确，但知识点尚未唯一确定
- OCR 能定位页码，但锚点与知识点仍需人工确认

判定条件：

- 主体对象和来源基本可信
- 事件类型可判断
- 至少已有一层知识上下文可用
- 尚存在局部歧义或缺口
- 明确标记 `requires_manual_review = true` 或进入待处理池

处理结果：

- 可保留为中间态事件
- 不直接更新关键学生状态
- 必须进入复核或补齐流程

### 9.3 失败

表示事件当前不满足主链路要求，不能以知识引用成功的名义向下游传播。

典型场景：

- 教材版本无法确定
- 主体对象不明确
- 时间冲突严重
- 知识引用跨版本冲突
- 文本/OCR 质量过低，无法稳定命中章节或知识点
- 引用对象不存在或已废弃

处理结果：

- 不进入成功链路
- 不更新 TWIN / GRAPH 主数据
- 进入异常池、待复核池或要求重传
- 失败原因必须可追溯

## 10. 映射状态与动作关系

| 契约状态 | 是否允许写入标准事件 | 是否允许进入下游主链路 | 是否需要人工复核 | 说明 |
|------|------|------|------|------|
| 成功 | 是 | 是 | 否 | 可作为正式事件 |
| 弱成功 | 是，中间态 | 否 | 是 | 可保留，不可直接放行 |
| 失败 | 否，或仅异常态存档 | 否 | 视情况 | 必须阻断主链路 |

---

## 11. 首轮骨架开发的放行约束

为支撑 KB + INGEST 首轮骨架开发，本契约要求至少满足以下条件：

1. KB 能稳定提供首轮试点范围内的 `curriculum_version_id`
2. KB 能稳定提供章节树、知识点、能力点、锚点的权威 ID 集合
3. INGEST 能基于输入源生成成功、弱成功、失败三类结果
4. `chapter_refs / knowledge_refs / anchor_refs / ability_refs` 的写入规则已统一
5. 映射失败、冲突、低置信度的处理路径已明确
6. 契约状态能支撑后续验收矩阵和端到端样例验证

---

## 12. 不做事项

本契约在首轮阶段明确不做以下内容：

- 不定义接口 URL 与报文格式
- 不定义数据库字段明细
- 不定义全量多地区、多教材并行自动映射
- 不定义复杂推理型知识补全
- 不允许 INGEST 绕过 KB 直接维护知识主数据

---

## 13. 结论

`KB-INGEST` 的协作关系在首轮开发阶段应被视为：

- KB 是权威提供方
- INGEST 是受约束引用方
- 成功 / 弱成功 / 失败必须分层处理
- 教材版本是所有知识引用的上位约束
- 知识引用必须可校验、可追溯、可阻断

在此契约下，KB + INGEST 可以进入首轮骨架开发；但在 API 细节、字段级协议和全量联调展开前，必须以本契约作为唯一的跨模块引用口径。
