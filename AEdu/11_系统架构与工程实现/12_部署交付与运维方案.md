## 1. 文档目的

本文档用于定义教育孪生项目的部署交付与运维方案，包括部署架构、环境划分、发布流程、运维能力、故障处理与数据安全要求。

本文档重点回答以下问题：

1. 系统如何部署才能保证稳定
2. 环境如何划分才能保证安全
3. 发布流程如何设计才能降低风险
4. 运维能力如何建设才能快速发现问题
5. 故障发生时如何快速恢复
6. 数据安全如何保障

---

## 2. 部署目标

### 2.1 稳定运行
确保系统 7×24 小时可用，支持试点学校持续使用。

### 2.2 快速恢复
故障发生时能在分钟级内恢复服务。

### 2.3 安全可控
数据权限隔离，敏感信息加密存储。

### 2.4 可扩展
支持从单试点到多学校的平滑扩容。

### 2.5 可观测
所有关键链路可监控、可告警、可追溯。

---

## 3. 部署架构

### 3.1 逻辑分层

建议系统按以下层次部署：

```text
┌─────────────────────────────────────────┐
│           API 网关层                      │
│    (请求路由、认证、限流、日志)           │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│          业务服务层                       │
│  (学生服务、事件服务、报告服务、推演服务)  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│          Agent 服务层                     │
│  (成长型 Agent、决策型 Agent、编排服务)   │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│          数据与存储层                     │
│  (图谱存储、关系数据库、文件存储、缓存)   │
└─────────────────────────────────────────┘
```

### 3.2 核心服务组件

| 组件 | 说明 | 部署建议 |
|------|------|----------|
| API 网关 | 统一入口、认证鉴权 | 高可用集群 |
| 学生状态服务 | StudentTwinAgent 读写 | 有状态服务 |
| 事件流服务 | 学习事件生成与分发 | 无状态服务 |
| 图谱服务 | 知识图谱读写 | 独立部署 |
| 报告服务 | 周报月报生成 | 异步任务 |
| 推演服务 | 干预模拟与比较 | 后台服务 |
| OCR 服务 | 文档识别 | 独立部署 |
| 文件服务 | 附件存储与访问 | 对象存储 |

## 4. 环境划分

### 4.1 环境定义

| 环境 | 用途 | 访问权限 |
|------|------|----------|
| dev | 开发调试 | 开发团队 |
| test | 测试验证 | 测试团队 |
| staging | 预发布/灰度 | 产品 + 试点学校 |
| prod | 生产环境 | 正式用户 |

### 4.2 环境隔离要求

- 数据库独立
- 配置文件独立
- API 密钥独立
- 网络隔离（prod 环境）
- 日志独立存储

## 5. 部署方式

### 5.1 云部署优先

推荐使用主流云平台（阿里云/腾讯云/AWS）：

- 弹性计算资源
- 托管数据库服务
- 对象存储服务
- 负载均衡
- 自动扩缩容

### 5.2 容器化部署

建议采用 Docker + Kubernetes：

```yaml
# 示例：服务部署配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: student-twin-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: student-twin
  template:
    spec:
      containers:
      - name: service
        image: edutwin/student-twin:v1.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
          limits:
            memory: "1Gi"
            cpu: "1000m"
```

### 5.3 关键服务高可用

| 服务 | 副本数 | 说明 |
|------|--------|------|
| API 网关 | ≥2 | 负载均衡 |
| 学生状态服务 | ≥2 | 会话保持 |
| 图谱服务 | ≥2 | 数据一致性 |
| 数据库 | 主从 | 读写分离 |

## 6. 发布流程

### 6.1 CI/CD 流水线

```text
代码提交
    ↓
自动构建
    ↓
单元测试
    ↓
集成测试
    ↓
构建镜像
    ↓
推送到镜像仓库
    ↓
部署到 test 环境
    ↓
自动化测试
    ↓
部署到 staging 环境
    ↓
人工验收
    ↓
灰度发布到 prod
    ↓
全量发布
```

### 6.2 发布检查清单

发布前必须确认：

- [ ] 所有测试通过
- [ ] 无 P0/P1 bug
- [ ] 数据库变更已评审
- [ ] 回滚方案已准备
- [ ] 监控告警已配置
- [ ] 相关团队已通知

### 6.3 灰度发布策略

| 阶段 | 范围 | 观察时长 |
|------|------|----------|
| 阶段 1 | 内部用户 | 1 小时 |
| 阶段 2 | 单试点学校 | 24 小时 |
| 阶段 3 | 30% 用户 | 48 小时 |
| 阶段 4 | 全量发布 | 持续监控 |

## 7. 运维能力

### 7.1 日志系统

**要求：**

- 所有服务统一日志格式
- 关键操作必须记录审计日志
- 日志集中存储（ELK/云日志服务）
- 日志保留 ≥90 天

**日志级别：**

| 级别 | 说明 | 示例 |
|------|------|------|
| ERROR | 错误，需要处理 | 服务调用失败 |
| WARN | 警告，需要关注 | 重试成功 |
| INFO | 信息，正常流程 | 事件生成成功 |
| DEBUG | 调试，开发使用 | 详细处理过程 |

### 7.2 监控告警

**核心监控指标：**

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| API 成功率 | <99% | 接口异常 |
| 响应延迟 P99 | >2s | 性能下降 |
| 错误率 | >1% | 系统异常 |
| CPU 使用率 | >80% | 资源不足 |
| 内存使用率 | >85% | 内存泄漏风险 |
| 磁盘使用率 | >80% | 存储不足 |
| 队列积压 | >1000 | 处理延迟 |

**告警渠道：**

- 企业微信/钉钉
- 短信
- 电话（P0 级）
- 邮件

### 7.3 链路追踪

建议接入链路追踪系统（SkyWalking/Jaeger）：

- 追踪请求完整路径
- 识别性能瓶颈
- 快速定位故障点

## 8. 故障处理

### 8.1 故障分级

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| P0 | 核心功能不可用 | 5 分钟 | 服务宕机、数据丢失 |
| P1 | 重要功能受损 | 30 分钟 | 部分接口失败 |
| P2 | 体验问题 | 2 小时 | 页面加载慢 |
| P3 | 轻微问题 | 24 小时 | UI 显示问题 |

### 8.2 故障处理流程

```text
故障发现
    ↓
告警通知
    ↓
快速响应（5 分钟内）
    ↓
问题定位
    ↓
紧急修复/回滚
    ↓
服务恢复
    ↓
故障复盘
    ↓
改进措施
```

### 8.3 回滚机制

**必须支持：**

- 服务版本回滚（10 分钟内完成）
- 配置回滚
- 数据库变更回滚（如适用）

## 9. 数据安全

### 9.1 权限隔离

- 多租户数据隔离
- 学校间数据不可见
- 角色化访问控制
- 最小权限原则

### 9.2 数据加密

| 场景 | 加密要求 |
|------|----------|
| 传输中 | TLS 1.2+ |
| 敏感数据 | 应用层加密 |
| 密码 | bcrypt/argon2 |
| API 密钥 | 加密存储 |

### 9.3 数据备份

| 数据类型 | 备份频率 | 保留周期 |
|----------|----------|----------|
| 数据库 | 每日全量 + 实时增量 | 30 天 |
| 文件存储 | 每日增量 | 90 天 |
| 日志 | 每日归档 | 90 天 |

### 9.4 灾备方案

- 跨区域备份（可选）
- 数据库主从切换
- 服务多可用区部署

## 10. 容量规划

### 10.1 单试点学校参考配置

| 资源 | 配置 | 说明 |
|------|------|------|
| CPU | 4 核 | 基础服务 |
| 内存 | 8GB | 应用服务 |
| 数据库 | 2 核 4GB | 单学校数据量 |
| 存储 | 50GB | 文件存储 |

### 10.2 扩容触发条件

- CPU 持续>70%
- 内存持续>80%
- 数据库连接数>80%
- 响应延迟 P99>2s

## 13. 结论

部署交付与运维方案的核心价值，不是让系统"能跑起来"，而是让系统能够：

- **稳定运行** - 持续可用
- **快速恢复** - 故障分钟级响应
- **安全可控** - 数据权限清晰
- **可扩展** - 支撑试点到规模化

它必须坚持：

- 环境隔离
- 自动化发布
- 全链路监控
- 故障快速响应
- 数据安全第一

否则系统无法支撑从试点到规模化运营的演进。
