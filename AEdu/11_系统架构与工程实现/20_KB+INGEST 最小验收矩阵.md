# KB+INGEST 最小验收矩阵

> 文档编号：ARCH-020  
> 版本：V1.0  
> 创建日期：2026-03-21  
> 最后更新：2026-03-21  
> 维护人：系统架构与工程实现负责人  
> 状态：已冻结（首轮骨架开发基线）  
> 用途：支撑 KB + INGEST 首轮骨架开发  
> 变更方式：后续修改需通过新一轮准入评审

---

## 1. 文档目的

本文档用于定义 KB + INGEST 两个模块在首轮骨架开发阶段的最小可验收集合、验收维度、验收矩阵与进出门槛，确保模块进入正式开发前有统一、可执行、可收口的验收口径。

本文档重点回答以下问题：

1. KB + INGEST 最少做到什么才算“可验收”
2. 正确性、一致性、完整性、可追溯性分别怎么落到模块对象上
3. 首轮骨架开发的入场门槛是什么
4. 首轮骨架开发完成后的出场门槛是什么
5. 哪些结果算通过，哪些结果只能继续保留为“条件允许”

---

## 2. 适用范围

### 2.1 适用模块

- KB 模块（知识底座）
- INGEST 模块（数据接入与事件流）

### 2.2 适用阶段

- 首轮骨架开发阶段
- 单地区、单学段、单学科、单教材版本的最小闭环阶段
- 正式开发放行前的模块级验收阶段

### 2.3 不适用范围

- 全量地区扩展验收
- 多学科并行验收
- 前台观察层和推演层的最终业务验收
- API 协议字段级兼容性验收
- 性能压测和大规模并发验收

---

## 3. 最小可验收集合定义

### 3.1 KB 最小可验收集合

必须至少包含以下对象和关系：

1. 一个冻结地区规则入口
2. 一套冻结教材版本
3. 一套章节树结构
4. 一组知识点
5. 一组能力点
6. 一组教材锚点
7. 一组核心依赖关系
8. 一个底座版本标识与变更记录入口

### 3.2 INGEST 最小可验收集合

必须至少包含以下处理能力：

1. 首轮 3 类输入源进入系统
2. 原始输入保留与追溯
3. 输入清洗与风险分级
4. 标准学习事件生成
5. KB 引用字段写入
6. 弱成功与失败事件分流
7. 人工复核任务生成与回流标记

### 3.3 模块闭环定义

本轮验收不要求跑通全部下游模块，但至少要形成以下闭环：

```text
输入 → 清洗 → 事件生成 → KB 引用 → 验收判定
```

也就是说，本轮必须证明：

- 输入能被稳定处理
- 事件能被标准化表达
- 事件能引用到 KB 的结构化对象
- 无法稳定处理的输入不会悄悄进入主链路

---

## 4. 验收维度定义

### 4.1 正确性
系统输出是否与冻结规则、源数据和定义文档一致。

### 4.2 一致性
同类对象、同类输入、同类引用是否遵循统一命名、统一口径、统一结果规则。

### 4.3 完整性
最小必需字段、必需对象、必需引用、必需审计信息是否齐全。

### 4.4 可追溯性
是否能从结构化结果追溯到原始输入、底座版本、规则版本和处理路径。

---

## 5. KB 模块最小验收矩阵

### 5.1 KB 对象验收矩阵

| 验收对象 | 正确性要求 | 一致性要求 | 完整性要求 | 可追溯性要求 | 通过标准 |
|------|------|------|------|------|------|
| 地区与教材绑定 | 地区、学段、年级、学科、教材版本绑定正确 | 同一范围内不出现多套冲突绑定 | 必须包含 `region_id`、`grade_level`、`subject`、`curriculum_version_id` | 能追溯到规则来源和冻结值表 | 冻结值 100% 命中 |
| 章节树 | 章节名称、层级、页码范围与教材一致 | 同级编号规则统一 | 章/节/小节骨架完整 | 可回溯到教材页码范围 | 首发教材章节覆盖率 100% |
| 知识点表 | 知识点归属章节正确、命名准确 | 编码规则、命名风格统一 | 核心知识点无明显漏项 | 可追溯到章节与教材来源 | 核心知识点覆盖率 ≥ 95% |
| 能力点表 | 能力点定义与学科场景匹配 | 能力层级表达统一 | 首发知识点至少映射一个能力点 | 可回溯到课程标准或内部定义口径 | 首发知识点能力点映射率 ≥ 90% |
| 教材锚点表 | 页码、位置、锚点类型正确 | 锚点类型命名统一 | 核心知识点必须有锚点 | 可追溯到教材页码和位置描述 | 核心知识点锚点覆盖率 ≥ 90% |
| 依赖关系表 | 前置/后置关系符合学科逻辑 | 依赖类型枚举统一 | 核心依赖链必须存在 | 能追溯到知识点对和标注理由 | 核心知识点依赖标注率 ≥ 80% |
| 底座版本信息 | 版本号与发布范围正确 | 版本编号风格统一 | 必须有版本号、发布时间、适用范围 | 可追溯到变更记录 | 首发版本信息齐全率 100% |

### 5.2 KB 模块级判定规则

| 判定项 | 要求 |
|------|------|
| P0 | 地区教材绑定、章节树、知识点表必须通过 |
| P1 | 能力点、锚点、依赖关系允许少量缺口，但不得影响首发闭环 |
| P2 | 误区、实验活动、题型扩展可后置，不作为首轮阻断项 |

### 5.3 KB 阻断条件

出现以下任一情况，KB 不得放行进入正式开发：

- 首发地区/教材版本未冻结
- 章节树骨架不完整
- 知识点无法稳定归属章节
- 锚点缺失导致事件无法回贴教材
- 同一教材范围内出现两套冲突编码口径
- 版本信息缺失，无法确认当前底座基线

## 6. INGEST 模块最小验收矩阵

### 6.1 INGEST 能力验收矩阵

| 验收对象 | 正确性要求 | 一致性要求 | 完整性要求 | 可追溯性要求 | 通过标准 |
|------|------|------|------|------|------|
| 原始输入留存 | 原始文本/文件未被覆盖 | 同类输入留存口径统一 | 必须保留 `raw_input_id` 与原始载体引用 | 可追溯到原始文本或文件 | 留存率 100% |
| 输入清洗 | 低风险规范化结果正确 | 日期、学科、对象规范化口径统一 | 必须输出清洗后中间态 | 可追溯到清洗规则版本 | 首发样本清洗成功率 ≥ 95% |
| 事件分类 | 事件类型与输入语义匹配 | 同类输入不出现随机漂移分类 | 必须输出 `event_type`、`event_subtype` | 可追溯到分类依据 | 首发样本一级事件类型准确率 ≥ 90% |
| 字段完整性 | 必填字段填写正确 | 同类事件必填口径一致 | `student_id`、`source_type`、`event_type`、`summary` 不得缺失 | 可追溯到来源与生成时间 | 必填字段完整率 ≥ 95% |
| KB 引用写入 | `chapter_refs`、`knowledge_refs`、`anchor_refs` 与 KB 数据对齐 | 同类事件引用口径统一 | 高价值事件必须至少命中 `knowledge_refs` 或 `chapter_refs` 之一 | 可追溯到 KB 版本和映射规则版本 | 首发样本有效引用率 ≥ 85% |
| 弱成功分流 | 低置信事件不直接进入强主链路 | 弱成功标记规则统一 | 必须写出风险标签和复核状态 | 可追溯到风险原因 | 弱成功事件标记率 100% |
| 异常与人工复核 | 高风险输入进入复核池 | 同类异常进入同类处理路径 | 必须生成异常编码和复核任务 | 可追溯到处理人和处理结果 | 高风险事件阻断率 100% |
| 审计字段 | 审计时间、解析器版本、规则版本正确 | 审计字段命名与写入规则统一 | `created_at`、`parser_version`、`rule_version` 必须可落 | 可追溯到处理链路 | 审计字段齐全率 100% |

### 6.2 INGEST 首轮输入覆盖要求

首轮最小验收至少覆盖以下 3 类场景：

| 输入场景 | 是否必须覆盖 | 说明 |
|------|------|------|
| 家长文本输入 | 是 | 用于验证文本解析与作业/反馈事件生成 |
| 老师反馈文本输入 | 是 | 用于验证教师反馈事件与弱事实表达 |
| 图片/扫描件 OCR 输入 | 是 | 用于验证 OCR 后清洗与锚点引用 |

### 6.3 INGEST 阻断条件

出现以下任一情况，INGEST 不得放行进入正式开发：

- 原始输入未保留
- 高风险异常未被阻断
- 低置信度事件直接当作有效事件进入主链路
- 事件必填字段大面积缺失
- KB 引用字段无法稳定写入
- 清洗、分类、映射过程无版本与审计信息

## 7. KB + INGEST 联合验收矩阵

| 联合验收项 | 说明 | 通过标准 |
|------|------|------|
| 知识引用闭环 | INGEST 生成的事件能够引用 KB 的章节/知识点/锚点 | 3 组样例中至少 2 组达到强引用成功，1 组进入弱成功或复核也必须合理 |
| 编码口径一致 | KB 编码与 INGEST 引用编码一致 | 不允许出现同义不同码或同码不同义 |
| 版本关联可追溯 | 事件知道自己引用了哪个 KB 版本、哪个规则版本 | 版本追溯率 100% |
| 异常不污染主链路 | 无法稳定映射的事件不得强入 | 高风险异常阻断率 100% |
| 骨架范围受控 | 本轮仅覆盖冻结的地区、年级、学科、教材版本与输入源 | 范围外数据全部标注为 `out_of_scope` 或转人工处理 |

## 9. 首轮骨架开发入场门槛

KB + INGEST 进入首轮骨架开发前，必须满足以下条件：

- [ ] 首发地区、学段、年级、学科、教材版本已冻结
- [ ] KB 首发切片定义已冻结
- [ ] INGEST 输入源映射表已冻结
- [ ] KB ↔ INGEST 接口契约已冻结
- [ ] 端到端样例集已准备
- [ ] 异常处理边界已明确
- [ ] 模块责任边界已明确

若以上任一项未完成，则只允许继续做预研，不允许进入正式骨架开发。

---

## 10. 首轮骨架开发出场门槛

KB + INGEST 从“骨架开发中”进入“正式允许继续开发”前，必须满足以下条件：

- [ ] KB 最小可验收集合全部具备
- [ ] INGEST 最小可验收集合全部具备
- [ ] 3 组端到端样例全部跑通并形成记录
- [ ] 联合引用闭环已验证
- [ ] 高风险异常阻断链路已验证
- [ ] 版本与审计追溯链路已验证
- [ ] 无 P0 阻断项
- [ ] P1 遗留项已被记录并明确不影响首轮闭环

### 10.1 放行规则

| 结论 | 判定标准 |
|------|------|
| 允许 | Entry Gate 全满足，Exit Gate 全满足，且无 P0 阻断项 |
| 条件允许 | Entry Gate 全满足，但 Exit Gate 仍有少量 P1 缺口，且不破坏主链路 |
| 不允许 | Entry Gate 未满足，或 Exit Gate 存在任一 P0 阻断项 |

---

## 11. 首轮不作为阻断项的内容

以下内容可记录为后续迭代项，但不作为本轮骨架开发阻断项：

- 多地区并行适配
- 多学科同时上线
- 全量 OCR 优化
- 复杂情绪/行为推断
- 推演层和观察层的最终业务输出
- 规模化性能压测
- 全量题型、误区、实验活动覆盖

---

## 12. 结论

KB + INGEST 的最小验收，不是要求一开始就把全量系统做完，而是要求先把可运行、可引用、可阻断、可追溯的最小闭环做实。

首轮骨架开发阶段，验收必须坚持四条底线：

- 正确性不过关，不放行
- 一致性不收口，不放行
- 完整性不足以支撑闭环，不放行
- 可追溯性缺失，不放行

只有在这四条底线被最小可验收集合共同满足时，KB + INGEST 才能从“条件允许”推进到“允许”。
