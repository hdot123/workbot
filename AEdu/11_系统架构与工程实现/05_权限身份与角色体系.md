## 1. 文档目的

本文档用于定义教育孪生项目中的权限身份与角色体系，包括身份对象、登录与绑定关系、角色定义、权限边界、访问控制原则与审计要求。

本文档重点回答以下问题：

1. 系统里到底有哪些身份对象
2. 家长、老师、学校管理员、运营人员分别能看什么、做什么
3. 权限应该按什么维度控制
4. 如何避免角色边界不清导致"看错人、改错人、越权看数据"
5. 如何让权限体系既能支撑试点快速落地，也能支撑后期规模化扩展

---

## 2. 权限身份与角色体系定义

权限身份与角色体系，是指系统对"谁是谁、属于哪个组织、具备什么角色、能访问哪些对象、能执行哪些动作"的统一建模和控制机制。

它的作用不是让系统显得更正式，而是确保教育场景里最敏感的那件事不会出问题：

**不同角色只能看到自己应该看到的内容，只能做自己应该做的动作。**

教育项目里，权限一旦乱掉，后果通常不是一个技术 bug，而是直接变成信任事故。

---

## 3. 建设目标

### 3.1 明确身份边界
系统中每个访问者都应有清晰身份对象和所属上下文。

### 3.2 明确角色边界
不同角色在不同场景中的可见范围和可操作范围必须清楚。

### 3.3 支撑最小权限原则
默认只给完成任务所必需的权限，不做宽权限放行。

### 3.4 支撑组织化管理
学校、班级、家庭、运营团队之间的关系应可被系统表达。

### 3.5 支撑审计与追责
关键访问和关键操作必须可审计、可追溯。

---

## 4. 设计原则

### 4.1 身份与角色分离
"这个人是谁"和"这个人当前以什么角色操作"不是一回事。

### 4.2 角色与权限分离
角色是业务语义，权限是系统控制，不应简单混成一张死表。

### 4.3 数据边界优先
先确定能看哪些对象，再确定能做什么动作。

### 4.4 最小权限优先
默认收紧，再按场景放开。

### 4.5 多租户上下文优先
权限判断必须始终带租户/学校上下文。

## 5. 核心身份对象建议

建议至少定义以下身份对象：

| 身份对象 | 编码 | 说明 |
|----------|------|------|
| 用户账号 | UserAccount | 系统统一账号对象，是登录、认证和审计的基础 |
| 家长身份 | ParentIdentity | 家长业务身份对象 |
| 老师身份 | TeacherIdentity | 老师业务身份对象 |
| 学校管理员身份 | SchoolAdminIdentity | 学校管理类业务身份对象 |
| 运营人员身份 | OperatorIdentity | 项目运营、实施、客服、数据处理人员的业务身份 |
| 系统服务身份 | SystemServiceIdentity | 系统内部服务、任务执行器、异步 worker 等机器身份 |

---

## 6. UserAccount

### 6.1 定义
系统统一账号对象，是登录、认证和审计的基础对象。

### 6.2 作用
- 登录入口
- 统一用户标识
- 绑定多个业务身份
- 记录登录方式与设备信息

### 6.3 说明
一个账号可能对应多个业务身份，例如某人既是家长又是老师，不建议把这些直接揉成一个混合身份。

### 6.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | string | 用户账号唯一标识 |
| username | string | 用户名/手机号 |
| login_channels | array[string] | 支持的登录方式 |
| status | string | 账号状态 (active/inactive/locked) |
| created_at | datetime | 创建时间 |
| last_login_at | datetime | 最后登录时间 |

---

## 7. ParentIdentity

### 7.1 定义
家长业务身份对象。

### 7.2 作用
- 查看绑定学生的观察结果
- 提交家庭反馈
- 查看周报/月报
- 查看低风险建议

### 7.3 边界
家长不应直接看到：
- 其他学生数据
- 班级全量对比明细
- 后台推演中间态
- 学校级管理信息

### 7.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| parent_id | string | 家长身份唯一标识 |
| user_id | string | 关联用户账号 |
| student_bindings | array[object] | 绑定学生列表 |
| contact_info | object | 联系方式 |
| status | string | 身份状态 |

---

## 8. TeacherIdentity

### 8.1 定义
老师业务身份对象。

### 8.2 作用
- 查看所授班级或所负责学生
- 查看知识点、题型、误区、风险提示
- 提交老师反馈
- 查看班级维度问题聚集
- 标记重点学生

### 8.3 边界
老师只应访问自己被授权的班级、年级、学科或学生对象。

### 8.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| teacher_id | string | 老师身份唯一标识 |
| user_id | string | 关联用户账号 |
| school_id | string | 所属学校 |
| class_bindings | array[string] | 绑定班级列表 |
| subject_list | array[string] | 任教学科 |
| role_type | string | 角色类型 (subject_teacher/class_head/grade_lead) |
| status | string | 身份状态 |

---

## 9. SchoolAdminIdentity

### 9.1 定义
学校管理类业务身份对象。

### 9.2 作用
- 管理学校级接入配置
- 查看学校整体运行情况
- 查看班级和年级级别聚合数据
- 管理学校内老师与班级绑定关系
- 处理学校实施期事务

### 9.3 边界
学校管理员默认不应查看超出学校范围的数据，也不应查看超出业务需要的学生敏感细节。

### 9.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| admin_id | string | 管理员身份唯一标识 |
| user_id | string | 关联用户账号 |
| school_id | string | 管理学校 |
| admin_level | string | 管理级别 |
| permissions | array[string] | 管理权限列表 |
| status | string | 身份状态 |

---

## 10. OperatorIdentity

### 10.1 定义
项目运营、实施、客服、数据处理人员的业务身份对象。

### 10.2 作用
- 处理异常输入
- 跟进试点学校问题
- 协助接入配置
- 查看运行指标和任务状态
- 做有限人工校准

### 10.3 边界
运营权限必须拆分，不应默认拥有所有学校、所有学生的完全读写权。

### 10.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| operator_id | string | 运营人员身份唯一标识 |
| user_id | string | 关联用户账号 |
| assigned_tenants | array[string] | 授权租户列表 |
| assigned_schools | array[string] | 授权学校列表 |
| permission_level | string | 权限级别 |
| status | string | 身份状态 |

---

## 11. SystemServiceIdentity

### 11.1 定义
系统内部服务、任务执行器、异步 worker 等机器身份对象。

### 11.2 作用
- 服务间调用鉴权
- 异步任务链执行
- 图谱写入
- 报告生成
- 监控采集

### 11.3 说明
机器身份必须独立管理，不能和人工账号混用。

### 11.4 核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| service_id | string | 服务身份唯一标识 |
| service_name | string | 服务名称 |
| service_type | string | 服务类型 |
| allowed_actions | array[string] | 允许执行的动作 |
| token | string | 服务调用令牌 |
| status | string | 服务状态 |

## 12. 角色建议

建议至少定义以下业务角色：

| 角色 | 编码 | 说明 |
|------|------|------|
| 家长 | parent | 查看绑定学生信息，提交家庭反馈 |
| 学科老师 | subject_teacher | 查看所授班级/学生，提交教学反馈 |
| 班主任 | class_head | 查看班级整体情况，协调家校沟通 |
| 年级负责人 | grade_lead | 查看年级聚合视图，班级比较 |
| 学校管理员 | school_admin | 管理学校配置，查看学校级运行 |
| 学校实施联络人 | school_liaison | 处理学校实施期事务 |
| 运营处理人 | operator | 处理异常输入，跟进试点问题 |
| 教学审核人 | teaching_reviewer | 审核知识映射、误区判断 |
| 系统管理员 | system_admin | 系统级配置和平台运维 |
| 后台服务角色 | backend_service | 系统内部服务执行 |

## 13. 家长角色

### 13.1 可见范围
| 内容 | 说明 |
|------|------|
| 自己绑定学生 | 仅查看自己孩子的相关信息 |
| 与自己相关的家庭反馈 | 自己提交的反馈记录 |
| 面向家长的观察结果 | 家长端观察页面和报告 |
| 家庭可执行建议 | 低风险家庭配合建议 |

### 13.2 可执行动作
| 动作 | 说明 |
|------|------|
| 查看首页和周报/月报 | 查看学生学习状态和报告 |
| 提交反馈 | 提交家庭学习情况反馈 |
| 补充家庭学习情况 | 补充系统未捕捉的信息 |
| 查看历史报告 | 查看历史周报/月报记录 |

### 13.3 不应拥有
- 学校管理权限
- 班级横向全量数据
- 后台推演配置权限

---

## 14. 学科老师角色

### 14.1 可见范围
| 内容 | 说明 |
|------|------|
| 所授班级或所授学生 | 按绑定关系确定范围 |
| 对应学科相关状态 | 本学科的学习状态 |
| 班级学科问题聚合 | 班级层面学科问题聚集 |
| 与教学相关的解释依据 | 知识点、题型、误区等 |

### 14.2 可执行动作
| 动作 | 说明 |
|------|------|
| 查看学生详情 | 查看所负责学生的详细信息 |
| 提交老师反馈 | 提交教学观察和反馈 |
| 标记重点学生 | 标记需要重点关注的学生 |
| 查看知识点、题型、误区 | 查看教学相关结构化信息 |
| 查看报告辅助信息 | 查看报告生成的辅助信息 |

### 14.3 权限边界
- 只能访问绑定班级/学科范围内的数据
- 不能访问其他班级的详细数据
- 不能修改系统配置

### 15.2 可执行动作

### 15.3 权限边界

### 17.1 可见范围
| 内容 | 说明 |

| 本校配置 | 学校基础配置信息 |
| 组织结构 | 学校组织结构信息 |

| 学校级运行与活跃度指标 | 学校运行指标 |
| 聚合性学生状态统计 | 学生状态聚合统计 |

### 17.2 可执行动作
| 动作 | 说明 |
|------|------|
| 管理学校基础配置 | 配置学校基本信息 |
| 确认学校接入信息 | 确认学校接入配置 |

## 18. 运营处理人角色

### 18.1 可见范围
| 内容 | 说明 |
|------|------|
| 被授权学校的实施与任务状态 | 授权范围内的学校状态 |
| 异常输入池 | 异常输入处理队列 |
| 任务失败与告警信息 | 任务运行状态 |
| 必要的对象定位信息 | 问题定位所需信息 |

### 18.2 可执行动作
| 动作 | 说明 |
|------|------|
| 异常输入分流 | 处理异常输入分流 |
| 人工确认与校准 | 执行人工校准操作 |
| 跟进试点问题 | 跟进试点学校问题 |
| 查看实施指标 | 查看实施相关指标 |

### 18.3 风险
运营角色最容易因为"方便处理问题"而被放得过宽，必须控制范围和审计强度。

### 18.4 权限边界
- 只能访问授权学校范围内的数据
- 不能随意修改学生核心数据
- 所有校准操作必须审计留痕

## 19. 教学审核人角色

### 19.1 定义
负责审核知识映射、误区判断、题型映射或高风险输出的人。

### 19.2 可执行动作
| 动作 | 说明 |
|------|------|
| 审核部分教学相关结果 | 审核知识映射、误区判断 |
| 修正高风险映射 | 修正教学相关高风险映射 |
| 参与问题复核 | 参与教学问题复核 |

### 19.3 说明
此角色不是试点初期必须有，但后续会很重要。

---

## 20. 系统管理员角色

### 20.1 定义
负责系统级配置和平台运维的内部角色。

### 20.2 可执行动作
| 动作 | 说明 |
|------|------|
| 管理租户与组织配置 | 管理租户和组织配置 |
| 管理系统参数 | 管理系统运行参数 |
| 查看全局运行状态 | 查看平台整体运行状态 |
| 执行高权限修复操作 | 执行系统级修复操作 |

### 20.3 原则
系统管理员不等于业务任意读写者，高敏感学生内容仍应受审计和最小必要访问控制。

## 21. 权限维度建议

建议权限判断至少结合以下五个维度：

| 维度 | 说明 | 示例字段 |
|------|------|----------|
| 身份维度 | 确认访问者是谁 | user_id, identity_type, login_channel, verified_status |
| 角色维度 | 确认访问者此刻以什么业务角色操作 | parent, teacher, class_head, school_admin, operator |
| 租户维度 | 确认访问发生在哪个租户/学校上下文下 | tenant_id, school_id, region_id |
| 组织范围维度 | 确认可访问对象范围 | 绑定学生范围、授课班级范围、年级范围、学校范围 |
| 动作维度 | 确认允许执行的动作类型 | read, create, update, submit, approve, correct, export, admin |

| 更新 | update | 更新现有对象 |
| 提交反馈 | submit | 提交反馈信息 |

| 校准 | correct | 人工校准 |
| 导出 | export | 导出数据 |
| 管理 | admin | 管理操作 |

## 27. 权限模型建议

建议采用：

### 27.1 RBAC 作为基础
角色驱动权限分配，便于理解和管理。

### 27.2 ABAC 作为补充
结合对象归属、租户、组织范围、绑定关系做细粒度控制。

即：
**基础用角色，关键边界靠属性。**

纯 RBAC 在教育场景里不够细，纯 ABAC 落地又太重，混合是更现实的路线。

### 27.3 权限判断流程
```text
接收请求
    ↓
验证身份 (身份维度)
    ↓
确认角色 (角色维度)
    ↓
验证租户上下文 (租户维度)
    ↓
检查对象范围 (组织范围维度)
    ↓
检查动作权限 (动作维度)
    ↓
返回允许/拒绝
```

## 34. 常见错误

### 34.1 把账号和角色写死绑定
后续一个人多个身份会非常麻烦。

**正确做法**：UserAccount 与业务身份分离，一个账号可绑定多个业务身份。

### 34.2 只按角色不按对象范围判断
老师可能因此看到不属于自己的学生。

**正确做法**：权限判断必须结合角色 + 对象范围 (班级/学科/学生绑定)。

### 34.3 运营默认全开权限
初期方便，后期非常危险。

**正确做法**：运营权限必须按授权学校/项目范围严格控制，所有操作审计留痕。

### 34.4 没有显式绑定表
最后全靠人工理解权限边界，系统本身什么都不懂。

**正确做法**：建立正式的绑定关系表，权限判断基于绑定表而非人工备注。

### 34.5 租户上下文缺失
查询不带租户 ID，导致跨租户数据泄露。

**正确做法**：所有查询必须默认带 tenant_id，repository/service 层自动注入租户过滤。

## 39. 风险与边界

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 不能忽视身份与角色分离 | 一人多身份场景会出问题 | 建立 UserAccount 与业务身份分离模型 |
| 不能忽视对象范围校验 | 角色对但范围错会导致越权 | 权限判断必须结合对象范围 |
| 不能忽视租户上下文 | 会导致跨租户数据泄露 | 所有查询强制带 tenant_id |
| 不能忽视运营权限控制 | 运营权限过宽风险极高 | 按授权范围严格控制，操作审计留痕 |
| 不能忽视审计留痕 | 出问题无法追溯 | 关键操作强制审计日志 |

## 40. 结论

权限身份与角色体系的核心价值，不是让系统多一层复杂控制，而是让每个角色在真实教育场景中都待在正确边界内。

它必须坚持：

| 原则 | 说明 |
|------|------|
| 身份、角色、权限分离 | 三者不应混为一谈 |
| 租户和组织范围显式 | 权限判断必须带租户/组织上下文 |
| 最小权限优先 | 默认收紧，按需放开 |
| 关键操作可审计 | 所有敏感操作必须留痕 |
| 绑定关系正式化 | 权限基于正式绑定表而非人工备注 |

否则系统后续所有"解释很专业、架构很完整"的东西，最后都可能被一次越权访问毁掉信任。
