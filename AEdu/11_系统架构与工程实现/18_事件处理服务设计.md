## 2. 服务定位

### 2.1 服务定义

**事件处理服务（Event Processing Service）** 是 INGEST 模块的**核心处理层**，负责接收来自各接入渠道的原始消息，解析消息内容，生成标准学习事件，并路由到下游服务。

### 2.2 核心定位

| 定位维度 | 描述 |
|---------|------|
| **核心处理层** | INGEST 模块的核心事件生成引擎 |
| **消息解析器** | 将原始消息解析为结构化数据 |
| **事件工厂** | 生成标准学习事件 |
| **路由中心** | 将事件路由到 TWIN、GRAPH 等下游服务 |

### 2.3 服务边界

**包含**：
- 原始消息消费（从消息队列）
- 消息内容解析
- 学习事件生成
- 事件校验
- 知识映射
- 事件路由
- 异常事件处理
- 人工复核任务生成

**不包含**：
- ❌ 原始消息接收（由微信/钉钉接入服务负责）
- ❌ 学生状态更新（由 TWIN 模块负责）
- ❌ 图谱记忆更新（由 GRAPH 模块负责）
- ❌ OCR 识别（由 OCR 服务负责）

## 3. 核心职责

### 3.1 核心职责

#### 3.1.1 消息消费
- 从消息队列消费来自微信、钉钉接入服务的原始消息
- 支持多队列并发消费
- 支持消息的 ACK 确认和重试机制

#### 3.1.2 消息解析
- 解析文本消息的语义内容
- 解析图片/文件消息的元数据
- 提取消息中的关键信息（学生、时间、内容、类型）

#### 3.1.3 事件生成
- 根据 INGEST-007 标准生成学习事件
- 支持 8 类学习事件（作业、考试、错题、任务、反馈、进度、干预等）
- 生成事件的唯一标识和批次标识

#### 3.1.4 事件校验
- 校验事件的完整性（必填字段）
- 校验事件的一致性（时间、对象、类型）
- 校验事件的合法性（知识节点存在性）

#### 3.1.5 知识映射
- 将事件内容映射到知识底座的知识点
- 将事件内容映射到能力点
- 建立事件与教材章节的关联

#### 3.1.6 事件路由
- 将标准学习事件路由到 TWIN 模块（状态更新）
- 将标准学习事件路由到 GRAPH 模块（图谱更新）
- 将事件路由到报告服务（报告生成）

#### 3.1.7 异常处理
- 识别无法自动处理的消息
- 生成人工复核任务
- 支持复核后的事件重新处理

### 3.2 服务原则

| 原则 | 说明 |
|------|------|
| **幂等处理** | 同一消息多次消费应生成相同事件 |
| **异步处理** | 事件生成与路由采用异步方式 |
| **可追溯** | 所有事件处理过程必须有日志记录 |
| **可恢复** | 处理失败的消息应支持重新消费 |
| **知识映射准确** | 知识映射应基于 KB 模块的权威数据 |

## 6. 数据流位置

### 6.1 数据流总览

```
原始消息 → 事件处理服务 → 标准学习事件 → TWIN/GRAPH 服务
```

### 6.2 数据流详细流程

```
1. 从消息队列消费原始消息
       ↓
2. 解析消息内容
       ↓
3. 识别实体（学生、时间、内容）
       ↓
4. 查询知识底座（知识点映射）
       ↓
5. 生成标准学习事件
       ↓
6. 校验事件完整性
       ↓
7. 路由到 TWIN/GRAPH 服务
       ↓
8. 记录处理日志
```

### 6.3 与其他服务的数据流

| 数据流 | 起点 | 终点 | 数据类型 |
|--------|------|------|---------|
| **消息消费流** | 消息队列 | 事件处理服务 | 原始消息 |
| **知识查询流** | 事件处理服务 | KB 服务 | 知识点查询 |
| **事件路由流** | 事件处理服务 | TWIN/GRAPH 服务 | 学习事件 |
| **复核任务流** | 事件处理服务 | 管理后台 | 复核任务 |
| **复核结果流** | 管理后台 | 事件处理服务 | 复核结果 |

## 7. 接口边界

### 7.1 消费接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **消息消费** | `consumeMessage` | 从消息队列消费消息 |
| **批量消费** | `consumeBatch` | 批量消费消息 |
| **重试消费** | `retryConsume` | 重试消费失败的消息 |

### 7.2 处理接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **事件生成** | `generateEvent` | 生成学习事件 |
| **事件校验** | `validateEvent` | 校验事件完整性 |
| **知识映射** | `mapKnowledge` | 映射知识点和能力点 |

### 7.3 路由接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **事件路由** | `routeEvent` | 路由事件到下游服务 |
| **TWIN 路由** | `routeToTwin` | 路由到 TWIN 模块 |
| **GRAPH 路由** | `routeToGraph` | 路由到 GRAPH 模块 |

### 7.4 复核接口

| 接口类型 | 接口名称 | 说明 |
|---------|---------|------|
| **创建复核任务** | `createReviewTask` | 创建人工复核任务 |
| **查询复核状态** | `queryReviewStatus` | 查询复核任务状态 |
| **处理复核结果** | `processReviewResult` | 处理复核后的结果 |

### 7.5 接口原则

- **幂等性**：所有接口必须支持幂等调用
- **异步处理**：事件生成和路由采用异步方式
- **可追溯**：所有接口调用必须有日志记录
- **可恢复**：处理失败的消息支持重新处理

## 8. 与其他模块的交互关系

### 8.1 与接入服务的交互

| 交互类型 | 说明 |
|---------|------|
| **消息消费** | 事件处理服务从消息队列消费接入服务的消息 |
| **状态同步** | 接入服务可查询事件处理状态 |

**数据流向**：接入服务 → 消息队列 → 事件处理服务（单向）

### 8.2 与 KB 服务的交互

| 交互类型 | 说明 |
|---------|------|
| **知识查询** | 事件处理服务查询 KB 服务获取知识点信息 |
| **知识映射** | 事件处理服务使用 KB 服务进行知识映射 |

**数据流向**：事件处理服务 → KB 服务（单向查询）

### 8.3 与 TWIN 服务的交互

| 交互类型 | 说明 |
|---------|------|
| **事件传递** | 事件处理服务将学习事件传递给 TWIN 服务 |
| **状态更新** | TWIN 服务使用事件更新学生状态 |

**数据流向**：事件处理服务 → TWIN 服务（单向）

### 8.4 与 GRAPH 服务的交互

| 交互类型 | 说明 |
|---------|------|
| **事件传递** | 事件处理服务将学习事件传递给 GRAPH 服务 |
| **图谱更新** | GRAPH 服务使用事件更新图谱记忆 |

**数据流向**：事件处理服务 → GRAPH 服务（单向）

### 8.5 与管理后台的交互

| 交互类型 | 说明 |
|---------|------|
| **复核任务** | 事件处理服务创建复核任务到管理后台 |
| **复核结果** | 管理后台返回复核结果给事件处理服务 |

**数据流向**：事件处理服务 ↔ 管理后台（双向）

### 8.6 交互总结

```
                事件处理服务
                     ↑
                消息队列
                /      \
          微信接入    钉钉接入
                     ↓
    ┌────────────────┼────────────────┐
    ↓                ↓                ↓
  KB 服务         TWIN 服务        GRAPH 服务
                     ↓
                管理后台（复核）
```

**特点**：
- 事件处理服务是 INGEST 模块的**核心处理层**
- 上游依赖接入服务，下游依赖 TWIN/GRAPH 服务
- 与 KB 服务是查询依赖关系
- 与管理后台是复核协作关系

### 10.3 开发建议

#### 10.3.1 优先级建议
- **P0（第一批）**：消息消费、事件生成、事件路由
- **P1（第二批）**：知识映射、事件校验、异常处理
- **P2（第三批）**：复核任务、性能优化、高级解析

#### 10.3.2 技术实现建议
1. **使用成熟的消息队列**：推荐使用 Kafka 或 RabbitMQ
2. **事件溯源设计**：所有事件应可追溯和重放
3. **幂等性保证**：使用事件 ID 保证幂等处理
4. **知识缓存**：缓存 KB 服务的数据，减少查询延迟
5. **日志完整**：实现完整的日志记录，便于追溯

#### 10.3.3 测试建议
1. **单元测试**：为所有接口编写单元测试
2. **集成测试**：测试与接入服务、KB 服务、TWIN/GRAPH 服务的集成
3. **压力测试**：测试高并发事件处理能力
4. **异常测试**：测试异常消息的处理能力
5. **回溯测试**：测试事件重放和回溯能力
