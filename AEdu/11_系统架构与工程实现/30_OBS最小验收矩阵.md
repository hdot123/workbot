## 1. 文档目的

本文档用于定义 OBS（观察层与产品层）模块在首轮骨架开发阶段的最小可验收集合、验收维度、验收矩阵与进出门槛，确保模块进入正式开发前有统一、可执行、可收口的验收口径。

本文档重点回答以下问题：

1. OBS 最少做到什么才算"可验收"
2. 正确性、一致性、完整性、可读性分别怎么落到模块对象上
3. 首轮骨架开发的入场门槛是什么
4. 首轮骨架开发完成后的出场门槛是什么
5. 哪些结果算通过，哪些结果只能继续保留为"条件允许"

---

## 2. 适用范围

### 2.1 适用模块

- OBS 模块（观察层与产品层）

### 2.2 适用阶段

- 首轮骨架开发阶段
- 单地区、单学段、单学科、单教材版本的最小闭环阶段
- 正式开发放行前的模块级验收阶段

### 2.3 不适用范围

- 全量地区扩展验收
- 多学科并行验收
- 学生端展示验收
- 学校端完整功能验收
- 性能压测和大规模并发验收

## 3. 最小可验收集合定义

### 3.1 家长端最小可验收集合

必须至少包含以下功能：

1. 家长端首页展示
2. 当前状态摘要展示
3. 近期变化展示
4. 关注点展示
5. 家长周报生成与展示
6. 基础反馈入口

### 3.2 老师端最小可验收集合

必须至少包含以下功能：

1. 学生详情页展示
2. 学生状态概览
3. 知识点掌握结构展示
4. 基础反馈入口

### 3.3 反馈闭环最小可验收集合

必须至少包含以下功能：

1. 反馈提交入口
2. 反馈类型分流
3. 状态校准类反馈处理
4. 反馈状态追踪

---

## 4. 验收维度定义

### 4.1 正确性

展示内容是否与上游 TWIN/GRAPH 输出一致。

### 4.2 一致性

同类展示、同类交互是否遵循统一风格、统一口径、统一规则。

### 4.3 完整性

最小必需功能、必需字段、必需交互是否齐全。

### 4.4 可读性

展示内容是否易于家长/老师理解，是否符合教育场景特点。

### 4.5 角色适配性

展示内容是否针对不同角色（家长/老师）进行了差异化适配。

## 9. 首轮骨架开发入场门槛

OBS 进入首轮骨架开发前，必须满足以下条件：

- [ ] 首发地区、学段、年级、学科、教材版本已冻结
- [ ] TWIN 模块准入已冻结（NAV-005）
- [ ] GRAPH 模块准入已冻结（NAV-006）
- [ ] TWIN-OBS 输出契约已冻结（ARCH-028）
- [ ] GRAPH-OBS 输出契约已冻结（ARCH-029）
- [ ] MVP 功能范围已冻结（家长端首页 + 周报、老师端学生详情页）
- [ ] 端到端样例集已准备

若以上任一项未完成，则只允许继续做预研，不允许进入正式骨架开发。

---

## 10. 首轮骨架开发出场门槛

OBS 从"骨架开发中"进入"正式允许继续开发"前，必须满足以下条件：

- [ ] 家长端首页最小可验收集合全部具备
- [ ] 家长周报最小可验收集合全部具备
- [ ] 老师端学生详情页最小可验收集合全部具备
- [ ] 反馈闭环最小可验收集合全部具备
- [ ] 3 组端到端样例全部跑通并形成记录
- [ ] 与 TWIN/GRAPH 联合消费闭环已验证
- [ ] 降级展示链路已验证
- [ ] 无 P0 阻断项
- [ ] P1 遗留项已被记录并明确不影响首轮闭环

### 10.1 放行规则

| 结论 | 判定标准 |
|------|----------|
| 允许 | Entry Gate 全满足，Exit Gate 全满足，且无 P0 阻断项 |
| 条件允许 | Entry Gate 全满足，但 Exit Gate 仍有少量 P1 缺口，且不破坏主链路 |
| 不允许 | Entry Gate 未满足，或 Exit Gate 存在任一 P0 阻断项 |

## 11. 首轮不作为阻断项的内容

以下内容可记录为后续迭代项，但不作为本轮骨架开发阻断项：

- 学校端完整功能
- 学生端展示
- 运营端展示
- 多地区并行适配
- 多学科同时上线
- 高风险建议展示（需依赖 SIM 模块）
- 性能压测和大规模并发
- 完整的 A/B 测试框架

---

## 12. 结论

OBS 的最小验收，不是要求一开始就把全量观察层做完，而是要求先把可展示、可消费、可降级、可反馈的最小闭环做实。

首轮骨架开发阶段，验收必须坚持五条底线：

- 正确性不过关，不放行
- 一致性不收口，不放行
- 完整性不足以支撑闭环，不放行
- 可读性不满足用户理解，不放行
- 角色适配性不符合教育场景，不放行

只有在这五条底线被最小可验收集合共同满足时，OBS 才能从"条件允许"推进到"允许"。
