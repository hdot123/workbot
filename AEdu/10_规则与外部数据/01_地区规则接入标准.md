# 地区规则接入标准

> 文档编号：RULE-001  
> 版本：V1.0  
> 创建日期：2024  
> 最后更新：待定  
> 维护人：规则与外部数据负责人

---

## 1. 文档目的

本文档用于定义教育孪生项目中"地区规则"的接入标准，包括规则范围、数据结构、版本管理、更新机制、校验要求与系统使用方式。

本文档重点回答以下问题：

1. 什么叫地区规则
2. 地区规则需要接入哪些内容
3. 地区规则如何结构化存储
4. 地区规则如何与知识底座、学生模型、推演层衔接
5. 如何避免规则接入混乱、过时和口径漂移

---

## 2. 地区规则定义

地区规则，是指学生所处地区在教育制度、考试制度、教材使用、选科要求、升学口径等方面的外部约束集合。

地区规则不是学生个体数据，也不是教材内容本身，而是教育孪生系统运行时必须遵守的制度边界。

典型包括：

- 学段与年级制度
- 教材适配规则
- 考试口径
- 选科规则
- 志愿填报规则
- 地方政策要求
- 阶段教学安排约束（如适用）

---

## 3. 地区规则接入目标

### 3.1 为学生建模提供制度边界
让 StudentTwinAgent 的状态解释和推演不脱离真实地区环境。

### 3.2 为知识底座提供适配依据
帮助确定某地区使用哪套教材、哪种章节结构、哪类规则口径。

### 3.3 为推演层提供约束条件
后台推演和决策支持必须在规则允许范围内运行。

### 3.4 为学校与试点提供一致口径
避免系统中不同页面、不同模块对同一地区规则各说各话。

---

## 4. 设计原则

### 4.1 规则优先于推演
凡是明确制度边界，必须先满足规则，再讨论优化方案。

### 4.2 来源可追溯
所有规则必须有来源文件、发布日期和版本说明。

### 4.3 结构化优先
规则不能只存 PDF 或长文说明，应提取成结构化字段。

### 4.4 版本化管理
政策会变化，系统不能假装世界永远静止。

### 4.5 地区最小单元明确
需明确规则到底适用于省、市、区、县还是学校级特例。

---

## 5. 地区规则范围

建议地区规则至少覆盖以下六类：

1. 地区基础信息
2. 学段与年级规则
3. 教材适配规则
4. 考试规则
5. 选科与升学规则
6. 政策与口径说明

---

## 6. 地区基础信息

### 6.1 作用
定义规则绑定的空间范围。

### 6.2 建议字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| region_id | string | 地区唯一标识 |
| country_code | string | 国家编码 |
| province_code | string | 省份编码 |
| city_code | string | 城市编码 |
| district_code | string | 区县编码 |
| region_name | string | 地区名称 |
| region_level | string | 地区层级（省/市/区/县） |
| valid_from | date | 生效开始时间 |
| valid_to | date/null | 生效结束时间 |
| status | string | active/deprecated |

### 6.3 说明
一个规则包必须明确绑定地区粒度，不能一会儿按省，一会儿按市，最后把系统搞成制度拼盘。

## 12. 标准字段建议

地区规则主表建议至少包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| rule_id | string | 是 | 规则唯一标识 |
| rule_type | string | 是 | 规则类型 |
| region_id | string | 是 | 地区标识 |
| rule_name | string | 是 | 规则名称 |
| rule_code | string | 是 | 规则编码 |
| rule_content | object | 是 | 结构化规则内容 |
| source_ref | string | 是 | 来源引用 |
| source_date | date/null | 否 | 来源发布日期 |
| valid_from | date/null | 否 | 生效开始时间 |
| valid_to | date/null | 否 | 生效结束时间 |
| status | string | 是 | active/deprecated/draft |
| priority | int | 否 | 优先级 |
| version | string | 是 | 版本号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

## 14. 规则编码建议

### 14.1 编码结构

建议编码结构：

```
REGION + RULE_TYPE + YEAR + SEQ
```

### 14.2 示例

| 编码 | 说明 |
|------|------|
| GD_SZ_EXAM_2024_001 | 深圳市 2024 年考试规则第 001 号 |
| BJ_SUBSEL_2024_002 | 北京市 2024 年选科规则第 002 号 |

### 14.3 要求
- 同地区同类型不重复
- 可读性强
- 支持按年份/版本区分

## 15. 接入流程建议

建议标准流程如下：

```
收集地区原始规则资料
    ↓
确认地区适用范围
    ↓
识别规则类型
    ↓
抽取结构化字段
    ↓
补充来源与生效时间
    ↓
建立规则版本
    ↓
质量校验
    ↓
接入规则库
    ↓
开放给知识底座/推演层引用
```

### 15.1 各阶段说明

| 阶段 | 说明 |
|------|------|
| 收集原始资料 | 收集政策文件、官方说明等原始资料 |
| 确认适用范围 | 明确规则适用的地区层级和范围 |
| 识别规则类型 | 将规则归类到对应的 rule_type |
| 抽取结构化字段 | 将规则内容转化为结构化字段 |
| 补充来源信息 | 记录来源文件、发布日期等 |
| 建立版本 | 建立规则版本记录 |
| 质量校验 | 校验完整性、准确性、一致性 |
| 接入规则库 | 将规则写入规则库 |
| 开放引用 | 向知识底座、推演层等开放使用 |

## 16. 规则质量要求

### 16.1 完整性

| 检查项 | 要求 |
|--------|------|
| 核心字段完整 | 必填字段不能空缺 |
| 规则内容完整 | 关键规则不能缺项 |
| 来源信息完整 | 来源和日期必须记录 |

### 16.2 准确性

| 检查项 | 要求 |
|--------|------|
| 与来源一致 | 规则内容必须与来源一致 |
| 无主观臆断 | 不能靠想当然补充 |
| 口径统一 | 同一地区规则口径一致 |

### 16.3 时效性

| 检查项 | 要求 |
|--------|------|
| 版本清晰 | 必须有版本标识 |
| 生效时间清晰 | 必须有生效时间说明 |
| 更新及时 | 政策变化后及时更新 |

### 16.4 可追溯性

| 检查项 | 要求 |
|--------|------|
| 来源可查 | 必须能回到来源文件 |
| 变更可查 | 版本变更有记录 |
| 影响可查 | 变更影响范围有记录 |

### 16.5 可结构化性

| 检查项 | 要求 |
|--------|------|
| 机器可读 | 关键规则转化为机器可消费字段 |
| 逻辑清晰 | 规则逻辑可被系统理解 |
| 可执行 | 规则可被系统执行校验 |

## 17. 版本管理要求

### 17.1 版本类型

| 版本类型 | 触发条件 | 示例 |
|----------|----------|------|
| 主版本 | 规则结构变化、重大政策变更 | V2.0.0 |
| 次版本 | 新增或补充规则 | V1.1.0 |
| 修订版本 | 纠错和说明更新 | V1.0.1 |

### 17.2 版本更新时必须记录

| 记录项 | 说明 |
|--------|------|
| 更新时间 | 版本更新时间 |
| 更新人 | 更新责任人 |
| 更新原因 | 为什么更新 |
| 影响范围 | 影响哪些模块和功能 |
| 是否影响既有推演结果 | 是否需要重新评估历史推演 |

### 17.3 回滚要求

如新规则接入错误，必须支持：
- 回退到上一稳定版本
- 记录回滚原因和时间
- 通知受影响的相关方

## 18. 规则冲突处理

### 18.1 典型冲突来源

| 冲突类型 | 说明 |
|----------|------|
| 来源冲突 | 同一地区不同来源口径不一致 |
| 层级冲突 | 市级和省级规则冲突 |
| 时间冲突 | 新旧政策交叠 |
| 特例冲突 | 学校特例与地区通则冲突 |

### 18.2 处理原则

| 优先级 | 原则 |
|--------|------|
| 1 | 规则来源优先级明确 |
| 2 | 生效时间优先（新规则优先） |
| 3 | 更细粒度地区优先于更大粒度地区 |
| 4 | 特殊规则必须显式标注 |
| 5 | 冲突无法自动解决时进入人工复核 |

### 18.3 冲突记录

冲突处理必须记录：
- 冲突内容
- 处理方式
- 处理人
- 处理时间
- 最终采用的规则

## 19. 与系统的关系

### 19.1 与知识底座

| 关系 | 说明 |
|------|------|
| 地区规则决定教材适配 | 某地区使用哪套教材版本 |
| 地区规则决定地区层绑定 | 学生绑定时参照地区规则 |
| 地区规则影响章节树适配 | 不同地区可能有不同章节要求 |

### 19.2 与 StudentTwinAgent

| 关系 | 说明 |
|------|------|
| 学生初始化依赖地区规则 | 创建学生时需绑定地区规则 |
| 状态解释依赖地区规则 | 状态解释不能脱离地区环境 |
| 推演准备字段受规则约束 | 推演字段受地区规则限制 |

### 19.3 与推演层

| 关系 | 说明 |
|------|------|
| 后台推演必须受规则约束 | 推演不能超出规则边界 |
| 志愿推演强依赖地区规则 | 志愿填报需严格遵循地区规则 |
| 风险预测需考虑规则因素 | 风险评估需考虑制度约束 |

### 19.4 与前台输出

| 关系 | 说明 |
|------|------|
| 前台不直接展示规则库 | 规则是后台约束 |
| 前台解释受规则影响 | 解释和边界受规则约束 |
| 报告生成需符合规则口径 | 报告不能违背规则 |

## 24. 风险与边界

### 24.1 不能把规则当真理

规则是制度约束，但政策会变化，系统需保持更新能力。

### 24.2 不能把规则当装饰

规则必须真正约束系统行为，不是摆在库里看的。

### 24.3 不能忽视地区差异

不同地区规则差异很大，不能用一套规则硬套所有地区。

### 24.4 不能忽视规则冲突

规则冲突必须记录和处理，不能假装不存在。

## 25. 结论

地区规则接入标准的核心，不是把政策资料堆进系统，而是把真实教育制度边界结构化地带进系统。

它必须坚持：

- **有来源** - 所有规则可追溯到来源文件
- **有时间** - 生效时间和版本清晰
- **有版本** - 支持版本管理和回滚
- **有结构** - 规则内容可被机器理解和执行
- **有约束力** - 规则真正约束系统行为

否则系统就会在最需要"尊重现实"的地方开始飘，推演和决策就会变成脱离制度边界的空中楼阁。
