## 一、核心错误

原稿第 5.1 节写了一句致命错误的话：

> 指挥官（Commander / 主线程）是任务分发的唯一决策者，**不在五个执行 bot 之中**

又写：

> 指挥官 ≠ pm-bot。pm-bot 是执行层 bot 之一，只负责 pm 阶段

**全部错误。正确理解：**

| 概念 | 原稿写法（错） | 正确理解 |
|------|----------------|----------|
| 主线程 bot | "不在五个执行 bot 之中"的抽象指挥官 | **codex 本身就是主线程 bot** |
| pm-bot | 被当作"只负责 pm 阶段"的执行 bot | pm-bot 是 codex 的工具 bot，负责 pm 阶段任务 |
| 指挥官 | 被描述为独立于所有 bot 之外的实体 | **指挥官 = codex = 主线程 bot** |

## 二、正确架构

### 2.1 一句话结论

**codex 才是主线程 bot。pm-bot、dev-bot、qa-bot、doc-bot、rea-bot 全都是 codex 的工具 bot。**

### 2.2 角色关系

```
codex（主线程 bot / 指挥官）
  ├── pm-bot（工具 bot：需求澄清、范围收敛、验收口径、任务拆解、数据采集）
  ├── dev-bot（工具 bot：实现、修复、最小验证）
  ├── qa-bot（工具 bot：测试、验收、阻断判断）
  ├── doc-bot（工具 bot：文档同步、验收材料、CE 同步准备）
  └── rea-bot（工具 bot：首轮审计 — 真实性、稳定性、一致性）
```

## 三、正确流程

### 3.1 标准执行流程

```
codex 接收用户任务
  → codex 判定阶段
  → codex 分派给对应工具 bot（pm-bot / dev-bot / qa-bot / doc-bot）
  → codex 自己做 CE 收口
```

各阶段：

| 阶段 | 工具 bot | 完成状态 | codex 下一步 |
|------|----------|----------|-------------|
| PM | `pm-bot` | `pm_ready` | 转派 dev-bot |
| Dev | `dev-bot` | `dev_done` | 转派 qa-bot |
| QA | `qa-bot` | `qa_done` | 转派 doc-bot |
| Doc | `doc-bot` | `doc_synced` | codex 自己做 CE |
| CE | codex 自己 | `ce_synced → done` | 关闭 CE issue |

### 3.2 审计流程（两轮）

当任务带有"审计 / 复核 / review / audit"属性时：

```
codex 收到审计需求
  → 第一轮：codex 分派 rea-bot 做只读审计
  → 第二轮：codex 自己复审
  → 两轮都通过才允许继续
```

| 轮次 | 执行者 | 方式 | 结论类型 |
|------|--------|------|----------|
| 第一轮 | `rea-bot`（工具 bot） | codex-plugin-cc 只读审查 | Findings / Evidence / Conclusion |
| 第二轮 | `codex`（主线程 bot） | codex 复审 | 通过 / 不通过 |

证据：`workspace/projects/AEdu/task-dispatch-playbook.md`
> 第二轮由 codex 复审；只有 rea-bot + codex 两轮都通过，才允许进入下一轮

### 3.3 codex 的 Intake 流程（10 步）

证据：`workspace/projects/AEdu/task-dispatch-playbook.md`

```
 1. codex 看本地代码和测试现场
 2. codex 看五张本地单（pm/dev/qa/doc/rea-task-list）
 3. codex 看 CE API 正式状态
 4. codex 判定阶段（pm/dev/qa/doc/rea/ce）
 5. codex 判定是否先走 rea-bot 首轮审计
 6. codex 选工具 bot
 7. codex 写清 write_scope
 8. codex 先写 assignment 真源
 9. codex 再发任务到目标 pane
10. codex 派发后立刻复查 pane，确认消息已真正提交
```

### 3.4 真源优先级

```
代码/测试现场 > 本地 task-list > CE
```

## 四、pm-bot 的正确定位

原稿把 pm-bot 描述为"产品经理角色，需求分析与数据采集"。这个描述太窄但方向没错，真正的错误是把 pm-bot 和 codex 的关系搞反了。

pm-bot 的正确定位：

- pm-bot 是 codex 的工具 bot 之一
- pm-bot 负责 PM 阶段任务：需求澄清、范围收敛、验收口径、任务拆解、数据采集
- pm-bot 不负责分派其他 bot（那是 codex 的活）
- pm-bot 不负责 CE 收口（那是 codex 的活）
- pm-bot 在 cmux 中占一个 pane（pane:1），和其他 4 个工具 bot 平级

证据：`workspace/projects/AEdu/pm-task-list.md`
> 本单只记录产品澄清、范围收敛、验收口径和任务拆解状态
> pm_ready 以后必须转入 dev / commander，不允许 PM 单直接写 done

这里的"commander"就是 codex。

## 五、CE 生命周期（不受影响）

这部分原稿理解正确，此处仅做确认。

证据：`workspace/memory/kb/decisions/2026-04-02-aedu-ce-lifecycle-rule.md`

状态流转链：

```
todo → in_progress → blocked → dev_done → qa_done → doc_synced → ce_synced → done
```

规则：

- CE 不给工具 bot 直接挂正式任务
- dev_done 后 CE issue 保持 opened
- qa_done 后 CE issue 保持 opened
- 只有 done 才允许关闭 CE issue
- QA、Doc 后续进度继续写回同一张 CE issue
- CE 收口由 codex 自己处理，不下放给工具 bot

## 七、勘误总结

| 错误类型 | 原稿错误 | 修正 |
|----------|----------|------|
| **根本性** | 把 codex 当成外部工具，虚构了独立于所有 bot 之外的抽象指挥官 | codex 就是主线程 bot，直接参与调度、审计、CE 收口 |

| **流程缺失** | 审计只写了两轮但没说清 codex 是第二轮的执行者 | codex 自己就是第二轮审计者（audit_round_2_owner = codex） |
| **术语错误** | "指挥官"被描述为一个非 bot 实体 | "指挥官" = "主线程" = codex，它就是一个 bot |
